## 本節導讀

這篇回答的是：View UI Plus 元件很多，第一輪到底應該先讀哪些？

目前已確認：

- [`src/components/`](../ViewUIPlus-SourceCode/src/components/) 第一層有 132 個 component 相關目錄。
- [`examples/routers/`](../ViewUIPlus-SourceCode/examples/routers/) 有 90 個 `.vue` 範例頁。
- [`types/`](../ViewUIPlus-SourceCode/types/) 有 84 個 `.d.ts` 型別檔。
- [`test/unit/specs/`](../ViewUIPlus-SourceCode/test/unit/specs/) 只有 9 個 spec 檔，集中在少數代表元件與工具。

這代表第一輪不適合按 `src/components/` 字母順序逐一閱讀。更穩定的方式是：

```text
先讀小元件建立基本模式
  -> 再讀布局 / 容器建立組合模式
  -> 再讀表單與輸入建立狀態同步模式
  -> 再讀複雜互動元件建立分支管理模式
  -> 再讀命令式 API 建立 service / instance 模式
  -> 最後讀大型資料元件與業務型元件
```

本篇不是元件總覽，也不是元件詳解；它是選樣與排序策略。真正的單一元件分析應使用 [00-04_元件分析模板](./00-04_元件分析模板.md)，完成後再用 [00-06_追蹤檢查表](./00-06_追蹤檢查表.md) 驗收。

當你遇到以下情境，可以回到這篇：

- 不知道下一個元件該讀哪個。
- 不確定某個元件應該精讀、略讀，還是暫緩。
- 想用少量元件覆蓋 View UI Plus 的主要設計模式。
- 讀大型元件前，想確認自己是否已經具備必要前置能力。

## 1. 選讀原則

### 1.1 不按字母順序讀

已確認：`src/components/` 底下有大量元件資料夾，且不同元件的複雜度差距很大。按字母順序讀會遇到幾個問題：

- 一開始就可能碰到大型或歷史包袱元件。
- 讀過很多檔案，卻沒有建立可複用的分析模型。
- 同類元件的共同模式分散，難以比較。
- 容易把 `index.js`、`.vue`、styles、types、examples、tests 之間的關係拆散。

合理推論：第一輪應以「建立讀碼能力」為目標，而不是以「覆蓋所有元件」為目標。

### 1.2 先讀有代表性的元件

代表性不是只看元件是否常用，而是看它能不能教會你一種模式：

| 模式 | 代表元件 | 能學到什麼 |
|---|---|---|
| 基礎元件模式 | Button | props、computed class、render、slot、mixin |
| 容器布局模式 | Layout、Row、Col、Grid | 組合元件、響應式 class、Less |
| 表單協作模式 | Form、FormItem、Input | provide/inject、mixin、驗證、v-model |
| 複雜選擇模式 | Select | multiple、filterable、remote、dropdown、options |
| 日期面板模式 | DatePicker | 面板拆分、日期工具、range、time spinner |
| 命令式服務模式 | Message、Notice、Modal | globalProperties、newInstance、config、destroy |
| 大型資料模式 | Table | columns、data、render 拆分、固定欄、CSV、summary |
| 導航協作模式 | Menu、Tabs、Breadcrumb | active 狀態、父子關係、子元件 |
| 彈層定位模式 | Tooltip、Poptip | trigger、placement、popper、transfer |
| 業務封裝模式 | Login、Exception、NumberInfo | 企業後台場景封裝 |

### 1.3 先讀有外部證據的元件

已確認：目前 `test/unit/specs/` 只有部分 spec，例如：

- [`button.spec.js`](../ViewUIPlus-SourceCode/test/unit/specs/button.spec.js)
- [`select.spec.js`](../ViewUIPlus-SourceCode/test/unit/specs/select.spec.js)
- [`date-picker.spec.js`](../ViewUIPlus-SourceCode/test/unit/specs/date-picker.spec.js)
- [`message.spec.js`](../ViewUIPlus-SourceCode/test/unit/specs/message.spec.js)
- [`table.spec.js`](../ViewUIPlus-SourceCode/test/unit/specs/table.spec.js)
- [`breadcrumb.spec.js`](../ViewUIPlus-SourceCode/test/unit/specs/breadcrumb.spec.js)
- [`affix.spec.js`](../ViewUIPlus-SourceCode/test/unit/specs/affix.spec.js)

合理推論：第一輪優先讀有 tests、examples、types 的元件，因為它們能讓你練習「實作 -> 型別 -> 範例 -> 測試」的交叉驗證。

### 1.4 先小後大

已確認：代表元件主檔大小差距明顯：

| 元件 | 主檔 | 行數 |
|---|---|---:|
| Button | [`button.vue`](../ViewUIPlus-SourceCode/src/components/button/button.vue) | 140 |
| Form | [`form.vue`](../ViewUIPlus-SourceCode/src/components/form/form.vue) | 139 |
| FormItem | [`form-item.vue`](../ViewUIPlus-SourceCode/src/components/form/form-item.vue) | 275 |
| Input | [`input.vue`](../ViewUIPlus-SourceCode/src/components/input/input.vue) | 435 |
| Select | [`select.vue`](../ViewUIPlus-SourceCode/src/components/select/select.vue) | 823 |
| DatePicker | [`picker.vue`](../ViewUIPlus-SourceCode/src/components/date-picker/picker.vue) | 846 |
| Table | [`table.vue`](../ViewUIPlus-SourceCode/src/components/table/table.vue) | 1567 |

合理推論：應先用 Button、Form、Input 建立方法，再進 Select、DatePicker、Table。直接讀 Table 容易同時遇到資料轉換、slot、render、fixed、summary、CSV、scroll 等多條線，初期成本太高。

### 2.2 略讀元件

略讀代表「要看入口、API、樣式與 examples，但不一定逐行精讀」。

| 類型 | 元件 | 略讀目標 |
|---|---|---|
| 基礎展示 | Icon、Tag、Badge、Alert、Divider | 補足基礎 class、slot、狀態樣式 |
| 輸入控件 | Checkbox、Radio、InputNumber、Switch、Rate、Slider | 比較 v-model 與表單 mixin 的共性 |
| 導航 | Breadcrumb、Dropdown、Steps、Timeline、PageHeader | 看父子組合與使用方式 |
| 資料展示 | List、DescriptionList、Card、Page、Tree | 看資料結構與 slot 配置 |
| 回饋 | Drawer、Spin、LoadingBar、Progress | 看可見狀態、loading、全域或局部使用 |
| 增強型 | Upload、Transfer、Cascader、TreeSelect、ColorPicker | 後期按專題選讀 |

### 2.3 暫緩元件

暫緩不是不讀，而是等到已有足夠前置模式後再讀。

| 類型 | 元件 | 暫緩理由 |
|---|---|---|
| 大型複雜元件 | Table、DatePicker、Select 的所有分支 | 初期只追主流程，支線後補 |
| 業務型元件 | Login、Exception、GlobalFooter、NumberInfo | 需要先理解基礎元件與布局 |
| 輔助包裝元件 | AvatarList、Description、SkeletonItem、TagSelectOption | 多半依附主元件，先讀主元件 |
| 底層支撐層 | `base/notification`、popper 相關檔 | 先由 Message / Notice / Tooltip 反向進入 |
| 產物或範例專用元件 | examples 中的展示頁 | 作為對照證據，不作第一閱讀入口 |

## 4. 第二輪建議順序

第二輪目標是補足同類元件比較能力。

| 類別 | 建議元件 | 比較重點 |
|---|---|---|
| 基礎展示 | Icon、Tag、Badge、Alert、Divider | class、slot、狀態樣式、低耦合 API |
| 表單輸入 | Checkbox、Radio、InputNumber、Switch、Rate、Slider | v-model 事件命名、disabled、size、表單 mixin |
| 導航 | Breadcrumb、Dropdown、Steps、Timeline、Page | 子元件協作、active、使用者點擊流程 |
| 彈層回饋 | Tooltip、Poptip、Drawer、Spin、LoadingBar | trigger、visible、transfer、全域 vs 局部 |
| 資料展示 | List、Tree、Page、DescriptionList | data 結構、slot、render 分支 |
| 增強型 | Upload、Transfer、Cascader、TreeSelect、ColorPicker | 非同步、複合資料、樹狀結構、上傳副作用 |

第二輪讀法：

```text
先選同類兩到三個元件
  -> 共用同一份快速模板
  -> 比較 props / events / slots / styles
  -> 找出共同模式與例外
  -> 把例外記入待驗證
```

| 5 | Poptip | 在 Tooltip 基礎上看內容型彈層 |
| 6 | Drawer / Spin / LoadingBar | 補其他回饋模式 |

已確認：[`message/index.js`](../ViewUIPlus-SourceCode/src/components/message/index.js)、[`notice/index.js`](../ViewUIPlus-SourceCode/src/components/notice/index.js)、[`modal/index.js`](../ViewUIPlus-SourceCode/src/components/modal/index.js) 都是讀命令式 API 的重要入口。

命令式 API 應使用這條線：

```text
src/index.js globalProperties
  -> components export
  -> service index.js
  -> getInstance / newInstance
  -> show / notice / remove / destroy
```

### 5.8 `18_業務型與增強型元件`

建議順序：

| 順序 | 元件 | 讀法 |
|---|---|---|
| 1 | Upload | 看檔案、非同步、列表狀態 |
| 2 | Transfer | 看左右資料搬移 |
| 3 | Cascader / TreeSelect | 看樹狀選擇與彈層 |
| 4 | ColorPicker | 看複合輸入與彈層 |
| 5 | Login / Exception / NumberInfo | 看企業後台場景封裝 |

合理推論：這些元件通常建立在前面章節的模式之上。若還沒有讀過表單、彈層、資料展示，直接讀它們會比較難形成穩定理解。

## 6. 按能力選元件

如果你的目標不是順讀，而是想補某種能力，可以用這張表。

| 想練的能力 | 建議元件 | 為什麼 |
|---|---|---|
| props -> class -> Less | Button、Tag、Badge | 小而直接 |
| render function | Button、Table 子檔 | Button 入門，Table 進階 |
| template 分支 | Input、Select、DatePicker | 分支較多，適合練狀態到 UI |
| v-model | Input、Select、DatePicker、Switch | 都有外部值與內部狀態同步 |
| provide / inject | Form / FormItem、Select、Table、Tabs | 父子協作代表 |
| mixin | Button、Input、Select、DatePicker | link、form、locale、globalConfig |
| directive | Select、DatePicker、v-resize / v-style examples | click outside、resize、style |
| 命令式 API | Message、Notice、Modal、LoadingBar | service / instance 模式 |
| popper / transfer | Tooltip、Poptip、Select、DatePicker | 彈層定位與 DOM 搬移 |
| 測試反推 | Button、Select、DatePicker、Message、Table | spec 證據較集中 |
| 大型元件拆分 | DatePicker、Table | 多檔案、多狀態、多分支 |
| 業務封裝 | Login、Exception、NumberInfo | 後台場景型元件 |

## 7. 每個代表元件要產出什麼

每讀一個代表元件，都應留下固定產物，否則第二輪會重新追。

| 元件類型 | 最低產出 |
|---|---|
| 基礎元件 | API 表、class 對照、render / template 主流程、types / examples / tests |
| 容器元件 | 子元素關係、class / style 規則、響應式策略 |
| 表單控件 | v-model 流程、表單 mixin、disabled / size、驗證觸發 |
| 複雜互動元件 | 主狀態、主流程、分支索引、暫緩支線 |
| 命令式 API | 全域掛載、service methods、instance、config、destroy |
| 大型資料元件 | 子模組地圖、data / columns 主流程、渲染拆分、測試缺口 |
| 業務型元件 | 依賴哪些基礎元件、封裝了什麼場景、可抽象能力 |

建議每篇代表元件筆記都包含：

- 源碼入口。
- 是否進入 `src/components/index.js`。
- 是否被 `src/index.js` install 註冊或全域掛載。
- props / events / slots / methods。
- data / computed / watch / methods / lifecycle。
- mixins / utils / base / directives。
- `prefixCls` 與 Less。
- types / examples / tests。
- 可仿寫點。
- 待驗證與暫緩分支。

## 8. 讀大型元件前的門檻

大型元件不是不能早讀，而是要先具備前置能力。

### 8.1 讀 Select 前

建議先完成：

- Button：知道基礎 props、computed、class、render。
- Input：知道 v-model 與表單控件基本模式。
- Form / FormItem：知道 `mixins/form` 的作用。

讀 Select 時第一輪只追：

- `modelValue` 如何初始化。
- 單選與多選的主流程。
- option 如何註冊與顯示。
- dropdown visible 如何控制。
- `update:modelValue` 與 `on-change` 如何觸發。

暫緩：

- remote 的所有細節。
- allow-create 的完整分支。
- keyboard 導航細節。
- transfer / popper 的深入實作。

### 8.2 讀 DatePicker 前

建議先完成：

- Input：知道輸入框與 model 同步。
- Select：知道 dropdown 與複雜選擇器。
- Form / FormItem：知道表單驗證入口。

讀 DatePicker 時第一輪只追：

- `picker.vue` 如何作為主入口。
- panel 如何切換。
- `modelValue` 如何 parse / format。
- range 如何和單值分流。
- `on-change`、`on-ok`、`update:modelValue` 如何觸發。

暫緩：

- 所有日期工具函式細節。
- time spinner 完整邏輯。
- 所有面板邊界情況。

### 8.3 讀 Table 前

建議先完成：

- Button / Tag：基礎 class 與 render。
- List / Page：資料展示與資料操作基本模式。
- Select / DatePicker：大型互動元件的分支索引方法。

讀 Table 時第一輪只追：

- props 中的 `columns`、`data`。
- data 如何變成 `rebuildData` 或渲染資料。
- header / body / cell 如何拆分。
- 固定欄與 scroll 只抓主線。
- CSV export 對應測試。

暫緩：

- drag / resize 細節。
- tree / expand 支線。
- summary 完整分支。
- fixed left / right 所有 DOM 計算細節。

## 9. 不同時間長度的選讀方案

| 時間 | 建議範圍 | 目標 |
|---|---|---|
| 30 分鐘 | Button | 建立最小元件讀法 |
| 半天 | Button + Layout / Row / Col | 建立基礎元件與容器模式 |
| 1 天 | Button + Form / FormItem + Input | 建立表單與輸入主線 |
| 2 天 | 再加 Select 或 DatePicker | 建立複雜互動元件讀法 |
| 3 天 | 再加 Message / Modal | 建立命令式 API 讀法 |
| 1 週 | Button、Layout、Form、Input、Select、DatePicker、Message、Modal、Table | 覆蓋主要元件庫模式 |
| 第二週 | Menu、Tabs、Tooltip、Poptip、Upload、Transfer、Tree | 補導航、彈層、增強型元件 |

合理推論：只要第一週能把代表元件精讀到可回查，後續讀其他元件會快很多。

## 10. 選讀決策樹

遇到一個新元件時，用這棵決策樹判斷讀法：

```text
這個元件有沒有 tests？
  -> 有：優先作為精讀候選
  -> 沒有：看 examples / types 是否完整

它是不是某類模式的第一個代表？
  -> 是：精讀
  -> 否：略讀或對照讀

它是不是大型元件？
  -> 是：先建立子模組地圖，只追主流程
  -> 否：可以直接用完整模板

它是不是命令式 API？
  -> 是：用命令式 API 模板，不套普通 .vue 模板
  -> 否：用元件模板

它是不是依附另一個主元件？
  -> 是：先讀主元件，再讀它
  -> 否：可獨立閱讀
```

## 11. 讀完後的驗收

每讀完一個代表元件，用 [00-06_追蹤檢查表](./00-06_追蹤檢查表.md) 驗收。

| 分數 | 標準 |
|---|---|
| 10 / 10 | 能交代入口、註冊、生效、API、狀態、事件、渲染、樣式、types、examples、tests、待驗證 |
| 9 / 10 | 主流程完整，少量分支待補 |
| 8 / 10 | 能回答本輪問題，但外部證據不足 |
| 6 到 7 / 10 | 有局部理解，尚未形成閉環 |
| 5 以下 | 只看過檔案，還不能作為筆記沉澱 |

代表元件筆記至少要能回答：

- 這個元件為什麼被選為代表？
- 它代表哪一種設計模式？
- 它和前一個已讀元件有什麼相同與不同？
- 它的主流程是什麼？
- 它有哪些支線暫緩？
- 後續讀同類元件時，哪些步驟可以複用？

## 12. 常見選讀錯誤

| 錯誤 | 問題 | 修正方式 |
|---|---|---|
| 一開始讀 Table | 同時遇到太多支線，難以建立基本方法 | 先讀 Button、Input、Select |
| 按字母順序讀 | 元件難度與分類不連續 | 改按模式與章節選讀 |
| 只讀常用元件 | 可能漏掉命令式 API、表單、彈層等重要模式 | 用代表性覆蓋 |
| 每個元件都精讀 | 成本太高，難以推進 | 精讀代表元件，略讀同類元件 |
| 每個大型元件都追完所有分支 | 主流程容易失焦 | 第一輪只追主流程，建立分支索引 |
| 忽略 tests | 少了行為規格證據 | 優先讀有 spec 的代表元件 |
| 忽略 types | 少了對外契約 | 每個代表元件都要對照 types |
| 忽略 examples | 少了使用場景 | examples 是實際用法證據 |
| 把業務型元件放太早 | 會被場景封裝干擾基礎模式 | 等基礎、表單、彈層讀完再看 |

## 14. 後續更新規則

這份策略不是一次寫死。每讀完一個代表元件，都要回來更新三件事：

1. 這個元件是否真的適合作為代表。
2. 同類元件是否應調整閱讀順序。
3. 哪些支線應移到 `00-09_問題索引.md` 或 `00-10_暫緩閱讀清單.md`。

更新時不要只改排序，還要補證據：

| 要調整的內容 | 需要補的證據 |
|---|---|
| 把元件升為精讀 | 它能代表新的模式，且有源碼 / examples / types / tests 支撐 |
| 把元件降為略讀 | 它和已讀代表元件高度重複 |
| 把元件暫緩 | 它依賴尚未讀過的前置概念 |
| 新增讀碼支線 | 能說明支線入口與暫緩理由 |

## 本節小結

View UI Plus 元件選讀的核心策略是：

> 不按字母順序讀，也不平均用力讀所有元件；先用 Button 建立基礎元件模型，再用 Layout / Row / Col 建立容器模型，用 Form / FormItem / Input 建立表單模型，用 Select / DatePicker 建立複雜互動模型，用 Message / Modal 建立命令式 API 模型，最後再讀 Table 與業務型元件。

這樣讀的好處是，每個代表元件都會產出一種可複用能力。後續再回頭讀其他元件時，就不是從零開始，而是在既有模型上做比較、補例外與整理設計模式。
