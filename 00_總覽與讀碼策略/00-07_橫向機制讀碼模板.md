# 00-07_橫向機制讀碼模板

> 版本定位：9.5 分版本  
> 適用對象：正在閱讀 View UI Plus 原始碼，並希望把「跨元件、跨目錄、跨層級」的機制整理成可回查、可仿寫、可重構、可應用到企業後台元件封裝的學習者。  
> 適用範圍：元件註冊機制、全域配置機制、`globalProperties`、`provide / inject`、`prefixCls`、表單驗證、命令式 API、彈層定位、國際化、自訂指令、工具函式、mixin、樣式主題、型別宣告、測試反推等橫向系統。  
> 不適用範圍：單一 Button、Input、Select、Table、Modal 等元件的完整讀碼筆記。這類內容應使用 `00-06_單一元件讀碼模板.md`。

---

## 1. 這份模板的用途

`00-06_單一元件讀碼模板.md` 解決的是「如何讀懂一個元件」。  
`00-07_橫向機制讀碼模板.md` 解決的是「如何讀懂一套機制」。

在 View UI Plus 這類企業級 Vue 3 元件庫中，真正有學習價值的地方不只在單一元件，而是在很多跨元件、跨檔案、跨層級的機制，例如：

```txt
1. app.use(ViewUIPlus) 如何安裝整套元件庫？
2. 每個元件如何被批量註冊？
3. 全域配置如何影響所有元件？
4. prefixCls 如何統一 class 命名？
5. Form 與 FormItem 如何協作完成驗證？
6. Message / Modal / Notice 這類命令式 API 如何動態建立？
7. Tooltip / Popover / Dropdown 這類彈層如何處理定位、顯示、隱藏與事件？
8. locale 國際化如何讓多個元件共用文案？
9. directive 如何封裝 DOM 行為？
10. utils / mixins / types 如何支撐整套元件庫？
```

如果只讀單一元件，容易停留在「這個元件怎麼寫」。  
如果能讀懂橫向機制，才會進入「一套元件庫怎麼設計」。

這份模板的目標是讓每一篇橫向機制筆記都能回答四個層次：

```txt
第一層：這個機制解決什麼問題？
第二層：使用者如何感受到這個機制？
第三層：源碼如何把這個機制串起來？
第四層：我如何把這個機制轉化成自己的企業後台封裝能力？
```

---

## 2. 什麼是「橫向機制」？

### 2.1 單一元件與橫向機制的差異

| 類型 | 關注點 | 範例 | 筆記模板 |
|---|---|---|---|
| 單一元件 | 一個元件如何設計與實作 | Button、Input、Select、Table、Modal | `00-06_單一元件讀碼模板.md` |
| 橫向機制 | 多個元件共用的底層設計 | install、global config、prefixCls、form validation、locale、command API | `00-07_橫向機制讀碼模板.md` |

單一元件偏「局部」。  
橫向機制偏「系統」。

單一元件通常從一個元件入口開始追。  
橫向機制通常要從多個入口交叉追蹤。

例如閱讀 `Input` 元件時，重點可能是：

```txt
1. props 如何設計？
2. v-model 如何同步？
3. clearable 如何實作？
4. prefix / suffix slot 如何實作？
5. focus / blur / input / change 事件如何派發？
```

但閱讀「表單驗證機制」時，重點就不是單一 `Input`，而是：

```txt
1. Form 如何管理 rules 與 model？
2. FormItem 如何知道自己對應哪個 prop？
3. Input 如何把 change / blur 通知 FormItem？
4. FormItem 如何觸發驗證？
5. 驗證錯誤如何回寫到 UI？
6. resetFields / validate / validateField 如何串起來？
```

這就是橫向機制。

---

### 2.2 橫向機制的判斷標準

當你看到一段設計符合以下任一條件，就應該用橫向機制模板來整理：

```txt
1. 被多個元件共用
2. 跨越多個資料夾
3. 需要多個元件互相協作
4. 對使用者暴露的是一套行為，而不是單一元件
5. 牽涉全域配置、插件安裝、樣式規範、國際化、型別或測試
6. 不讀整條流程就很難理解單一元件為什麼這樣設計
```

常見橫向機制如下：

| 機制 | 典型特徵 | 代表問題 |
|---|---|---|
| 插件安裝機制 | `install(app)`、`app.use()`、全域註冊 | 元件庫如何被 Vue 應用安裝？ |
| 元件註冊機制 | 批量註冊、單元件 install、name | 元件如何被全域使用？ |
| 全域配置機制 | config、globalProperties、provide/inject | 全域尺寸、prefix、locale 如何被所有元件取得？ |
| 樣式命名機制 | prefixCls、class helper、Less 變數 | class 名稱如何統一？ |
| 表單驗證機制 | Form、FormItem、rules、model、validate | 輸入元件如何參與驗證？ |
| 命令式 API 機制 | Message、Notice、Modal confirm | `$Message.success()` 如何建立畫面？ |
| 彈層定位機制 | Tooltip、Popover、Dropdown、Transfer | overlay 如何定位、掛載、關閉？ |
| 自訂指令機制 | directive、DOM event、outside click | DOM 行為如何從元件中抽離？ |
| 國際化機制 | locale、i18n、語系包 | 元件文案如何多語系化？ |
| 工具函式機制 | utils、dom、assist、date、type | 重複邏輯如何集中管理？ |
| mixin 共用邏輯 | mixins、shared options | 多元件共同行為如何複用？ |
| 型別宣告機制 | `types`、component props、plugin type | 使用者如何獲得型別提示？ |
| 測試反推機制 | unit test、snapshot、interaction test | 如何從測試推回元件規格？ |

## 3. 橫向機制讀碼的基本原則

### 3.1 不要從工具函式開始讀

很多橫向機制最後都會追到工具函式，例如：

```txt
src/utils
src/mixins
src/directives
src/locale
src/styles
types
```

但不要一開始就讀工具函式，否則容易陷入細節。

正確順序是：

```txt
使用者行為
→ 對外 API
→ 入口檔案
→ 核心資料流
→ 參與元件
→ 共用工具
→ 樣式與型別
→ 測試與 Demo
```

先理解「這個機制被誰使用、解決什麼問題」，再讀細節。

---

### 3.2 先抓主流程，再抓邊界分支

橫向機制常常有很多分支，例如：

```txt
1. 全域註冊與按需引入
2. 宣告式使用與命令式使用
3. controlled 狀態與 uncontrolled 狀態
4. hover 觸發與 click 觸發
5. blur 驗證與 change 驗證
6. 預設語系與自訂語系
7. 預設 prefixCls 與自訂 prefixCls
```

不要一開始就追所有分支。

先建立一條最小主流程：

```txt
輸入
→ 中間處理
→ 輸出
```

例如表單驗證的最小主流程：

```txt
使用者輸入
→ Input 觸發 change / blur
→ FormItem 接收驗證訊號
→ FormItem 根據 prop 找 rules
→ 執行 validator
→ 更新 validateState / validateMessage
→ 畫面顯示錯誤訊息
```

等這條主流程清楚後，再補：

```txt
1. resetFields
2. validateField
3. async validator
4. nested prop
5. rule trigger
6. disabled / required / message
```

---

### 3.3 橫向機制要畫流程，不只摘錄程式碼

單一元件筆記可以有較多 API 表格。  
橫向機制筆記一定要有流程圖或流程文字。

最低限度要整理出：

```txt
1. 入口在哪裡？
2. 資料從哪裡來？
3. 經過哪些物件或函式？
4. 被哪些元件接收？
5. 最後如何反映到畫面或使用者 API？
```

推薦使用這種文字流程圖：

```txt
app.use(ViewUIPlus)
→ ViewUIPlus.install(app)
→ 遍歷 components
→ app.component(component.name, component)
→ 掛載全域方法 / 全域配置
→ 使用者可在 template 中使用元件
```

或：

```txt
$Message.success(options)
→ 建立 Message instance
→ normalize options
→ render vnode
→ mount 到 container
→ 加入 message queue
→ timeout 後關閉
→ destroy / remove DOM
```

---

### 3.4 橫向機制要追「關係」，不是只追「檔案」

讀橫向機制時，最重要的不是知道有幾個檔案，而是知道它們的關係。

每篇筆記都應該整理出：

```txt
1. 入口檔案
2. 核心資料結構
3. 核心函式
4. 參與元件
5. 依賴工具
6. 輸出效果
```

例如：

```txt
Form 驗證機制
├── 入口：Form / FormItem
├── 使用者輸入：Input / Select / Checkbox / DatePicker
├── 核心資料：model / rules / prop / validateState / validateMessage
├── 核心行為：validate / resetFields / validateField
├── 事件來源：change / blur
├── 工具依賴：async-validator / path get helper / type helper
└── 輸出效果：錯誤訊息、狀態 class、回呼結果
```

## 21. 最終總結

`00-07_橫向機制讀碼模板.md` 的核心價值是：

```txt
不要只看一個元件怎麼寫，
而要看一套元件庫如何把共同問題抽成可複用的機制。
```

閱讀橫向機制時，最重要的不是把所有源碼都貼進筆記，而是要整理出：

```txt
1. 這個機制解決什麼問題
2. 使用者如何使用它
3. 源碼入口在哪裡
4. 參與者有哪些
5. 資料如何流動
6. 狀態如何管理
7. 樣式如何呈現
8. 型別如何支撐
9. 邊界如何處理
10. 我如何仿寫與應用到企業後台
```

如果 `00-06` 是「看懂單一零件」，那 `00-07` 就是「看懂整台機器的傳動系統」。

等你能讀懂這些橫向機制，你就不只是會用 View UI Plus，而是會開始理解：

```txt
一套企業級 Vue 3 元件庫是如何被設計出來的。
```
