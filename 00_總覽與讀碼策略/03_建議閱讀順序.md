# 03_建議閱讀順序

> 本篇任務：把前面的「專案定位」與「源碼總目錄地圖」轉成一條可以真正執行的讀碼路線。  
> 這不是單純排序，而是規定：**先讀什麼、為什麼先讀、看到什麼程度算完成、卡住時要退回哪一層。**

---

## 0. 本篇結論

View UI Plus 的建議閱讀順序不是：

```txt
從 src/components 第一個資料夾開始一路看完
```

而應該是：

```txt
專案對外身份
  ↓
打包入口與開發入口
  ↓
Vue Plugin 安裝入口
  ↓
元件匯出清單
  ↓
examples 使用者視角
  ↓
簡單元件建立模式感
  ↓
共用層回補：mixins / utils / directives / styles / locale
  ↓
父子協作型元件
  ↓
表單與資料輸入元件
  ↓
命令式回饋與彈層元件
  ↓
資料型、重互動、重狀態元件
  ↓
types / test / build / dist 工程配套
```

用一句話說：

> **先知道這套庫怎麼被安裝與使用，再看元件怎麼被集中匯出，接著用簡單元件建立寫法模式，最後才進入大型元件與工程配套。**

第一輪最短路線建議是：

```txt
package.json
  ↓
vite.config.js
  ↓
vite.config.dev.js
  ↓
src/index.js
  ↓
src/components/index.js
  ↓
examples/main.js
  ↓
src/components/button/
  ↓
src/components/alert/ 或 src/components/tag/
  ↓
src/mixins/form.js + src/mixins/link.js
  ↓
src/utils/assist.js
  ↓
src/styles/index.less + src/styles/components/button.less
  ↓
Form / Select / Modal / Message
  ↓
Table / Tree / Upload / DatePicker
```

---

## 1. 本篇源碼核對依據

這份閱讀順序不是憑感覺排的，而是依照 View UI Plus 的實際源碼責任排列。

| 源碼位置 | 實際角色 | 對閱讀順序的影響 |
|---|---|---|
| `README.md` | 專案對外定位為基於 Vue 3 的 UI component library / front-end solution | 先確認它不是業務系統，而是元件庫 |
| `package.json` | 宣告 npm 套件名稱、入口 `main`、型別入口 `typings`、發布檔案、開發與建置 scripts | 第一輪要先知道它如何被安裝、打包、發布 |
| `vite.config.js` | library build 的入口指向 `src/index.js`，輸出 UMD / ES 產物到 `dist` | 證明 `src/index.js` 是正式打包入口，不只是普通檔案 |
| `vite.config.dev.js` | 開發 root 指向 `examples`，並透過 alias 讓 examples 可引用 `src` | 證明 examples 是開發與驗證元件的重要入口 |
| `src/index.js` | 匯出 components、定義 `install(app, opts)`、註冊元件、註冊 directives、掛載全域 API | 源碼閱讀必須優先理解它 |
| `src/components/index.js` | 集中 export 對外元件 | 它是元件庫的 public component surface |
| `examples/main.js` | 建立 Vue app、匯入 `../src/index`、`app.use(ViewUIPlus)`、建立大量元件 demo routes | 它能從使用者角度反推元件 API 與展示方式 |
| `src/components/button/` | 包含 `button.vue`、`button-group.vue`、`index.js` | Button 是第一個適合深讀的代表性基礎元件 |
| `src/mixins/form.js` | 透過 inject 連接 Form / FormItem 狀態與事件 | 讀表單元件前要先補這一層 |
| `src/mixins/link.js` | 抽象 `to`、`replace`、`target`、router / window 跳轉邏輯 | 讀 Button / Link / 可跳轉元件時要回補 |
| `src/styles` | 統一管理 Less 入口、元件樣式、變數與 mixins | 不應等所有元件讀完才看，讀完簡單元件後就要回補 |
| `types` | 對外 TypeScript 型別契約 | 應在理解元件 API 後再讀，否則容易只看到宣告看不到設計意圖 |
| `build` / `dist` | 建置腳本與打包產物 | 放後面；不是第一輪學源碼的入口 |

## 2. 閱讀順序的核心原則

### 2.1 先讀「對外入口」，再讀「內部實作」

元件庫不是一般頁面專案。它的核心不是某個 route 或某個業務頁，而是：

```txt
使用者 import / app.use
        ↓
套件入口 src/index.js
        ↓
集中匯出 src/components/index.js
        ↓
單一元件實作 src/components/*
```

所以第一輪不應該直接鑽 `Table`、`DatePicker`、`Upload`，而是先看：

```txt
package.json
vite.config.js
src/index.js
src/components/index.js
```

這樣你才知道後面每個元件最後會如何被對外暴露。

---

### 2.2 先用 examples 取得使用者視角

只看 `src/components/button/button.vue`，你會知道 Button 怎麼寫。  
但先看 `examples/main.js` 與 `examples/routers/button.vue`，你會更早知道：

- Button 對使用者暴露哪些 props？
- demo 中最常被展示的是哪些能力？
- 元件是被全域註冊後直接使用，還是局部 import？
- 命令式 API 例如 Message / Modal / Notice 是怎麼被呼叫？
- directives 例如 `v-style`、`v-resize`、`v-line-clamp` 有沒有 demo？

這會讓你讀源碼時比較不會陷入「我知道程式碼怎麼跑，但不知道為什麼這樣設計」。

---

### 2.3 先讀簡單元件，建立模式感

第一批元件要選「能代表元件庫寫法，但不至於太重」的元件。建議先讀：

```txt
Button
Alert
Tag
Divider
Space
Card
```

不要一開始就讀：

```txt
Table
DatePicker
Tree
Upload
Cascader
Transfer
```

原因是重型元件通常同時牽涉：

- 複雜狀態
- 多個子元件
- 大量事件
- 複雜 keyboard / mouse / scroll 行為
- 資料轉換
- overlay / popup / virtual / async 等進階互動
- 跨檔案工具函式

如果你還沒有看過簡單元件，會很難判斷哪些是「View UI Plus 的固定寫法」，哪些才是「該重型元件自己的複雜度」。

---

### 2.4 共用層不要太早，也不要太晚

`mixins`、`utils`、`directives`、`styles`、`locale` 都很重要，但不建議一開始就逐行看。

比較好的時機是：

```txt
讀完 src/index.js
讀完 src/components/index.js
讀完 Button / Alert / Tag 這類簡單元件
        ↓
開始回補共用層
```

因為這時候你已經看過幾個真實使用點，回頭看共用層才知道它們解決什麼問題。

---

### 2.5 每一階段都要有產出，不要只說「看過了」

讀碼不是看完就好。每一階段至少要留下以下其中一種產出：

```txt
入口流程圖
元件匯出表
元件 API 表
檔案依賴表
核心流程筆記
可仿寫設計
問題回查索引
```

如果沒有產出，兩週後很容易只剩下「好像看過」。

## 5. 第 1 階段：理解打包入口與開發入口

### 5.1 建議閱讀檔案

```txt
vite.config.js
vite.config.dev.js
```

### 5.2 `vite.config.js` 要看什麼

你要觀察：

```txt
build.lib.entry
build.lib.name
build.outDir
rollupOptions.external
output format
entryFileNames
```

重點結論是：

```txt
正式打包入口是 src/index.js
輸出產物在 dist
Vue 被視為外部依賴 external
產物包含 UMD / ES 類型
```

這說明：

> `src/index.js` 是真正的 library 入口，所以它必須在很前面讀。

### 5.3 `vite.config.dev.js` 要看什麼

你要觀察：

```txt
root: examples
server.port
resolve.alias
```

重點結論是：

```txt
開發站入口在 examples
examples 可以透過 alias 或相對路徑引用 src
```

這說明：

> examples 不是可有可無的展示資料夾，而是元件開發時的使用者視角驗證場。

### 5.4 這一階段的筆記產出

建議建立：

```txt
01_入口與安裝機制/01_vite_config_library_build.md
01_入口與安裝機制/02_vite_config_dev_examples.md
```

最低完成標準：

```md
## 正式打包入口
- 源碼：vite.config.js
- library entry：src/index.js
- 輸出：dist

## 開發入口
- 源碼：vite.config.dev.js
- dev root：examples
- 作用：讓 examples 直接驗證 src 內元件
```

## 6. 第 2 階段：理解 Vue Plugin 入口

### 6.1 建議閱讀檔案

```txt
src/index.js
```

這是第一個真正必須深讀的源碼檔。

### 6.2 建議閱讀順序

讀 `src/index.js` 時，不要從第一行逐字卡住。建議先切成七塊：

```txt
1. export * from './components'
2. import * as components from './components'
3. import localeFile from './locale/index'
4. import directives
5. 建立 directives map
6. 建立 ViewUI map 與 iButton / iInput 等別名
7. install(app, opts)
8. 掛載 globalProperties
9. 匯出 version / locale / i18n / lang / default API
```

### 6.3 這一階段要看懂什麼

你要能說出：

```txt
app.use(ViewUIPlus)
  ↓
執行 install(app, opts)
  ↓
處理 locale / i18n
  ↓
遍歷 ViewUI 註冊全域元件
  ↓
遍歷 directives 註冊全域指令
  ↓
設定 app.config.globalProperties.$VIEWUI
  ↓
掛載 $Spin / $Loading / $Message / $Notice / $Modal 等命令式 API
```

### 6.4 特別觀察點

#### 觀察點一：為什麼要有 `ViewUI` map？

`ViewUI` 不只是直接等於 components，它還補了一批 `iXxx` 別名：

```txt
iButton
iCircle
iCol
iContent
iForm
iInput
iMenu
iSelect
iTable
...
```

這代表它要相容或支援某些命名使用習慣。

#### 觀察點二：為什麼 `$VIEWUI` 很重要？

`$VIEWUI` 是全域設定容器，例如：

```txt
size
capture
transfer
cell
menu
modal
tabs
select
colorPicker
cascader
tree
datePicker
timePicker
typography
space
image
```

後面讀 Button 時，你會看到 Button 的 `size` 會讀全域 `$VIEWUI.size`。這就是入口設定與單一元件實作的連接點。

#### 觀察點三：為什麼 Message / Modal / Notice 要全域掛載？

因為這類元件不一定只靠 template 使用，它們常常是命令式呼叫：

```txt
this.$Message
this.$Modal
this.$Notice
```

所以它們和普通 Button / Input 的閱讀方式不同。

### 6.5 這一階段的筆記產出

建議建立：

```txt
01_入口與安裝機制/03_src_index_install流程.md
01_入口與安裝機制/04_globalProperties_全域能力掛載.md
01_入口與安裝機制/05_directives_全域指令註冊.md
```

最低完成標準：

```md
# src/index.js install 流程

## 原始碼位置
- src/index.js

## 核心流程
1. 匯入 components
2. 匯入 locale
3. 匯入 directives
```

## 7. 第 3 階段：理解元件對外表面

### 7.1 建議閱讀檔案

```txt
src/components/index.js
```

### 7.2 這一階段要看懂什麼

這個檔案的重點不是每一行都很難，而是它回答一個關鍵問題：

> View UI Plus 對外提供哪些元件名稱？

你要整理出：

```txt
基礎元件：Button / Icon / Divider / Space / Tag

資料展示：Table / List / Card / Description / Tree / Timeline
回饋元件：Modal / Message / Notice / Drawer / Spin / Alert

```

因為你要先知道：

```txt
哪些元件是對外 exported component

哪些命名是一組父子元件
哪些其實是命令式 API 的背後實作
```

例如：

```txt
Button + ButtonGroup
Collapse + Panel
Tabs + TabPane
Menu + MenuItem + MenuGroup + Submenu
Select + Option + OptionGroup
Checkbox + CheckboxGroup
Radio + RadioGroup
Timeline + TimelineItem
List + ListItem + ListItemMeta
```

這些不能只當作單一孤立元件看。

### 8.1 建議閱讀檔案

```txt
examples/main.js
```

### 8.2 讀 examples/main.js 時要看什麼

你要觀察：

```txt
createApp(App)
createRouter(...)
routes: [{ path, component: () => import(...) }]
import ViewUIPlus from '../src/index'
app.use(ViewUIPlus)
app.use(router)
app.mount('#app')
```

這裡的價值是：

```txt
examples 直接引用 src/index.js

```

代表開發者在本地開發時，是用 examples 當作測試與展示入口。

### 8.3 讀 demo router 時要看什麼

每個 demo router 不要當成一般頁面讀，而要當成「API 使用案例」讀。

例如讀 `examples/routers/button.vue` 時，你要整理：

```txt
Button 展示了哪些 type？
Button 展示了哪些 size？
Button 是否展示 loading？
Button 是否展示 icon？
Button 是否展示 shape？
Button 是否展示 long / ghost？
Button 是否展示 ButtonGroup？
```

然後再回到 `src/components/button/button.vue` 對照：

```txt
demo 用到的 prop 是否都有在 props 中定義？
demo 觸發的事件是否都有 emits / methods？
demo 出現的 class 效果是否能在 styles 中找到？
```

## 17. 每次讀單一元件的固定流程

無論讀 Button、Input、Select、Table，都用同一套流程。

```txt
1. 先看 examples demo
2. 再看 components/index.js 是否 export
3. 看該元件資料夾 index.js
4. 看主元件 .vue
5. 看子元件
6. 看 mixins / utils 依賴
7. 看 styles/components/*.less
8. 看 types 對應宣告
9. 最後才看 test 或 build 關聯
```

對每個元件都建立同一張表：

## 18. 卡住時的回退策略

### 18.1 讀大型元件看不懂

不要硬撐。退回：

```txt
components/index.js
  ↓
examples demo
  ↓
元件資料夾檔案樹
  ↓
props / emits
  ↓
只追一條流程
```

例如 Table 看不懂時，不要一次追 selection、sort、filter、scroll。先只追：

```txt
data 如何被渲染成 row / cell
```

---

### 18.2 mixins / utils 看不懂

退回使用它的元件。

```txt
不知道 mixins/form.js 為什麼存在
  ↓
回去看 Input / FormItem / Form
```

```txt
不知道 mixins/link.js 為什麼存在
  ↓
回去看 Button / Link
```

---

### 18.3 styles 看不懂

退回元件 computed classes。

```txt
不知道 button.less 的 class 怎麼來
  ↓
回去看 button.vue 的 classes computed
```

樣式閱讀順序永遠是：

```txt
元件 class 產生方式
  ↓
styles/components/*.less
  ↓
styles/custom.less 變數
  ↓
styles/mixins/*
```

---

### 18.4 examples 看不懂

退回 `examples/main.js` 的路由表。

```txt
某個 demo 頁太多內容
  ↓
先確認 route path
  ↓
找到對應 component
  ↓
只挑最小 demo 區塊
  ↓
回源碼對照 props
```

## 19. 不建議的閱讀方式

### 19.1 不建議從 `dist` 開始

`dist` 是打包結果，適合驗證產物，不適合學源碼。

### 19.2 不建議從 Table 開始

Table 是重型元件，會把資料、狀態、事件、layout、render、style 全部混在一起。第一輪直接讀 Table，容易只看到複雜度，看不到元件庫模式。

### 19.3 不建議只看 README 與文件

README / docs 能知道怎麼用，但不能知道：

```txt
元件如何註冊
全域 API 如何掛載
props 如何驗證
class 如何組裝
樣式如何集中管理
父子元件如何協作
```

### 19.4 不建議把每個檔案都逐行讀完

元件庫太大，逐行讀完不是最有效的方法。正確方式是：

```txt
先建立地圖
再挑代表性元件
再抽出模式
再回補共用層
最後進入重型元件
```
