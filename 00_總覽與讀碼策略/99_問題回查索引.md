# 99_問題回查索引

> 本篇任務：把你讀 View UI Plus 原始碼時產生的問題，整理成一個可以長期回查、補洞、驗證、複習的 **問題資料庫**。  
> 它不是單純 FAQ，也不是名詞表，而是用來回答：**我之前卡過什麼？答案在哪裡？對應哪段源碼？後續要補讀哪篇筆記？**

---

## 0. 本篇結論

`99_問題回查索引.md` 是整套筆記的最後入口，也是最常用的回查入口。

前面的筆記分工如下：

| 筆記 | 主要任務 |
|---|---|
| `00_完整目錄樹.md` | 管整套筆記架構 |
| `01_ViewUIPlus_專案定位.md` | 判斷 View UI Plus 是什麼類型的專案 |
| `02_源碼總目錄地圖.md` | 建立真實源碼地圖 |
| `03_建議閱讀順序.md` | 決定先讀什麼、後讀什麼 |
| `04_元件庫核心問題清單.md` | 定義讀碼時應該追問哪些問題 |
| `05_如何用_AI_輔助讀源碼.md` | 定義 AI 協助讀碼 SOP |
| `06_AI_提問模板.md` | 提供可直接複製的 prompt |
| `07_源碼覆蓋對照表.md` | 追蹤哪些源碼已讀、讀多深、筆記是否合格 |
| `99_問題回查索引.md` | 追蹤所有問題、答案狀態、對應源碼與回查路徑 |

所以這篇的核心不是：

```txt
我有哪些問題？
```

而是：

```txt
這個問題屬於哪一類？
它對應 View UI Plus 哪個真實源碼位置？
目前答案是源碼事實、個人推論，還是待驗證？
下次遇到類似問題時，應該回查哪篇筆記？
這個問題是否已經被整理成可複習的知識？
```

## 1. 源碼核對基準

本索引以 View UI Plus 官方原始碼為核對對象。

目前已確認的讀碼核心基準如下：

| 區域 | 代表源碼 | 對本索引的意義 |
|---|---|---|
| 專案定位 | `README.md` | 判斷 View UI Plus 是 Vue 3 元件庫與前端解決方案 |
| 套件入口 | `src/index.js` | 回查 `app.use`、`install`、全域註冊、全域配置、全域 API |
| 元件匯出 | `src/components/index.js` | 回查哪些元件是 public component surface |
| 使用入口 | `examples/main.js` | 回查 examples 如何使用 `app.use(ViewUIPlus)` 啟動整套庫 |
| 核心源碼 | `src/components/` | 回查單一元件 props、emits、slots、狀態、渲染與互動 |
| 共用機制 | `src/mixins/`、`src/utils/` | 回查跨元件複用邏輯 |
| 指令能力 | `src/directives/` | 回查全域 directive 註冊與使用場景 |
| 語系能力 | `src/locale/` | 回查 locale、i18n、lang 如何串接 |
| 樣式系統 | `src/styles/` | 回查 Less 入口、變數、mixin、元件樣式 |
| 型別契約 | `types/` | 回查使用者側 TypeScript API |
| 工程建置 | `package.json`、`vite.config.js`、`build/` | 回查 library build、style build、language pack build |

### 1.1 這篇不能接受的問題寫法

以下寫法太空泛，不適合作為長期索引：

```txt
Button 怎麼寫？
Table 好複雜。
Message 是什麼？
樣式怎麼管理？
```

### 1.2 這篇應該接受的問題寫法

問題必須可以回到源碼：

```txt
Q-COMP-BUTTON-001：Button 為什麼透過 mixinsLink 決定渲染 button 或 a？
Q-ARCH-002：src/index.js 的 install(app, opts) 做了哪些全域註冊？
Q-CMD-001：$Message 為什麼掛到 app.config.globalProperties，而不是只 export 元件？
Q-STYLE-003：元件樣式是集中由 src/styles/index.less 匯入，還是由元件內部各自引入？
Q-I18N-002：opts.locale 與 opts.i18n 分別在安裝流程中扮演什麼角色？
```

## 2. 問題狀態規則

所有問題都要有狀態。否則問題越累積越多，最後會變成雜訊。

| 狀態 | 名稱 | 說明 | 是否可視為完成 |
|---|---|---|---|
| `Q0` | 未整理 | 只是臨時想到，還沒分類 | 否 |
| `Q1` | 已分類 | 已知道它屬於架構、元件、樣式、工程等哪一類 | 否 |
| `Q2` | 已定位 | 已找到對應源碼位置 | 否 |
| `Q3` | 已初答 | 已有初步答案，但還未完全驗證 | 否 |
| `Q4` | 已驗證 | 答案已對照源碼或官方用法驗證 | 是 |
| `Q5` | 已沉澱 | 已整理成可複習筆記、可遷移設計、或仿寫練習 | 是 |

### 2.1 問題完成標準

一個問題至少要到 `Q4`，才算真正回答完成。

`Q4` 最低要求：

```txt
1. 有清楚問題。
2. 有問題分類。
3. 有對應源碼位置。
4. 有答案摘要。
5. 有「源碼事實 / 我的推論 / 待驗證」區分。
6. 有回查筆記連結。
```

`Q5` 進階要求：

```txt
1. 能整理成固定筆記段落。
2. 能抽象成設計原則。
3. 能用在自己仿寫元件或重構程式碼。
4. 能回填到 04、06、07 或具體元件筆記。
```

## 3. 問題分類代碼

建議每個問題都用固定代碼，方便搜尋。

| 代碼 | 分類 | 適用問題 |
|---|---|---|
| `ARCH` | 架構 / 插件入口 | `app.use`、`install`、全域註冊、全域配置 |
| `EXPORT` | 對外匯出面 | `components/index.js`、命名、public API surface |
| `COMP` | 單一元件 | Button、Input、Select、Table、DatePicker 等 |
| `API` | 元件 API 設計 | props、emits、slots、v-model、expose |
| `STATE` | 狀態管理 | open/close、active、selected、checked、loading、disabled |
| `COMM` | 元件通訊 | provide/inject、父子協作、複合元件協議 |
| `CMD` | 命令式元件 | Message、Notice、Modal、LoadingBar、Spin |
| `DIR` | 指令 | resize、line-clamp、style 類 directive |
| `MIXIN` | mixin / 共用邏輯 | link、form、emitter、可重用行為 |
| `UTIL` | 工具函式 | DOM、型別判斷、驗證、格式化、helper |
| `STYLE` | 樣式系統 | Less、變數、mixin、元件樣式入口、主題覆蓋 |
| `I18N` | 語系 / 國際化 | locale、i18n、lang、語系包 |
| `EXAMPLE` | examples / demo | router demo、使用者視角、互動案例 |
| `TYPE` | TypeScript 型別 | `types/`、`.d.ts`、對外型別契約 |
| `BUILD` | 建置 / 發布 | Vite library mode、package scripts、dist、style build |
| `TEST` | 測試 | test coverage、測試缺口、回歸保護 |
| `VUE` | Vue 基礎 | computed、watch、nextTick、slots、directive、globalProperties |
| `REFACTOR` | 可遷移設計 | 可學走、可仿寫、可套用到自己專案的設計 |

### 3.1 問題 ID 命名規則

建議格式：

```txt
Q-{分類}-{主題}-{序號}
```

範例：

```txt
Q-ARCH-INSTALL-001
Q-EXPORT-COMPONENTS-001
Q-COMP-BUTTON-001
Q-API-VMODEL-001
Q-CMD-MESSAGE-001
Q-STYLE-LESS-001
Q-I18N-LOCALE-001
Q-BUILD-VITE-001
```

命名原則：

- `分類` 用上表代碼。
- `主題` 用源碼或機制名稱。
- `序號` 固定三位數，方便排序。
- 不要用中文當 ID，中文寫在問題標題即可。

## 5. P0 必建問題索引

以下問題是第一輪讀 View UI Plus 時最值得建立的索引。它們對應整套元件庫的骨架。

### 5.1 架構入口問題

| 問題 ID | 問題 | 對應源碼 | 狀態 | 回查筆記 |
|---|---|---|---|---|
| `Q-ARCH-INSTALL-001` | `app.use(ViewUIPlus)` 背後執行什麼流程？ | `src/index.js`、`examples/main.js` | Q2 | `03`、`04`、`05` |
| `Q-ARCH-INSTALL-002` | `install(app, opts)` 為什麼需要防止重複安裝？ | `src/index.js` | Q1 | `04` |
| `Q-ARCH-GLOBAL-001` | 為什麼要把 `$VIEWUI` 掛到 `app.config.globalProperties`？ | `src/index.js` | Q2 | `04` |
| `Q-ARCH-GLOBAL-002` | `$VIEWUI` 目前管理哪些全域配置？ | `src/index.js` | Q2 | `07` |
| `Q-ARCH-DIRECTIVE-001` | 全域 directive 是在哪裡被註冊的？ | `src/index.js`、`src/directives/` | Q2 | `07` |
| `Q-ARCH-DAYJS-001` | 為什麼 `$Date` 會掛到全域？ | `src/index.js` | Q1 | `04` |

#### 第一輪答案摘要

- `examples/main.js` 可以看到使用者側透過 `app.use(ViewUIPlus)` 啟動整套元件庫。
- `src/index.js` 是核心安裝入口，負責匯入 components、directive、locale、dayjs 與 package version。
- `install(app, opts)` 會註冊元件、註冊 directive，並掛載 `$VIEWUI`、`$Message`、`$Modal`、`$Notice`、`$Spin`、`$Loading` 等全域能力。
- 這類問題要優先讀，因為它解釋了「View UI Plus 作為 Vue plugin」的使用模型。

### 5.2 元件匯出問題

| 問題 ID | 問題 | 對應源碼 | 狀態 | 回查筆記 |
|---|---|---|---|---|
| `Q-EXPORT-COMPONENTS-001` | `src/components/index.js` 匯出的元件是否都會被全域註冊？ | `src/components/index.js`、`src/index.js` | Q2 | `02`、`07` |
| `Q-EXPORT-COMPONENTS-002` | 為什麼部分元件有 `iButton`、`iTable` 這類別名？ | `src/index.js` | Q2 | `04` |
| `Q-EXPORT-COMPONENTS-003` | 哪些匯出是基礎元件，哪些是命令式能力？ | `src/components/index.js` | Q1 | `07` |
| `Q-EXPORT-COMPONENTS-004` | `components/index.js` 是使用者 API 邊界，還是內部組裝檔？ | `src/components/index.js` | Q1 | `02` |

#### 第一輪答案摘要

- `src/components/index.js` 是 View UI Plus 對外元件匯出總表。
- `src/index.js` 會 `import * as components from './components'`，再透過 `ViewUI` 物件集中註冊。
- 所以讀單一元件前，應先知道它是否真的出現在 `components/index.js` 的對外匯出清單中。

### 5.3 Button 入門元件問題

| 問題 ID | 問題 | 對應源碼 | 狀態 | 回查筆記 |
|---|---|---|---|---|
| `Q-COMP-BUTTON-001` | Button 的 props 如何定義外觀、尺寸、狀態與行為？ | `src/components/button/button.vue` | Q1 | 後續 Button 筆記 |
| `Q-COMP-BUTTON-002` | Button 為什麼會用 mixin 處理 link 與 form 行為？ | `button.vue`、`src/mixins/` | Q1 | 後續 mixin 筆記 |
| `Q-COMP-BUTTON-003` | Button 如何決定 class name？ | `button.vue` | Q1 | 後續 Button 筆記 |
| `Q-COMP-BUTTON-004` | Button 為什麼使用 render function，而不是純 template？ | `button.vue` | Q1 | 後續 Button 筆記 |
| `Q-COMP-BUTTON-005` | Button 的 loading、disabled、icon、children 如何組合渲染？ | `button.vue` | Q1 | 後續 Button 筆記 |

#### 第一輪答案摘要

Button 適合作為第一個深讀元件，因為它同時包含：

- props validator
- computed class 組裝
- mixin 複用
- slot / children 渲染
- icon / loading 狀態
- button / link 行為分支
- render function

它的複雜度適中，比 Table、Select、DatePicker 更適合入門。

- 全域 API 設計
- programmatic UI
- instance 管理
- DOM 掛載與銷毀

## 8. 依 Vue 基礎觀念回查

這一區不是 View UI Plus 特有知識，而是你讀原始碼時常會遇到的 Vue 機制。

| Vue 概念 | 在 View UI Plus 可能出現的問題 | 優先處理方式 |
|---|---|---|
| `computed` | 為什麼 class、style、狀態要用 computed？ | 先理解資料衍生，再看元件實作 |
| `watch` | 為什麼要監聽 props 或內部狀態？ | 找狀態同步或副作用 |
| `nextTick` | 為什麼要等 DOM 更新？ | 找 DOM measurement、focus、popup 更新 |
| `emits` | 事件是狀態同步還是通知？ | 對照 props 與 v-model |
| `slots` | slot 是版型擴充還是資料傳遞？ | 看 render/template 如何插入 children |
| `provide/inject` | 為什麼不用層層 props？ | 通常出現在複合元件協作 |
| `directive` | 為什麼寫成 directive，不放在元件裡？ | 看是否是可跨元件複用的 DOM 行為 |
| `globalProperties` | 為什麼 `$Message` 可以掛全域？ | 回查 `src/index.js` |
| `h()` / render function | 為什麼不用 template？ | 看動態 tag、動態 children、程式化渲染需求 |
| `Teleport` | 彈層是否需要脫離當前 DOM 結構？ | 回查 Modal、Drawer、Poptip、Tooltip 等彈層類元件 |

### 8.1 Vue 問題的處理順序

遇到 Vue 基礎問題時，不要直接記成「View UI Plus 很難」。

先做三步：

```txt
1. 把問題抽象成 Vue 概念。
2. 用最小範例確認這個 Vue 概念。
3. 回到 View UI Plus 看它如何工程化使用。
```

範例：

```txt
我看不懂 Button 的 render function
→ 先查 Vue 的 h() 與 render function
→ 再回 button.vue 看它為何需要動態組 children、icon、loading、tag
→ 最後整理成 Q-COMP-BUTTON-004
```

## 12. 問題資料庫範例

### Q-ARCH-INSTALL-001：`app.use(ViewUIPlus)` 背後執行什麼流程？

| 欄位 | 內容 |
|---|---|
| 狀態 | Q2 已定位 |

| 對應筆記 | `01`、`02`、`03`、`04`、`07` |
| 關鍵字 | app.use, install, plugin, component registration, globalProperties |

#### 問題

`examples/main.js` 中呼叫 `app.use(ViewUIPlus)` 後，View UI Plus 到底做了哪些事情？

#### 源碼事實

- `examples/main.js` 建立 Vue app 後呼叫 `app.use(ViewUIPlus)`。
- `src/index.js` export `install`。
- `install(app, opts)` 會註冊 components。
- `install(app, opts)` 會註冊 directives。
- `install(app, opts)` 會設定 `$VIEWUI`。
- `install(app, opts)` 會掛載 `$Message`、`$Notice`、`$Modal` 等全域能力。

#### 我的推論

View UI Plus 把自己設計成 Vue plugin，讓使用者只要 `app.use(ViewUIPlus)` 就能取得完整元件、全域方法、語系設定與指令能力。

#### 待驗證

- 補讀 `src/components/index.js`，確認所有匯出與註冊關係。
- 補讀 `src/directives/`，確認每個 directive 實際用途。
- 補讀 `src/locale/index.js`，確認 locale 與 i18n 的行為。
