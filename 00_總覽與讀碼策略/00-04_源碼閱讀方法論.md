# 00-04_源碼閱讀方法論

> 適用專案：View UI Plus  
> 適用對象：已有 Vue 開發經驗，想透過開源元件庫提升「元件設計能力、讀碼能力、企業後台封裝能力」的人  
> 本篇定位：這不是某一個元件的源碼解析，而是閱讀 View UI Plus 全專案時要遵守的「方法論、流程、模板與檢核標準」。

---

## 0. 這篇筆記要解決什麼問題？

讀 View UI Plus 這類元件庫，很容易遇到幾個問題：

1. 專案檔案很多，不知道從哪裡開始。
2. 每個元件都有 props、emits、slots、樣式、工具函式、mixin，容易看得很散。
3. 只看懂單行程式碼，但沒有看懂元件設計思路。
4. 只會使用元件 API，卻無法自己封裝企業後台元件。
5. 讀完後沒有產出，過幾天就忘記。
6. 過度依賴 AI 逐行解釋，卻沒有建立自己的讀碼判斷能力。

所以本篇的目的，是建立一套固定方法：

```txt
先理解定位
再找到入口
再分析 API
再追內部狀態
再看互動流程
再看共用機制
最後抽象成自己的設計模式與仿寫能力
```

簡單說：

```txt
不要只是「看懂 View UI Plus」，
而是要透過 View UI Plus 學會「怎麼設計一套 Vue 3 元件庫」。
```

---

## 1. 源碼閱讀的核心原則

### 1.1 先使用，再讀碼

不要一開始就衝進 `src/components` 看源碼。

正確順序應該是：

```txt
官方文件 / examples
↓
實際使用方式
↓
元件 API 表面行為
↓
源碼入口
↓
內部實作
↓
設計模式
↓
仿寫
```

原因是：元件庫的源碼不是單純的函式邏輯，而是「為了支援某種使用方式」才設計出來的。

如果你不知道使用者怎麼用，就很難理解為什麼 props 要這樣設計、為什麼 emits 要這樣命名、為什麼內部狀態要這樣拆。

---

### 1.2 先看主線，再看分支

閱讀源碼時，不要一開始就追每個細節。

優先看：

```txt
1. 入口在哪裡？
2. 使用者傳入什麼？
3. 元件內部產生什麼狀態？
4. 畫面如何渲染？
5. 使用者操作後會觸發什麼事件？
6. 事件如何改變狀態？
7. 狀態如何再反映到畫面？
```

先把主線看懂，再補：

```txt
1. 邊界條件
2. 錯誤處理
3. disabled / readonly / loading
4. size / type / status
5. clearable / filterable / multiple
6. teleport / popup / z-index
7. locale / prefixCls / theme
8. TypeScript 型別
```

---

### 1.3 先 API，後實作

元件庫的本質是「對外 API 設計」。

所以讀單一元件時，要先問：

```txt
這個元件對使用者暴露了什麼能力？
```

包含：

```txt
1. Props：使用者可以控制什麼？
2. Emits：元件會通知使用者什麼？
3. Slots：使用者可以自訂哪裡？
4. Methods：是否暴露命令式方法？
5. v-model：是否支援雙向綁定？
6. Provide / Inject：是否參與父子協作？
7. Global Config：是否受全域配置影響？
```

等 API 面理解後，再看內部如何實作。

---

### 1.4 先關係，後細節

View UI Plus 這類元件庫通常不是單一檔案完成所有事情，而是由多個層次組成：

```txt
元件本體
↓
子元件
↓
mixin / composable / utils
↓
directives
↓
locale
↓
styles
↓
types
↓
examples
↓
test
```

讀碼時不要只盯著一個 `.vue` 檔案，要畫出它和其他模組的關係。

---

### 1.5 先產出，再追求完整

不要要求自己第一次就讀完整個元件。

每次讀碼至少產出一個明確成果：

```txt
1. 一張流程圖
2. 一份 API 表
3. 一個狀態流轉表
4. 一個最小仿寫版本
5. 一個設計模式總結
6. 一個企業後台應用範例
```

如果沒有產出，只是「看過」，很快就會忘記。

---

## 2. View UI Plus 的四層閱讀視角

閱讀 View UI Plus 時，不要只用「檔案」角度看，要切成四個視角。

---

### 2.1 使用者視角：我怎麼用這個元件？

例如：

```vue
<template>
  <Button type="primary" loading @click="handleSubmit">
    送出
  </Button>
</template>
```

使用者視角要關注：

```txt
1. 這個元件解決什麼 UI 問題？
2. 使用者最常怎麼寫？
3. 有哪些 props？
4. 有哪些事件？
5. 有哪些 slot？
6. 是否支援 v-model？
7. 是否支援全域配置？
8. 是否支援樣式客製化？
```

這是理解源碼的起點。

---

### 2.2 框架視角：這個元件如何被安裝到 Vue App？

例如：

```ts
import { createApp } from 'vue'
import ViewUIPlus from 'view-ui-plus'
import 'view-ui-plus/dist/styles/viewuiplus.css'

createApp(App).use(ViewUIPlus).mount('#app')
```

框架視角要關注：

```txt
1. ViewUIPlus 對外 export 了什麼？
2. install 方法在哪裡？
3. 每個元件如何被 app.component 註冊？
4. 全域配置如何傳入？
5. globalProperties 是否被掛載？
6. 命令式 API 是否掛到 app.config.globalProperties？
```

這個視角對應後續章節：

```txt
06_src_入口與插件機制
07_元件註冊機制
08_全域配置與globalProperties
09_元件庫整體設計模式
```

---

### 2.3 元件設計視角：這個元件內部如何運作？

元件設計視角要關注：

```txt
1. template / render 如何組織？
2. props 如何定義？
3. emits 如何定義？
4. data / ref / reactive 狀態有哪些？
5. computed 負責推導什麼？
6. watch 負責同步什麼？
7. methods / function 負責處理哪些互動？
8. 是否有 provide / inject？
9. 是否依賴其他子元件？
10. 是否依賴 utils / mixins？
```

這是閱讀 `Button`、`Input`、`Form`、`Table`、`Modal`、`Select` 等元件時的主視角。

---

### 2.4 系統設計視角：這套元件庫抽象出了什麼共用機制？

系統設計視角要關注：

```txt
1. prefixCls 如何統一 class 命名？
2. locale 如何處理國際化？
3. form / form-item 如何協作？
4. modal / message / notice 如何實作命令式 API？
5. directives 如何封裝 DOM 行為？
6. mixins 如何抽取共用邏輯？
7. utils 如何支撐多個元件？
8. styles 如何支撐主題與樣式一致性？
9. types 如何對外提供型別支援？
10. test 如何反推元件規格？
```

這個視角對應後續高階專題：

```txt
28_高階專題_元件API設計
29_高階專題_受控與非受控狀態
30_高階專題_彈層與定位系統
31_高階專題_命令式API設計
32_高階專題_表單架構設計
33_高階專題_樣式主題與PrefixCls
34_高階專題_Mixin與CompositionAPI重構
35_高階專題_測試反推元件規格
```

## 3. 單一元件源碼閱讀流程

每次閱讀一個元件，都固定使用這個流程。

---

### 3.1 第一步：先看使用場景

先不要看源碼，先整理這個元件的使用場景。

範例：

```txt
元件：Input

使用場景：
1. 一般文字輸入
2. 密碼輸入
3. 搜尋輸入
4. 可清除輸入
5. 禁用狀態
6. readonly 狀態
7. 搭配 FormItem 驗證
8. 搭配 prefix / suffix icon
9. 支援 v-model
```

這一步要回答：

```txt
這個元件為什麼存在？
```

---

### 3.2 第二步：整理對外 API

用表格整理 props、emits、slots。

範例：

```md
## Props

| prop | 類型 | 預設值 | 用途 | 分類 |
|---|---|---|---|---|
| modelValue | string / number | - | 綁定值 | 狀態 |
| disabled | boolean | false | 是否禁用 | 行為 |
| size | string | default | 尺寸 | 樣式 |
| clearable | boolean | false | 是否可清除 | 行為 |

## Emits

| event | 觸發時機 | 參數 |
|---|---|---|
| update:modelValue | 值變化時 | value |
| change | change 事件時 | event |
| focus | focus 時 | event |
| blur | blur 時 | event |

## Slots

| slot | 用途 |
|---|---|
| prefix | 前綴內容 |
| suffix | 後綴內容 |
```

整理 API 時，建議把 props 分類：

```txt
1. 狀態型 props：modelValue、value、visible
2. 樣式型 props：type、size、shape、status
3. 行為型 props：disabled、readonly、clearable、filterable
4. 資料型 props：data、options、columns
5. 結構型 props：placement、appendToBody、transfer
6. 進階型 props：render、formatter、validator
```

---

### 3.3 第三步：找到源碼入口

常見尋找方式：

```bash
# 找元件入口
rg "Button" src

# 找 install
rg "install" src

# 找元件 export
rg "export.*Button" src

# 找樣式 class 前綴
rg "prefixCls" src

# 找 modelValue / update:modelValue
rg "modelValue|update:modelValue" src
```

如果沒有安裝 `ripgrep`，也可以用 VS Code 全域搜尋。

源碼入口通常要記錄：

```md
## 源碼入口

- 元件主檔：
- install 檔：
- 樣式檔：
- 子元件：
- 依賴的 mixin：
- 依賴的 utils：
- 相關 types：
- 相關 examples：
- 相關 test：
```

---

### 3.4 第四步：閱讀元件結構

閱讀元件時，先看大結構：

```txt
1. template / render
2. script
3. props
4. emits
5. setup / data
6. computed
7. watch
8. methods
9. lifecycle
10. style class
```

不要一開始就追每個 function 的細節。

先畫出元件骨架：

```txt
Component
├── Props
├── Emits
├── Slots
├── Internal State
├── Computed State
├── Event Handlers
├── Child Components
├── Utils
└── Styles
```

---

### 3.5 第五步：分析內部狀態

每個元件都要整理「狀態表」。

範例：

```md
## 內部狀態表

| 狀態名稱 | 來源 | 是否由 props 控制 | 何時改變 | 影響哪裡 |
|---|---|---|---|---|
| currentValue | props.modelValue | 是 | input/change | input 顯示值 |
| focused | DOM event | 否 | focus/blur | class 狀態 |
| isDisabled | props + form context | 是 | props/context 改變 | 是否可操作 |
```

這一步非常重要，因為 UI 元件的核心就是：

```txt
狀態 → 畫面
互動 → 事件
事件 → 狀態
狀態 → 畫面更新
```

---

### 3.6 第六步：分析事件流程

每個互動都要畫流程。

範例：Input 輸入流程

```txt
使用者輸入
↓
原生 input event
↓
元件 handleInput
↓
取得 event.target.value
↓
更新內部狀態
↓
emit update:modelValue
↓
父層資料更新
↓
props.modelValue 傳回元件
↓
畫面重新渲染
```

事件流程要特別關注：

```txt
1. 原生事件和自定義事件如何對應？
2. 是否有防抖 / 節流？
3. 是否有 compositionstart / compositionend？
4. 是否有 disabled / readonly 阻擋？
5. 是否有表單驗證觸發？
6. 是否有 emit 多個事件？
7. 是否有同步內部狀態與外部 props？
```

---

### 3.7 第七步：分析受控與非受控狀態

很多元件都會遇到這個問題：

```txt
值到底由父層控制，還是元件自己控制？
```

閱讀時要判斷：

```txt
1. 是否有 v-model？
2. 是否有 defaultValue / defaultVisible 這類初始值？
3. props 改變時，內部狀態是否同步？
4. 內部狀態改變時，是否 emit 給父層？
5. 父層不更新 props 時，畫面會不會更新？
```

判斷公式：

```txt
受控元件：
資料來源主要是 props，元件透過 emit 通知父層更新。

非受控元件：
資料主要存在元件內部，父層只給初始值或不參與控制。

混合型元件：
同時支援外部控制與內部預設行為。
```

---

### 3.8 第八步：分析父子協作

像 `Form / FormItem`、`Menu / MenuItem`、`Select / Option`、`Table / Column` 這類元件，不是單一元件能看懂的。

要整理父子關係：

```txt
父元件提供什麼？
↓
子元件注入什麼？
↓
子元件何時向父元件註冊？
↓
父元件如何管理子元件？
↓
父元件如何通知子元件？
↓
子元件如何回報狀態？
```

常見協作方式：

```txt
1. provide / inject
2. parent / children 註冊
3. event bus
4. props 傳遞
5. slot scope
6. context object
7. shared mixin
```

筆記模板：

```md
## 父子協作

| 角色 | 職責 | 傳遞方式 |
|---|---|---|
| Form | 管理 model、rules、validate 方法 | provide |
| FormItem | 註冊欄位、執行單欄驗證 | inject |
| Input | 觸發 blur/change 驗證 | event / dispatch |
```

---

### 3.9 第九步：分析樣式與 class 設計

元件庫的樣式不是隨便寫的。

閱讀樣式時要看：

```txt
1. class 前綴如何產生？
2. base class 是什麼？
3. modifier class 是什麼？
4. 狀態 class 是什麼？
5. size class 如何處理？
6. type class 如何處理？
7. disabled / active / focused / error 如何處理？
8. Less 變數如何被使用？
9. 樣式是否和 props 對應？
```

建議整理成：

```md
## Class 對照表

| class | 來源 | 觸發條件 | 用途 |
|---|---|---|---|
| ivu-btn | 固定 base class | 永遠存在 | 基礎樣式 |
| ivu-btn-primary | props.type | type = primary | 主色按鈕 |
| ivu-btn-loading | props.loading | loading = true | loading 狀態 |
| ivu-btn-disabled | props.disabled | disabled = true | 禁用樣式 |
```

---

### 3.10 第十步：分析邊界條件

只看 happy path 不夠。

每個元件都要問：

```txt
1. disabled 時事件是否還會觸發？
2. loading 時是否阻止重複操作？
3. value 為 null / undefined / 空字串時如何處理？
4. props 類型錯誤會怎樣？
5. slot 缺失時如何顯示？
6. 子元件不存在時是否會報錯？
7. 元件卸載時是否清理事件？
8. 彈層是否會清理 DOM？
9. 非同步流程是否有 race condition？
10. 鍵盤操作是否支援？
```

這一步可以搭配 test 或 examples 反推。

---

### 3.11 第十一步：抽象成設計模式

看完源碼後，不要只寫「它做了什麼」，還要寫「它屬於什麼模式」。

例如：

```txt
Button：
- props 驅動樣式模式
- loading 狀態阻斷互動模式
- slot 作為內容擴展模式

Input：
- v-model 受控狀態模式
- 原生事件包裝模式
- FormItem 驗證觸發模式

Form：
- 父子協作模式
- provide / inject 表單上下文模式
- 集中式 validate 管理模式

Message：
- 命令式 API 模式
- 動態掛載元件模式
- 單例容器管理模式

Table：
- 資料驅動渲染模式
- columns schema 配置模式
- render function 擴展模式
```

這才是讀碼真正的價值。

---

### 3.12 第十二步：做最小仿寫

每讀完一個核心元件，都應該做一個 mini 版本。

仿寫不是照抄，而是保留核心設計。

範例：

```txt
讀完 Button：
產出 MiniButton

讀完 Input：
產出 MiniInput

讀完 Form / FormItem：
產出 MiniForm

讀完 Modal：
產出 MiniModal

讀完 Message：
產出 MiniMessage

讀完 Table：
產出 MiniTable
```

最小仿寫標準：

```txt
1. API 名稱可以簡化
2. 樣式可以簡化
3. 邊界條件可以少一點
4. 但核心資料流不能省
5. 核心互動流程不能省
6. 核心設計模式要保留
```

## 4. 橫向機制源碼閱讀流程

有些內容不能用「單一元件」的方式讀，而要用「橫向機制」的方式讀。

例如：

```txt

3. prefixCls 機制

5. 彈層定位機制

7. 國際化機制
8. 自訂指令機制
9. mixin 共用邏輯
10. TypeScript 對外宣告機制
```

---

### 4.1 橫向機制閱讀公式

```txt
使用入口

↓
核心流程
↓
參與模組
↓
狀態流向
↓
擴展點
↓
邊界處理
↓

↓

```

---

### 4.2 橫向機制筆記模板


---

### 4.3 橫向機制分析範例：命令式 API

命令式 API 常見於：

```txt
Message
Notice
Modal.confirm
LoadingBar
```

閱讀時要問：

```txt
1. 使用者是怎麼呼叫的？
2. 呼叫的是函式還是元件？
3. 函式如何建立 Vue 元件實例？
4. DOM 掛載到哪裡？
5. 多個 message 如何排隊？
6. 關閉時如何移除？
7. 是否有單例容器？
8. 是否掛到 globalProperties？
9. 是否支援 Promise / callback？
10. SSR 場景如何處理？
```

命令式 API 的核心不是 template，而是：

```txt
函式呼叫
↓
建立配置
↓
動態建立元件
↓
掛載 DOM
↓
顯示 UI
↓
更新 / 關閉
↓
銷毀 DOM
```

---

### 4.4 橫向機制分析範例：表單驗證

表單驗證要從三層看：

```txt
Form：管理整份表單
FormItem：管理單一欄位
Input / Select / DatePicker：觸發欄位事件
```

要追的流程：

```txt
使用者輸入
↓
Input emit change / blur
↓
FormItem 接收到驗證觸發
↓
根據 prop 找 model 中的值
↓
根據 rules 找驗證規則
↓
執行 validator
↓
更新 validateState / validateMessage
↓
顯示錯誤樣式與錯誤文字
↓
Form.validate 彙總所有 FormItem 結果
```

這種章節不要只看一個檔案，要跨檔案追。

## 9. 常見讀碼錯誤

### 9.1 一開始就讀最難的元件

不建議一開始就讀：

```txt
Table
Tree
Select
DatePicker
Upload
```

這些元件依賴大量狀態與子元件，初期容易卡住。

建議順序：

```txt
Button
Icon
Input
Checkbox / Radio
Form / FormItem
Modal
Message
Select
Table
DatePicker
Tree
```

---

### 9.2 只抄 API，不分析設計

錯誤筆記：

```txt
Button 有 type、size、loading、disabled。
```

更好的筆記：

```txt
Button 的 API 可以分成三類：
1. 樣式型：type、size、shape
2. 狀態型：loading、disabled
3. 結構型：icon、slot

其中 loading 不只是樣式狀態，也會影響互動行為，因此它屬於「樣式 + 行為」混合型 props。
```

---

### 9.3 只看元件，不看 examples

examples 是理解元件行為最快的方式。

因為 examples 通常會展示：

```txt
1. 最常見用法
2. 邊界用法
3. 組合用法
4. 官方想強調的 API
5. 文件沒有講清楚的互動細節
```

---

### 9.4 只看主檔，不看 utils / mixins

元件庫常把真正重要的邏輯抽到：

```txt
src/utils
src/mixins
src/directives
src/locale
```

如果只看元件主檔，會看不懂很多設計。

---

### 9.5 只問 AI，不自己畫流程

AI 解釋完後，你一定要自己補三個東西：

```txt
1. 流程圖
2. 狀態表
3. 仿寫版本
```

否則很容易變成「聽懂了，但寫不出來」。
