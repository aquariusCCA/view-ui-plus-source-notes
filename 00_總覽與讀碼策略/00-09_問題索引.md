## 所屬章節

`00_總覽與讀碼策略`

## 本節導讀

這篇是 View UI Plus 源碼閱讀時的問題管理入口。讀源碼一定會遇到暫時不懂、證據不足、需要跨檔案查證、需要之後回頭驗證的問題；如果這些問題只散落在每篇筆記底部，很快就會變成「記得有看過，但想不起來在哪裡」。

本篇要解決的是：

1. 讀碼時遇到問題，應該如何記錄？
2. 哪些問題要立刻追，哪些可以暫緩？
3. 一個問題要補到什麼程度，才算可以結案？
4. 如何從問題快速回到對應章節、源碼入口、外部證據與下一步？
5. 如何避免問題索引變成只有問題、沒有行動的清單？

本篇的核心觀念是：

> 問題索引不是疑問堆放處，而是下一次讀碼的入口表。  
> 每個問題都應該有分類、狀態、證據缺口、下一入口與結案條件。

如果 `00-06_追蹤檢查表.md` 用來判斷「本輪源碼是否追完」，`00-08_學習里程碑.md` 用來判斷「本階段是否完成」，那本篇就是用來管理「還沒追完、需要再回來」的所有問題。

## 建議回查情境

當你遇到以下情境，可以回到這篇：

- 讀碼過程中有問題，但不想打斷當前主線。
- 某個結論只有命名推測，還沒有源碼證據。
- source、types、examples、tests 或 dist 之間出現不一致。
- 不確定某個問題應該先查入口、元件本體、樣式、型別、範例還是測試。
- 想知道哪些問題會阻塞目前階段，哪些可以放進暫緩清單。
- 完成一個里程碑後，要整理待驗證問題。
- 第二輪閱讀時，要從未解問題反推下一個閱讀入口。

## 證據標記

本篇延續 00 章的證據標記：

| 標記 | 意義 |
|---|---|
| 已確認 | 已從目前倉庫中的原始碼、設定檔、型別、範例、測試或產物直接讀到 |
| 合理推論 | 由已確認事實推導出的理解，但仍應保留推論依據 |
| 待驗證 | 目前缺少源碼、外部對照、執行結果或測試證據 |

| 轉入暫緩 | 問題仍有價值，但不影響當前主線，應移到 `00-10_暫緩閱讀清單.md` |

## 0. 五分鐘使用方式

如果只是讀碼時臨時記問題，先照這個最小流程。

```text
遇到問題
  -> 用一句話寫清楚
  -> 標記來源章節與源碼位置
  -> 判斷是否阻塞當前主線
  -> 不阻塞就放入問題索引
  -> 補上下一次回來時的第一入口
  -> 之後用證據結案或轉入暫緩
```

最小記錄格式：

| 欄位 | 必填原因 |
|---|---|
| 問題 | 沒有明確問題，就無法追蹤 |
| 所屬階段 | 方便連回 `00-08_學習里程碑.md` |
| 關聯章節 | 方便從筆記回查 |
| 第一入口 | 下次不用重新找起點 |
| 目前缺口 | 知道還差 source、types、examples、tests、runtime 驗證中的哪一種 |
| 狀態 | 區分待定位、待驗證、證據衝突、已結案、轉入暫緩 |

快速判斷：

| 狀況 | 放哪裡 |
|---|---|
| 會阻塞當前主流程 | 留在當前章節筆記，先追完 |
| 不阻塞但需要後續查證 | 放入本篇 |
| 已確定本輪不追，但日後可能要追 | 放入 `00-10_暫緩閱讀清單.md` |
| 已解決且有證據 | 留在本篇已結案區，並回填到相關章節 |
| 只是臨時想法，無法形成問題 | 先不收錄，避免索引變成雜訊 |

## 2. 問題生命週期

每個問題都應該在狀態之間移動，而不是永遠停在「待查」。

```text
新問題
  -> 待定位
  -> 待查證
  -> 待對照
  -> 待結論
  -> 已結案

特殊分支：
  -> 證據衝突
  -> 轉入暫緩
  -> 作廢
```

| 狀態 | 意義 | 下一步 |
|---|---|---|
| 新問題 | 剛被記下，只有問題描述 | 補所屬階段、關聯章節、第一入口 |
| 待定位 | 還不知道該從哪個檔案開始查 | 回到 `00-05` 或本篇第 9 節找入口 |
| 待查證 | 已有入口，但還沒讀源碼證據 | 打開 source，補已確認與合理推論 |
| 待對照 | 已讀 source，但還缺 types、examples、tests 或 dist | 補外部證據 |
| 待結論 | 證據已足夠，但還沒寫成結論 | 回填相關章節筆記 |
| 已結案 | 問題已有結論、證據、回查位置 | 移入已結案區或保留摘要 |
| 證據衝突 | 不同來源不一致 | 記錄衝突來源，避免硬下結論 |
| 轉入暫緩 | 有價值但不影響當前主線 | 移到 `00-10_暫緩閱讀清單.md` |
| 作廢 | 問題前提錯誤或已不適用 | 標明作廢原因，不直接刪除重要脈絡 |

狀態更新規則：

- 每次讀碼後至少處理一個問題的狀態。
- 問題不能長期只有「新問題」而沒有第一入口。
- 若一個問題超過一輪學習都沒有處理，應判斷是轉入暫緩，還是拆成更小問題。
- 證據衝突不要急著合併成單一結論，先保留衝突來源。

## 3. 優先級判斷

問題有優先級。不是所有問題都值得立刻追。

| 優先級 | 定義 | 例子 | 處理方式 |
|---|---|---|---|
| P0 | 阻塞當前主線，不解決就無法繼續 | 不知道 `install` 入口在哪裡 | 當場追完或縮小問題 |
| P1 | 影響本階段通關 | 不懂 FormItem 如何觸發 validate | 放入本篇並安排本階段處理 |
| P2 | 重要但不阻塞本輪 | Select remote 分支細節 | 放入問題索引或暫緩清單 |
| P3 | 好奇或延伸問題 | 某個歷史設定為什麼保留 | 只有在有明確入口時才收錄 |

優先級判斷表：

| 問題類型 | 預設優先級 | 調整條件 |
|---|---|---|
| 找不到入口 | P0 | 若只是支線入口，可降為 P2 |
| 主流程不懂 | P0 或 P1 | 影響當前章節通關時是 P0 |
| 型別與實作不一致 | P1 | 若不影響目前主題，可降為 P2 |
| tests 缺口 | P2 | 若正在寫測試反推章，可升為 P1 |
| 大型元件分支 | P2 | 若分支是主流程必要條件，可升為 P1 |
| 歷史設定疑問 | P3 | 若影響 build 或 publish，可升為 P1 |

## 5. 問題 ID 與命名規則

問題 ID 要能排序、能搜尋、能回填。

建議格式：

```text
Q-YYYYMMDD-NN
```

例如：

| ID | 意義 |
|---|---|
| `Q-20260505-01` | 2026-05-05 記錄的第 1 個問題 |
| `Q-20260505-02` | 2026-05-05 記錄的第 2 個問題 |
| `Q-20260506-01` | 2026-05-06 記錄的第 1 個問題 |

標題建議使用「對象 + 行為 + 疑問」：

| 不佳標題 | 較佳標題 |
|---|---|
| Form 問題 | FormItem validate 如何被 Form 統一觸發？ |
| Select 看不懂 | Select 的 option 是如何註冊到父層的？ |
| Message | Message service 如何建立並銷毀 notification instance？ |
| 樣式 | Button 的 computed class 如何對應到 Less selector？ |

## 6. 問題分類

問題先分類，後續才容易批次處理。

| 分類 | 適用問題 | 常見入口 |
|---|---|---|
| 工程 / build | dev、build、dist、發布、依賴 | `package.json`、`vite.config.js`、`build/` |
| 入口 / install | `app.use`、插件安裝、default export | `src/index.js` |
| 元件註冊 | 單元件如何匯出、集中註冊、全域可用 | `src/components/index.js`、單元件 `index.js` |
| 普通元件 | props、events、slots、render、class | `src/components/<component>/` |
| 表單 / v-model | model 同步、FormItem、validate | Input、Form、FormItem、Select |
| 複雜互動 | dropdown、panel、range、multiple、remote | Select、DatePicker、Cascader |
| 大型資料元件 | columns、data、render、summary、CSV | Table |
| 導航 / 父子協作 | active、parent、child registration | Menu、Tabs、Breadcrumb |
| 彈層 / 定位 | trigger、visible、placement、transfer、popper | Tooltip、Poptip、Modal、Select |
| 命令式 API | service、instance、destroy、globalProperties | Message、Notice、Modal、LoadingBar |
| directives | 指令定義、註冊、生效場景 | `src/directives`、`src/index.js` |
| utils / mixins | 共用工具、共用邏輯、上下文 | `src/utils`、`src/mixins` |
| locale | 語系、格式化、install options | `src/locale`、`src/index.js` |

## 9. 證據衝突問題

這一區放 source、types、examples、tests 或 dist 之間不一致的問題。

| ID | 衝突主題 | 來源 A | 來源 B | 目前判斷 | 下一步 |
|---|---|---|---|---|---|
|  |  |  |  |  |  |

常見衝突類型：

| 衝突類型 | 例子 | 處理方式 |
|---|---|---|
| source vs types | 實作支援某 prop，但 types 沒宣告 | 標記缺口，補查 examples / tests |
| source vs examples | examples 使用方式和實作預期不同 | 以 source 為執行真相，再查是否是舊範例 |
| source vs tests | 測試固定的行為與目前實作不一致 | 標記為高優先級衝突 |
| source vs dist | dist 產物與 source 不一致 | 確認 dist 是否為舊產物 |
| 筆記 vs source | 既有筆記描述和 source 不一致 | 回修筆記，不把舊筆記當真相 |

結論規則：

- 不要把衝突問題直接寫成已確認。
- 先保留每個來源的具體位置。
- 若只能合理推論，要寫清推論依據。
- 若需要執行測試或 Demo 才能確認，要標記為待驗證。

## 10. 已結案問題

已結案不代表問題可以刪掉。保留摘要能幫助之後回查。

| ID | 問題 | 結論摘要 | 證據位置 | 回填位置 |
|---|---|---|---|---|
|  |  |  |  |  |

已結案標準：

| 檢查項 | 合格標準 |
|---|---|
| 問題清楚 | 能用一句話說明原本疑問 |
| 證據足夠 | 至少有 source 證據，必要時有 types、examples、tests 對照 |
| 結論明確 | 能回答原問題，不只說「已讀」 |
| 缺口處理 | 剩餘分支已標記為待驗證或暫緩 |
| 回填完成 | 已寫回相關章節筆記或元件分析筆記 |

## 11. 轉入暫緩清單

有些問題不是不重要，而是不該在當前階段追。這類問題應該轉入 `00-10_暫緩閱讀清單.md`。

| ID | 問題 | 暫緩理由 | 之後回來時的第一入口 | 建議回來時機 |
|---|---|---|---|---|
|  |  |  |  |  |

應轉入暫緩的典型情況：

| 問題類型 | 暫緩原因 |
|---|---|
| 大型元件支線 | 例如 Table fixed / summary / tree / expand 細節，初期會打斷主流程 |
| 外部庫內部實作 | 例如 popper.js、async-validator 的完整原理，不是第一輪核心 |
| 歷史設定 | 例如 Vue CLI 與 Vite 並存原因，若不影響主線可暫緩 |
| dist 細節 | minified 產物不適合第一輪逐行讀 |
| 邊界案例 | 如果不影響主流程，可等第二輪補 |

轉入暫緩時不能只寫「之後再看」。至少要補：

- 暫緩理由。
- 目前已確認到哪裡。
- 之後回來時第一入口。
- 回來條件，例如「讀完 Table 主流程後」、「進入彈層專題時」。

## 13. 按證據缺口整理問題

有時候問題不是分類不明，而是缺少某種證據。可以用這張表反推下一步。

| 缺口 | 代表情境 | 下一步 |
|---|---|---|
| 缺 source | 只有章節描述或命名推測 | 打開對應 source，確認定義位置 |
| 缺註冊 | 只看到 `.vue`，不知道是否對外可用 | 查單元件 `index.js`、`src/components/index.js`、`src/index.js` |
| 缺生效場景 | 知道能力存在，但不知道誰觸發 | 查 examples、tests、父元件或使用方 |
| 缺樣式 | 只看到 class，不知道樣式來源 | 查 component less、styles 入口、variables、mixins |
| 缺型別 | 不知道對外 API 是否有型別 | 查 `types/` 與 global properties 宣告 |
| 缺 examples | 不知道實際怎麼使用 | 查 `examples/routers/<component>.vue` |
| 缺 tests | 不知道行為是否被固定 | 查 `test/unit/specs` |
| 缺 runtime 驗證 | source 靜態閱讀不足 | 執行 examples 或 tests，必要時記為待驗證 |
| 證據衝突 | 多個來源不一致 | 放入第 9 節，不急著結案 |

## 14. 問題拆分規則

一個問題太大，通常就無法結案。遇到大問題時要拆。

| 大問題 | 拆分方式 |
|---|---|
| Select 怎麼運作？ | 拆成 modelValue、option 註冊、dropdown visible、multiple、filterable、remote |
| Table 怎麼運作？ | 拆成 columns、data、header、body、cell、fixed、summary、CSV |
| Form 怎麼驗證？ | 拆成 Form fields、FormItem rules、validate 觸發、async-validator、錯誤顯示 |
| Modal 怎麼運作？ | 拆成普通 Modal、confirm service、instance 建立、show / remove、mask / scroll |
| 樣式系統怎麼運作？ | 拆成入口、variables、mixins、common、animation、component less |

拆分原則：

- 每張問題卡只回答一個可追蹤問題。
- 問題標題要能指向第一入口。
- 如果需要跨三個以上大模組，先拆成主問題與子問題。
- 子問題完成後，再回到主問題整理總結。

## 15. 問題結案標準

一個問題能結案，至少要符合這些條件。

| 檢查項 | 10 / 10 標準 |
|---|---|
| 問題明確 | 能用一句話描述，不含模糊詞 |
| 分類正確 | 已標明工程、入口、元件、樣式、types、tests 等分類 |
| 狀態正確 | 狀態不是長期停在新問題 |
| 第一入口明確 | 下次回查能直接打開檔案或章節 |
| 證據足夠 | 至少有 source；必要時補 styles、types、examples、tests |
| 結論可回答問題 | 結論不是摘要，而是回答原問題 |
| 缺口已處理 | 剩餘缺口轉入待驗證或暫緩 |
| 回填完成 | 相關章節或元件筆記已同步更新 |
| 下一步清楚 | 若未完全結案，知道下一次查哪裡 |

不能結案的情況：

| 情況 | 原因 |
|---|---|
| 只有「看起來像」 | 命名推測不是證據 |
| 只有一個 source 片段 | 可能缺註冊或生效場景 |
| 只查了 types | types 不是執行時真相 |
| 只查了 examples | examples 不是完整規格 |
| 只查了 tests | tests 只代表被測行為，不代表所有行為 |
| 還有證據衝突 | 必須先記錄衝突來源 |

## 18. 週期性整理規則

問題索引要定期整理，否則會失去回查價值。

| 時機 | 要做的整理 |
|---|---|
| 每次讀完一篇章節 | 把待驗證問題加入本篇 |
| 每次完成一個代表元件 | 檢查問題是否已結案、轉入暫緩或拆分 |
| 每次完成一個里程碑 | 用 `00-08` 檢查本階段問題是否阻塞通關 |
| 每週或每輪學習結束 | 清理作廢問題、更新已結案、同步根目錄索引 |
| 開始第二輪閱讀前 | 以 P0 / P1 問題決定下一輪入口 |

整理時優先做三件事：

1. 把沒有第一入口的問題補上入口。
2. 把太大的問題拆小。
3. 把不影響當前主線的問題轉入暫緩。

## 19. 10 / 10 問題索引檢查表

用這張表判斷本篇是否維持在可用狀態。

| 檢查項 | 滿分標準 |
|---|---|
| 每個問題都有 ID | 可排序、可搜尋、可引用 |

| 證據衝突獨立管理 | 不把衝突寫成確定結論 |

## 本節小結

問題索引的價值不在於記下很多疑問，而在於讓每個疑問都能回到行動：

```text
問題
  -> 分類
  -> 狀態
  -> 優先級
  -> 第一入口
  -> 證據缺口
  -> 下一步
  -> 結案或暫緩
```

只要每個問題都能沿著這條線移動，讀碼過程中的卡點就不會變成壓力或雜訊，而會變成下一輪閱讀、補證據、寫總結與做仿寫的具體入口。
