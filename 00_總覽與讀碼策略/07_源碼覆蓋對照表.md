> 本篇任務：把「我目前讀到 View UI Plus 哪裡」變成一張可以追蹤、回查、補洞的表。  
> 它不是單純列目錄，而是用來管理 **源碼覆蓋率、讀碼深度、筆記產出、待補檔案、下一步閱讀順序**。

## 0. 本篇結論

這份筆記系統要避免變成「看起來很完整，但其實很多地方只是列標題」。

所以 `07_源碼覆蓋對照表` 的核心不是：

```txt
這個目錄對應哪篇筆記？
```

而是：

```txt
這個源碼區域我讀到什麼程度？
是否有對照到真實檔案？
是否產出可回查筆記？
是否能說出它在 View UI Plus 裡的責任？
下一個應該補讀哪個檔案？
```

本篇要和前面幾篇形成關係：

## 1. 源碼核對基準

本表以 View UI Plus 官方 GitHub 原始碼為核對對象。

### 1.1 專案根目錄核對

目前根目錄可觀察到的主要區域包含：

```txt
.github/
assets/
build/
dist/
examples/
src/
test/
types/
package.json
vite.config.js
vite.config.dev.js
vue.config.js
tsconfig.json
README.md
README-CN.md
```

這表示 View UI Plus 不是只有 `src/components`。如果只讀元件，會漏掉：

- 套件如何被建置與發布
- 樣式如何被打包
- 語系如何被打包
- examples 如何作為使用者視角
- types 如何作為對外 API 契約
- dist 如何作為最終交付產物

---

### 1.2 `src` 核心區域核對

`src` 是第一主戰場，應優先追蹤下列區域：

```txt
src/index.js
src/components/
src/directives/
src/locale/
src/mixins/
src/styles/
src/utils/
```

其中：

- `src/index.js` 是插件安裝入口。
- `src/components/index.js` 是元件對外匯出總表。
- `src/components/*` 是元件實作主體。
- `src/directives/*` 是全域指令與 DOM 行為抽象。
- `src/locale/*` 是國際化能力。
- `src/mixins/*` 是跨元件共用邏輯。
- `src/styles/*` 是 Less 樣式系統。
- `src/utils/*` 是工具函式與底層輔助能力。

---

### 1.3 入口流程核對

從目前源碼可以確認，View UI Plus 的使用鏈大致是：

```txt
使用者 app.use(ViewUIPlus)
        ↓
進入 src/index.js 的 install(app, opts)
        ↓
匯入 src/components/index.js 的元件集合
        ↓
Object.keys(ViewUI).forEach(... app.component(...))
        ↓
Object.keys(directives).forEach(... app.directive(...))
        ↓
掛載 app.config.globalProperties.$VIEWUI
        ↓
掛載 $Message / $Modal / $Notice / $Spin / $Loading 等全域能力
```

所以覆蓋表的 P0 項目一定要包含：

```txt
package.json
vite.config.js
examples/main.js
src/index.js
src/components/index.js
```

如果這五個檔案還沒建立筆記，就不應該急著宣稱自己「已經理解 View UI Plus 架構」。

## 2. 覆蓋狀態定義

這張表的狀態不要只寫「已讀 / 未讀」，那太粗。建議使用以下狀態。

| 狀態 | 代號 | 定義 | 是否算完成 |
|---|---|---|---|
| 未開始 | S0 | 還沒看過源碼 | 否 |
| 已定位 | S1 | 知道檔案位置與大致用途 | 否 |
| 粗讀完成 | S2 | 看過主要結構，但還不能解釋細節 | 否 |
| 深讀完成 | S3 | 能說出 import、export、props、methods、computed、render、流程與依賴 | 部分算 |
| 已驗證 | S4 | 重要結論都有對照源碼、examples 或 types | 算 |
| 已產出正式筆記 | S5 | 已形成可回查 Markdown 筆記，且能串到 `99_問題回查索引` | 算完成 |

建議最低完成標準：

```txt
重要源碼至少要到 S4。
核心源碼必須到 S5。
非核心源碼可以停在 S2 或 S3。
```

## 3. 讀碼優先級定義

| 優先級 | 定義 | 代表區域 |
|---|---|---|
| P0 | 不讀就無法理解整套庫 | `package.json`、`vite.config.js`、`examples/main.js`、`src/index.js`、`src/components/index.js` |
| P1 | 元件庫核心能力 | `src/components`、`src/styles`、`src/locale`、`src/mixins`、`src/utils` |
| P2 | 進階理解與工程配套 | `src/directives`、`types`、`build`、`test` |
| P3 | 輔助觀察，不主讀 | `dist`、`assets`、`.github`、舊式或過渡設定檔 |

判斷原則：

```txt
會影響使用者如何 import / app.use / 使用元件 / 使用樣式 / 使用全域 API 的，優先級高。
只影響開發流程、發布流程、展示資源的，優先級次之。
只是打包結果或靜態資源的，不應該優先深讀。
```

## 6. P0 必讀檔案覆蓋表

這一節是第一輪一定要完成的範圍。

| 檔案 | 優先級 | 目標狀態 | 為什麼必讀 | 讀完要能回答 |

| `vite.config.js` | P0 | S4 | 它定義 library build 的 entry 與輸出格式 | `src/index.js` 如何變成 `dist/viewuiplus.min.js`？ |
| `examples/main.js` | P0 | S4 | 它是使用者視角與開發視角的入口 | examples 如何 `app.use(ViewUIPlus)`？ |
| `src/index.js` | P0 | S5 | 它是 View UI Plus 的插件入口 | `install(app, opts)` 註冊了哪些元件、指令與全域能力？ |
| `src/components/index.js` | P0 | S5 | 它決定對外 component surface | 哪些元件被 export？哪些是元件組合成員？ |

完成 P0 後，你應該能畫出這條主線：

```txt
package.json
  ├─ main -> dist/viewuiplus.min.js
  ├─ typings -> types/index.d.ts
  └─ scripts -> build:prod / build:style / build:lang

vite.config.js
  └─ lib.entry -> src/index.js

examples/main.js
  └─ app.use(ViewUIPlus)

src/index.js
  ├─ export * from './components'
  ├─ import * as components from './components'
  ├─ install(app, opts)
  ├─ app.component(...)
  ├─ app.directive(...)
  └─ app.config.globalProperties.$VIEWUI / $Message / $Modal / ...

src/components/index.js
  └─ export { default as X } from './x'
```

## 9. 型別與對外 API 覆蓋表

`types` 的閱讀目的不是學 TypeScript 語法，而是確認：

```txt
源碼實作提供的能力，對外到底如何被宣告？
```

| 區域 | 優先級 | 建議狀態 | 必問問題 | 產出物 |
|---|---:|---|---|---|
| `types/index.d.ts` | P2 | S4 | 對外型別總入口如何組織？ | types 入口筆記 |
| 元件型別宣告 | P2 | S4 | 元件 props / methods 是否有對外型別？ | 元件 API 契約對照表 |
| 全域 API 型別 | P2 | S4 | `$Message`、`$Modal`、`$Notice` 等是否有型別？ | 全域 API 型別表 |
| plugin 型別 | P2 | S3 | `app.use(ViewUIPlus)` 的型別如何描述？ | plugin 型別筆記 |

讀 types 時不要孤立閱讀，應該對照：

```txt
src/components/button/button.vue
        ↔
types/button.d.ts 或對應型別區域

src/components/message
        ↔
全域 $Message 型別宣告

src/index.js
        ↔
types/index.d.ts
```

## 10. 樣式系統覆蓋表

View UI Plus 使用 Less 組織樣式，因此樣式筆記不能只寫「有樣式」。要追蹤：

- 樣式入口
- 變數
- mixin
- common class
- animation
- components style
- 元件 class name 與樣式檔對照

| 樣式區域 | 優先級 | 建議狀態 | 閱讀目的 | 產出物 |
|---|---:|---|---|---|
| `src/styles/index.less` | P1 | S4 | 樣式總入口 | Less 匯入圖 |
| `src/styles/base.less` | P1 | S3 | 基礎 reset / base style | base 筆記 |
| `src/styles/custom.less` | P1 | S3 | 自訂覆蓋或主題入口 | custom 筆記 |
| `src/styles/common/*` | P1 | S3 | 通用樣式 | common class 表 |
| `src/styles/mixins/*` | P1 | S4 | 樣式 mixin 與複用 | mixin 表 |
| `src/styles/animation/*` | P2 | S3 | transition / animation | 動畫筆記 |
| `src/styles/components/*` | P1 | S4 | 元件樣式對照 | 元件 class 對照表 |
| `src/styles/color/*` | P2 | S3 | 顏色生成 / 色彩系統 | color 筆記 |

單一元件樣式筆記最低要回答：

```txt
1. 元件 render/template 產生哪些 class？
2. 對應的 Less 檔案在哪裡？
3. 哪些 class 是狀態 class？例如 disabled、active、focused、loading。
4. 哪些樣式來自全域變數或 mixin？
5. 這種 class 設計可以怎麼學到自己的專案？
```

## 11. examples 覆蓋表

## 18. 不能算覆蓋完成的情況

以下情況不要把狀態標為 S4 或 S5。

| 情況 | 為什麼不算 |
|---|---|

| 只看元件，不看 examples | 不知道使用者視角 |
| 只看元件，不看 styles | 不知道 class 與樣式如何對應 |
| 只看元件，不看 types | 不知道對外 API 契約 |
| 只看單檔，不追 import/export | 不知道它在整體架構的位置 |
| 只複製 AI 回答，沒有人工驗證 | 容易留下幻覺內容 |
