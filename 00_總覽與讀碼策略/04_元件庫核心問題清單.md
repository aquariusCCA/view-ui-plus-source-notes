# 04_元件庫核心問題清單

> 本篇任務：把「讀 View UI Plus 時應該問什麼」整理成一套可執行、可回查、可延伸的問題框架。  
> 它不是單純的問題列表，而是要求每一題都能對應到：**源碼位置、觀察重點、判斷標準、筆記產出**。

---

## 0. 本篇結論

讀 View UI Plus 這類 UI 元件庫，最重要的不是把每個元件都看完，而是能回答下面這句話：

```txt
這套元件庫如何從「一批 Vue 元件」變成「可被 app.use 安裝、可被 import 使用、可被樣式與型別支撐、可被 examples 驗證」的完整前端基礎設施？
```

所以核心問題不能只問：

```txt
Button 有哪些 props？
Table 怎麼寫？
Form 怎麼驗證？
```

而要追問：

```txt
這個 props 為什麼這樣設計？
它如何被 types、examples、styles、locale、mixins 支撐？
它是單一元件自己的邏輯，還是整套元件庫的共同設計模式？
它最後如何透過 src/index.js 被 app.use(ViewUIPlus) 暴露給使用者？
```

本篇將問題分成 8 組：

| 分組 | 核心目的 | 優先級 |
|---|---|---|
| A. 入口與安裝問題 | 看懂 `app.use(ViewUIPlus)` 背後做了什麼 | P0 |
| B. 對外 API 邊界問題 | 看懂哪些東西是 public surface，哪些只是內部實作 | P0 |
| C. 單一元件設計問題 | 看懂 Props / Emits / Slots / class / render / 狀態 | P0 |
| D. 複合元件協作問題 | 看懂 Form、Select、Collapse、Tabs 這類父子協作 | P1 |
| E. 命令式 API 問題 | 看懂 Message、Modal、Notice、LoadingBar 等非模板式能力 | P1 |
| F. 共用層問題 | 看懂 mixins、utils、directives、locale、styles 如何支撐元件 | P1 |
| G. 工程交付問題 | 看懂 package、Vite library build、types、build scripts、dist | P1 |
| H. 讀碼品質問題 | 確保每篇筆記不是空泛心得，而是能回查源碼 | P0 |

一句話版本：

> **先問入口，再問邊界；先問元件 API，再問內部狀態；先問共用抽象，再問工程交付。**

## 1. 本篇源碼核對依據

這份問題清單不是一般 UI 元件庫通用題庫，而是依照 View UI Plus 的實際源碼結構設計。

| 源碼位置 | 已確認角色 | 本篇會用來追問什麼 |
|---|---|---|
| `README.md` | 專案定位為 Vue 3 UI component library / front-end solution | 這不是業務系統，問題要圍繞元件庫而不是頁面流程 |
| `package.json` | 宣告 `main`、`typings`、`files`、`scripts`、dependencies | 套件如何被安裝、打包、發布、提供型別 |
| `vite.config.js` | library build entry 指向 `src/index.js`，輸出 UMD / ES | 為什麼 `src/index.js` 是正式打包入口 |
| `vite.config.dev.js` | 開發環境以 `examples` 為主 | 為什麼 examples 是驗證元件 API 的重要入口 |
| `examples/main.js` | 匯入 `../src/index`，建立 Vue app，`app.use(ViewUIPlus)`，建立大量 demo routes | 從使用者視角反推元件 API 與主要場景 |
| `src/index.js` | 匯出 components，定義 `install(app, opts)`，註冊 components / directives，掛載 `$VIEWUI` 與多個全域 API | 入口、安裝、全域能力、全域配置 |
| `src/components/index.js` | 集中 export 對外元件 | public component surface、整包安裝與按需引入基礎 |
| `src/components/button/button.vue` | 代表性基礎元件：Options API、mixins、props validator、computed classes、render function | 單一元件 API 設計與源碼筆記範例 |
| `src/components/button/index.js` | 對 Button 做簡單 default export | 單一元件入口與集中匯出之間的關係 |
| `src/mixins/form.js` | inject Form / FormItem，回傳 disabled 狀態，轉發 blur / change | 表單元件協作、父子通訊、驗證事件傳遞 |
| `src/mixins/link.js` | 抽象 `to`、`replace`、`target`、router / window 跳轉 | Button / Link 這類可跳轉元件的共用能力 |
| `src/locale/index.js` | 預設 zh-CN，提供 `use`、`t`、`i18n`，並整合 vue-i18n 情境 | 語系切換、全域翻譯、元件如何取文字 |
| `src/directives` | 包含 `clickoutside.js`、`line-clamp.js`、`resize.js`、`style.js`、`transfer-dom.js` 等 | 為什麼有些行為不用元件，而用 directive 封裝 |
| `src/styles` | 包含 `animation`、`color`、`common`、`components`、`mixins`、`base.less`、`custom.less`、`index.less` | 樣式入口、元件樣式、Less 變數、mixin、主題覆蓋 |
| `types` | 多個 `.d.ts` 型別檔與 `types/index.d.ts` 入口 | 對外 API 契約如何被 TypeScript 描述 |
| `build` | 建置樣式與語系包相關腳本 | 工程交付如何補齊 JS 之外的產物 |

## 2. 這份問題清單怎麼用

### 2.1 不要一次回答所有問題

這篇不是考卷，不需要一次寫完。正確用法是：

```txt
讀一個源碼區塊
  ↓
挑對應的 3～5 題
  ↓
用源碼回答
  ↓
寫成筆記
  ↓
把仍不確定的地方丟到 99_問題回查索引.md
```

例如讀 `src/components/button/button.vue` 時，不要同時研究 Table、Modal、build。你只需要回答：

```txt
Button 的 props 怎麼設計？
Button 如何組裝 class？
Button 為什麼混入 link / form？
Button 何時渲染 button，何時渲染 a？
Button 如何受到全域 $VIEWUI.size 影響？
```

---

### 2.2 每一題都要有四個產出

每個問題最少要寫出四件事：

```md
## 問題
這段源碼解決什麼問題？

## 原始碼位置
- src/xxx
- src/yyy

## 我看到的證據
- 具體變數
- 具體函式
- 具體 props / emits / computed / methods
- 具體 import / export

## 我的判斷
- 它屬於對外 API、內部實作，還是共用抽象？
- 這個設計可以學走什麼？
- 還有哪些地方沒有驗證？
```

如果只有「我覺得這裡是在做註冊」，但沒有標明 `src/index.js` 裡對應到 `app.component(...)` 或 `app.directive(...)`，這就不是合格源碼筆記。

---

### 2.3 問題狀態標記

每個問題建議加一個狀態，方便後續回查：

| 狀態 | 意義 | 使用時機 |
|---|---|---|
| `未讀` | 還沒看源碼 | 只先保留問題 |
| `已定位` | 已找到源碼位置，但還沒完整分析 | 知道要看哪裡 |
| `部分回答` | 能回答主流程，但分支還不清楚 | 適合先記錄，不必卡住 |
| `已回答` | 已能用源碼說明流程與設計取捨 | 可放入正式筆記 |
| `待驗證` | 推論合理，但還缺 examples / types / 測試佐證 | 不能寫成確定結論 |
| `已抽象` | 已整理成可學走的設計模式 | 可以進入仿寫或重構 |

## 3. A 組：入口與安裝問題（P0）

這一組問題是第一輪必答。它決定你是否真的理解 View UI Plus 作為「元件庫」如何被使用，而不是只會看單一元件。

---

### A01. `app.use(ViewUIPlus)` 背後到底做了什麼？

| 項目 | 內容 |
|---|---|
| 優先級 | P0 |
| 對應源碼 | `src/index.js`、`examples/main.js`、`vite.config.js` |
| 核心概念 | Vue Plugin、install、全域註冊、全域配置 |

要回答的問題：

1. `examples/main.js` 是如何匯入 View UI Plus 的？
2. `app.use(ViewUIPlus)` 最終會呼叫哪個 `install`？
3. `install(app, opts)` 裡面做了哪些事？
4. 安裝流程中註冊了哪些 components？
5. 安裝流程中註冊了哪些 directives？
6. 安裝流程中掛了哪些 `app.config.globalProperties`？
7. `install.installed` 的存在代表什麼？
8. `opts.locale`、`opts.i18n`、`opts.size`、`opts.transfer` 這類選項如何進入元件庫？

觀察重點：

```txt
src/index.js
  ├─ export * from './components'
  ├─ import * as components from './components'
  ├─ const directives = { ... }
  ├─ const ViewUI = { ...components, iButton, iInput, iTable, ... }
  ├─ export const install = function(app, opts = {}) { ... }
  ├─ Object.keys(ViewUI).forEach(key => app.component(key, ViewUI[key]))
  ├─ Object.keys(directives).forEach(key => app.directive(key, directives[key]))
  ├─ app.config.globalProperties.$VIEWUI = { ... }
  └─ app.config.globalProperties.$Message / $Modal / $Notice / ...
```

筆記產出標準：

```md
## app.use(ViewUIPlus) 安裝流程圖

createApp(App)
  ↓
app.use(ViewUIPlus)
  ↓
呼叫 src/index.js default export 裡的 install
  ↓
套用 locale / i18n
  ↓
註冊所有 ViewUI components
  ↓
註冊 directives
  ↓
掛載 $VIEWUI 全域配置
  ↓
掛載 $Message / $Modal / $Notice / $Spin / $Loading 等全域 API
```

常見錯誤：

- 只寫「它會註冊元件」，但沒有指出 `app.component` 的來源。
- 只看 `components/index.js`，沒有看 `src/index.js` 如何把它們掛到 Vue app。
- 看到 `$Message` 就直接說它是 Vue Plugin，沒有追到底層檔案。

### A02. 為什麼 `src/index.js` 是正式入口，不只是普通檔案？

| 項目 | 內容 |
|---|---|
| 優先級 | P0 |
| 對應源碼 | `vite.config.js`、`package.json`、`src/index.js` |
| 核心概念 | library entry、npm main、dist 產物 |

要回答的問題：

1. `vite.config.js` 的 library entry 指向哪裡？
2. `package.json` 的 `main` 指向哪個打包產物？
3. `package.json` 的 `files` 發布哪些資料夾？
4. `src/index.js` 是開發入口、打包入口，還是兩者都有關？
5. 使用者 import `view-ui-plus` 時，和開發時 examples import `../src/index` 有什麼差異？

判斷標準：

```txt
如果一個檔案同時被：
- library build 當作 entry
- examples 當作開發時入口
- package 產物 indirectly 指向
- default export API

那它就是元件庫的正式入口。
```

筆記產出標準：

```md
## 正式入口判斷

View UI Plus 的正式源碼入口是 `src/index.js`，理由如下：
1. `vite.config.js` 的 lib.entry 指向它。
2. `examples/main.js` 開發時直接 import 它。
3. 它 default export 的 API 內含 install，因此可被 app.use 使用。
4. 它匯出 components、version、locale、i18n、lang 等對外能力。
```

### A03. 全域配置 `$VIEWUI` 是如何被建立，又如何被元件讀取？

| 項目 | 內容 |
|---|---|
| 優先級 | P0 |
| 對應源碼 | `src/index.js`、`src/components/button/button.vue`、其他讀取 `$VIEWUI` 的元件 |
| 核心概念 | globalProperties、全域預設值、元件 props default |

要回答的問題：

1. `$VIEWUI` 裡有哪些全域配置？
2. `size`、`capture`、`transfer`、`cell`、`menu`、`modal`、`tabs`、`select`、`datePicker`、`typography` 等配置分別服務哪些元件？
3. 元件是透過 `this.$VIEWUI`，還是透過 `getCurrentInstance().appContext.config.globalProperties` 取得配置？
4. 全域配置與單一元件 props 發生衝突時，以誰為準？
5. 哪些元件有讀 `$VIEWUI`，哪些元件完全不依賴它？

以 Button 為例，應該觀察：

```txt
src/components/button/button.vue
  ├─ size prop 的 default 使用 getCurrentInstance()
  ├─ 從 appContext.config.globalProperties 取得 global
  └─ 如果 global.$VIEWUI.size 不存在，回落到 'default'
```

筆記產出標準：

```md
## 全域配置與局部 props 的關係

以 Button 的 size 為例：
- 如果使用者傳入 size，使用局部 props。
- 如果沒有傳入 size，default function 讀取全域 `$VIEWUI.size`。
- 如果全域 size 也不存在，回到 `default`。

這代表 View UI Plus 用 `$VIEWUI` 提供全域預設，但仍保留單一元件 props 的覆蓋能力。
```

待驗證問題：

- 其他元件是否都遵守「局部 props 優先，全域配置次之，元件預設最後」？
- 是否有元件直接讀 `this.$VIEWUI`？
- 是否有元件對 `$VIEWUI` 的欄位名稱依賴很深，導致重構成本高？

### A04. directives 是怎麼被統一註冊的？

| 項目 | 內容 |

要回答的問題：

1. `src/directives` 裡有哪些 directive？
2. `src/index.js` 實際註冊了哪些 directive？
3. 為什麼有些行為要用 directive，而不是寫成 component？
4. `style.js` 為什麼可以拆成 `display`、`width`、`height`、`margin`、`padding`、`font`、`color`、`bg-color`？
5. `resize`、`line-clamp` 這類 directive 對應到什麼使用場景？
6. examples 裡是否有對應 demo routes？

觀察重點：

```txt
src/index.js
  └─ directives = {
       display,
       width,
       height,
       margin,
       padding,
       font,
       color,
       bg-color,
```

```txt
存在於 directives 目錄
≠
一定被 install 全域註冊
```

```md

| resize.js | 是 | resize | 尺寸變化監聽 | examples/routers/v-resize.vue |
| line-clamp.js | 是 | line-clamp | 多行省略 | examples/routers/v-line-clamp.vue |
| transfer-dom.js | 待驗證 | 待追蹤 | 彈層轉移 DOM | 待追蹤 |
```

### A05. 語系系統是如何接入安裝流程的？

| 項目 | 內容 |
|---|---|
| 優先級 | P0 |
| 對應源碼 | `src/index.js`、`src/locale/index.js`、`src/locale/lang/*`、使用 `t` 的元件 |
| 核心概念 | locale、i18n、deepmerge、全域文字 |

要回答的問題：

1. 預設語系是哪一個？
2. `install(app, opts)` 如何處理 `opts.locale`？
3. `install(app, opts)` 如何處理 `opts.i18n`？
4. `localeFile.use` 與 `localeFile.i18n` 分別負責什麼？
5. `t(path, options)` 如何根據 path 取出語系文字？
6. 如果使用者傳入 vue-i18n，會如何 merge local message？
7. 元件內部如何呼叫翻譯函式？是 mixin、function import，還是其他方式？

觀察重點：

```txt
src/locale/index.js
  ├─ import zhLang from './lang/zh-CN'
  ├─ let lang = zhLang
  ├─ use(l) => lang = l || lang
  ├─ i18n(initI18n) => vuei18n = initI18n
  └─ t(path, options) => 先嘗試 i18nHandler，再 fallback 到 lang path
```

筆記產出標準：

```md
## 語系系統流程

install(app, opts)
  ↓
如果 opts.locale 存在：localeFile.use(opts.locale)
  ↓
如果 opts.i18n 存在：localeFile.i18n(opts.i18n)
  ↓
元件呼叫翻譯函式
  ↓
優先走 vue-i18n / $t
  ↓
fallback 到 View UI Plus 自己的 lang 物件
```

待驗證問題：

- 哪些元件直接使用 `t`？
- 語系 key 命名是否和元件名一致？
- types 是否有描述 locale API？

## 4. B 組：對外 API 邊界問題（P0）

這一組問題用來分辨「使用者可以依賴的 API」與「作者內部實作細節」。這對讀開源專案很重要，因為你不能把每個內部函式都當成穩定 API。

---

### B01. `src/components/index.js` 到底代表什麼？

| 項目 | 內容 |
|---|---|
| 優先級 | P0 |
| 對應源碼 | `src/components/index.js`、`src/index.js`、`types/index.d.ts` |
| 核心概念 | public component surface、集中匯出、整包安裝、按需引入 |

要回答的問題：

1. 哪些元件被 `src/components/index.js` export？
2. 被 export 的名稱是否就是使用者 import 時的名稱？
3. `src/index.js` 是否直接 `export * from './components'`？
4. `install` 是否直接拿 `components` 物件逐一 `app.component`？
5. `types/index.d.ts` 是否與 `src/components/index.js` 對齊？
6. 如果某個資料夾存在於 `src/components`，但沒有被 `components/index.js` export，它是否算對外元件？

判斷標準：

```txt
被 src/components/index.js export
  ↓
被 src/index.js 聚合
  ↓
被 install 註冊
  ↓
可能成為 public component surface
```

但仍要補看：

```txt
types 是否提供型別
examples 是否提供 demo
docs 是否提供文件
```

因為「源碼 export」不一定等於「官方鼓勵直接使用」。

---

### B02. 元件別名 `iButton`、`iInput`、`iTable` 代表什麼相容性設計？

| 項目 | 內容 |
|---|---|
| 優先級 | P1 |
| 對應源碼 | `src/index.js` |
| 核心概念 | alias、相容性、命名衝突 |

要回答的問題：

1. `ViewUI` 物件裡為什麼除了 `...components`，還額外定義 `iButton`、`iInput`、`iTable` 等？
2. 這些 `iXxx` 是否是為了解決 HTML 原生標籤或舊版 iView 命名習慣？
3. 它們是否和原本 `Button`、`Input`、`Table` 指向同一個元件？
4. examples 或文件中是否仍有使用 `i-button` / `i-input` 類似寫法？
5. 類似 alias 是否會增加 API 維護成本？

筆記產出標準：

```md
## 元件別名策略

`src/index.js` 裡的 `ViewUI` 不只是展開 components，還額外提供部分 `iXxx` 別名。
這代表 View UI Plus 除了提供標準元件名稱，也可能保留歷史相容或命名衝突解法。

後續要驗證：
- docs 是否建議使用 iXxx？
- types 是否描述 iXxx？
- examples 是否實際使用 iXxx？
```

### B03. 單一元件資料夾的 `index.js` 是 install 入口還是單純 export？

| 項目 | 內容 |
|---|---|

不合格範例：

合格範例：

```md
Button 的 `size` prop 使用 validator 限制為 `small / large / default`，default function 會讀取 `appContext.config.globalProperties.$VIEWUI.size`。
這說明 View UI Plus 用全域 `$VIEWUI` 提供元件預設尺寸，但仍允許單一元件覆蓋。
```

### H02. 這篇筆記是否區分「事實」與「推論」？

## 12. 單篇元件筆記最低完成標準

之後你每讀一個元件，至少要完成下面這份模板。

```md
# 元件名稱：Button

## 1. 原始碼位置
- src/components/button/button.vue
- src/components/button/index.js
- src/styles/components/button.less
- types/button.d.ts
- examples/routers/button.vue

## 2. 元件定位
Button 是基礎操作元件，主要負責觸發使用者行為，並支援類型、尺寸、圖示、loading、disabled、跳轉與表單 disabled 協作。

## 3. 對外 API
| 類型 | 名稱 | 用途 | 預設值 | 備註 |
|---|---|---|---|---|
| prop | type | 控制按鈕語意樣式 | default | validator 使用 oneOf |
| prop | size | 控制尺寸 | 全域或 default | 會讀 $VIEWUI.size |
| emit | click | 點擊通知 | - | handleClickLink 觸發 |
| slot | default | 按鈕內容 | - | render 中包 span |

## 4. 核心流程
使用者點擊 Button
  ↓
handleClickLink(event)
  ↓
$emit('click', event)
  ↓
如果有 to，交給 mixinsLink.handleCheckClick
  ↓
根據 router / window / target 決定跳轉方式

## 5. 共用依賴
- mixins/link.js：可跳轉能力
- mixins/form.js：表單 disabled 與 blur/change 協作
- utils/assist.js：oneOf validator
- Icon：loading / icon 顯示

## 6. class contract
| class | 來源 | 用途 |
|---|---|---|
| ivu-btn | 固定 | 基礎樣式 |
| ivu-btn-${type} | type prop | 語意樣式 |
| ivu-btn-loading | loading prop | loading 狀態 |

## 7. 可學走的設計
- props validator 與 class 命名形成穩定契約。
- 全域配置只作為 default，不取代局部 props。
- 跨元件能力抽到 mixin，例如 link / form。
- 根節點需要動態 button / a 時，可以用 render function。

## 8. 待驗證
- types/button.d.ts 是否完整對應 button.vue props。
- examples/routers/button.vue 是否涵蓋所有主要 props。
- button.less 是否完整覆蓋所有 class contract。
```

## 14. 本篇使用時的警戒線

讀 View UI Plus 時，最容易犯 5 個錯：

### 錯誤一：只看元件，不看入口

只看 `button.vue`，你會知道 Button 怎麼寫。  
但不看 `src/index.js`，你不知道 Button 如何被整包安裝、全域註冊、成為使用者可用的元件。

---

### 錯誤二：只看源碼，不看 examples

源碼告訴你「能做什麼」。  
examples 告訴你「作者希望使用者怎麼用」。

讀 API 設計時，兩者都要看。

---

### 錯誤三：把內部實作當成對外 API

內部 method、computed、mixin 欄位不一定是穩定 API。  
能否視為對外 API，要至少交叉核對：

```txt
components/index.js
examples
types
docs
```

---

### 錯誤四：沒有區分「已確認」與「推論」

例如你看到 `$Message` 被掛到 globalProperties，只能確認：

```txt
$Message 被掛到全域
```

不能直接確認：

```txt
$Message 底層是如何建立 DOM、如何管理佇列、如何銷毀 instance
```

那些要繼續追 `src/components/message/index.js`。

---

### 錯誤五：看到複雜元件就直接硬啃

Table、Tree、DatePicker、Upload、Transfer 這些元件第一輪不要逐行讀。  
應該先用 Button / Alert / Tag / Form / Message 建立模式感，再進入重型元件。
