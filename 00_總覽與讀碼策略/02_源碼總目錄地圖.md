# 02_源碼總目錄地圖

> 本篇任務：把 View UI Plus 的真實 repo 目錄轉成一張「可用來讀碼、回查、寫筆記」的導航地圖。  
> 它不是單純列出資料夾，而是要回答：每個區塊負責什麼、要先讀哪裡、讀完要產出什麼、哪些地方不適合第一輪深入。

---

## 0. 本篇結論

View UI Plus 的源碼可以用四層來理解：

```txt
ViewUIPlus/
├─ src/                 # P0：元件庫本體，真正的讀碼主戰場
├─ examples/            # P1：範例站，從使用者視角反推元件 API
├─ types/               # P1：TypeScript 對外型別契約
├─ build/               # P2：樣式包、語系包與建置輔助腳本
├─ test/                # P2：行為驗證與邊界案例
├─ dist/                # P3：打包產物，不建議作為讀碼入口
├─ assets/              # P3：靜態資源
├─ .github/             # P3：GitHub 專案協作設定
└─ package / vite / config files
                         # P1：理解專案如何開發、打包、發布
```

如果你的目標是學會讀 View UI Plus 原始碼，第一輪不要從 `dist`、`types` 或複雜元件開始。最穩定的路線是：

```txt
package.json
  ↓
vite.config.js / vite.config.dev.js
  ↓
src/index.js
  ↓
src/components/index.js
  ↓
簡單元件：Button / Alert / Tag / Divider
  ↓
共用層：mixins / utils / directives / styles
  ↓
中型元件：Input / Form / Select / Modal / Message
  ↓
重型元件：Table / DatePicker / Tree / Upload
  ↓
examples / types / test / build
```

## 1. 本篇源碼核對基準

以下是本篇用來校準目錄地圖的主要依據。後續如果 repo 結構變動，應該先更新這一節，再更新後面的地圖。

| 區塊 | 實際觀察到的內容 | 對讀碼的意義 |
|---|---|---|
| 根目錄 | `.github`、`assets`、`build`、`dist`、`examples`、`src`、`test`、`types`、`package.json`、`vite.config.js`、`vite.config.dev.js`、`vue.config.js` 等 | View UI Plus 不是單一 Vue App，而是可開發、可展示、可測試、可打包、可發布的元件庫專案 |
| `package.json` | `name` 為 `view-ui-plus`，`main` 指向 `dist/viewuiplus.min.js`，`typings` 指向 `types/index.d.ts`，`files` 包含 `dist`、`src`、`types` | 這是一個 npm library，而不是純展示專案 |
| `package.json scripts` | `dev`、`dev2`、`build`、`build:prod`、`build:style`、`build:lang`、`lint` | 專案同時處理開發、正式 JS 打包、樣式打包、語系包打包與 lint |
| `vite.config.js` | library build 入口是 `src/index.js`，輸出到 `dist`，並輸出 UMD 與 ES 格式 | `src/index.js` 是正式打包入口，不只是普通檔案 |
| `vite.config.dev.js` | dev root 指向 `examples`，alias `@` 指向 `src` | 開發時透過 examples 直接引用源碼驗證元件 |
| `src` | `components`、`directives`、`locale`、`mixins`、`styles`、`utils`、`index.js` | 這是元件庫的核心骨架 |
| `src/index.js` | 匯出 components、定義 `install(app, opts)`、註冊元件與 directive、掛載 `$VIEWUI` 與 `$Message`、`$Modal`、`$Notice` 等全域 API | 這是 Vue Plugin 入口與全域能力組裝層 |
| `src/components/index.js` | 集中 export 大量元件，例如 Button、Input、Select、Table、Modal、Message、Tree、Upload 等 | 這是 public component surface，讀元件庫時必看 |
| `src/styles` | `animation`、`color`、`common`、`components`、`mixins`、`base.less`、`custom.less`、`index.less` | 樣式不是散在各元件，而是有集中系統 |
| `types` | 大量 `.d.ts`，以及 `index.d.ts` | 對外 API 需要型別契約支撐 |

## 2. 閱讀優先級標記

| 標記 | 意義 | 怎麼處理 |
|---|---|---|
| `P0` | 第一輪必讀 | 必須寫完整筆記，能回查、能仿寫、能講清楚流程 |
| `P1` | 第二輪核心 | 先理解定位，等代表性元件讀完後再補細節 |
| `P2` | 有需求再深入 | 先知道用途，遇到相關問題再讀 |
| `P3` | 不主讀 | 不適合作為學習源碼的入口，只需要知道存在理由 |

這篇的重點不是把所有目錄都逐行讀完，而是建立判斷：

```txt
我要讀的是元件實作？
我要讀的是公共邏輯？
我要讀的是樣式系統？
我要讀的是打包流程？
我要讀的是對外 API 型別？
```

不同問題要進入不同目錄。

## 4. 根目錄各區塊怎麼讀

### 4.1 `src/`：元件庫本體，第一優先

`src` 是整套 View UI Plus 的本體。讀 View UI Plus，最終一定會回到這裡。

你可以把 `src` 看成七個子系統：

| 子系統 | 位置 | 負責什麼 | 優先級 |
|---|---|---|---|
| 插件入口 | `src/index.js` | 安裝、全域註冊、全域 API、語系、版本資訊 | P0 |
| 元件實作 | `src/components` | 所有對外元件與部分內部支撐元件 | P0 |
| 指令系統 | `src/directives` | DOM 行為增強，例如 resize、line-clamp、style directive | P1 |
| 語系系統 | `src/locale` | 語系包、格式化、locale use / i18n 接口 | P1 |
| 共用 mixin | `src/mixins` | form、link、locale、globalConfig、emitter 等跨元件邏輯 | P1 |
| 樣式系統 | `src/styles` | Less 入口、變數、mixin、元件樣式、主題覆蓋 | P1 |
| 工具函式 | `src/utils` | DOM、date、csv、keyCode、textarea height、transfer queue 等工具 | P1 |

讀完 `src` 的最低產出：

```md
## 我應該能回答
1. app.use(ViewUIPlus) 背後做了什麼？
2. 一個元件如何被匯出、註冊、使用？
3. 全域 API 如 $Message、$Modal 從哪裡掛上去？
4. 共用邏輯放在 mixins、utils、directives 還是 styles？
5. 樣式如何從 src/styles 進入 dist/styles/viewuiplus.css？
```

---

### 4.2 `examples/`：使用者視角，不是業務系統

`examples` 是開發與展示用的範例站。它不是正式產品，也不是後台業務系統。

實際結構可以先看：

```txt
examples/
├─ components/       # 各元件展示頁或示例組件
├─ public/fonts/     # 範例站用字型資源
├─ routers/          # demo 路由
├─ app.vue           # 範例站根組件
├─ index.html        # dev HTML 入口
└─ main.js           # examples 啟動入口
```

讀 `examples` 的目的不是學業務流程，而是反推：

| 想知道什麼 | 去 examples 看什麼 |
|---|---|
| 元件最常見怎麼用 | 對應元件 demo |
| Props / Slots / Events 的使用情境 | demo 中的模板與事件綁定 |
| 複雜元件的互動情境 | Table、Select、DatePicker、Upload 等展示頁 |
| 本地開發如何載入源碼 | `examples/main.js`、`vite.config.dev.js` |
| 元件文件與實作有沒有落差 | demo 寫法 vs `src/components` 實作 |

讀完 `examples` 的最低產出：

```md
## 使用者視角整理
- 這個元件最常見的 3 種用法
- 對外使用時最重要的 props
- 對外使用時最重要的 events
- 哪些能力是 demo 有展示，但源碼還沒看懂的
```

---

### 4.3 `types/`：對外 API 型別契約

`types` 不是第一個入口，但它非常適合用來核對「元件庫對 TypeScript 使用者承諾了什麼」。

它的讀法不是逐行背型別，而是反查：

| 使用場景 | 讀法 |
|---|---|
| 看元件是否有型別 | 找 `types/button.d.ts`、`types/input.d.ts` 等 |
| 看全域 API 是否有型別 | 看 `types/index.d.ts` |
| 看實作與對外型別是否一致 | 對照 `src/components/*` 與 `types/*.d.ts` |
| 看元件庫如何支援 TS 使用者 | 看 `package.json` 的 `typings` 與 `types/index.d.ts` |

讀完 `types` 的最低產出：

```md
## 型別契約核對
- 實作檔案：src/components/xxx
- 型別檔案：types/xxx.d.ts
- 型別是否覆蓋主要 props / methods / install API
- 實作與型別是否有明顯落差
```

---

### 4.4 `build/`：發布輔助，不是元件實作

`build` 目前可以先分成三個檔案看：

```txt
build/
├─ build-style.js        # 樣式建置
├─ locale.js             # 語系包建置輔助
└─ vite.lang.config.js   # 語系相關 Vite build 設定
```

它的價值在於理解：View UI Plus 不只要把 JS 打包出來，還要處理樣式與語系的輸出。

第一輪不用逐行研究 `build`，但你要先知道它回答的是這些問題：

```txt
styles 怎麼從 Less 變成可發布 CSS？
locale 怎麼單獨輸出？
package.json 的 build:style / build:lang 分別呼叫什麼？
```

---

### 4.5 `dist/`：結果，不是設計意圖

`dist` 是打包後的產物。它可以用來確認最後輸出長什麼樣，但不適合用來學原始碼設計。

不要從 `dist` 開始讀，原因是：

- 模組邊界已經被打包器處理過。
- 可讀性比 `src` 低。
- 很多命名、拆分、註解、原始設計意圖會變得不清楚。
- 你會看到結果，但看不清楚作者為什麼這樣拆。

只有在下列情況才回來看 `dist`：

```txt
我要確認 npm package 最終輸出檔名。
我要確認 UMD / ES build 實際結果。
我要確認 CSS 產物是否存在。
我要 debug 使用者安裝後載入的檔案。
```

---

### 4.6 `test/`：從測試反推穩定行為

`test` 不是第一輪主線，但它對理解「哪些行為是作者認為不能壞的」很有價值。

讀測試時不要只看語法，要問：

```txt
這個測試在保護哪個元件行為？
這個行為是 props、event、slot、DOM class，還是互動狀態？
這個測試是否揭露了源碼裡看不懂的分支？
```

適合在你讀完某個元件後，再回頭找它是否有測試。

---

### 4.7 根目錄設定檔：理解工程入口

這些檔案不用每個都逐行背，但一定要知道它們控制什麼。

| 檔案 | 角色 | 讀碼重點 |
|---|---|---|
| `package.json` | npm 套件描述與 scripts | `main`、`typings`、`files`、`scripts`、dependencies |
| `vite.config.js` | 正式 library build | build entry、output formats、external vue、dist 輸出 |
| `vite.config.dev.js` | examples 開發環境 | dev root、port、alias `@ -> src` |
| `vue.config.js` | Vue CLI 配置 | 偏歷史或相容用途，第二輪再看 |
| `tsconfig.json` | TS 設定 | 型別與編譯設定，搭配 `types` 看 |
| `.eslintrc.js` / `tslint.json` | lint 設定 | 了解風格限制即可 |
| `.npmignore` | npm 發布排除規則 | 和 `package.json files` 一起看 |

## 5. `src` 目錄詳細地圖

### 5.1 `src/index.js`：整套元件庫的插件入口

`src/index.js` 是 View UI Plus 最值得第一個讀的檔案。它回答的是：

> 使用者呼叫 `app.use(ViewUIPlus)` 時，這套元件庫如何被安裝進 Vue App？

它大致可以拆成六個區塊：

| 區塊 | 原始碼角色 | 讀碼問題 |
|---|---|---|
| components 匯入與匯出 | `export * from './components'`，並 `import * as components` | 所有元件如何集中到入口？ |
| locale 匯入 | `localeFile` | 插件安裝時如何接入 locale / i18n？ |
| directives 匯入與組裝 | `line-clamp`、`resize`、`style` 等 | 哪些 directive 會被全域註冊？ |
| `ViewUI` 物件 | 合併 components，並提供 `iButton`、`iInput` 等別名 | 為什麼有些元件有 `iXxx` 別名？ |
| `install(app, opts)` | 註冊元件、註冊指令、掛載 `$VIEWUI` 與全域 API | 插件真正做了什麼？ |
| API export | `version`、`locale`、`i18n`、`lang`、`install`、components | 對外可 import 的能力有哪些？ |

`src/index.js` 讀完後，至少要能畫出這條線：

```txt
package.json main / vite lib entry
  ↓
src/index.js
  ↓
components + directives + locale + globalProperties
  ↓
使用者 app.use(ViewUIPlus)
  ↓
全域元件 / 全域指令 / $Message / $Modal / $VIEWUI
```

---

### 5.2 `src/components/`：所有元件與內部元件的核心區

`src/components` 是最大區塊，但不要一口氣從上到下讀。你要先分清楚三件事：

1. 哪些是對外公開元件。
2. 哪些是內部支撐資料夾。
3. 哪些元件適合第一輪讀，哪些應該晚點讀。

#### 5.2.1 `src/components/index.js`：public component surface

`src/components/index.js` 集中匯出大量元件，例如：

```txt
Affix / Alert / Anchor / AutoComplete / Avatar / Badge / Breadcrumb
Button / Calendar / Card / Carousel / Cascader / Checkbox / Collapse
ColorPicker / DatePicker / Drawer / Dropdown / Form / Grid / Icon / Image
Input / InputNumber / Layout / List / LoadingBar / Menu / Message / Modal
Notice / Notification / Page / Panel / Poptip / Progress / Radio / Rate
Row / Scroll / Select / Skeleton / Slider / Space / Spin / Split / Steps
Switch / Table / TablePaste / Tabs / Tag / TimePicker / Tooltip / Transfer
Tree / TreeSelect / Typography / Upload / WordCount
```

這份匯出清單的讀碼價值很高，因為它告訴你：

```txt
哪些元件是 View UI Plus 對外承諾的一部分？
哪些目錄雖然存在，但不一定等於 public component？
新增一個元件時，需要在哪裡補 export？
```

#### 5.2.2 內部支撐目錄不要誤判成一般元件

`src/components` 中不只有對外元件，也有一些偏內部支撐的資料夾，例如：

```txt
base/
i18n/
locale/
options/
```

讀碼時要避免把它們和 Button、Input、Table 這類元件混在一起。比較好的筆記標記方式是：

```md
## 元件性質
- public component：是 / 否
- internal helper：是 / 否
- 是否出現在 src/components/index.js：是 / 否
- 是否有 styles/components 對應樣式：是 / 否
```

#### 5.2.3 元件資料夾常見結構

單一元件資料夾通常可以先用這個角度拆：

```txt
src/components/button/
├─ index.js          # 對外匯出與 install 註冊點，若有的話
├─ button.vue        # 元件本體
└─ 其他子檔案         # 大型元件才會有更多拆分
```

不要只看 `.vue`。對元件庫來說，`index.js`、樣式檔、型別檔與 demo 都是同一個元件 API 的一部分。

#### 5.2.4 建議先讀的元件分類

| 分類 | 代表元件 | 建議優先級 | 為什麼 |
|---|---|---|---|
| 最小基礎元件 | `Button`、`Alert`、`Tag`、`Divider` | P0 | 容易建立元件庫基本寫法 |
| 布局元件 | `Row`、`Col`、`Layout`、`Header`、`Sider`、`Content`、`Footer`、`Grid` | P1 | 可理解 class、slot、樣式系統如何配合 |
| 表單元件 | `Input`、`InputNumber`、`Checkbox`、`Radio`、`Switch`、`Select`、`Form`、`FormItem` | P0 / P1 | 牽涉 v-model、驗證、表單上下文，是前端很核心的能力 |
| 命令式元件 | `Message`、`Modal`、`Notice`、`LoadingBar`、`Spin` | P1 | 可學全域 API、動態掛載、狀態管理 |
| 浮層互動 | `Tooltip`、`Poptip`、`Dropdown`、`Drawer` | P1 | 可學定位、click outside、transfer-dom、z-index |
| 資料展示 | `Table`、`List`、`Tree`、`Timeline`、`Collapse`、`Panel`、`Carousel` | P1 / P2 | 互動與狀態較多，適合第二輪 |
| 日期與選擇器 | `DatePicker`、`TimePicker`、`Cascader`、`TreeSelect`、`ColorPicker` | P2 | 子模組多，依賴工具與浮層能力 |
| 上傳與複雜輸入 | `Upload`、`Transfer`、`TablePaste` | P2 | 邊界多，等共用層熟悉後再讀 |
| Pro / 業務型元件 | `Login`、`Auth`、`Captcha`、`GlobalFooter`、`FooterToolbar`、`UserName` | P2 | 偏應用場景封裝，適合後段補 |

### 5.3 `src/directives/`：DOM 行為增強層

實際目錄包含：

```txt
src/directives/
├─ clickoutside.js
├─ line-clamp.js
├─ resize.js
├─ style.js
├─ transfer-dom.js
└─ v-click-outside-x.js
```

這一層負責處理不一定適合放進單一 component 的 DOM 行為。

| 檔案 | 可能負責的問題 | 讀碼時問什麼 |
|---|---|---|
| `style.js` | display、width、height、margin、padding、font、color、bg-color 等樣式型 directive | 為什麼要用 directive 操作樣式，而不是全部交給 class？ |
| `resize.js` | 元素 resize 監聽 | 哪些元件需要尺寸變化通知？ |
| `line-clamp.js` | 文字行數截斷 | 和 Ellipsis / Typography 類能力是否有關？ |
| `transfer-dom.js` | DOM 掛載位置轉移 | Modal、Drawer、Tooltip 等浮層為什麼需要 transfer？ |
| `clickoutside.js` / `v-click-outside-x.js` | 點擊外部偵測 | 下拉、浮層、選單如何判斷外部點擊？ |

注意：`src/index.js` 中全域註冊的 directive，不一定等於 `src/directives` 內所有檔案都直接註冊。讀碼時要回到 `src/index.js` 確認實際安裝路徑。

---

### 5.4 `src/mixins/`：跨元件共用邏輯

實際目錄包含：

```txt
src/mixins/
├─ emitter.js
├─ form.js
├─ globalConfig.js
├─ link.js
└─ locale.js
```

這是讀 View UI Plus 很重要的一層。很多元件看起來很短，是因為共用邏輯被放進 mixin。

| mixin | 主要價值 | 可能被哪些元件使用 |
|---|---|---|
| `emitter.js` | 跨層級事件派發或元件通訊 | Form、FormItem、Menu、Select 類需要上下文溝通的元件 |
| `form.js` | 表單狀態、disabled、size、驗證相關共用能力 | Input、Select、Checkbox、Radio、Switch、DatePicker |
| `globalConfig.js` | 讀取 `$VIEWUI` 全域配置 | 需要全域 size、icon、transfer 等設定的元件 |
| `link.js` | link / router-link / a 標籤行為抽象 | Button、MenuItem、Link、Cell 等可跳轉元件 |
| `locale.js` | 語系方法或文字取得 | DatePicker、Page、Table、Select 等有文案的元件 |

讀元件時只要看到 mixin，就應該在筆記中補上：

```md
## mixins 依賴
- 使用了哪些 mixin？
- 這些 mixin 替元件補了哪些 props / methods / computed？
- 如果不看 mixin，會誤解哪一段元件行為？
```

### 5.5 `src/utils/`：工具函式與底層輔助

實際目錄包含：

```txt
src/utils/
├─ assist.js
├─ calcTextareaHeight.js
├─ canUseDom.js
├─ csv.js
├─ date.js
├─ dom.js
├─ index.js
├─ keyCode.js
├─ random_str.js
├─ styleCheck.js
└─ transfer-queue.js
```

這一層不要死背，要用「元件遇到問題時回查」的方式讀。

| 檔案 | 常見用途 | 適合搭配哪類元件看 |
|---|---|---|
| `dom.js` | DOM class、style、事件、位置等輔助 | Tooltip、Poptip、Modal、Drawer、Select |
| `date.js` | 日期處理 | DatePicker、TimePicker、Calendar |
| `csv.js` | CSV 匯出或解析 | Table 類資料輸出功能 |
| `calcTextareaHeight.js` | textarea 高度計算 | Input textarea / autosize |
| `keyCode.js` | 鍵盤事件 key code 常數 | Select、DatePicker、Cascader、Tree、Menu |
| `transfer-queue.js` | 浮層或 transfer 排程管理 | Modal、Drawer、Tooltip、Poptip |
| `canUseDom.js` | SSR / DOM 可用性判斷 | 任何直接使用 DOM 的元件或 directive |
| `styleCheck.js` | 瀏覽器樣式能力檢測 | 樣式相容性相關功能 |
| `assist.js` | 通用輔助函式集合 | 多數元件都可能間接依賴 |

讀法建議：

```txt
不要先從 utils 開始硬讀。
先讀一個元件，遇到 import from utils，再回來追該工具函式。
這樣你會知道工具函式是為了解決哪個真實問題而存在。
```

### 5.6 `src/styles/`：樣式系統與主題骨架

實際目錄包含：

```txt
src/styles/
├─ animation/
├─ color/
├─ common/
├─ components/
├─ mixins/
├─ README.md
├─ base.less
├─ copyright.less
├─ custom.less
└─ index.less
```

樣式系統的讀法要從入口開始：

| 位置 | 角色 | 讀碼問題 |
|---|---|---|
| `index.less` | 樣式總入口 | 最終打包 CSS 會引入哪些樣式？ |
| `base.less` | 基礎樣式 | 全域 reset、字體、body、基礎 class 如何定義？ |
| `custom.less` | 可覆蓋變數或客製化入口 | 使用者如何客製主題？ |
| `components/` | 各元件樣式 | 元件 class 與 Less 規則如何對應？ |
| `mixins/` | Less mixin | 重複樣式邏輯如何抽象？ |
| `animation/` | 動畫 | Modal、Message、Dropdown 等如何過渡？ |
| `color/` | 色彩系統 | 主色、狀態色、色階如何集中管理？ |
| `common/` | 全域通用樣式 | 共用 class、輔助樣式如何放置？ |

讀元件時，樣式筆記不能只寫「有 class」。要至少補：

```md
## 樣式對照
- 元件根 class 是什麼？
- 對應 Less 檔在哪裡？
- 狀態 class 有哪些？例如 disabled / focused / active / error / loading
- 是否依賴全域變數、mixin、animation？
```

### 5.7 `src/locale/`：國際化與文案系統

實際目錄包含：

```txt
src/locale/
├─ lang/
├─ format.js
├─ index.js
└─ lang.js
```

這一層的讀法是從 `src/index.js` 的安裝參數切入：


你要追的問題是：

```txt
opts.locale 如何影響元件文案？
opts.i18n 如何接入外部 i18n？
內建 lang 包如何被使用？
元件內部如何拿到翻譯文字？
```

適合搭配這些元件一起讀：

```txt
DatePicker / TimePicker / Calendar / Page / Table / Select / Modal
```

## 6. 從使用者使用方式反推目錄關係

### 6.1 全量引入

使用者常見寫法：

```js
import { createApp } from 'vue'
import ViewUIPlus from 'view-ui-plus'
import 'view-ui-plus/dist/styles/viewuiplus.css'

createApp(App).use(ViewUIPlus).mount('#app')
```

這段背後對應的源碼路徑是：

```txt
view-ui-plus package
  ↓
package.json main / typings
  ↓
dist/viewuiplus.min.js
  ↓
由 src/index.js 打包而來
  ↓
install(app, opts)
  ↓
app.component / app.directive / globalProperties
  ↓
dist/styles/viewuiplus.css
  ↓
由 src/styles 打包而來
```

### 6.2 按需引入

使用者也可能這樣：

```js
import { Button, Modal } from 'view-ui-plus'
```

這時要回頭看：

```txt
src/components/index.js
  ↓
export { default as Button } from './button'
export { default as Modal } from './modal'
  ↓
各元件資料夾 index.js
  ↓
元件本體 .vue / js
```

所以 `src/components/index.js` 不是普通清單，而是元件庫對外 API 的核心邊界。

## 9. 初學者最容易誤讀的地方

### 9.1 把 `examples` 當成正式業務系統

`examples` 是 demo，不是產品。它的價值是反推元件用法，不是學業務架構。

### 9.2 把 `dist` 當成源碼

`dist` 是輸出結果，不是最佳閱讀入口。學設計要讀 `src`。

### 9.3 只看 `.vue`，不看 `index.js`

元件庫中 `index.js` 常常負責匯出與安裝。只看 `.vue` 會漏掉元件如何對外暴露。

### 9.4 只看元件，不看 styles

View UI Plus 是 UI 元件庫，樣式系統是核心之一。元件的 class、狀態 class、Less 變數與 mixin 都要對照。

### 9.5 只看元件本身，不追 mixins / utils / directives

很多行為不是寫在元件本體，而是由共用層支援。例如 form 行為、link 行為、全域配置、DOM transfer、resize、click outside。

### 9.6 一開始就讀 Table

Table 很有價值，但不適合當第一個元件。你還沒熟悉入口、匯出、樣式、共用層之前，直接讀 Table 很容易迷路。

## 11. 本篇完成標準

讀完這篇後，你不需要記住所有元件名稱，但應該能做到：

1. 看到 `src/index.js`，知道它是 Vue Plugin 入口。
2. 看到 `src/components/index.js`，知道它是 public component surface。
3. 看到 `src/components/*`，知道要分辨 public component 與 internal helper。
4. 看到 `src/styles`，知道樣式系統需要和元件一起讀。
5. 看到 `src/mixins`、`src/utils`、`src/directives`，知道它們是跨元件共用能力。
6. 看到 `examples`，知道它是 demo 與使用者視角，不是業務系統。
7. 看到 `types`，知道它是對 TypeScript 使用者的 API 契約。
8. 看到 `build` 和 `dist`，知道前者是建置輔助，後者是輸出產物。

一句話總結：

> **這篇不是要你把 View UI Plus 目錄背起來，而是讓你看到任一檔案時，都能判斷它在元件庫架構中的位置、讀碼優先級，以及應該產出什麼筆記。**
