## 本節導讀

這篇是 View UI Plus 原始碼閱讀的第一張地圖。它不追單一元件細節，而是先回答三個問題：

1. 每個主要目錄負責什麼？
2. 一個元件如何從 `src` 進入套件入口、被註冊，最後被使用者使用？
3. 後續讀碼時，應該從哪裡開始、何時查樣式、型別、範例與測試？

本篇會先把「工程建置線」和「插件安裝線」接起來看。工程建置線回答原始碼如何進入 build；插件安裝線回答 `app.use(ViewUIPlus)` 後元件與全域 API 如何被註冊。

工程建置線：

```text
package.json build
  -> build:prod  -> vite.config.js -> src/index.js -> dist/viewuiplus.min*.js
  -> build:style -> build/build-style.js -> src/styles/index.less -> dist/styles/viewuiplus.css
  -> build:lang  -> build/vite.lang.config.js -> src/locale/lang -> dist/locale
```

插件安裝與元件使用線：

```text
app.use(ViewUIPlus)
  -> src/index.js install()
  -> src/components/index.js
  -> src/components/<component>/index.js
  -> src/components/<component>/<component>.vue
```

這兩條線合起來回答的是：View UI Plus 如何從原始碼變成可建置、可安裝、可全域使用元件與命令式 API 的 Vue 3 元件庫。

## 你會在這篇學到什麼

- `package.json`、`vite.config.js`、`src/index.js` 之間的入口關係。

- 哪些區域適合第一輪先讀，哪些區域應該暫緩深挖。

## 0. 30 秒複習

只想快速抓回本篇核心時，記住這六句：

1. `package.json` 同時告訴你發布入口、型別入口與 build scripts。
2. `build:prod` 只負責 JS library build；樣式與語系分別由 `build:style`、`build:lang` 補齊。
3. `vite.config.js` 的 library entry 是 `src/index.js`，所以讀插件入口要回到原始碼，不要從壓縮後的 `dist` 開始。
4. `src/index.js` 的 `install()` 會註冊 components、註冊被納入 `directives` map 的指令，並掛載 `$VIEWUI` 與命令式 API。
5. `src/components/index.js` 是元件是否進入對外 API 的集中清單；單一元件仍要回到自己的資料夾看實作。
6. `types`、`examples`、`test` 不是主入口，但它們是反推對外契約、使用方式與行為規格的重要證據。

## 5. 打包與產物主線

View UI Plus 的正式建置可以拆成三條主線。

### 5.1 JavaScript library build

已確認：[`vite.config.js`](../ViewUIPlus-SourceCode/vite.config.js) 中：

- `build.lib.entry` 指向 `src/index.js`。
- library name 是 `ViewUIPlus`。
- Rollup output 同時輸出：
  - UMD：`viewuiplus.min.js`
  - ES：`viewuiplus.min.esm.js`
- `vue` 被設定為 external，UMD globals 是 `Vue`。

合理推論：要理解整個元件庫如何對外暴露，第一個入口是 [`src/index.js`](../ViewUIPlus-SourceCode/src/index.js)，不是 `dist/`。

### 5.2 樣式 build

已確認：[`build/build-style.js`](../ViewUIPlus-SourceCode/build/build-style.js) 中：

- 樣式入口是 `src/styles/index.less`。
- Less 會經過 `less`、`autoprefixer`、`cleanCSS`。
- 最後輸出 `dist/styles/viewuiplus.css`。
- 字型檔從 `src/styles/common/iconfont/fonts/*.*` 複製到 `dist/styles/fonts`。

已確認：[`src/styles/index.less`](../ViewUIPlus-SourceCode/src/styles/index.less) 的第一層 import 順序是：

```less
@import "./custom";
@import "./base";
@import "./mixins/index";
@import "./common/index";
@import "./animation/index";
@import "./components/index";
```

合理推論：讀樣式應從 `src/styles/index.less` 進入，再依序看變數 / 基礎樣式、mixins、common、animation、components，而不是從 `dist/styles/viewuiplus.css` 開始。

### 5.3 語系 build

已確認：[`build/vite.lang.config.js`](../ViewUIPlus-SourceCode/build/vite.lang.config.js) 會掃描 `src/locale/lang` 下的檔案，把每個語系檔做成 Rollup input，並輸出到 `dist/locale`。

已確認：[`src/locale/index.js`](../ViewUIPlus-SourceCode/src/locale/index.js) 是語系執行時邏輯，負責：

- 預設載入 `zh-CN`。
- 提供 `t(path, options)`。
- 提供 `use(l)` 切換語系。
- 提供 `i18n(initI18n)` 接入外部 i18n。

### 6.1 src/index.js：整個套件的入口

已確認：[`src/index.js`](../ViewUIPlus-SourceCode/src/index.js) 是目前 library build 的入口。它同時是定義位置、註冊位置與對外入口。

| 角度 | 在這裡看到什麼 |
|---|---|
| 定義位置 | `install`、`version`、`locale`、`i18n`、`lang` |
| 註冊位置 | `app.component`、`app.directive`、`app.config.globalProperties` |
| 對外入口 | default API、named exports、`export * from './components'` |

`install(app, opts = {})` 內已確認有幾個核心動作：

1. 若 `opts.locale` 存在，呼叫 `localeFile.use(opts.locale)`。
2. 若 `opts.i18n` 存在，呼叫 `localeFile.i18n(opts.i18n)`。
3. 遍歷 `ViewUI`，用 `app.component(key, ViewUI[key])` 全量註冊元件。
4. 遍歷 `directives`，用 `app.directive(key, directives[key])` 註冊內建指令。
5. 建立 `$VIEWUI` 全域配置。
6. 掛載 `$Spin`、`$Loading`、`$Message`、`$Notice`、`$Modal`、`$ImagePreview`、`$Copy`、`$ScrollIntoView`、`$ScrollTop`。
7. 掛載 `$Date = dayjs`。

因此，`src/index.js` 是後續三個專題的共同入口：

- 元件註冊機制。
- 全域配置與 `globalProperties`。
- 命令式 API 掛載。

### 6.2 src/components/index.js：元件匯出總表

已確認：[`src/components/index.js`](../ViewUIPlus-SourceCode/src/components/index.js) 是元件匯出的集中清單。它回答的是「哪些元件 / 服務被納入 View UI Plus 對外 API」。

單一元件的基礎追蹤路線是：

```text
src/components/index.js
  -> src/components/<component>/index.js
  -> src/components/<component>/<component>.vue
  -> src/styles/components/<component>.less
  -> types/<component>.d.ts
  -> examples/routers/<component>.vue
  -> test/unit/specs/<component>.spec.js
```

其中後三步不一定每個元件都有對應檔案，必須逐一確認。

以 Button 為例，已確認 [`src/components/button/index.js`](../ViewUIPlus-SourceCode/src/components/button/index.js) 是簡單 re-export：

```js
import Button from './button.vue';

export default Button;
```

以 Message 為例，已確認 [`src/components/message/index.js`](../ViewUIPlus-SourceCode/src/components/message/index.js) 不是 `.vue` 元件轉發，而是包裝 `info`、`success`、`warning`、`error`、`loading`、`config`、`destroy` 等命令式方法，並依賴 `../base/notification` 建立通知實例。

合理推論：讀任一 component 前，先判斷它屬於「普通 Vue 元件」、「元件群組入口」、「命令式服務」或「底層支撐層」，可以避免誤讀 `index.js` 的角色。

## 8. directives、mixins、utils 的位置感

這三個區域不是主入口，但常常是讀懂元件行為的關鍵。

### 8.1 directives

已確認：[`src/directives/`](../ViewUIPlus-SourceCode/src/directives/) 放指令實作；[`src/index.js`](../ViewUIPlus-SourceCode/src/index.js) 只會全域註冊 `directives` 物件中列出的指令。

入口中已確認註冊的指令包含：

- `display`
- `width`
- `height`
- `margin`
- `padding`
- `font`
- `color`
- `bg-color`
- `resize`
- `line-clamp`

合理推論：如果在 examples 或使用文件中看到 `v-resize`、`v-line-clamp` 或樣式快捷指令，應回到 `src/directives/` 查具體行為。

注意：`src/directives/` 目錄下還有其他指令檔，例如 `clickoutside.js`、`transfer-dom.js`、`v-click-outside-x.js`。不要只因為檔案位於 `src/directives/`，就直接判定它會被 `app.use(ViewUIPlus)` 全域註冊；必須回到 `src/index.js` 的 `directives` map 或具體元件使用點確認。

### 8.2 mixins

已確認：[`src/mixins/`](../ViewUIPlus-SourceCode/src/mixins/) 包含：

- `emitter.js`
- `form.js`
- `globalConfig.js`
- `link.js`
- `locale.js`

閱讀價值：

| mixin | 優先關注 |
|---|---|
| `emitter.js` | 跨層事件派發與接收 |
| `form.js` | 表單元件如何和 Form / FormItem 協作 |
| `globalConfig.js` | 元件如何讀取 `$VIEWUI` 全域配置 |
| `locale.js` | 元件如何使用翻譯函式 |
| `link.js` | 連結、跳轉或類 link 行為 |

### 8.3 utils

已確認：[`src/utils/`](../ViewUIPlus-SourceCode/src/utils/) 包含 DOM、日期、CSV、textarea 高度、keyCode、transfer queue 等工具檔。

合理推論：讀大型元件時，如果看到大量資料處理或 DOM 邏輯，不應只停在 `.vue`，而要追進 `src/utils/` 確認實際行為。

## 9. 樣式系統地圖

樣式原始碼在 [`src/styles/`](../ViewUIPlus-SourceCode/src/styles/)，正式 build 入口是 [`src/styles/index.less`](../ViewUIPlus-SourceCode/src/styles/index.less)。

| 區域 | 作用 | 讀碼時機 |
|---|---|---|
| `custom.less` | 樣式變數與客製設定入口 | 查主題變數、prefix、尺寸、顏色 |
| `base.less` | 基礎樣式入口 | 查全域基礎規則 |
| `mixins/` | Less mixin | 查按鈕、input、select、tooltip 等共用樣式生成 |
| `common/` | normalize、layout、article、iconfont 等共用樣式 | 查全域 class、iconfont、reset |
| `animation/` | fade、move、slide、ease、loop 等動畫 | 查 transition name 對應動畫 |
| `components/` | 各元件對應樣式 | 查元件 class 與狀態 class |
| `color/` | 色彩生成與 tinyColor 相關工具 | 查色彩計算邏輯 |

單一元件樣式建議對照方式：

```text
src/components/button/button.vue
  -> template / computed class
  -> src/styles/components/button.less
  -> src/styles/mixins/button.less
  -> src/styles/custom.less
```

待驗證：某些元件的樣式可能不只依賴同名 `components/<component>.less`，還可能依賴 common、mixins、animation 或其他元件樣式。讀單一元件時要從 class 名稱與 Less import 逐步確認。

## 10. 型別宣告地圖

TypeScript 型別在 [`types/`](../ViewUIPlus-SourceCode/types/)。入口是 [`types/index.d.ts`](../ViewUIPlus-SourceCode/types/index.d.ts)。

已確認：`types/index.d.ts` 做了幾件事：

- `export * from './viewuiplus.components'`，集中匯出元件型別。
- 宣告 `ViewUIPlusGlobalOptions`，對應 `$VIEWUI` 的全域配置。
- 宣告 `ViewUIPlusInstallOptions`，對應 `install(app, options)` 的 options。
- 擴充 `@vue/runtime-core` 的 `ComponentCustomProperties`，補上：
  - `$VIEWUI`
  - `$Spin`
  - `$Loading`
  - `$Message`
  - `$Notice`
  - `$Modal`
  - `$ImagePreview`
  - `$Copy`
  - `$ScrollIntoView`
  - `$ScrollTop`
  - `$Date`
- 宣告 `install: (app: App, options?: ViewUIPlusInstallOptions) => void`。

合理推論：分析單一元件時，可以把 `types/<component>.d.ts` 當成「對外 API 清單」，再回 `.vue` 檢查 props、events、slots、methods 是否一致。

待驗證：型別宣告不一定完整等同於執行時行為。若型別、README、examples、測試與 `.vue` 實作不一致，應以實作與測試為優先證據，並記錄衝突。

## 12. 常見混淆與實務排查

這一節用來避免初期讀碼最常見的錯誤心智模型。讀大型前端元件庫時，很多問題不是出在單一 `.vue`，而是出在入口、build、樣式、型別與範例之間沒有對齊。

| 容易混淆的地方 | 正確看法 | 排查方式 |
|---|---|---|
| `build` 等於 `vite build` | `build` 是三段流程：JS、樣式、語系；`vite.config.js` 只覆蓋 JS library build | 產物缺 JS 看 `build:prod`；缺 CSS 看 `build:style`；缺語系檔看 `build:lang` |
| `src/index.js` 和 `src/components/index.js` 是同一種入口 | `src/index.js` 是套件插件入口；`src/components/index.js` 是元件與服務匯出清單 | 追 `app.use` 看 `src/index.js`；追某元件是否對外匯出看 `src/components/index.js` |
| `src/directives/` 下的所有檔案都會全域註冊 | 只有被 `src/index.js` 的 `directives` map 收進去的指令會在 `install` 中註冊 | 先看 `directives` map，再看具體元件是否局部使用其他指令 |
| `types/<component>.d.ts` 等於執行時真相 | types 是對外型別契約，不保證和實作完全一致 | types、`.vue`、examples、test 若不一致，先記錄證據衝突，再以實作與測試優先確認 |
| examples 是源碼設計入口 | examples 更適合驗證使用方式，不適合取代實作閱讀 | 先讀元件實作，再用 examples 確認作者預期的使用情境 |
| dist 可以拿來學設計 | dist 是產物驗證入口，不是設計閱讀入口 | 只用 dist 確認輸出檔名、CSS、語系包，不逐行讀壓縮產物 |

工程實務中可以用這張表快速定位問題：

| 現象 | 優先檢查 | 不要一開始就做 |
|---|---|---|
| `app.use(ViewUIPlus)` 後元件不可用 | `src/index.js` 的 `ViewUI`、`app.component`、`src/components/index.js` 匯出鏈 | 直接翻 `dist/viewuiplus.min.js` |
| 命令式 API 例如 `$Message` 不存在 | `src/index.js` 的 `app.config.globalProperties` 掛載清單與對應 service | 只看元件 `.vue` |
| 元件有結構但樣式不對 | `build:style`、`src/styles/index.less`、component less、是否載入 `dist/styles/viewuiplus.css` | 只追 props 或 methods |
| TypeScript 報 API 型別錯 | `types/index.d.ts`、`types/<component>.d.ts`、實作 props / methods 是否一致 | 只改 runtime 程式碼 |
| Demo 可用但專案不可用 | examples 的引入方式、全域註冊方式、樣式載入與版本差異 | 直接假設元件實作壞掉 |

對 Java 全端學習來說，這張地圖的價值在於建立「前端套件如何被建置、發布、安裝與除錯」的工程視角。之後在 Spring Boot / Java 後端提供 API、前端用元件庫渲染資料時，若畫面、樣式、型別或互動出問題，可以先判斷問題落在前端套件入口、樣式產物、型別契約、Demo 使用方式，還是後端資料本身，而不是把所有 UI 異常都歸因於單一元件實作。

## 14. 單一元件追蹤模板

後續讀任何元件，都可以先照這個順序縮小範圍：

```text
1. 先查 src/components/index.js
   確認它是否被對外 export。

2. 再查 src/components/<component>/index.js
   判斷它是 re-export、元件群組入口，還是命令式服務。

3. 讀主要 .vue 或 index.js
   確認 props、emits、slots、data、computed、watch、methods。

4. 追 mixins / utils / base
   不要把共用邏輯漏掉。

5. 對照 src/styles/components/<component>.less
   把 class、狀態、動畫、prefixCls 對起來。

6. 對照 types/<component>.d.ts
   檢查對外 API 型別。

7. 對照 examples 與 test
   驗證使用方式與行為規格。
```

每次讀完至少留下五件事：

- 源碼位置。
- 對外 API。
- 內部狀態與事件流。
- 樣式與 class 來源。
- 可仿寫點與待驗證問題。

## 本節小結

View UI Plus 的第一層地圖可以濃縮成一句話：

> `package.json` 告訴你套件如何發布與建置，`vite.config.js` 指向 JS build 入口 `src/index.js`，`src/index.js` 負責插件安裝、元件註冊、指令註冊與全域 API 掛載，`src/components/index.js` 匯總元件，單一元件資料夾提供實作，`src/styles`、`types`、`examples`、`test` 則分別用來對照樣式、型別、使用方式與行為規格。

```text
安裝與註冊線 -> 元件匯出線 -> 單元件分析線 -> 樣式線 -> 對外規格線
```

這樣可以避免一開始陷入大量檔案細節，並把每次閱讀都累積成可回查、可延伸、可仿寫的筆記。
