## 1. 本篇筆記的目的

閱讀 View UI Plus 這類企業級 Vue 3 元件庫時，真正困難的地方通常不是「看不懂某一行程式碼」，而是容易把不同層級的概念混在一起。

例如：

- 把「全域安裝」誤解成「只能全域使用」。
- 把 `props` 誤解成所有狀態來源。
- 把 `$Message` 這類命令式 API 誤解成普通元件。
- 把 `globalProperties` 誤解成 Vuex、Pinia 或全域狀態管理。
- 把 `dist` 打包產物誤解成最適合讀碼的源碼。
- 把 `types` 型別宣告誤解成真正的 runtime 邏輯。
- 把 `examples` 看成文件，而不是「使用場景與規格反推資料」。

所以這篇筆記的目的不是介紹所有概念，而是建立一份「容易混淆概念的對照表」，幫助後續讀碼時快速判斷：

```txt
我現在看到的程式碼，屬於哪一層？
它是在處理註冊？配置？狀態？渲染？樣式？型別？還是打包？
```

---

## 2. 使用方式

建議遇到以下情況時回來查這篇：

```txt
1. 讀到一段程式碼，不知道它屬於哪個架構層。
2. 分不清楚某個 API 是元件 API、全域 API 還是命令式 API。
3. 看見 props、emits、slots、v-model 時，不知道該怎麼分類。
4. 看見 globalProperties、install、component、directive 時，不知道它們的責任邊界。
5. 看見 Form / FormItem / rules / validator 時，不知道誰負責什麼。
6. 看見 Modal / Message / Notice / Tooltip / Popover 時，不知道它們的差異。
7. 看見 Less、prefixCls、className、style directive 時，不知道樣式系統如何分層。
8. 看見 types、dist、src、examples、test 時，不知道該優先讀哪裡。
```

閱讀方式：

```txt
先看「對照表」
再看「判斷標準」
最後回到源碼驗證
```

不要只背結論，因為不同元件可能會有不同實作方式。這篇的作用是建立判斷框架。

---

## 3. 總判斷公式

讀到任何一段源碼時，先用下面這個公式判斷：

```txt
使用者輸入
  ↓
元件 API
  ↓
狀態來源
  ↓
內部邏輯
  ↓
渲染輸出
  ↓
副作用
  ↓
對外結果
```

對應問題如下：

| 判斷層級 | 要問的問題 |
|---|---|
| 使用者輸入 | 使用者是透過 props、slot、v-model、事件、全域配置，還是命令式方法使用它？ |
| 元件 API | 這是公開 API，還是內部私有實作？ |
| 狀態來源 | 狀態來自父層、自己、Form、全域配置，還是 DOM？ |
| 內部邏輯 | 是格式轉換、驗證、事件協調、定位、動畫，還是註冊？ |
| 渲染輸出 | 最後輸出的是 DOM、VNode、彈層、樣式 class，還是服務實例？ |
| 副作用 | 有沒有操作 DOM、註冊事件、定時器、全域實例、body 節點、document 事件？ |
| 對外結果 | 使用者感知到的是 UI 變化、事件回傳、Promise、callback，還是全域方法？ |

# Part A：專案定位與工程層易混淆概念

## 4. View UI Plus vs View Design vs iView vs View UI

| 概念 | 核心意思 | 容易混淆點 | 讀碼時的正確理解 |
|---|---|---|---|
| View UI Plus | 基於 Vue 3 的 UI 元件庫 | 以為它只是舊版 iView 換名字 | 應以目前 repo 與目前套件版本為準 |
| View Design | 設計體系 / 品牌體系 | 以為它就是 npm 套件名稱 | 它更像整體產品與設計系統名稱 |
| iView | Vue 2 時代常見的 UI library 名稱 | 拿 Vue 2 寫法直接套 Vue 3 | 可以參考歷史，但不能直接當成現在源碼邏輯 |
| View UI | 舊版或相關命名 | 和 View UI Plus 混用 | 源碼閱讀時要確認 package name 與 import name |

讀碼提醒：

```txt
不要用「以前 iView 怎麼做」直接推論 View UI Plus。
View UI Plus 是 Vue 3 版本的元件庫，要以目前 source code、package.json、README 為準。
```

## 5. UI 元件庫 vs Design System vs Admin Template vs 業務元件庫

| 概念 | 責任 | 例子 | 你讀碼時要學什麼 |
|---|---|---|---|
| UI 元件庫 | 提供通用 UI 元件 | Button、Input、Form、Table、Modal | 通用 API 設計、元件封裝、樣式系統 |
| Design System | 統一設計語言 | 色彩、字體、間距、元件規範 | 設計 token、命名規則、主題一致性 |
| Admin Template | 提供後台專案骨架 | Layout、Router、Permission、Dashboard | 專案架構、權限、路由、頁面組織 |
| 業務元件庫 | 封裝公司內部場景 | SearchForm、CrudTable、DictSelect | 如何把通用元件轉成業務開發效率工具 |

容易混淆：

```txt
View UI Plus 本身主要是 UI 元件庫，不等於完整後台系統。
但你可以透過閱讀它，學會如何封裝自己的企業後台元件庫。
```

你的學習轉化目標應該是：

```txt
View UI Plus 通用元件設計
  ↓
Mini 元件庫仿寫
  ↓
企業後台業務元件封裝
```

## 6. `src` vs `dist` vs `types` vs `examples` vs `test`

| 目錄 | 主要用途 | 適合用來做什麼 | 不適合做什麼 |
|---|---|---|---|
| `src` | 原始碼 | 理解真實設計與內部邏輯 | 不要只看檔名就推測行為 |
| `dist` | 打包產物 | 分析發佈格式、使用者實際引用結果 | 不適合當第一讀碼入口 |
| `types` | TypeScript 型別宣告 | 反推對外 API 形狀 | 不代表 runtime 一定完全一致 |
| `examples` | 使用範例 | 反推元件使用場景與規格 | 不要當成內部實作 |
| `test` | 測試案例 | 反推邊界條件與預期行為 | 不一定覆蓋所有場景 |

讀碼優先順序建議：

```txt
先看 examples / docs 了解使用方式
再看 src 理解實作
再看 types 對照 API
最後看 test 反推規格與邊界情況
```

## 7. GitHub Release vs npm 版本 vs package.json 版本 vs CDN 版本

| 概念 | 說明 | 容易混淆點 | 正確做法 |
|---|---|---|---|
| GitHub Release | repo 維護者在 GitHub 上標記的版本 | 以為它一定等於 npm latest | 不一定同步 |
| npm 版本 | 使用者 `npm install` 取得的版本 | 以為和 repo 頁面顯示完全一致 | 以 npm package 或 lockfile 為安裝基準 |
| package.json version | repo/package 中宣告的版本 | 以為所有平台都即時更新 | 需要確認目前分支與發佈狀態 |
| CDN 版本 | unpkg/jsDelivr 上提供的套件版本 | 以為 latest 永遠和本地一致 | 要固定版本避免讀碼與實際使用落差 |

讀碼建議：

```txt
正式寫筆記時，要記錄：
1. 閱讀日期
2. repo 分支或 commit
3. npm 套件版本
4. 本地 lockfile 版本
```

這樣之後版本變動時，筆記才不會失真。

## 8. 原始碼閱讀 vs API 使用

| 概念 | 目的 | 產出 |
|---|---|---|
| API 使用 | 知道怎麼用元件完成畫面 | 使用範例、常見配置 |
| 原始碼閱讀 | 知道元件為什麼這樣設計 | 設計模式、架構理解、仿寫能力 |

容易混淆：

```txt
會用 Button，不等於懂 Button 設計。
會用 Form，不等於懂 Form/FormItem 協作。
會用 Message，不等於懂命令式 API。
```

你的筆記應該避免只寫：

```txt
Button 可以設定 type、size、disabled。
```

而要寫成：

```txt
Button 的 props 分成三類：
1. 視覺類：type、shape、size
2. 狀態類：disabled、loading
3. 行為類：htmlType、to、replace、target

這反映出基礎元件 API 設計時，通常會把外觀、狀態、行為拆開。
```

# Part B：入口、安裝與全域註冊易混淆概念

## 9. `src/index.js` vs `src/components/index.js`

| 概念 | 主要責任 | 可以學到什麼 |
|---|---|---|
| `src/index.js` | 元件庫總入口、install、directives、globalProperties、locale、version | UI library 如何被 Vue app 安裝 |
| `src/components/index.js` | 彙整並 export 各個元件 | 元件庫如何管理元件出口 |
| 單一元件目錄 | 元件本身實作 | 單一元件 API、狀態、渲染、樣式 |

判斷方式：

```txt
如果你想知道 app.use(ViewUIPlus) 做了什麼：
看 src/index.js

如果你想知道有哪些元件被對外暴露：
看 src/components/index.js

如果你想知道 Button 怎麼實作：
看 src/components/button
```

容易混淆：

```txt
src/components/index.js 不是 install 的全部。
它只是元件匯出清單之一。
真正全域註冊、directive 註冊、globalProperties 掛載，通常要回到 src/index.js 觀察。
```

---

## 10. `install(app, opts)` vs `app.use(ViewUIPlus)` vs `import { Button }`

| 寫法 | 代表意思 | 特性 |
|---|---|---|
| `app.use(ViewUIPlus)` | 安裝整個元件庫插件 | 會觸發 install |
| `install(app, opts)` | 元件庫提供給 Vue 呼叫的安裝函式 | 裡面通常處理全域註冊與全域配置 |
| `import { Button } from 'view-ui-plus'` | 單獨引入某個元件 | 不一定會執行整包 install |
| `app.component('Button', Button)` | 手動註冊單一元件 | 使用者自己控制註冊範圍 |

容易混淆：

```txt
import 元件不等於 install 整個套件。
install 整個套件不等於每個地方都必須全域使用。
```

讀碼時要問：

```txt
這個元件是透過全域 install 註冊？
還是使用者可以單獨 import？
這兩種使用方式會不會走不同路徑？
```

---

## 11. 全域元件註冊 vs 區域元件註冊

| 概念 | 寫法 | 優點 | 缺點 |
|---|---|---|---|
| 全域註冊 | `app.component(name, component)` | 使用方便，不用每頁 import | 可能增加全域命名負擔 |
| 區域註冊 | `import Button from ...` 後在局部使用 | 更明確，可配合 tree-shaking | 使用成本較高 |
| 自動按需註冊 | 透過工具掃描元件使用 | 使用方便又能減少體積 | 依賴額外工具與 resolver |

讀 View UI Plus 時要觀察：

```txt
1. install 是否會遍歷所有元件？
2. 是否存在元件別名？
3. 是否支援 named export？
4. 文件推薦的是全量引入還是局部引入？
5. 打包產物是否支援按需？
```

## 12. `app.component` vs `app.directive` vs `app.config.globalProperties`

| API | 責任 | 使用者感知 |
|---|---|---|
| `app.component` | 註冊全域元件 | 模板中可以直接使用 `<Button />` |
| `app.directive` | 註冊全域指令 | 模板中可以使用 `v-resize`、`v-line-clamp` 等 |
| `app.config.globalProperties` | 掛載全域方法或物件 | Options API 可用 `this.$Message`、`this.$Modal` 等 |
| `provide` | 向子孫元件提供依賴 | 元件樹內部協作 |
| `inject` | 取得上層提供的依賴 | 表單、配置、父子協作常見 |

容易混淆：

```txt
component 是給模板標籤用。
directive 是給 v-xxx 行為用。
globalProperties 是給實例方法或全域服務用。
provide/inject 是給元件樹內部協作用。
```

這幾個都是「全域或跨層能力」，但責任不同。

---

## 13. `$VIEWUI` 全域配置 vs 單一元件 props

| 概念 | 作用範圍 | 適合放什麼 |
|---|---|---|
| `$VIEWUI` 全域配置 | 整個應用或元件庫預設值 | size、transfer、capture、某些元件預設 icon |
| props | 單一元件實例 | 當前元件的具體狀態與行為 |
| component default value | 元件內部預設值 | 沒傳 props 時的 fallback |
| provide/inject config | 某個元件樹範圍 | Form、Menu、Table 這類父子協作資料 |

容易混淆：

```txt
全域配置不是所有 props 的替代品。
props 通常優先描述當前元件實例。
全域配置通常描述全站共用預設值。
```

判斷順序：

```txt
單一元件 props
  ↓
上層 provide/inject
  ↓
$VIEWUI 全域配置
  ↓
元件內部預設值
```

但實際優先順序要以源碼為準。

## 14. 元件名稱 vs 元件別名

| 概念 | 例子 | 意義 |
|---|---|---|
| 原始元件名稱 | `Button`、`Input`、`Table` | 元件本身名稱 |
| 別名 | `iButton`、`iInput`、`iTable` | 可能為了避免原生標籤或歷史相容 |
| 檔案名稱 | `button`、`input`、`table` | 源碼目錄命名 |
| 文件名稱 | Button、Input、Table | 使用者文件中的名稱 |

容易混淆：

```txt
元件名稱、檔案名稱、註冊名稱、別名可能不是完全一樣。
讀碼時要追：
檔案 export 了什麼？
components/index.js export 了什麼？
src/index.js 註冊時用了什麼 key？
```

筆記建議增加一欄：

```txt
| 源碼檔案 | export 名稱 | install 註冊名稱 | 文件使用名稱 | 是否有別名 |
```

## 15. Props vs Attrs

| 概念 | 來源 | 是否明確宣告 | 用途 |
|---|---|---|---|
| props | 父層傳入 | 元件內明確宣告 | 元件公開 API |
| attrs | 父層傳入但未被 props 接收 | 不一定宣告 | 轉交到內部 DOM 或子元件 |
| DOM attributes | 真正落在 HTML 上的屬性 | HTML 層級 | 影響瀏覽器行為 |

容易混淆：

```txt
父層傳進來的東西，不一定都是 props。
有些會進入 attrs，可能被元件轉交到 input、button、a 等內部元素。
```

讀碼時要看：

```txt
1. props 宣告了哪些欄位？
2. inheritAttrs 是否被關閉？
3. attrs 是否被手動綁到內部元素？
4. 哪些屬性會被過濾、轉換或覆蓋？
```

---

## 16. Props vs Emits

| 概念 | 方向 | 功能 |
|---|---|---|
| props | 父 → 子 | 父層控制子元件輸入 |
| emits | 子 → 父 | 子元件通知父層發生事件 |
| v-model | 父 ↔ 子 | props + emits 的約定組合 |

容易混淆：

```txt
props 是輸入。
emits 是輸出。
v-model 是一種雙向資料同步協議。
```

讀元件時可以畫成：

```txt
Parent state
  ↓ props
Component internal logic
  ↓ emits
Parent update
```

---

## 17. Emits vs 原生 DOM Event

| 概念 | 觸發來源 | 使用者如何接 |
|---|---|---|
| 原生 DOM event | button、input、div 等 DOM | `@click`、`@input` |
| component emit | 元件內主動 `$emit` | `<MyComp @change="...">` |
| 包裝後事件 | 元件接收 DOM event 後整理再 emit | Select change、Input clear、Modal ok |

容易混淆：

```txt
使用者寫 @click，不代表一定是原生 click。
如果事件是在元件層 emit 出來，它就是元件自訂事件。
```

讀碼時要追：

```txt
DOM event
  ↓
內部 handler
  ↓
emit 對外事件
```

## 18. Slots vs Props

| 概念 | 適合傳什麼 | 例子 |
|---|---|---|
| props | 結構化資料、狀態、配置 | `type="primary"`、`:disabled="true"` |
| slots | 自訂內容、結構、局部 UI | Button 文字、Table cell、Modal footer |
| scoped slot | 子元件把資料交給父層決定怎麼渲染 | Table column slot、Select option slot |

容易混淆：

```txt
props 控制元件。
slot 擴展元件內容。
scoped slot 讓使用者接管局部渲染。
```

判斷方式：

```txt
如果使用者只是要改狀態或行為 → props
如果使用者要替換一段畫面結構 → slot
如果使用者要拿到元件內資料後自訂畫面 → scoped slot
```

---

## 19. `v-model` vs `modelValue` vs `value`

| 概念 | 所屬背景 | 說明 |
|---|---|---|
| `v-model` | 使用者語法 | 雙向綁定語法糖 |
| `modelValue` | Vue 3 常見約定 | v-model 預設 prop 名稱 |
| `update:modelValue` | Vue 3 常見約定 | v-model 預設更新事件 |
| `value` / `input` | Vue 2 常見或歷史相容寫法 | 舊版元件庫常見 |
| 自訂 model prop/event | 元件自訂 | 需要看源碼確認 |

讀碼提醒：

```txt
不要預設所有 Vue 3 元件庫都完全使用 modelValue。
從 Vue 2 遷移而來的元件庫，可能保留部分 value / input / change 等歷史 API。
```

讀元件時要確認：

```txt
1. props 中是否有 modelValue？
2. props 中是否有 value？
3. emits 中是否有 update:modelValue？
4. emits 中是否有 input、change？
5. 文件中的 v-model 對應哪個 prop/event？
```

---

## 20. 受控狀態 vs 非受控狀態

| 概念 | 狀態擁有者 | 特性 |
|---|---|---|
| 受控狀態 | 父元件 | 子元件透過 props 接收狀態，透過 emits 要求父層更新 |
| 非受控狀態 | 子元件自己 | 子元件自己維護內部狀態 |
| 混合模式 | 父子都參與 | 初始值由父層給，後續可能由內部維護 |

容易混淆：

```txt
有 v-model 通常偏受控。
但元件內仍可能有暫存狀態，例如：
- inputValue
- visible
- focused
- hovering
- selectedOptions
- cachedValue
```

讀碼時要分清楚：

```txt
外部真實值：model / value / selected
內部暫存值：inputValue / currentValue / visible / focused
```

## 21. Computed vs Watch vs Method

| 概念 | 用途 | 適合場景 |
|---|---|---|
| computed | 根據資料推導結果 | className、disabled 狀態、格式化顯示 |
| watch | 監聽資料變化並產生副作用 | 同步 props、重新驗證、更新 popper |
| method | 主動執行動作 | click、focus、validate、open、close |

容易混淆：

```txt
computed 是推導。
watch 是反應。
method 是動作。
```

讀碼時可以問：

```txt
這段邏輯是純推導？
還是資料變化後要做事？
還是使用者操作後才執行？
```

---

## 22. Mixin vs Utility Function vs Composable

| 概念 | 型態 | 特性 |
|---|---|---|
| mixin | Options API 共用配置 | 可以混入 props、data、methods、computed、lifecycle |
| utility function | 普通函式 | 輸入資料，輸出結果或執行工具邏輯 |
| composable | Composition API 共用邏輯 | 通常以 `useXxx()` 形式組織可組合狀態與方法 |

容易混淆：

```txt
mixin 不是單純工具函式。
mixin 會影響元件本身的選項與生命週期。
utility function 不應該偷偷改變元件結構。
composable 通常更明確回傳狀態與方法。
```

重構時的方向：

```txt
老式 mixin
  ↓
抽出純 utility
  ↓
抽成 useXxx composable
  ↓
保留清楚的輸入與輸出
```

# Part D：Form 與表單驗證易混淆概念

---

## 23. Form vs FormItem

| 概念 | 主要責任 |
|---|---|
| Form | 管理整個表單模型、rules、validate、reset、submit |
| FormItem | 管理單一欄位的 label、prop、驗證狀態、錯誤訊息 |
| Input / Select / DatePicker | 真正輸入資料的控制元件 |
| async-validator | 常見的規則驗證工具 |

容易混淆：

```txt
Input 不應該知道整個表單。
FormItem 通常知道自己對應哪個欄位。
Form 負責整體 model 和 rules。
```

典型協作：

```txt
Form
  ├── model
  ├── rules
  └── FormItem
        ├── prop
        ├── validateState
        ├── validateMessage
        └── Input / Select / DatePicker
```

---

## 24. `model` vs `rules` vs `prop`

| 概念 | 所在位置 | 意義 |
|---|---|---|
| `model` | Form | 整個表單資料物件 |
| `rules` | Form | 驗證規則集合 |
| `prop` | FormItem | 對應 model 中哪個欄位 |
| `value` / `modelValue` | Input 類元件 | 單一輸入元件目前值 |

容易混淆：

```txt
FormItem 的 prop 不是值本身。
prop 是欄位路徑，用來從 Form model 和 rules 中找到對應資料。
```

例子：

```vue
<Form :model="form" :rules="rules">
  <FormItem prop="username">
    <Input v-model="form.username" />
  </FormItem>
</Form>
```

對應關係：

```txt
Form model: form
Form rules: rules
FormItem prop: username
Input value: form.username
```

---

## 25. `validate` vs `validateField` vs `resetFields`

| 方法 | 作用範圍 | 使用時機 |
|---|---|---|
| `validate` | 整個表單 | submit 前檢查全部欄位 |
| `validateField` | 指定欄位 | blur/change 後檢查單一欄位 |
| `resetFields` | 整個或部分欄位 | 清空錯誤並重置初始值 |
| `clearValidate` | 驗證狀態 | 只清錯誤，不一定重置資料 |

容易混淆：

```txt
reset 不只是清錯誤，可能還會回到初始值。
clearValidate 通常只處理驗證狀態。
validate 是檢查，不是提交。
```

讀碼時要追：

```txt
1. validate 回傳 callback 還是 Promise？
2. validateField 如何找到對應 FormItem？
3. resetFields 是否保存 initialValue？
4. clearValidate 是否影響 model？
```

---

## 26. Trigger：`blur` vs `change`

| trigger | 觸發時機 | 常見場景 |
|---|---|---|
| `blur` | 失去焦點 | Input、Textarea |
| `change` | 值變動完成 | Select、DatePicker、Checkbox |
| manual | 手動呼叫 validate | submit、特殊流程 |

容易混淆：

```txt
不是所有欄位都適合 blur。
Select、DatePicker 這類元件通常更偏 change。
```

讀碼時要看：

```txt
輸入元件在什麼時候通知 FormItem？
FormItem 根據什麼 trigger 過濾 rules？
```

# Part E：彈層、命令式 API 與定位易混淆概念

---

## 27. Modal 元件 vs `$Modal` 命令式 API

| 概念 | 使用方式 | 特性 |
|---|---|---|
| Modal 元件 | `<Modal v-model="visible" />` | 宣告式，狀態通常由父層控制 |
| `$Modal` | `this.$Modal.confirm({...})` | 命令式，由方法建立或控制彈窗 |
| Modal 內部實例 | 源碼建立的彈窗容器或動態元件 | 使用者通常不直接接觸 |

容易混淆：

```txt
Modal 元件是模板中的 UI。
$Modal 是一個服務 API。
它們可能共用部分底層元件，但使用方式與狀態管理不同。
```

讀碼時要分清楚：

```txt
使用者宣告 visible
  → Modal component

使用者呼叫 confirm/info/success/error
  → Modal service / command API
```

---

## 28. Message vs Notice vs Modal vs Spin vs LoadingBar

| 概念 | 常見用途 | 互動性 | 存在時間 |
|---|---|---|---|
| Message | 短提示 | 低 | 自動消失 |
| Notice | 通知 | 中 | 可停留較久 |
| Modal | 對話框 | 高 | 需要使用者處理 |
| Spin | 區塊或全局載入狀態 | 低 | 跟任務狀態相關 |
| LoadingBar | 頁面頂部進度提示 | 低 | 跟路由或請求進度相關 |

容易混淆：

```txt
都是回饋元件，但不是同一類。
Message 解決「短暫提示」。
Notice 解決「通知展示」。
Modal 解決「阻斷式決策」。
Spin/LoadingBar 解決「等待狀態」。
```

企業後台選型：

```txt
成功提示：Message
審核通知：Notice
刪除確認：Modal.confirm
查詢 loading：Spin
路由切換/全局請求：LoadingBar
```

---

## 29. 宣告式 API vs 命令式 API

| 概念 | 寫法 | 狀態管理 |
|---|---|---|
| 宣告式 API | `<Modal v-model="visible" />` | 狀態在父層資料中 |
| 命令式 API | `$Message.success('完成')` | 透過方法產生副作用 |
| 混合模式 | 元件 + service 共用底層能力 | 需要拆清楚入口 |

容易混淆：

```txt
宣告式 API 比較像「畫面狀態」。
命令式 API 比較像「執行一個動作」。
```

讀碼時要問：

```txt
這個 API 是回傳 VNode？
還是建立實例？
還是操作既有 singleton？
還是只是改變已存在元件的狀態？
```

---

## 30. Singleton Container vs 多實例元件

| 概念 | 意義 | 常見位置 |
|---|---|---|
| Singleton Container | 全域只建立一個容器，後續往裡面加 item | Message、Notice |
| 多實例元件 | 每使用一次就有一個元件實例 | Button、Input、FormItem |
| 動態建立實例 | 呼叫 API 時臨時建立 | Modal.confirm、Message |

容易混淆：

```txt
看到 Message.newInstance，不代表每則訊息都是完全獨立的一整套 app。
可能是共用容器，再動態插入 message item。
```

讀碼時要追：

```txt
1. 是否有 instance 變數快取？
2. 是否有 getInstance / newInstance？
3. 是否將節點 append 到 document.body？
4. 是否提供 destroy / close / remove？
```

---

## 31. Tooltip / Poptip / Popover / Dropdown

| 概念 | 主要用途 | 互動複雜度 |
|---|---|---|
| Tooltip | 文字提示 | 低 |
| Poptip | 小型確認或提示 | 中 |
| Popover | 自訂浮層內容 | 中 |
| Dropdown | 下拉選單與選項 | 中高 |

容易混淆：

```txt
它們都可能用到定位系統，但語意不同。
不要因為底層都跟 popper / position 有關，就把它們視為同一種元件。
```

讀碼時要分：

```txt
觸發方式：hover / click / focus / contextmenu
定位方式：top / bottom / left / right
內容來源：文字 / slot / menu item
關閉條件：click outside / blur / manual / select
```

---

## 32. `transfer` vs Teleport vs appendToBody

| 概念 | 意義 |
|---|---|
| transfer | 元件庫常用設定名，表示彈層是否轉移到 body 或其他容器 |
| Teleport | Vue 3 官方提供的跨 DOM 掛載能力 |
| appendToBody | 手動把 DOM append 到 body |
| popper container | 定位系統使用的浮層容器 |

容易混淆：

```txt
transfer 是元件庫層的概念。
Teleport 是 Vue 提供的技術。
appendToBody 是 DOM 操作方式。
三者可能達到相似效果，但層級不同。
```

讀碼時不要直接假設：

```txt
看到 transfer 就等於一定用了 Teleport。
要看實際源碼如何搬移節點或渲染浮層。
```

# Part F：資料展示與複合元件易混淆概念

---

## 33. Table data vs columns vs slot render

| 概念 | 意義 | 控制權 |
|---|---|---|
| data | 表格資料來源 | 使用者提供 |
| columns | 欄位結構描述 | 使用者提供，元件解讀 |
| slot render | 自訂儲存格內容 | 使用者接管局部渲染 |
| internal state | 排序、篩選、展開、選取 | 元件內部維護或同步 |

容易混淆：

```txt
data 是資料。
columns 是資料如何被呈現。
slot 是使用者接管某一格怎麼畫。
```

企業封裝對應：

```txt
View UI Plus Table
  ↓
CrudTable
  ↓
欄位 schema
  ↓
查詢 / 分頁 / 排序 / 操作欄封裝
```

---

## 34. Select vs Option

| 概念 | 責任 |
|---|---|
| Select | 管理選中值、展開狀態、搜尋、清空、鍵盤操作 |
| Option | 描述一個可選項 |
| OptionGroup | 分組 |
| Dropdown layer | 展示選項列表 |
| FormItem | 表單驗證協作 |

容易混淆：

```txt
Option 不是單純靜態 DOM。
它通常需要和 Select 父層協作，例如註冊、選中、disabled、label、value。
```

讀碼時要追：

```txt
1. Option 如何被 Select 收集？
2. 點擊 Option 後誰更新 value？
3. Select 如何對外 emit change？
4. 搜尋時 Option 如何過濾？
5. clear 時如何同步 FormItem 驗證？
```

---

## 35. Tree / Cascader / TreeSelect

| 概念 | 核心資料結構 | 關鍵難點 |
|---|---|---|
| Tree | 樹狀節點 | 展開、選中、半選、懶載入 |
| Cascader | 分級選擇 | 路徑、級聯、hover/click 展開 |
| TreeSelect | Tree + Select | 樹狀資料與下拉選擇合併 |

容易混淆：

```txt
它們都處理樹狀資料，但互動模型不同。
Tree 偏展示與操作樹。
Cascader 偏路徑選擇。
TreeSelect 偏下拉選擇中的樹狀選項。
```

讀碼時要看：

```txt
資料結構
選中值格式
展開狀態
父子關聯
懶載入
搜尋策略
```

# Part G：樣式、主題與 PrefixCls 易混淆概念

---

## 36. prefixCls vs CSS class vs BEM

| 概念 | 意義 |
|---|---|
| prefixCls | 元件庫統一 class 前綴 |
| CSS class | 最終套在 DOM 上的 class |
| BEM | 一種 class 命名方法論 |
| Less 變數 | 樣式值的抽象，例如顏色、間距 |
| theme | 整體視覺主題 |

容易混淆：

```txt
prefixCls 不是 theme。
prefixCls 主要處理命名一致性與衝突隔離。
theme 處理視覺風格。
Less 變數處理樣式值。
```

例子：

```txt
prefixCls：ivu-btn
狀態 class：ivu-btn-primary、ivu-btn-disabled
Less 變數：@primary-color
theme：整套顏色與樣式配置
```

---

## 37. className vs style vs directive style

| 概念 | 適合處理什麼 |
|---|---|
| className | 狀態化、可複用樣式 |
| inline style | 動態數值或少量覆蓋 |
| style directive | 把常見樣式控制封裝成指令 |
| Less 檔案 | 元件主樣式與主題變數 |
| CSS variable | runtime 主題切換常見方案，但要看專案是否使用 |

容易混淆：

```txt
不是所有動態樣式都應該用 inline style。
不是所有樣式配置都應該做成 props。
不是所有樣式能力都屬於 theme。
```

讀碼時問：

```txt
這個樣式是元件狀態？
還是使用者自訂？
還是全域主題？
還是一次性 DOM 樣式？
```

---

## 38. Scoped style vs 全域樣式

| 概念 | 特性 |
|---|---|
| scoped style | 樣式只作用於當前 SFC 範圍 |
| 全域樣式 | 可作用於整個元件庫 |
| UI library style | 通常需要被使用者全域引入 |
| theme style | 可能需要覆蓋全局變數或重新編譯 |

容易混淆：

```txt
元件庫通常不能完全依賴 scoped style。
因為元件 class、主題、彈層、全域樣式都需要跨元件作用。
```

---

## 39. Icon prop vs customIcon vs slot

| 概念 | 使用方式 | 適合場景 |
|---|---|---|
| icon prop | 傳入 icon 名稱 | 快速配置 |
| customIcon | 傳入自訂 icon 設定 | 全域或特殊替換 |
| icon slot | 自訂整段渲染 | 高度客製 |
| render function | 程式化自訂渲染 | 高階場景 |

容易混淆：

```txt
icon prop 是配置。
slot 是渲染擴展。
customIcon 通常是更高層的自訂入口。
```

讀碼時要看：

```txt
icon 的優先順序：
slot > customIcon > icon prop > default icon
```

實際順序仍要以各元件源碼為準。

# Part H：型別、打包與測試易混淆概念

---

## 40. TypeScript 型別宣告 vs Runtime Props

| 概念 | 發生時間 | 責任 |
|---|---|---|
| TypeScript 型別 | 編譯期 / 開發期 | 提供 IDE 提示與型別檢查 |
| runtime props | 執行期 | Vue 實際接收與驗證 props |
| 文件 API 表 | 文件層 | 告訴使用者如何使用 |
| 測試案例 | 驗證層 | 確認行為符合預期 |

容易混淆：

```txt
types 寫了，不代表 runtime 一定照做。
runtime 支援，不代表 types 一定完整。
文件寫了，也可能和源碼版本有落差。
```

9.5 分筆記應該做三方對照：

```txt
| API | docs | types | source props/emits | test/examples | 備註 |
```

---

## 41. `main` / `module` / `types` / `files`

| package.json 欄位 | 用途 |
|---|---|
| `main` | 套件主要入口，常見於 Node / bundler 解析 |
| `module` | ESM 入口，若有提供可支援更佳 tree-shaking |
| `types` / `typings` | TypeScript 型別宣告入口 |
| `files` | npm 發佈時包含哪些檔案 |
| `scripts` | 開發、打包、樣式建置、測試等命令 |

容易混淆：

```txt
package.json 的入口是「使用者安裝後怎麼被解析」。
src/index.js 是「源碼中元件庫如何組織」。
dist 是「打包之後使用者實際拿到的產物」。
```

---

## 42. Build script vs Runtime logic

| 概念 | 發生時間 | 例子 |
|---|---|---|
| build script | 打包期間 | Vite build、gulp build style |
| runtime logic | 使用者瀏覽器執行時 | component render、event、validate |
| dev server | 開發期間 | examples 啟動、熱更新 |
| lint/test | 品質檢查期間 | ESLint、Mocha、Karma |

容易混淆：

```txt
build 檔案不是元件執行邏輯。
runtime 邏輯不一定會出現在 build 腳本。
```

讀碼時先問：

```txt
這段程式碼是在開發時跑？
打包時跑？
還是使用者瀏覽器中跑？
```

---

## 43. Test as Spec vs Test as Implementation Detail

| 概念 | 意義 |
|---|---|
| Test as Spec | 測試描述元件應該有什麼行為 |
| Implementation Detail | 測試依賴內部實作細節 |
| Examples as Usage | 範例展示使用者會怎麼使用 |
| Docs as Contract | 文件描述對使用者承諾的 API |

容易混淆：

```txt
測試不是只用來看有沒有 bug。
對讀碼者而言，測試是反推元件規格的工具。
```

讀測試時要整理：

```txt
1. 這個元件有哪些必要行為？
2. 哪些邊界情況被測到？
3. 哪些事件必須被觸發？
4. 哪些 class 或 DOM 結構被期待？
5. 哪些行為沒有被測到？
```

# Part I：常見讀碼誤區對照

---

## 44. 「看懂一段程式碼」vs「看懂設計」

| 層級 | 代表能力 |
|---|---|
| 看懂語法 | 知道這段 JS/Vue 在做什麼 |
| 看懂流程 | 知道資料如何流動 |
| 看懂責任 | 知道這段程式碼屬於哪一層 |
| 看懂設計 | 知道為什麼要這樣拆 |
| 能仿寫 | 能用同樣思路做出簡化版本 |
| 能重構 | 能指出它的限制並提出替代方案 |

9.5 分筆記不只寫：

```txt
這個方法會設定 visible 為 true。
```

而要寫：

```txt
這個方法屬於命令式 API 的狀態切換入口。
它把使用者呼叫方法轉換成內部彈層顯示狀態。
這代表該元件同時支援宣告式與命令式使用情境。
```

---

## 45. 「元件越複雜越值得先讀」是錯的

| 元件 | 是否適合一開始讀 | 原因 |
|---|---|---|
| Button | 適合 | API 簡單，容易理解元件設計基本型 |
| Input | 適合 | 可學 v-model、事件、表單協作 |
| Form/FormItem | 適合中期 | 父子協作與驗證複雜 |
| Table | 不建議太早 | API 面很廣，容易迷失 |
| Modal/Message | 中期後 | 需要先理解彈層與命令式 API |
| Tree/Cascader | 後期 | 資料結構與互動狀態複雜 |

正確策略：

```txt
先讀能建立框架感的元件。
再讀能建立橫向機制理解的元件。
最後讀最複雜的資料與彈層元件。
```

---

## 46. 「和 Element Plus 很像」不代表實作一樣

| 面向 | 可能相似 | 可能不同 |
|---|---|---|
| API 命名 | Button、Input、Form、Table | props 名稱、事件名稱、slot 名稱 |
| 使用方式 | Vue 元件形式 | install、全域配置、命令式 API |
| 樣式 | class + theme | prefix、Less/Sass、變數設計 |
| 表單驗證 | rules、validator | Form/FormItem 協作細節 |
| 彈層 | Modal、Message | 實例管理、容器策略、關閉流程 |

讀碼提醒：

```txt
可以用其他 UI library 做對照，但不要用它們替代 View UI Plus 的源碼證據。
```

---

## 47. 「全域 API」不等於「全域狀態管理」

| 概念 | 說明 |
|---|---|
| `$Message` | 全域可呼叫的提示服務 |
| `$Modal` | 全域可呼叫的對話框服務 |
| `$VIEWUI` | 元件庫全域配置 |
| Pinia/Vuex | 應用資料狀態管理 |
| provide/inject | 元件樹內依賴傳遞 |

容易混淆：

```txt
globalProperties 讓你在元件實例上存取某些方法或配置。
它不等於完整狀態管理方案。
```

---

## 48. 「有 types」不等於「源碼是 TypeScript」

| 概念 | 說明 |
|---|---|
| JavaScript source | 源碼主要用 JS 寫 |
| Type declarations | 額外提供 `.d.ts` 給 TS 使用者 |
| TypeScript source | 原始碼本身用 TS 寫 |
| JSDoc | 用註解補充型別資訊 |

容易混淆：

```txt
types 目錄存在，只能說明它提供型別宣告。
不能直接推論整個專案源碼都是 TypeScript。
```

讀碼時要分：

```txt
source implementation
type declaration
documentation
```

三者都要對照。

---

## 49. 「指令」vs「元件」

| 概念 | 使用方式 | 責任 |
|---|---|---|
| 元件 | `<Button />` | 封裝 UI 結構、狀態與行為 |
| 指令 | `v-resize`、`v-line-clamp` | 直接增強 DOM 行為 |
| 工具函式 | `formatDate()` | 純邏輯或輔助邏輯 |
| mixin | 注入元件選項 | 共用元件內部邏輯 |

容易混淆：

```txt
指令通常更靠近 DOM。
元件通常更靠近 UI 抽象。
工具函式通常不直接關心 Vue 生命週期。
mixin 會改變元件本身能力。
```

---

## 50. 「可配置」vs「可擴展」vs「可覆寫」

| 概念 | 使用方式 | 例子 |
|---|---|---|
| 可配置 | 透過 props/options 改變行為 | `size`、`type`、`disabled` |
| 可擴展 | 透過 slot/render 接管局部 | Table cell slot |
| 可覆寫 | 透過樣式、全域配置或自訂元件改掉預設 | prefixCls、theme、customIcon |

容易混淆：

```txt
props 多，不一定代表擴展性強。
slot 設計好，才是真正允許使用者接管局部 UI。
全域配置適合預設值，不適合塞所有業務差異。
```

企業封裝時要問：

```txt
這個需求應該做成：
1. props？
2. slot？
3. render function？
4. global config？
5. 業務元件外層封裝？
```

## 51. 一張表快速回查

| 容易混淆的兩個概念 | 一句話區分 |
|---|---|
| `src/index.js` vs `components/index.js` | 前者是元件庫入口與 install，後者是元件 export 清單 |
| `app.component` vs `app.directive` | 前者註冊元件，後者註冊 `v-xxx` 指令 |
| `globalProperties` vs `provide/inject` | 前者掛到 app 實例原型能力，後者用於元件樹依賴傳遞 |
| `$VIEWUI` vs props | 前者是全域預設配置，後者是單一元件實例配置 |
| props vs attrs | props 是明確宣告 API，attrs 是未被 props 接收的外部屬性 |
| emits vs DOM event | emits 是元件對外事件，DOM event 是原生元素事件 |
| slot vs props | slot 換內容結構，props 改狀態與配置 |
| v-model vs value | v-model 是語法協議，value 是可能的具體 prop |
| controlled vs uncontrolled | 受控由父層管狀態，非受控由元件自己管 |
| computed vs watch | computed 推導資料，watch 監聽變化做副作用 |
| mixin vs util | mixin 注入元件選項，util 是普通工具函式 |
| Form vs FormItem | Form 管整體 model/rules，FormItem 管單一欄位驗證 |
| model vs prop | model 是資料物件，prop 是欄位路徑 |
| validate vs resetFields | validate 是檢查，resetFields 是重置 |
| Modal vs `$Modal` | Modal 是宣告式元件，`$Modal` 是命令式服務 |
| Message vs Notice | Message 短提示，Notice 通知展示 |
| Tooltip vs Popover | Tooltip 偏文字提示，Popover 偏自訂浮層內容 |
| transfer vs Teleport | transfer 是元件庫配置概念，Teleport 是 Vue 技術 |
| prefixCls vs theme | prefixCls 管 class 命名，theme 管視覺風格 |
| Less variable vs props | Less variable 是樣式值，props 是元件 API |
| types vs source | types 是型別宣告，source 是實際執行邏輯 |
| examples vs test | examples 看使用場景，test 看預期行為 |
| build script vs runtime | build script 打包時跑，runtime 在瀏覽器執行 |
| dist vs src | dist 是產物，src 是源碼 |
| UI library vs admin template | UI library 給元件，admin template 給後台架構 |

## 52. 概念混淆時的五步檢查法

當你分不清楚某段程式碼時，照下面五步檢查：

```txt
第一步：它在哪個目錄？
- src？
- components？
- directives？
- mixins？
- utils？
- types？
- examples？
- test？
- build？
- dist？

第二步：它何時執行？
- 開發時？
- 打包時？
- app.use 時？
- 元件 render 時？
- 使用者操作時？
- 表單驗證時？
- 彈層建立時？

第三步：它服務誰？
- 使用者 API？
- 元件內部？
- 父子協作？
- 全域配置？
- 樣式系統？
- 型別提示？
- 測試驗證？

第四步：它的輸入輸出是什麼？
- props → DOM？
- event → emit？
- options → globalProperties？
- model/rules → validate result？
- command call → dynamic instance？

第五步：它能不能被仿寫成最小版本？
- 可以：表示你真的理解它的責任。
- 不可以：表示你可能只看懂語法，還沒看懂設計。
```
