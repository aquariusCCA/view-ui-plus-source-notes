> 學習對象：已有 Vue 開發經驗，想透過 View UI Plus 系統性理解 Vue 3 元件庫設計、企業後台元件封裝與高互動前端工程能力的人。

---

## 1. 這篇筆記的定位

這篇不是「View UI Plus 元件清單」。

這篇的真正定位是：

> 建立一套元件閱讀優先級地圖，幫助我知道哪些元件應該先讀、哪些元件應該後讀、哪些元件適合做高階專題、哪些元件適合拿來仿寫與企業後台封裝。

讀元件庫時最容易犯的錯，是從 `src/components` 看到什麼就讀什麼，結果會出現幾個問題：

1. 一開始就讀太複雜的元件，例如 `Table`、`DatePicker`、`Select`、`Cascader`，很容易被細節淹沒。
2. 太早看命令式元件，例如 `Message`、`Notice`、`Modal`，但還沒理解元件註冊、全域 API、動態掛載。
3. 只看單一元件，沒有看出元件庫背後的共同規律。
4. 只看 UI 畫面，沒有看出 Props、Emits、Slots、v-model、provide/inject、globalProperties、prefixCls、mixin、utils、types 之間的協作。
5. 讀完很多元件，卻沒有轉化成自己能仿寫或封裝企業元件的能力。

所以這篇要解決的核心問題是：

```txt
我應該按照什麼順序閱讀 View UI Plus 的元件，
才能用最短路徑理解一套 Vue 3 企業級元件庫的設計方式？
```

---

## 2. 分類原則：不要只照元件功能分類，要照「學習價值」分類

一般 UI 元件庫會把元件分成：

```txt
基礎元件
布局元件
表單元件
資料展示元件
導航元件
回饋元件
業務型元件
```

這種分類有用，但對「讀源碼」還不夠。

因為讀源碼時，我更應該關心的是：

| 分類維度 | 要問的問題 |
|---|---|
| API 設計價值 | 這個元件能不能讓我學到 Props / Emits / Slots / v-model 的設計？ |
| 狀態管理價值 | 這個元件有沒有內部狀態、受控狀態、非受控狀態？ |
| 父子協作價值 | 這個元件有沒有和父層、子層元件協作？ |
| 橫向機制價值 | 這個元件有沒有牽涉全域配置、locale、utils、mixins、directives？ |
| 互動行為價值 | 這個元件有沒有滑鼠、鍵盤、focus、blur、scroll、resize、outside click 等互動？ |
| DOM 操作價值 | 這個元件有沒有操作真實 DOM、定位、掛載、銷毀、轉移節點？ |
| 企業後台價值 | 這個元件在企業後台是不是高頻使用？ |
| 仿寫價值 | 這個元件是否適合自己做 mini version？ |
| 難度 | 這個元件適不適合目前階段閱讀？ |

因此，本篇採用的是：

```txt
功能分類 + 學習價值分類 + 難度分級 + 企業後台使用頻率
```

## 3. 元件閱讀優先級評分模型

閱讀 View UI Plus 元件時，可以用下面這個模型幫元件打分。

每個維度 1～5 分：

| 維度 | 說明 |
|---|---|
| 使用頻率 | 企業後台是否常用 |
| API 密度 | Props / Emits / Slots / v-model 是否豐富 |
| 狀態複雜度 | 是否有內部狀態、雙向綁定、狀態同步 |
| 協作複雜度 | 是否和其他元件形成父子或跨層協作 |
| 橫向機制 | 是否牽涉全域配置、locale、utils、mixins、directives、types |
| 互動複雜度 | 是否牽涉 keyboard、mouse、focus、scroll、resize、outside click |
| 仿寫價值 | 是否適合做 mini component 練習 |
| 企業封裝價值 | 是否能轉成 SearchForm、CrudTable、DialogForm、DictSelect 等後台元件 |

### 推薦優先級判斷

```txt
優先級 A：
高使用頻率 + 高學習價值 + 可拆解 + 可仿寫

優先級 B：
有一定難度，但能學到核心模式

優先級 C：
使用場景較特殊，適合補齊視野

優先級 D：
偏業務增強或邊緣功能，最後再讀
```

## 4. View UI Plus 元件閱讀總策略

我的總策略是：

```txt
先讀簡單、通用、可觀察 API 設計的元件；
再讀表單與輸入元件；
再讀父子協作與驗證系統；
再讀資料展示與複雜互動；
再讀彈層、定位、命令式 API；
最後讀業務型與增強型元件。
```

不要一開始就追求「看最難的」。

正確順序應該是：

```txt
Button / Icon
    ↓
Input / Radio / Checkbox
    ↓
Form / FormItem
    ↓
Select / DatePicker / Cascader
    ↓
Modal / Message / Notice / Drawer
    ↓
Table / Page / Tree / Upload
    ↓
業務型元件與高階專題
```

# 第一類：基礎通用元件

## 6. 基礎通用元件的定位

基礎通用元件通常是最適合入門讀碼的元件。

它們的特點是：

1. API 相對清楚。
2. 狀態不一定複雜。
3. 容易看到 props 如何控制 class、style、slot。
4. 容易觀察元件命名、prefixCls、class 組合方式。
5. 適合用來建立「讀單一元件」的基本能力。

### 代表元件

```txt
Button
ButtonGroup
Icon
Alert
Badge
Divider
Progress
Circle
Tag
Link
Typography / Paragraph / Text 類元件
```

> 實際元件名稱請以當前版本的 `src/components` 為準。

---

## 7. 基礎通用元件推薦閱讀順序

### 7.1 第一個元件：Button

`Button` 是最適合拿來當 View UI Plus 讀碼起點的元件。

原因：

1. 使用頻率極高。
2. API 容易理解。
3. 通常會有 `type`、`size`、`disabled`、`loading`、`icon`、`shape` 等 props。
4. 能觀察 class 如何根據 props 動態組合。
5. 能觀察 slot 如何承載文字內容。
6. 能觀察 click 事件如何處理。
7. 能觀察全域 size 設定是否影響元件。
8. 很適合仿寫 `MiniButton`。

閱讀重點：

```txt
1. props 如何設計？
2. class 如何組合？
3. disabled / loading 狀態如何影響行為？
4. icon 如何插入？
5. slot 如何渲染？
6. ButtonGroup 如何影響 Button？
7. 是否使用 mixin 或 prefixCls？
```

仿寫產出：

```txt
MiniButton
MiniButtonGroup
```

---

### 7.2 第二個元件：Icon

`Icon` 適合用來理解「基礎渲染型元件」。

閱讀重點：

```txt
1. icon 名稱如何轉成 class？
2. size / color 如何處理？
3. 是否支援自訂 class？
4. 是否只是純展示，還是也承擔互動角色？
5. 在 Button、Menu、Select、DatePicker 等元件中如何被複用？
```

仿寫產出：

```txt
MiniIcon
```

---

### 7.3 第三個元件：Alert

`Alert` 適合用來理解：

```txt
1. type 狀態如何控制樣式？
2. closeable 如何設計？
3. showIcon 如何設計？
4. slot 和 props 如何共同決定內容？
5. close 事件如何 emit？
```

仿寫產出：

```txt

```

---

### 7.4 第四個元件：Badge

`Badge` 適合用來理解「包裹型元件」。

閱讀重點：

```txt
1. 如何包裹預設 slot？
2. count / dot / overflowCount 如何控制展示？
3. offset 如何影響定位？

```

### 7.5 第五個元件：Progress / Circle

`Progress` 和 `Circle` 適合用來理解「資料驅動視覺展示」。

閱讀重點：

```txt
1. percent 如何控制畫面？

3. strokeWidth / trailColor / strokeColor 如何設計？
```

# 第三類：表單與輸入元件

## 12. 表單與輸入元件的定位

表單與輸入元件是 View UI Plus 最值得深挖的一類。

原因：

1. 企業後台最常見。
2. 涉及大量 v-model。
3. 涉及受控與非受控狀態。
4. 涉及 focus、blur、change、input、clear 等事件。
5. 涉及 Form / FormItem 協作。
6. 涉及驗證狀態、錯誤提示、disabled、readonly、size。
7. 很適合轉成企業級封裝元件。

# 第四類：表單協作與驗證系統

## 15. Form / FormItem 的特殊地位

`Form` 和 `FormItem` 不應該只放在「表單元件」裡，它們應該被視為一套橫向系統。

它們通常會牽涉：

```txt
model
rules
validate
validateField
resetFields
clearValidate
FormItem 收集
prop path
async-validator
錯誤訊息
驗證狀態
子元件 blur / change 觸發驗證
provide / inject
```

這是 View UI Plus 讀碼中最重要的專題之一。

---

## 16. Form / FormItem 推薦閱讀時機

不建議一開始就讀 `Form / FormItem`。

建議前置順序：

```txt
Button
Input
RadioGroup
CheckboxGroup
Select 初步
    ↓
Form
FormItem
    ↓
表單驗證系統專題
```

如果還沒讀過 Input，就直接讀 Form，很容易不知道表單元件如何觸發驗證。

如果還沒讀過 provide/inject 或 mixin，也會看不懂 FormItem 如何收集與協作。

---

## 17. Form / FormItem 閱讀重點

```txt
1. Form 如何接收 model？
2. Form 如何接收 rules？
3. FormItem 如何知道自己的 prop？
```

# 第十九類：本篇的最終結論

## 69. 最重要的閱讀原則

```txt
再把這些模式轉化成自己的企業後台元件封裝能力。
```

```txt
4. 再進入表單驗證
```
