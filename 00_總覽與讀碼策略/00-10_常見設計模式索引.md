# 00-10_ViewUIPlus 常見設計模式索引

> 筆記定位：這份文件不是「設計模式名詞表」，而是 View UI Plus 源碼閱讀時的「模式索引」。  
> 目標是讓我在閱讀 Button、Input、Form、Modal、Message、Table、Select、Dropdown、Tooltip 等元件時，能快速判斷：這段程式碼背後使用的是什麼設計思路？它解決什麼問題？我能不能把它轉化成自己的企業後台元件封裝能力？

---

## 0. 這篇筆記要解決什麼問題？

閱讀元件庫源碼時，最容易犯的錯是：

1. 只看懂單一檔案，卻不知道它在整體架構中的角色。
2. 只記得 props、methods、watch，卻沒有看出背後的設計模式。
3. 只會說「這裡用了 mixin」、「這裡用了 globalProperties」，但不知道為什麼這樣設計。
4. 看到 Message、Modal、Notice 這種命令式 API 時，只知道用法，卻不知道它如何動態建立實例。
5. 看到 Form / FormItem / Input 的關係時，只看局部元件，沒有理解父子協作機制。
6. 看到 prefixCls、locale、directives、utils、types 時，不知道它們是元件庫級別的橫向能力。

所以這份筆記的核心任務是：

```txt
把 View UI Plus 中反覆出現的設計思路整理成索引，
讓我在讀每個元件時都能快速對照、分類、提煉、仿寫。
```

## 2. 先分清楚：這裡的「設計模式」不是只指 GoF

View UI Plus 裡面會出現的模式，大多不是傳統 GoF 的 Singleton、Factory、Observer 那種教科書式模式，而是前端元件庫常見的「工程設計模式」。

我會把它們分成六類：

```txt
1. 工程架構模式
   install、export、register、build、dist、types

2. 元件 API 模式
   props、emits、slots、v-model、controlled / uncontrolled

3. 元件協作模式
   parent-child、provide/inject、mixin、event dispatch、form item 協作

4. 命令式服務模式
   Message、Notice、Modal、LoadingBar、Spin

5. 橫向能力模式
   global config、locale、prefixCls、directive、utils、style、types

6. 企業封裝轉化模式
   SearchForm、CrudTable、DialogForm、DictSelect、PermissionButton
```

簡單說：

```txt
GoF 設計模式偏向「物件導向設計」；
元件庫設計模式偏向「UI 系統如何組織、擴展、協作、封裝」。
```

## 4. Plugin Install 模式

### 4.1 模式定義

Plugin Install 模式是 Vue 元件庫最核心的入口模式。

它的目的是讓使用者可以這樣使用整套元件庫：

```js
import { createApp } from 'vue'
import ViewUIPlus from 'view-ui-plus'
import 'view-ui-plus/dist/styles/viewuiplus.css'

const app = createApp(App)

app.use(ViewUIPlus, {
  size: 'default',
  transfer: true
})
```

背後的核心概念是：

```txt
把所有元件、指令、全域配置、命令式服務，統一收斂到 install(app, opts)。
```

### 4.2 在 View UI Plus 中的觀察位置

建議閱讀：

```txt
src/index.js
```

重點觀察：

```txt
1. install 如何接收 app 與 opts
2. 是否防止重複 install
3. 如何批量註冊 components
4. 如何批量註冊 directives
5. 如何設定 app.config.globalProperties
6. 如何暴露 version / locale / i18n / lang
```

### 4.3 要問自己的問題

```txt
1. install 是整個元件庫唯一入口嗎？
2. 使用者 app.use(ViewUIPlus) 時，實際發生了哪些事？
3. components 是從哪裡收集來的？
4. directives 是從哪裡收集來的？
5. opts 會如何影響全域配置？
6. Message、Modal、Notice 這些命令式 API 是在 install 裡掛載的嗎？
7. 如果我只想按需引入 Button，是否還需要 install 整套元件庫？
```

### 4.4 可以學到什麼？

這個模式適合你未來仿寫：

```txt
MiniViewUI.install(app, options)
MiniViewUI.Button
MiniViewUI.Input
MiniViewUI.Form
MiniViewUI.Message
MiniViewUI.Modal
```

你可以從最小版本開始：

```js
const components = [MiniButton, MiniInput]

export default {
  install(app) {
    components.forEach(component => {
      app.component(component.name, component)
    })
  }
}
```

### 4.5 企業後台轉化

在企業內部，你可以用這個模式做：

```txt
1. 內部元件庫 install
2. 全域業務元件註冊
3. 全域指令註冊
4. 全域權限方法掛載
5. 全域字典工具掛載
6. 全域日期格式化工具掛載
```

例如：

```js
app.use(BankAdminUI, {
  permission: true,
  dict: true,
  defaultTableSize: 'small'
})
```

## 5. Barrel Export 模式

### 5.1 模式定義

Barrel Export 是指用一個統一入口檔，把大量模組集中 export 出去。

在元件庫中，它解決的問題是：

```txt
使用者不需要知道每個元件實際檔案路徑，
只要從統一入口 import 即可。
```

例如：

```js
import { Button, Input, Modal } from 'view-ui-plus'
```

### 5.2 在 View UI Plus 中的觀察位置

建議閱讀：

```txt
src/components/index.js
src/index.js
```

常見形式：

```js
export { default as Button } from './button'
export { default as Input } from './input'
export { default as Form } from './form'
```

### 5.3 要問自己的問題

```txt
1. 每個元件是否都有統一 export？
2. export 名稱是否等於元件 name？
3. 有沒有因為歷史相容而保留別名？
4. 是否支援 import { Button } from 'view-ui-plus'？
5. 是否支援整套 app.use(ViewUIPlus)？
6. Barrel export 是否會影響 tree-shaking？
```

### 5.4 可以學到什麼？

Barrel Export 是元件庫對外 API 的第一層。

你在讀它時不要只看「export 了什麼」，而要看：

```txt
這個元件庫希望對使用者暴露哪些穩定 API？
```

### 5.5 企業後台轉化

你自己的內部元件庫也應該有：

```txt
src/components/index.ts
src/directives/index.ts
src/utils/index.ts
src/index.ts
```

而不是讓使用者寫：

```js
import SearchForm from '@/components/form/search-form/SearchForm.vue'
```

更好的方式是：

```js
import { SearchForm, CrudTable, DialogForm } from '@bank/admin-ui'
```

## 6. Component Registry 模式

### 6.1 模式定義

Component Registry 模式是指把元件集中成一個 map 或 object，然後批量註冊。

概念如下：

```js
const components = {
  Button,
  Input,
  Form,
  FormItem,
  Modal
}

Object.keys(components).forEach(key => {
  app.component(key, components[key])
})
```

### 6.2 解決的問題

```txt
1. 避免每個元件都手動 app.component
2. 讓 install 邏輯簡潔
3. 方便統一處理元件別名
4. 方便後續擴充元件清單
```

### 6.3 在 View UI Plus 中的觀察重點

```txt
1. components 來源
2. ViewUI 物件如何組合
3. 是否存在 iButton、iInput、iTable 等別名
4. install 時是否遍歷 ViewUI
```

### 6.4 企業後台轉化

你可以設計：

```js
const components = {
  SearchForm,
  CrudTable,
  DialogForm,
  DictSelect,
  PermissionButton
}

export function install(app) {
  Object.entries(components).forEach(([name, component]) => {
    app.component(name, component)
  })
}
```

## 7. Alias Compatibility 模式

### 7.1 模式定義

Alias Compatibility 是指為了相容歷史 API 或避免名稱衝突，額外提供元件別名。

例如 View UI Plus 中可觀察到類似：

```txt
Button 之外，也可能存在 iButton
Input 之外，也可能存在 iInput
Table 之外，也可能存在 iTable
```

### 7.2 解決的問題

```txt
1. 避免與原生標籤或其他元件名衝突
2. 相容舊版 iView / View UI 使用習慣
3. 讓使用者有過渡方案
```

### 7.3 要問自己的問題

```txt
1. 這個別名是為了相容舊 API 還是避免衝突？
2. 別名是否會增加維護成本？
3. 文件是否推薦使用原名還是別名？
4. 未來是否可能廢棄？
```

### 7.4 企業後台轉化

在企業後台元件庫中可以用，但要謹慎。

例如：

```txt
BankTable
CrudTable
BTable
```

如果別名太多，會造成團隊風格混亂。

建議規則：

```txt
1. 核心元件只保留一個正式名稱
2. 舊名稱只作相容，不鼓勵新程式使用
3. 文件明確標示 deprecated
```

## 8. Global Config 模式

### 8.1 模式定義

Global Config 模式是指元件庫允許使用者在安裝時傳入全域配置，然後元件內部可以讀取這些配置。

例如：

```js
app.use(ViewUIPlus, {
  size: 'small',
  transfer: true,
  modal: {
    maskClosable: false
  }
})
```

### 8.2 解決的問題

```txt
1. 不用每個元件都重複傳 prop
2. 建立整套系統的預設行為
3. 讓企業專案可以統一 UI 規格
4. 讓 Modal、Select、Tree、DatePicker 等元件有統一預設值
```

### 8.3 在 View UI Plus 中的觀察位置

建議閱讀：

```txt
src/index.js
src/mixins/globalConfig.js
```

重點觀察：

```txt
1. $VIEWUI 的資料結構
2. size、capture、transfer 如何設定
3. cell、menu、modal、tabs、select、colorPicker、cascader、tree 等如何被配置
4. 元件內如何透過 globalConfig 讀取全域配置
```

### 8.4 要問自己的問題

```txt
1. 哪些設定適合放全域？
2. 哪些設定應該只由單一元件 prop 控制？
3. 全域配置與局部 prop 衝突時，誰優先？
4. 全域配置是否有預設值？
5. 配置格式是否容易擴充？
6. 是否需要 TypeScript 型別保護？
```

### 8.5 企業後台轉化

非常值得學。

企業後台常見全域配置：

```txt
1. tableSize
2. formLabelWidth
3. dialogWidth
4. dateFormat
5. emptyText
6. permissionMode
7. dictSource
8. uploadMaxSize
9. requestErrorHandler
10. defaultPageSize
```

例如：

```js
app.use(AdminUI, {
  table: {
    size: 'small',
    pageSize: 20,
    stripe: true
  },
  form: {
    labelWidth: 120,
    validateTrigger: 'blur'
  }
})
```

## 9. globalProperties 掛載模式

### 9.1 模式定義

Vue 3 中可以透過：

```js
app.config.globalProperties.xxx = something
```

把全域方法或服務掛到元件實例上。

使用者可以在 Options API 中透過：

```js
this.$Message.success('成功')
this.$Modal.confirm(...)
```

### 9.2 解決的問題

```txt
1. 讓 Message、Notice、Modal 這類服務可以全域呼叫
2. 讓日期工具、配置物件、工具函式可以全域取得
3. 降低業務程式中重複 import 的成本
```

### 9.3 在 View UI Plus 中的觀察重點

View UI Plus 在 install 中會掛載多個全域能力，例如：

```txt
$VIEWUI
$Spin
$Loading
$Message
$Notice
$Modal
$ImagePreview
$Copy
$ScrollIntoView
$ScrollTop
$Date
```

### 9.4 要問自己的問題

```txt
1. 哪些東西適合掛 globalProperties？
2. 掛太多是否會污染全域命名空間？
3. Composition API 中是否還適合依賴 this.$xxx？
4. 是否需要額外提供 composable，例如 useMessage()？
5. 測試時如何 mock 全域服務？
```

### 9.5 企業後台轉化

銀行後台或企業後台常見：

```js
this.$message.success('儲存成功')
this.$modal.confirm(...)
this.$auth.hasPermission('user:create')
this.$dict.getLabel('currency', value)
this.$download.excel(...)
```

但建議你在新專案中更偏向：

```js
const message = useMessage()
const auth = useAuth()
```

原因是 Composition API 裡 composable 更容易型別化、測試與重構。

## 10. Locale / i18n Adapter 模式

### 10.1 模式定義

Locale / i18n Adapter 模式是指元件庫不直接綁死某一套國際化實作，而是提供一層語言包與 i18n adapter。

### 10.2 解決的問題

```txt
1. 元件文字可以多語系
2. 日期、空狀態、提示文字可以國際化
3. 使用者可以接入自己的 i18n 函式
4. 元件庫不必強依賴特定業務專案的 i18n 架構
```

### 10.3 在 View UI Plus 中的觀察位置

建議閱讀：

```txt
src/locale
src/components/locale
src/mixins/locale.js
src/index.js 中的 locale / i18n / lang
```

### 10.4 要問自己的問題

```txt
1. 語言包資料結構長什麼樣？
2. locale.use 做了什麼？
3. i18n 函式如何注入？
4. 元件內如何取得翻譯文字？
5. 如果使用者沒有設定語系，預設語系是什麼？
6. DatePicker / TimePicker / Page / Table 等元件是否共用語系機制？
```

### 10.5 企業後台轉化

即使你的公司暫時只做中文系統，也值得學這個模式。

因為它可以轉化成：

```txt
1. 統一錯誤訊息
2. 統一欄位 label
3. 統一空狀態文案
4. 統一操作按鈕文字
5. 統一表單驗證訊息
```

## 11. PrefixCls 樣式前綴模式

### 11.1 模式定義

PrefixCls 是 UI 元件庫常見的樣式命名模式。

概念是：

```txt
元件 class 不直接散落命名，
而是由統一前綴組合出來。
```

例如：

```txt
ivu-btn
ivu-input
ivu-form
ivu-modal
ivu-select
```

### 11.2 解決的問題

```txt
1. 避免 class 命名衝突
2. 建立元件庫統一識別
3. 方便主題樣式覆蓋
4. 方便元件內狀態 class 組合
5. 方便使用者查 DOM 時辨識元件結構
```

### 11.3 要問自己的問題

```txt
1. 每個元件是否有固定 prefixCls？
2. 狀態 class 如何命名？例如 disabled、active、focused、error
3. 尺寸 class 如何命名？例如 small、large
4. 類型 class 如何命名？例如 primary、dashed、text
5. class 是寫死還是 computed 出來？
6. prefixCls 是否可以全域自訂？
```

### 11.4 企業後台轉化

你自己的內部元件庫也應該有一致前綴。

例如：

```txt
bank-btn
bank-table
bank-search-form
bank-dialog-form
bank-dict-select
```

或：

```txt
admin-btn
admin-table
admin-form
```

不要讓 class 變成：

```txt
my-button
custom-table
searchBox
form-area
```

這樣長期很難維護。

## 12. Props Driven UI 模式

### 12.1 模式定義

Props Driven UI 是 Vue 元件最基本、也是元件庫最重要的 API 設計模式。

概念是：

```txt
元件外觀、狀態、行為，主要由 props 驅動。
```

例如：

```vue
<Button type="primary" size="small" disabled loading>
  提交
</Button>
```

### 12.2 解決的問題

```txt
1. 讓元件使用方式宣告化
2. 讓 UI 狀態可預測
3. 讓元件 API 清楚
4. 讓文件、測試、型別都能圍繞 props 展開
```

### 12.3 讀碼觀察重點

讀每個元件時都要整理 props：

```txt
1. 外觀型 props
   type、size、shape、icon、color

2. 狀態型 props
   disabled、loading、active、checked、visible

3. 行為型 props
   clearable、filterable、multiple、transfer、closable

4. 資料型 props
   modelValue、value、data、columns、options

5. 高階擴展 props
   render、formatter、customIcon、validate
```

### 12.4 要問自己的問題

```txt
1. 這個 prop 是控制樣式還是行為？
2. prop 預設值是否合理？
3. prop 類型是否明確？
4. prop 是否可以被全域配置覆蓋？
5. prop 命名是否和其他元件一致？
6. 有沒有多個 prop 控制同一件事？
```

### 12.5 企業後台轉化

企業後台封裝元件時，要避免 props 失控。

例如 SearchForm：

```vue
<SearchForm
  :schemas="schemas"
  :loading="loading"
  :label-width="120"
  :collapsed="collapsed"
  @search="handleSearch"
  @reset="handleReset"

```

不要把所有業務條件都硬寫在元件內部。

## 13. Emits Event Contract 模式

### 13.1 模式定義

Emits Event Contract 是指元件透過事件向外部通知狀態變化或使用者操作。

例如：

```vue
<Input
  :model-value="keyword"
  @update:model-value="keyword = $event"
  @on-enter="handleSearch"
/>
```

### 13.2 解決的問題

```txt
1. 元件不直接操作外部狀態
2. 父層可以決定如何回應使用者操作
3. 元件內外形成清楚的事件契約
4. 支援 v-model
```

### 13.3 讀碼觀察重點

```txt
1. 事件名稱是否一致？
2. 事件觸發時機是否明確？
3. 事件參數是否穩定？
4. 是否有 update:modelValue？
5. 是否有 before / after 類型事件？
6. 是否有 on-change / on-input / on-clear / on-focus / on-blur？
```

### 13.4 企業後台轉化

你在封裝元件時，要把事件設計成「業務語意」或「UI 語意」。

例如：

```vue
<CrudTable
  @query="handleQuery"
  @reset="handleReset"
  @create="openCreateDialog"
  @edit="openEditDialog"
  @delete="handleDelete"
/>
```

比起：

```vue
<CrudTable
  @click-button-a="..."
  @click-button-b="..."
/>
```

前者更容易維護。

## 15. v-model 雙向綁定模式

### 15.1 模式定義

v-model 是表單元件最核心的資料同步模式。

Vue 3 中常見形式：

```vue
<Input v-model="form.name" />
```

背後等價於：

```vue
<Input
  :model-value="form.name"
  @update:model-value="form.name = $event"
/>
```

### 15.2 解決的問題

```txt
1. 讓表單元件使用簡潔
2. 統一外部資料與內部輸入狀態
3. 降低父元件管理事件的成本
```

### 15.3 讀碼觀察重點

```txt
1. 元件使用 modelValue 還是 value？
2. 事件是 update:modelValue 還是 input/change？
3. 內部是否有 currentValue？
4. prop 變化時是否同步內部狀態？
5. 內部輸入時是否 emit 更新事件？
6. 是否支援多個 v-model？
```

### 15.4 常見陷阱

```txt
1. 直接修改 prop
2. 內部狀態與外部 prop 不同步
3. change / input / update:modelValue 時機混亂
4. blur 後才更新導致 UI 延遲
5. 表單驗證觸發時機不一致
```

## 16. Controlled / Uncontrolled 狀態模式

### 16.1 模式定義

受控與非受控狀態是元件庫非常重要的設計概念。

```txt
受控狀態：
狀態由父層傳入，元件只負責顯示與 emit。

非受控狀態：
狀態由元件自己管理，父層不一定介入。
```

### 16.2 常見例子

```txt
Input:
- value / modelValue 由父層控制

Modal:
- visible 可以由父層控制
- 命令式 Modal.confirm 則由內部服務控制

Tabs:
- activeKey 可以父層控制
- 也可以元件內部維護目前 active tab

Select:
- selected value 由 v-model 控制
- dropdown open 狀態可能由內部控制
```

### 16.3 要問自己的問題

```txt
1. 這個狀態誰擁有？
2. 父層是否可以完全控制？
3. 元件是否有 defaultValue？
4. 外部 prop 改變時，內部狀態是否同步？
5. 內部狀態改變時，是否通知外部？
6. 是否會出現雙重資料來源？
```

### 16.4 企業後台轉化

封裝 SearchForm 時：

```txt
form model 通常應該受控；
collapsed 展開收合狀態可以內部管理，也可以讓父層控制。
```

封裝 DialogForm 時：

```txt
visible 通常建議受控；
loading 可以由父層或元件內部管理，但要明確。
```

## 33. Config Normalize 模式

## 40. Schema Driven Component 模式

### 40.1 模式定義

Schema Driven Component 是企業後台常見進階模式。

概念是：

```txt
不是手寫大量 template，
而是用 schema 描述 UI。
```

例如：

```js
const schemas = [
  {
    field: 'customerName',
    label: '客戶名稱',
    component: 'Input',
    props: {
      clearable: true
    }
  },
  {
    field: 'status',
    label: '狀態',
    component: 'Select',
    options: statusOptions
  }
]
```

然後：

```vue
<SearchForm :schemas="schemas" v-model="query" />
```

### 40.2 和 View UI Plus 的關係

View UI Plus 本身是通用元件庫，不一定直接做 schema-driven。

但你可以從它學到：

```txt
1. 每個基礎元件 API 如何設計
2. Form / FormItem 如何協作
3. Select / Option 如何處理 options
4. validation 如何接入
5. slot / render 如何提供擴展
```

然後把這些能力組合成企業級 schema form。

```txt
1. 全域狀態過多
2. 命令式 API 沒有 destroy
3. directive 沒有清理 event listener
4. transfer DOM 沒有處理 unmount
5. mixin 命名衝突
6. props 過多導致 API 膨脹
7. slot / render / prop 職責混亂
8. Form 驗證與輸入事件觸發時機不一致
```
