## 本節導讀

`00-01_源碼總地圖` 解決的是「檔案在哪裡、各區塊負責什麼」。這一節解決的是「應該照什麼順序讀，讀到什麼程度可以停，下一步接去哪裡」。

View UI Plus 的源碼量不小，如果從 `src/components/` 第一個資料夾一路往下讀，很容易變成看過很多檔案，卻沒有形成穩定理解。更有效的方式是沿著幾條主線分段閱讀：

```text
工程建置線
  -> 入口安裝線
  -> 元件註冊線
  -> 代表性元件分析線
  -> 樣式系統線
  -> 型別 / 範例 / 測試反推線
  -> 高階設計線
```

每條線都只追到能回答當前問題的程度，再把細節交給後續章節展開。

- 工程建置線

## 建議回查情境

當你遇到以下情境，可以回到這篇：

- 不知道下一個應該讀哪個章節或哪個源碼檔。
- 讀元件時分支越追越多，需要重新收斂主線。
- 想判斷某個主題該先看工程設定、入口、元件本體、樣式、型別，還是測試。
- 想安排一輪完整的 View UI Plus 源碼學習進度。
- 想確認某個階段讀完後，至少應該能回答哪些問題。

## 0. 快速選路表

如果只是回查，不必從第一條路線順讀。先用下面三張表判斷「現在要走哪條線、從哪個檔案開始、讀到什麼程度可以停」。

### 0.1 用問題選路線

| 你現在的問題 | 優先路線 | 第一個入口 | 這輪要產出的結果 |
|---|---|---|---|
| 想知道專案如何啟動、建置、發布 | 工程建置線 | [`package.json`](../ViewUIPlus-SourceCode/package.json) | 一張 build / dev / publish 流程圖 |
| 想知道 `app.use(ViewUIPlus)` 做了什麼 | 入口安裝線 | [`src/index.js`](../ViewUIPlus-SourceCode/src/index.js) | `install` 的動作拆解 |
| 想確認某個元件是否對外可用 | 元件註冊線 | [`src/components/index.js`](../ViewUIPlus-SourceCode/src/components/index.js) | 定義、匯出、註冊鏈 |
| 想讀懂某個元件的 props / events / slots | 元件分析線 | `src/components/<component>/index.js` | 元件 API、狀態、事件、渲染與樣式主流程 |
| 想知道某個 class、動畫或主題變數從哪裡來 | 樣式系統線 | [`src/styles/index.less`](../ViewUIPlus-SourceCode/src/styles/index.less) 或 component less | class 到 Less selector / 變數 / mixin 的對照 |
| 想確認外部用法或行為規格 | 型別 / 範例 / 測試反推線 | [`types/`](../ViewUIPlus-SourceCode/types/)、[`examples/`](../ViewUIPlus-SourceCode/examples/)、[`test/`](../ViewUIPlus-SourceCode/test/) | 實作、型別、範例、測試的一致性檢查 |
| 想抽象出可仿寫的元件庫設計能力 | 高階設計線 | 已讀過的代表元件筆記 | 設計模式、適用情境、可仿寫點 |

### 0.2 用時間選讀法

| 時間 | 建議讀法 | 停止條件 |
|---|---|---|
| 5 分鐘 | 只看本篇 `0.1` 與第 1 節總表 | 能選出下一個入口檔 |
| 15 分鐘 | 看對應路線的「先讀什麼」與「什麼時候可以停」 | 能列出本輪閱讀邊界 |
| 30 分鐘 | 照單一路線追完整個最小閉環 | 能留下已確認、合理推論、待驗證 |
| 半天 | 完成一個階段，例如工程建置線或入口安裝線 | 能更新對應章節筆記 |
| 一輪學習 | 照第 9 節階段表推進 | 能從總覽、入口、元件、樣式、型別、測試一路交叉驗證 |

### 0.3 每條路線的完成產物

| 路線 | 完成後應留下 |
|---|---|
| 工程建置線 | `package.json` scripts、Vite build、style build、locale build 的流程圖 |
| 入口安裝線 | `install` 拆解表、全域配置與命令式 API 掛載清單 |
| 元件註冊線 | `components/index.js` 到 `src/index.js` 的註冊鏈 |
| 元件分析線 | 單一元件分析筆記，至少包含 API、狀態、事件、渲染、樣式 |
| 樣式系統線 | `prefixCls`、class、Less、變數、mixin、animation 對照 |
| 型別 / 範例 / 測試反推線 | types、examples、tests 與實作的一致性或衝突記錄 |
| 高階設計線 | 可仿寫模式、適用場景、反例與待驗證問題 |

## 2. 路線一：工程建置線

### 2.1 先讀什麼

| 順序 | 源碼位置 | 已確認事實 | 本階段要回答 |
|---|---|---|---|
| 1 | [`package.json`](../ViewUIPlus-SourceCode/package.json) | `main` 是 `dist/viewuiplus.min.js`，`typings` 是 `types/index.d.ts`，`files` 包含 `dist`、`src`、`types` | 套件發布入口、型別入口、發布範圍是什麼？ |
| 2 | [`vite.config.js`](../ViewUIPlus-SourceCode/vite.config.js) | `build.lib.entry` 指向 `src/index.js`，輸出 UMD 與 ES | JS library build 從哪裡開始？ |
| 3 | [`build/build-style.js`](../ViewUIPlus-SourceCode/build/build-style.js) | 從 `src/styles/index.less` 編譯到 `dist/styles/viewuiplus.css` | 樣式如何生成 dist CSS？ |
| 4 | [`build/vite.lang.config.js`](../ViewUIPlus-SourceCode/build/vite.lang.config.js) | 掃描 `src/locale/lang`，輸出到 `dist/locale` | 語系包如何被獨立建置？ |
| 5 | [`vite.config.dev.js`](../ViewUIPlus-SourceCode/vite.config.dev.js)、[`vue.config.js`](../ViewUIPlus-SourceCode/vue.config.js) | examples 同時存在 Vite dev 與 Vue CLI 設定 | 本地 Demo 如何啟動？ |

### 2.2 閱讀目標

讀完工程建置線，不需要理解每個 Rollup 選項，但要能回答：

- `npm run build` 實際分成哪幾段？
- 正式 JS 打包入口為什麼是 `src/index.js`？
- 樣式入口為什麼是 `src/styles/index.less`？
- 語系包為什麼會輸出到 `dist/locale`？
- `dist/` 是產物觀察區，為什麼不適合初期逐行閱讀？

### 2.3 什麼時候可以停

本線讀到能畫出這條流程就可以停：

```text
package.json scripts
  -> vite.config.js -> src/index.js -> dist/viewuiplus.min.js / viewuiplus.min.esm.js
  -> build/build-style.js -> src/styles/index.less -> dist/styles/viewuiplus.css
  -> build/vite.lang.config.js -> src/locale/lang -> dist/locale
```

細節交給 [04_build_打包腳本與建置流程](../04_build_打包腳本與建置流程/README.md) 與 [05_dist_打包產物分析](../05_dist_打包產物分析/README.md)。

## 3. 路線二：入口安裝線

### 3.1 先讀什麼

入口安裝線從 [`src/index.js`](../ViewUIPlus-SourceCode/src/index.js) 開始。

已確認：`src/index.js` 中包含以下角色：

| 位置 | 作用 |
|---|---|
| `export * from './components'` | 對外 re-export 所有 components |
| `import * as components from './components'` | 收集 components 給 `ViewUI` 與 `install` 使用 |
| `directives` 物件 | 收集內建指令 |
| `ViewUI` 物件 | 收集 components，並補上 `iButton` 等別名 |
| `install(app, opts = {})` | 插件安裝主函式 |
| `app.component` | 全量註冊元件 |
| `app.directive` | 註冊內建指令 |
| `app.config.globalProperties` | 掛載 `$VIEWUI` 與命令式 API |
| default `API` | 對外預設匯出 |

### 3.2 閱讀目標

讀完入口安裝線，要能回答：

- `app.use(ViewUIPlus, opts)` 進入哪個函式？
- `opts.locale`、`opts.i18n` 如何進入 locale 系統？
- 元件如何透過 `app.component` 被註冊？
- 內建指令如何透過 `app.directive` 被註冊？
- `$VIEWUI` 保存哪些全域配置？
- `$Message`、`$Notice`、`$Modal`、`$Loading`、`$Date` 等全域屬性在哪裡掛上？

### 3.3 什麼時候可以停

能把 `install` 拆成以下五段，就可以停：

```text
locale 設定
  -> components 註冊
  -> directives 註冊
  -> $VIEWUI 全域配置
  -> 命令式 API / $Date 掛載
```

深入分析交給 [06_src_入口與插件機制](../06_src_入口與插件機制/README.md) 與 [08_全域配置與globalProperties](../08_全域配置與globalProperties/README.md)。

## 4. 路線三：元件註冊線

### 4.1 先讀什麼

元件註冊線的固定入口是 [`src/components/index.js`](../ViewUIPlus-SourceCode/src/components/index.js)。

已確認：這個檔案集中 `export { default as Xxx } from './xxx'`，例如：

- `Button`
- `Form`
- `Input`
- `Select`
- `DatePicker`
- `Table`
- `Modal`
- `Message`
- `Notice`
- `LoadingBar`
- `ImagePreview`

### 4.2 閱讀目標

這條線要回答的是「元件如何進入 View UI Plus 的對外 API」：

```text
src/components/<component>/index.js
  -> src/components/index.js
  -> src/index.js 的 components
  -> ViewUI
  -> install 裡的 app.component
```

但要注意：不是每個 `index.js` 都只是轉發 `.vue`。

| 類型 | 例子 | 讀法 |
|---|---|---|
| 簡單 re-export | [`button/index.js`](../ViewUIPlus-SourceCode/src/components/button/index.js) | 直接進入 `.vue` |
| 元件群組入口 | `tabs`、`breadcrumb`、`timeline` | 要看父子元件如何一起匯出或協作 |
| 命令式服務 | [`message/index.js`](../ViewUIPlus-SourceCode/src/components/message/index.js)、`notice`、`loading-bar` | 要看 instance 建立、方法包裝、銷毀 |
| 底層支撐層 | `base/notification`、`base/popper.js` | 要看被誰依賴，不一定直接對外使用 |

### 4.3 什麼時候可以停

能回答以下問題就可以停：

- 某個元件是否出現在 `src/components/index.js`？
- 它的 `src/components/<component>/index.js` 是轉發、群組入口，還是命令式服務？
- 它最後是否會被 `src/index.js` 的 `install` 全量註冊？
- 若是命令式 API，它是否還會掛到 `globalProperties`？

深入分析交給 [07_元件註冊機制](../07_元件註冊機制/README.md) 與 [10_src-components_元件總覽](../10_src-components_元件總覽/README.md)。

## 5. 路線四：代表性元件分析線

### 5.1 不要從所有元件開始

`src/components/` 底下元件很多。合理推論：第一輪不應按字母或資料夾順序讀，而是選代表性元件建立模式。

建議順序如下：

| 階段 | 代表元件 | 為什麼先讀 | 要建立的能力 |
|---|---|---|---|
| 1 | `button` | 結構小，適合看最小元件模式 | props、class、slots、events |
| 2 | `layout`、`row`、`col`、`grid` | 結構清楚，適合看容器與響應式 | 組合元件、樣式 class |
| 3 | `form`、`form-item`、`input` | 是表單系統入口 | v-model、驗證、上下文協作 |
| 4 | `select` | 互動分支多但仍可追 | 下拉、選項、狀態同步、彈層 |
| 5 | `date-picker` | 檔案拆分明顯 | 面板、日期工具、範圍選擇 |
| 6 | `modal`、`message`、`notice` | 命令式 API 代表 | instance、全域 API、銷毀流程 |
| 7 | `table` | 大型元件代表 | 資料轉換、render 拆分、slot、summary、export |
| 8 | `login`、`exception`、`number-info` | 業務型元件代表 | 如何基於基礎元件封裝後台場景 |

### 5.2 單一元件固定路線

每次讀單一元件，都照這條線走：

```text
src/components/index.js
  -> src/components/<component>/index.js
  -> src/components/<component>/<main>.vue 或 index.js
  -> mixins / utils / base 依賴
  -> src/styles/components/<component>.less
  -> types/<component>.d.ts
  -> examples/routers/<component>.vue
  -> test/unit/specs/<component>.spec.js
```

後三步不一定存在，要逐一確認。已確認目前 `test/unit/specs` 中只有部分代表性 spec，例如 `button`、`select`、`table`、`message`、`date-picker`、`breadcrumb`、`affix`。

### 5.3 每個元件至少要留下什麼

每篇元件筆記至少保留：

- 源碼位置。
- 對外 API：props、events、slots、methods。
- 內部狀態：data、computed、watch。
- 互動流程：使用者操作 -> 事件 -> 狀態變化 -> UI 更新。
- 共用依賴：mixins、utils、base、directives。
- 樣式來源：`prefixCls`、class、Less 檔案。
- 型別 / example / test 對照。
- 可仿寫點與待驗證問題。

深入模板交給 `00-04_元件分析模板.md`。

## 6. 路線五：樣式系統線

### 6.1 先讀什麼

已確認：[`src/styles/index.less`](../ViewUIPlus-SourceCode/src/styles/index.less) 是樣式入口，第一層 import 順序是：

```less
@import "./custom";
@import "./base";
@import "./mixins/index";
@import "./common/index";
@import "./animation/index";
@import "./components/index";
```

建議閱讀順序：

| 順序 | 位置 | 要回答 |
|---|---|---|
| 1 | `src/styles/index.less` | 樣式入口如何組合？ |
| 2 | `src/styles/custom.less` | 主題變數在哪裡？ |
| 3 | `src/styles/mixins/` | 共用 Less mixin 如何被元件樣式使用？ |
| 4 | `src/styles/common/` | reset、layout、iconfont 等全域樣式在哪裡？ |
| 5 | `src/styles/animation/` | transition name 對應哪些動畫？ |
| 6 | `src/styles/components/` | 單一元件樣式在哪裡？ |

### 6.2 和元件分析怎麼接

讀元件時不要把樣式放到最後才補。比較穩定的方式是：

```text
.vue template / computed class
  -> prefixCls / class name
  -> src/styles/components/<component>.less
  -> mixins / custom variables
```

例如讀 Button 時，先看 `button.vue` 如何產生 class，再回到 `src/styles/components/button.less` 對照 selector。

### 6.3 什麼時候可以停

能回答以下問題即可：

- 樣式入口如何 import 各層樣式？
- 單一元件 class 如何對應到 Less 檔？
- 變數與 mixin 大致從哪裡來？
- transition name 對應的動畫可能在哪裡找？

深入分析交給 [23_src-styles_樣式系統](../23_src-styles_樣式系統/README.md) 與 [33_高階專題_樣式主題與PrefixCls](../33_高階專題_樣式主題與PrefixCls/README.md)。

## 7. 路線六：型別 / 範例 / 測試反推線

### 7.1 型別線

已確認：[`types/index.d.ts`](../ViewUIPlus-SourceCode/types/index.d.ts) 是型別入口，並匯出 `viewuiplus.components`，同時宣告 install options 與 Vue global properties。

讀單一元件時，型別線用來回答：

- 對外 props 或 options 是否有型別宣告？
- 命令式 API 是否有型別宣告？
- `$VIEWUI`、`$Message`、`$Modal` 等 global properties 是否有宣告？
- 型別與 `.vue` 實作是否一致？

### 7.2 範例線

已確認：[`examples/main.js`](../ViewUIPlus-SourceCode/examples/main.js) 使用 `createApp`、`createRouter`，並從 `../src/index` 載入 View UI Plus。[`examples/app.vue`](../ViewUIPlus-SourceCode/examples/app.vue) 會 import `../src/styles/index.less`，並提供大量 router link。

範例線用來回答：

- 元件在 Demo 中怎麼使用？
- props、events、slots 的常見組合是什麼？
- 指令與命令式 API 在 examples 中怎麼觸發？

### 7.3 測試線

已確認：[`test/unit/specs`](../ViewUIPlus-SourceCode/test/unit/specs/) 目前包含部分代表性測試，例如：

- `affix.spec.js`
- `breadcrumb.spec.js`
- `button.spec.js`
- `date-picker.spec.js`
- `date-picker-utils.spec.js`
- `message.spec.js`
- `select.spec.js`
- `table.spec.js`
- `time-spinner.spec.js`

測試線用來回答：

- 哪些行為被測試固定下來？
- 哪些 props / events 是作者明確期待的？
- 哪些邊界條件有被覆蓋？

待驗證：不是每個元件都有測試。若找不到 spec，不代表元件不重要，只代表需要改用 examples、types 與實作本身交叉驗證。

## 8. 路線七：高階設計線

高階設計線不是一開始就讀，而是在你已經讀過代表性元件後，用來抽象出可復用的設計模式。

| 高階主題 | 先讀代表 | 要抽出的設計能力 |
|---|---|---|
| 元件 API 設計 | `button`、`input`、`select`、`modal` | props、events、slots、methods 如何分層 |
| 受控與非受控狀態 | `input`、`select`、`tabs`、`collapse` | `modelValue` / 內部狀態 / watch 如何協作 |
| 彈層與定位系統 | `tooltip`、`poptip`、`select`、`date-picker`、`base/popper.js` | trigger、placement、transfer、popper |
| 命令式 API | `message`、`notice`、`modal`、`loading-bar` | 全域方法、instance、銷毀與配置 |
| 表單架構 | `form`、`form-item`、`input`、`select` | 驗證、上下文、欄位狀態 |
| 樣式主題與 prefixCls | 多數元件 + `src/styles` | class 命名、Less 變數、主題替換 |
| mixin 與重構 | `src/mixins` + 代表元件 | Options API 共用邏輯如何抽出 |
| 測試反推規格 | `button.spec.js`、`select.spec.js`、`table.spec.js` | 如何用測試描述元件行為 |

合理推論：高階章節最適合在「至少讀過 3 個簡單元件、2 個複雜元件、1 個命令式 API」之後再寫，否則容易變成抽象概念，缺少源碼證據。

## 9. 第一輪建議進度

如果要安排一輪完整學習，可以用下面的順序：

| 階段 | 建議閱讀 | 完成標準 |
|---|---|---|
| 第 0 階段 | `00-01`、`00-02`、`00-03` | 知道地圖、路線與方法 |
| 第 1 階段 | `02`、`03`、`04`、`05` | 能說明工程設定、build 與 dist 產物 |
| 第 2 階段 | `06`、`07`、`08`、`09` | 能說明 install、元件註冊、全域配置與整體模式 |
| 第 3 階段 | `10`、`11`、`12` | 能讀懂基礎元件與布局元件 |
| 第 4 階段 | `13`、`14`、`15` | 能讀懂表單、輸入、資料展示與資料操作 |
| 第 5 階段 | `16`、`17`、`18` | 能讀懂導航、彈層、命令式 API、業務型元件 |

## 10. 遇到問題時怎麼選路線

讀碼時可以用問題反推路線：

| 你想問的問題 | 優先走哪條線 |
|---|---|
| 這個套件怎麼打包？ | 工程建置線 |
| 為什麼 `app.use` 後元件可用？ | 入口安裝線 + 元件註冊線 |
| 某個元件有沒有被對外匯出？ | 元件註冊線 |
| 某個 prop / event 實際怎麼運作？ | 元件分析線 |
| 為什麼某個 class 或動畫生效？ | 樣式系統線 |
| `$Message`、`$Modal` 怎麼呼叫？ | 入口安裝線 + 命令式 API 高階線 |
| 型別和實作不一致怎麼辦？ | 型別線 + 元件分析線 |
| Demo 裡某個用法從哪裡來？ | 範例線 + 元件分析線 |
| 測試覆蓋了什麼行為？ | 測試反推線 |
| 想仿寫一個元件庫能力 | 高階設計線 |

## 11. 初期不要做的事

以下做法容易讓讀碼失焦：

- 不要一開始逐行讀 `dist/viewuiplus.min.js`。
- 不要按 `src/components/` 字母順序讀完所有元件。
- 不要只看 `.vue`，卻不查 `index.js`、mixins、utils、styles、types。
- 不要一看到大型元件就追完所有分支；先抓入口、主狀態、主事件。
- 不要把 examples 當成唯一規格；它只是使用方式證據之一。
- 不要把 types 當成唯一真相；型別需要和實作、測試互相驗證。
- 不要在總覽章塞入所有細節；總覽章負責路線，細節交給後續章節。
