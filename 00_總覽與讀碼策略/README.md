# 00_總覽與讀碼策略 README

> 本目錄是整套 View UI Plus 讀碼筆記的「導航台」。  
> 它不負責記錄單一元件細節，而是用來建立：學習目標、讀碼路線、筆記規範、問題清單、模板、檢核表與 AI 輔助流程。

---

## 1. 目錄定位

`00_總覽與讀碼策略` 是整套 View UI Plus 筆記系統的起點。

它的核心任務不是分析某一個元件，而是先回答下面幾個問題：

1. 我為什麼要讀 View UI Plus？
2. 我讀 View UI Plus 想獲得什麼能力？
3. 我應該用什麼順序讀？
4. 我讀單一元件時要看哪些面向？
5. 我讀橫向機制時要追哪些路徑？
6. 我如何把讀碼轉成自己的筆記、仿寫與企業後台封裝能力？
7. 我如何避免 AI 輔助讀碼時產生錯誤理解？
8. 我如何檢查自己是否真的讀懂？

換句話說，本目錄是後續 `01` 到 `39` 所有筆記的「方法論總綱」。

---

## 2. 本目錄不做什麼

本目錄不應該放過多細節源碼分析。

例如，以下內容不適合放在本目錄：

- `Button` 元件完整源碼分析
- `Input` 元件 props 細節
- `FormItem` 驗證流程完整拆解
- `Message` 命令式 API 具體實作細節
- `Table` 欄位渲染與資料處理流程
- `src/index.js` 每一行完整註解
- `build` 腳本逐行分析
- `Less` 變數逐項整理

這些內容應放在後續對應章節中。

本目錄只負責提供「怎麼讀、讀什麼、如何記、如何驗收」的框架。

- 定義學習範圍
- 定義不優先研究的內容

- repo 主要目錄概念
- 技術棧分層
- 與後續章節的關係

這篇不是細讀 `package.json`，而是先從宏觀角度理解它是一個什麼類型的開源專案。

```txt

```

使用 AI 時，必須提供：

1. 元件名稱或機制名稱
2. 對應源碼片段
3. 目前想解決的問題
4. 希望輸出的格式

```txt

並最後整理成我的單一元件讀碼模板格式。
```

## 8. 9.5 分筆記標準

本目錄所有筆記應符合以下標準。

### 8.1 不是摘要，而是方法

9.5 分筆記不能只是「這個檔案在說什麼」，而要能指導後續行動。

例如，不只寫：

```txt
FormItem 負責表單欄位驗證。
```

而要寫：

```txt
讀 FormItem 時應觀察：
1. 它如何取得上層 Form 的 rules
2. 它如何根據 prop 找到對應規則
3. 它如何觸發 validate
4. 它如何保存 validateState 與 validateMessage
5. 它如何讓 Input / Select 等子元件觸發驗證
```

---

### 8.2 要能回查

筆記應該能回答未來問題，例如：

```txt
我忘記 Message 和 Modal 的命令式 API 差在哪裡？
我應該看哪一篇？
```

答案應該可以透過本目錄定位到：

```txt
00-10_常見設計模式索引.md
00-14_易混淆概念對照表.md
31_高階專題_命令式API設計
17_回饋_彈層_命令式元件
```

---

### 8.3 要能支援仿寫

讀完一篇筆記後，應該能知道下一步能仿寫什麼。

例如：

```txt
讀完 Button：仿寫 MiniButton
讀完 Input：仿寫 MiniInput
讀完 Form/FormItem：仿寫 MiniForm
讀完 Message：仿寫 MiniMessage
讀完 Modal：仿寫 MiniModal
讀完 Table：仿寫 MiniTable
```

---

### 8.4 要能支援企業後台封裝

每篇筆記最後都應該回答：

```txt
這個設計可以如何用在企業後台？
```

例如：

```txt
Form / FormItem → SearchForm、DialogForm
Table → CrudTable
Select → DictSelect、RemoteSelect
Button → PermissionButton
Modal → DialogForm、ConfirmDialog
Message → request error handler、global feedback service
```

## 9. 本目錄和 AI 的協作原則

使用 AI 時應遵守下面原則。

### 9.1 AI 不是源碼真相

AI 的分析只能作為輔助，最後仍要回到源碼驗證。

必須養成下面流程：

```txt
AI 分析
    ↓
回到源碼確認
    ↓
修正筆記
    ↓
補充自己的理解
    ↓
產出仿寫任務
```

---

### 9.2 不讓 AI 直接跳到結論

不建議直接問：

```txt
View UI Plus 的 Form 是怎麼設計的？
```

應該拆成：

```txt
1. Form 的對外 API 是什麼？
2. FormItem 如何註冊到 Form？
3. rules 如何被讀取？
4. validate 如何被觸發？
5. validateState 如何影響畫面？
6. Input / Select 如何和 FormItem 互動？
```

---

### 9.3 AI 輸出必須轉成自己的筆記

AI 給出的分析不能直接貼進筆記。

應再整理成固定格式：

```txt
1. 元件定位
2. 使用方式
3. 源碼入口
4. 核心流程
5. 設計亮點
6. 易混淆點
7. 可仿寫點
8. 企業後台應用
9. 待確認問題
```

## 11. 推薦的核心元件優先順序

若只挑最有學習價值的元件，建議先讀：

```txt
1. Button
2. Icon
3. Input
4. Form
5. FormItem
6. Select
7. Modal
8. Message
9. Table
10. DatePicker
```

理由：

- `Button`：適合理解基礎元件 API 設計
- `Icon`：適合理解資源與 class 設計
- `Input`：適合理解 v-model、事件、表單互動
- `Form`：適合理解父層協調者角色
- `FormItem`：適合理解驗證與欄位狀態
- `Select`：適合理解複合元件與下拉互動
- `Modal`：適合理解彈層與受控狀態
- `Message`：適合理解命令式 API
- `Table`：適合理解資料展示與複雜渲染
- `DatePicker`：適合理解複雜狀態與時間資料處理

## 14. 常見錯誤

### 錯誤一：一開始就讀最難的元件

不建議一開始就讀：

```txt
Table
DatePicker
Upload
Tree
Cascader
```

應先從：

```txt
Button
Icon
Input
Form
FormItem
Message
Modal
```

建立基礎模式感。

---

### 錯誤二：只看 template，不看 API 設計

元件庫的核心不只是畫面，而是 API 設計。

讀每個元件時都要看：

```txt
props
emits
slots
v-model
methods
expose
types
```

---

### 錯誤三：只看單一元件，不看橫向機制

很多設計不是藏在單一元件，而是分散在：

```txt
utils
mixins
directives
locale
styles
types
install
config
```

所以要搭配 `00-07_橫向機制讀碼模板.md`。

---

### 錯誤四：只讀不仿寫

只讀源碼容易產生「我好像懂了」的錯覺。

每讀完一個核心元件，至少要做一個最小仿寫版本。

---

### 錯誤五：把 AI 分析直接當答案

AI 很適合輔助整理，但不一定知道你手上版本的實際源碼。

正確流程是：

```txt
提供源碼 → 讓 AI 分析 → 回源碼驗證 → 修正筆記 → 產出自己的理解
```
