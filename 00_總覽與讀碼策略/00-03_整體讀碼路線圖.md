# 00-03_整體讀碼路線圖

> 筆記定位：本篇是整套 View UI Plus 讀碼筆記的「導航圖」。  
> 目標不是直接解釋某個元件，而是幫自己建立一條從「跑起專案」到「理解元件庫設計」，最後能「仿寫與封裝企業後台元件」的完整路線。

---

## 0. 本篇筆記的作用

在閱讀 View UI Plus 這類元件庫時，最常見的問題不是「完全看不懂程式碼」，而是：

```txt
1. 檔案很多，不知道從哪裡開始看。
2. 一開始就鑽進某個元件，卻看不到整體架構。
3. 看懂了局部程式碼，但不知道它在整個元件庫裡扮演什麼角色。
4. 只記錄程式碼細節，沒有形成可複用的設計方法。
5. 看完原始碼後，仍然不會仿寫自己的元件庫或企業後台封裝。
```

所以本篇要解決的核心問題是：

```txt
我應該按照什麼順序閱讀 View UI Plus？
每個階段要看哪些目錄與機制？
每個階段的完成標準是什麼？
讀完後應該產出什麼筆記或練習成果？
```

這篇不是細節筆記，而是「讀碼地圖」。後續所有章節都應該能回到這篇找到自己的位置。

---

## 1. 整體讀碼總原則

View UI Plus 是基於 Vue 3 的企業級 UI 元件庫與前端解決方案。它不是單一頁面專案，而是一套可安裝、可註冊、可打包、可被業務專案引用的元件庫。

因此，閱讀它不能只用「看 Vue 單檔元件」的方式，而應該使用「元件庫工程」的角度。

---

### 1.1 不要一開始就從 `src/components` 亂讀

這是最容易犯的錯。

如果一開始就打開 `Button.vue`、`Input.vue`、`Table.vue`，可能會遇到這些問題：

```txt
1. 不知道這個元件是如何被匯出的。
2. 不知道它是全域註冊還是局部引入。
3. 不知道 prefixCls 從哪裡來。
4. 不知道 mixin、utils、locale、types 的關係。
5. 不知道樣式檔與元件檔怎麼對應。
6. 不知道 examples 和 test 如何輔助理解元件。
```

所以正確順序應該是：

```txt
先看工程入口，再看元件；
先看註冊機制，再看單一元件；
先看共用規則，再看特殊實作；
先建立地圖，再深入細節。
```

---

### 1.2 讀碼要分成「縱向」與「橫向」

閱讀元件庫不能只看單一元件。要同時做兩種閱讀：

```txt
縱向閱讀：從一個元件的使用方式一路追到源碼、樣式、事件、狀態、型別。
橫向閱讀：從一個共用機制出發，看它如何影響多個元件。
```

例如：

```txt
縱向閱讀範例：
Button → props → class → event → loading → icon → style → export → install

橫向閱讀範例：
prefixCls → config → globalProperties → mixin → class 命名 → styles
```

後續的筆記目錄剛好可以對應這兩種讀法：

```txt
10～18：主要是縱向閱讀元件。
19～24：主要是橫向閱讀共用機制。
28～35：主要是抽象出高階設計模式。
36～38：主要是轉化成自己的仿寫與企業實戰。
```

---

### 1.3 每一階段都要有「產出物」

只讀原始碼很容易產生錯覺：好像理解了，但其實只是看過。

因此每一階段都要有具體產出。

```txt
讀工程結構 → 產出專案目錄地圖
讀 package.json → 產出腳本與依賴說明
讀入口檔 → 產出 app.use 流程圖
讀元件 → 產出元件 API 反推表
讀表單系統 → 產出 Form / FormItem 協作流程圖
讀命令式元件 → 產出 Message / Modal 建立流程圖
讀樣式系統 → 產出 prefixCls 與 class 命名規則
讀測試 → 產出元件規格反推清單
讀完一輪 → 產出 Mini View UI Plus 仿寫專案
```

這樣讀碼才會真正轉化為自己的能力。

## 18. 橫向機制的標準讀碼流程

每讀一套機制，建議照下面流程。

```txt
Step 1：先找使用場景
Step 2：找入口檔
Step 3：找核心資料結構
Step 4：找哪些元件依賴它
Step 5：追蹤資料流
Step 6：追蹤事件流
Step 7：追蹤樣式或 DOM 影響
Step 8：整理優點與限制
Step 9：思考 Composition API 重構可能
Step 10：仿寫最小機制
```

例如讀表單驗證：

```txt
使用場景：<Form :model :rules>
入口：Form / FormItem
資料結構：model / rules / fields
依賴元件：Input / Select / Radio / Checkbox
資料流：model value → prop → validate
事件流：blur / change → FormItem validate
輸出：validateState / validateMessage
仿寫：MiniForm
```

## 20. 容易踩坑的地方

### 20.1 把「會用元件」誤認為「懂元件設計」

會寫：

```vue
<Button type="primary">提交</Button>
```

不代表懂 Button 的設計。

真正要懂的是：

```txt
1. type 如何對應 class
2. loading 如何影響 disabled 與 click
3. icon 與 slot 如何組合
4. size 如何與全域配置互動
5. styles 如何維護一致性
```

---

### 20.2 太早讀 Table

Table 很重要，但不適合一開始精讀。

原因：

```txt
1. 狀態多
2. DOM 結構複雜
3. columns API 複雜
4. 涉及排序、篩選、選取、固定欄位
5. 可能牽涉寬度計算與滾動同步
```

建議先讀：

```txt
Button → Input → Form → Modal → Message → Tooltip → Table
```

---

### 20.3 只看 Vue 檔，不看 styles

元件庫的元件設計有一半體現在樣式系統。

如果只看 `.vue` 或 `.js`，會漏掉：

```txt
1. 狀態 class
2. disabled 樣式
3. active 樣式
4. hover / focus 樣式
5. size 樣式
6. error 樣式
7. prefixCls 命名規則
8. 主題變數
```

---

### 20.4 只看源碼，不看 examples / test

源碼告訴你「如何實作」，examples 和 test 告訴你「它被期待如何使用」。

好的讀法是：

```txt
先看 examples 知道怎麼用；
再看源碼知道怎麼做；
再看 test 知道哪些行為必須保證。
```

---

### 20.5 盲目照抄 mixin

如果看到 mixin，不要馬上認為這是現代 Vue 3 最佳實踐。

要判斷：

```txt
1. 這是不是從 Vue 2 時代延續下來的寫法？
2. 它是否造成隱式依賴？
3. 能不能改寫成 composable？
4. 改寫後是否更容易測試與維護？
```

## 21. 對我的學習策略總結

以我的目標來說，讀 View UI Plus 不是為了變成 View UI Plus 維護者，而是為了提升三種能力：

```txt
1. Vue 3 元件設計能力
2. 前端元件庫工程能力
3. 企業後台封裝能力
```

因此最合理的主線是：

```txt
先跑起專案
  ↓
理解 install / 註冊 / 全域配置
  ↓
精讀 Button / Input / Form / Modal / Message / Table
  ↓
橫向整理 styles / utils / directives / types / locale
  ↓
抽象成 API 設計、狀態設計、表單架構、命令式 API、彈層定位
  ↓
仿寫 MiniUI
  ↓
封裝企業後台常用元件
```

如果用一句話總結這份路線圖：

```txt
不要把 View UI Plus 當成一本 API 字典來讀，
而要把它當成一套 Vue 3 企業級元件庫工程範本來拆解、反推、仿寫與內化。
```
