# 00-05_元件庫核心問題清單

> 版本：9.5 分版本  
> 適用專案：View UI Plus  
> 適用階段：閱讀 `src/components`、`src/directives`、`src/utils`、`src/mixins`、`src/locale`、`src/styles`、`types`、`examples`、`test` 之前與之中  
> 本篇定位：這不是單純的「問題列表」，而是一套用來閱讀 View UI Plus 的「提問系統」。它的目標是讓你每讀一個元件、每看一個機制，都能從「看懂程式碼」進階到「看懂設計決策」，最後轉化成自己的企業後台元件封裝能力。

---

## 0. 這篇筆記要解決什麼問題？

讀 View UI Plus 這類元件庫時，最常見的問題不是「看不懂某一行程式碼」，而是：

```txt
我看了很多檔案，卻不知道它們為什麼要這樣設計。
```

你可能會遇到以下情況：

1. 看懂了 props，但不知道為什麼 API 要這樣命名。
2. 看懂了 template，但不知道元件結構背後的抽象邏輯。
3. 看懂了 watch / computed，但不知道它們在解決什麼狀態同步問題。
4. 看懂了 emit，但不知道事件設計是否合理。
5. 看懂了 class，但不知道 prefixCls、主題、樣式隔離的設計價值。
6. 看懂了工具函式，但不知道它在整個元件庫中扮演什麼角色。
7. 看懂了 Form / Modal / Table 的局部程式碼，但沒有整理成自己未來可復用的封裝模式。
8. 依賴 AI 解釋每段源碼，但自己沒有一套固定的讀碼判斷框架。

所以本篇的核心目標是建立一組固定問題：

```txt
每次讀元件，都用同一批問題檢查。
每次讀機制，都用同一批問題追主線。
每次整理筆記，都把答案轉化成自己的設計能力。
```

簡單說：

```txt
不要只問：「這段程式碼在做什麼？」
而要問：「這段程式碼為什麼存在？它解決了什麼元件庫設計問題？」
```

## 1. 本篇的使用方式

這份問題清單不是要求你每次全部回答完，而是依照閱讀階段使用。

### 1.1 讀碼前：用來建立閱讀目標

讀一個元件前，先問：

```txt
1. 這個元件在元件庫中屬於哪一類？
2. 這個元件最核心的使用場景是什麼？
3. 我這次要重點學 API 設計、狀態管理、互動流程，還是樣式系統？
4. 我讀完後要產出什麼？筆記？流程圖？仿寫版本？企業封裝版本？
```

如果沒有先定義目標，讀碼很容易變成散亂翻檔案。

---

### 1.2 讀碼中：用來追蹤主線

讀碼過程中，優先使用這五個問題：

```txt
1. 使用者傳入了什麼？
2. 元件內部維護了什麼？
3. 畫面根據什麼渲染？
4. 使用者操作後觸發什麼？
5. 元件如何通知外部或更新自身？
```

這五個問題可以對應到 Vue 元件的主線：

```txt
Props / Slots / v-model
↓
Internal State
↓
Computed / Watch
↓
Template / Render
↓
Events / Emits / Methods
```

---

### 1.3 讀碼後：用來檢查是否真的讀懂

讀完後，不要只說「我懂了」。你應該能回答：

```txt
1. 這個元件的 API 設計重點是什麼？
2. 它的狀態同步難點在哪裡？
3. 它有沒有父子元件協作？
4. 它有沒有共用工具、mixin、directive 或全域配置？
5. 它有沒有可以抽象成企業後台封裝的地方？
6. 如果我要仿寫最小版，需要保留哪些核心能力？
7. 如果我要重構它，可以優先改善哪裡？
```

## 2. 元件庫讀碼的總問題地圖

讀 View UI Plus 不能只用「單一元件」視角，還要同時具備「元件庫整體設計」視角。

建議把問題分成 12 大類：

```txt
1. 專案定位問題
2. 元件分類問題
3. API 設計問題
4. 狀態管理問題
5. 渲染結構問題
6. 互動事件問題
7. 父子協作問題
8. 橫向機制問題
9. 樣式主題問題
10. 型別設計問題
11. 測試與規格問題
12. 企業後台轉化問題
```

這 12 類問題會貫穿你後面的所有章節：

```txt
10_src-components_元件總覽
11_基礎通用元件
13_表單與輸入元件
14_表單驗證系統專題
15_資料展示與資料操作元件
17_回饋_彈層_命令式元件
20_src-utils_工具函式
21_src-mixins_混入與共用邏輯
23_src-styles_樣式系統
24_types_型別宣告
28~35_高階專題
36~38_仿寫與企業封裝
```

## 3. 第一層問題：專案定位與學習目標

這一層問題用來確認你不是在「盲目看源碼」。

### 3.1 專案定位問題

閱讀任何源碼之前，先問：

```txt
1. View UI Plus 的主要使用場景是什麼？
2. 它偏向通用元件庫，還是偏企業中後台元件庫？
3. 它和單純的業務專案有什麼不同？
4. 它的核心價值是 UI 元件、工程解決方案，還是企業後台開發體驗？
5. 它的使用者是一般前端開發者、企業後台開發者，還是元件庫維護者？
6. 這個專案中哪些地方是「元件庫通用能力」？
7. 哪些地方是「View Design 自己的設計取向」？
8. 哪些地方可以直接學，哪些地方只適合參考？
```

### 3.2 學習目標問題

你讀它的目的不是成為 View UI Plus 維護者，而是提升自己的前端能力。

所以要問：

```txt
1. 我讀這個元件，是為了學使用方式，還是學設計方式？
2. 我讀這個機制，是為了理解源碼，還是為了以後自己封裝？
3. 這個知識能不能轉化成企業後台能力？
4. 這個知識能不能幫我寫出更好的 Vue 3 元件？
5. 這個知識能不能幫我設計更穩定的表單、彈層、Table、SearchForm、DialogForm？
```

### 3.3 讀碼邊界問題

不是所有源碼都值得同等深度閱讀。

你要問：

```txt
1. 這個檔案是核心主線，還是邊角輔助？
2. 這個元件在企業後台常用嗎？
3. 這個邏輯是元件庫通用邏輯，還是歷史包袱？
4. 這個設計值得仿寫，還是只要知道即可？
5. 我目前階段是否需要深入這裡？
```

判斷標準：

```txt
高優先：Button、Input、Form、FormItem、Select、Table、Modal、Message、Tooltip、Upload
中優先：Menu、Tabs、Dropdown、DatePicker、Tree、Drawer、Notice、Page
低優先：少用元件、特殊功能元件、暫時不影響主線的邊緣邏輯
```

## 4. 第二層問題：元件分類與角色定位

讀元件之前，一定要先分類。不同類型的元件，閱讀重點完全不同。

### 4.1 元件分類問題

```txt
1. 這是基礎展示元件嗎？例如 Button、Icon、Typography。
2. 這是表單輸入元件嗎？例如 Input、Select、DatePicker、Checkbox、Radio。
3. 這是表單架構元件嗎？例如 Form、FormItem。
4. 這是資料展示元件嗎？例如 Table、List、Tree、Card。
5. 這是導航元件嗎？例如 Menu、Tabs、Breadcrumb、Dropdown。
6. 這是彈層元件嗎？例如 Modal、Drawer、Tooltip、Poptip、Popover。
7. 這是命令式元件嗎？例如 Message、Notice、Modal.confirm。
8. 這是布局元件嗎？例如 Row、Col、Layout、Space。
9. 這是增強型元件嗎？例如 Upload、Transfer、Cascader、AutoComplete。
10. 這是工具型能力嗎？例如 directive、utils、locale、style、types。
```

### 4.2 元件角色問題

```txt
1. 這個元件是被使用者直接使用，還是被其他元件內部使用？
2. 它是葉子元件，還是容器元件？
3. 它是單一職責元件，還是複合型元件？
4. 它是否需要跟父層元件協作？
5. 它是否會對子元件提供上下文？
6. 它是否需要操作 DOM？
7. 它是否依賴全域配置？
8. 它是否依賴樣式系統或 prefixCls？
9. 它是否有無障礙、鍵盤操作、focus 管理需求？
10. 它是否有非同步、遠端資料、loading、錯誤狀態？
```

### 4.3 元件複雜度判斷問題

```txt
1. 這個元件有沒有 v-model？
2. 有沒有多個狀態來源？
3. 有沒有巢狀子元件？
4. 有沒有遞迴結構？
5. 有沒有 popper / floating / teleport 類需求？
6. 有沒有表單驗證整合？
7. 有沒有大量 props 組合？
8. 有沒有異步行為？
9. 有沒有複雜 keyboard / mouse interaction？
10. 有沒有性能問題，例如大量資料、虛擬列表、頻繁渲染？
```

如果以上問題命中越多，代表這個元件越值得深挖。

## 5. 第三層問題：API 設計

元件庫的本質是 API 設計。API 設計比內部實作更重要。

### 5.1 Props 設計問題

讀 props 時，不要只記有哪些欄位，要問：

```txt
1. 這個 prop 是控制樣式、狀態，還是行為？
2. 這個 prop 的命名是否直覺？
3. 這個 prop 是布林開關、枚舉值、物件配置，還是函式？
4. 這個 prop 有沒有預設值？預設值代表什麼設計取向？
5. 這個 prop 是單純轉成 class，還是會影響內部邏輯？
6. 這個 prop 是否會和其他 props 互斥？
7. 這個 prop 是否會產生組合爆炸？
8. 這個 prop 是否應該拆成多個小 prop？
9. 這個 prop 是否應該改成 slot 或 render function？
10. 這個 prop 是否適合作為企業後台二次封裝的配置項？
```

可以把 props 分類成：

```txt
1. 外觀型 props：type、size、shape、status、icon、ghost、long
2. 狀態型 props：disabled、loading、checked、selected、visible、active
3. 資料型 props：value、modelValue、data、options、columns、list
4. 行為型 props：clearable、filterable、multiple、transfer、closable
5. 配置型 props：placement、trigger、rules、labelWidth、format
6. 回呼型 props：beforeUpload、filterMethod、render、onChange 類函式
```

### 5.2 Emits 設計問題

```txt
1. 這個事件是使用者操作事件，還是狀態變更事件？
2. 事件名稱是否符合 Vue 習慣？
3. 事件參數是否足夠？
4. 事件參數是否過度暴露內部實作？
5. change、input、update:modelValue、select、clear、open、close 的分工是否清楚？
6. 同一個操作是否會觸發多個事件？觸發順序是什麼？
7. 事件是否會在 disabled / readonly 狀態下被阻止？
8. 事件是否支援表單驗證聯動？
9. 事件是否方便企業後台做追蹤、紀錄、二次封裝？
10. 如果我自己設計，會保留哪些事件？會刪除哪些事件？
```

### 5.3 Slots 設計問題

```txt
1. 這個元件暴露了哪些插槽？
2. slot 是用來改文字、改圖示、改結構，還是改渲染邏輯？
3. slot scope 傳出的資料是否足夠？
4. slot 是否會破壞元件內部一致性？
5. slot 和 prop 的邊界在哪裡？
6. 這個需求應該用 prop 解決，還是 slot 解決？
7. slot 是否適合企業後台客製化？
8. 如果 slot 過多，會不會讓元件難以維護？
9. slot 是否需要搭配 class / style / prefixCls？
10. 如果我要仿寫，哪些 slot 是最小版本必須保留的？
```

### 5.4 v-model 設計問題

```txt
1. 這個元件是否支援 v-model？
2. 綁定的是 value、modelValue，還是其他欄位？
3. 內部狀態和外部值如何同步？
4. props 改變時，內部狀態是否會更新？
5. 內部操作改變時，是否會 emit 給外部？
6. 是否存在受控 / 非受控狀態？
7. 是否支援多個 v-model？
8. 是否有 lazy、debounce、composition event 等輸入法問題？
9. clear、reset、defaultValue 時如何處理？
10. 在 FormItem 驗證中，值變化如何觸發 validate？
```

## 6. 第四層問題：內部狀態與資料流

讀 Vue 元件，不能只看 template。真正的難點通常在狀態同步。

### 6.1 狀態來源問題

```txt
1. 這個元件有哪些狀態？
2. 哪些狀態來自 props？
3. 哪些狀態是元件內部自己維護？
4. 哪些狀態來自父層 provide / inject？
5. 哪些狀態來自全域配置？
6. 哪些狀態來自 DOM？
7. 哪些狀態來自非同步資料？
8. 哪些狀態只是中間衍生值？
9. 哪些狀態應該用 computed 而不是 data？
10. 哪些狀態可能造成同步問題？
```

### 6.2 狀態分類問題

可以把狀態分成：

```txt
1. 顯示狀態：visible、active、expanded、opened、focused
2. 選擇狀態：selected、checked、currentValue、activeKey
3. 輸入狀態：inputValue、query、searchText、isComposing
4. 載入狀態：loading、uploading、pending
5. 驗證狀態：validateState、validateMessage、isRequired
6. 位置狀態：placement、x、y、width、height
7. 快取狀態：cachedOptions、flattenData、normalizedColumns
8. 過渡狀態：animating、mounted、destroyed
```

讀碼時要問：

```txt
這個狀態存在的必要性是什麼？
它是單一真相來源，還是某個值的副本？
```

### 6.3 computed / watch 問題

```txt
1. computed 是在封裝衍生狀態，還是在簡化 template？
2. watch 是在同步 props 到內部，還是在處理副作用？
3. watch 是否可能造成循環更新？
4. watch 是否需要 immediate？為什麼？
5. watch 是否需要 deep？deep 是否有性能風險？
6. computed 是否過度複雜？是否應該拆工具函式？
7. computed 是否依賴太多資料來源？
8. watch 中是否有 DOM 操作？是否應該放到 nextTick？
9. watch 是否和 FormItem 驗證或彈層定位有關？
10. 如果改成 Composition API，可以如何拆分？
```

### 6.4 受控與非受控問題

這是高階元件非常常見的設計重點。

```txt
1. 元件的值由外部完全控制嗎？
2. 元件是否允許自己維護內部狀態？
3. defaultValue 和 value / modelValue 的關係是什麼？
4. 外部改變 props 時，內部如何同步？
5. 內部改變時，是否一定通知外部？
6. visible / open 類狀態是否可控？
7. 表單元件 reset 時，應該回到哪個值？
8. 外部傳入 undefined / null / 空字串時如何處理？
9. 受控模式下，元件是否還能自己改狀態？
10. 這個設計對企業封裝 SearchForm / DialogForm 有什麼啟發？
```

## 7. 第五層問題：渲染結構與模板設計

### 7.1 Template 結構問題

```txt
1. template 最外層元素是什麼？為什麼？
2. class 如何組合？
3. 哪些區塊是固定結構？
4. 哪些區塊是根據 props 條件渲染？
5. 哪些區塊支援 slot 替換？
6. 哪些區塊需要 transition？
7. 哪些區塊需要 teleport / transfer？
8. 哪些區塊需要 v-if，哪些應該用 v-show？
9. 是否有遞迴渲染？
10. 是否有性能風險，例如大量 v-for？
```

### 7.2 Class 組合問題

```txt
1. class 是否以 prefixCls 為基礎？
2. 狀態 class 如何命名？
3. 尺寸 class 如何命名？
4. 類型 class 如何命名？
5. disabled / active / selected / error / focused 如何反映到 class？
6. class 是寫在 template，還是透過 computed 組合？
7. class 組合是否容易維護？
8. 樣式與邏輯是否過度耦合？
9. 企業二次封裝時是否容易覆蓋樣式？
10. 如果要支援主題，class 設計是否足夠穩定？
```

### 7.3 條件渲染問題

```txt
1. v-if 的條件是否來自 props、state，還是 computed？
2. v-if 是否會影響元件生命週期？
3. v-show 是否更適合保留狀態？
4. 彈層元件是否需要延遲渲染？
5. 表單元件是否需要保留輸入狀態？
6. 切換時是否會重置子元件？
7. 是否需要 destroy-on-close 類能力？
8. 是否有 transition 進出場問題？
9. 是否會造成焦點遺失？
10. 是否會影響表單驗證或 DOM 測量？
```

## 8. 第六層問題：互動事件與使用者行為

元件庫要處理大量使用者互動，這也是你想強化的前端能力之一。

### 8.1 滑鼠事件問題

```txt
1. 元件有哪些 click / mouseenter / mouseleave / mousedown / mouseup 行為？
2. 點擊事件是在根元素處理，還是子元素處理？
3. 是否需要 stopPropagation？
4. 是否需要 preventDefault？
5. disabled 狀態下是否阻止互動？
6. 點擊外部是否要關閉？
7. 多個彈層巢狀時，點擊判斷如何處理？
8. hover 和 click 觸發是否可以切換？
9. 拖曳、選取、框選是否需要特殊處理？
10. 事件處理是否會影響性能？
```

### 8.2 鍵盤事件問題

```txt
1. 元件是否支援 Enter / Esc / Tab / Arrow keys？
2. Select / DatePicker / Menu / Dropdown 是否支援鍵盤導航？
3. Esc 是否關閉彈層？
4. Enter 是否確認選擇？
5. Arrow keys 是否移動 active item？
6. Tab 是否符合自然焦點順序？
7. 快捷鍵是否會和瀏覽器預設行為衝突？
8. 是否需要 preventDefault？
9. 是否需要處理輸入法 composition？
10. 鍵盤支援是否能提升可用性與無障礙？
```

### 8.3 Focus / Blur 問題

```txt
1. 元件是否需要維護 focused 狀態？
2. focus 時 class 是否改變？
```

## 18. 筆記品質評分標準

這份問題清單也可以用來評估你的筆記品質。

### 18.1 低分筆記：5 分以下

特徵：

```txt
1. 只貼源碼。
2. 只翻譯每一行在做什麼。
3. 沒有說明元件定位。
4. 沒有整理 API。
5. 沒有說明設計原因。
6. 沒有企業後台應用思考。
```

範例：

```txt
這裡定義了一個 disabled prop。
這裡 emit change。
這裡 computed class。
```

這種筆記短期看起來有用，但很快會忘。

---

### 18.2 中分筆記：6～8 分

特徵：

```txt
1. 有整理元件用途。
2. 有列出 props / emits / slots。
3. 有說明核心流程。
4. 有簡單分析狀態。
5. 有提到一些設計模式。
```

但缺點是：

```txt
1. 還沒有橫向比較。
2. 還沒有轉化成自己的封裝能力。
3. 還沒有反推出測試與規格。
```

---

### 18.3 高分筆記：9 分以上

特徵：

```txt
1. 能說清楚元件定位。
2. 能分類 API 設計。
3. 能畫出資料流與事件流。
4. 能說明內外狀態同步。
5. 能指出父子協作或橫向機制。
6. 能說明樣式與 prefixCls 設計。
7. 能對照 examples / test 反推規格。
8. 能整理設計模式。
9. 能提出最小仿寫版本。
10. 能轉化成企業後台封裝方向。
```

9.5 分筆記應該做到：

```txt
不只知道 View UI Plus 怎麼寫，
還能知道為什麼這樣寫，
以及自己在企業專案中要如何取用、改寫、封裝。
```

## 19. 常見錯誤提問方式

### 19.1 錯誤一：只問這段程式碼是什麼意思

低品質問題：

```txt
這段程式碼是什麼意思？
```

更好的問題：

```txt
這段程式碼在這個元件的 API、狀態同步、事件流程中扮演什麼角色？
如果拿掉它，會影響哪些使用場景？
```

---

### 19.2 錯誤二：一開始就要求逐行解釋

逐行解釋不是不能用，但不能一開始就用。

更好的順序：

```txt
先請 AI 幫我整理主線流程。
再針對不懂的分支逐行解釋。
最後請 AI 幫我轉成設計模式筆記。
```

---

### 19.3 錯誤三：只看單一元件，不看共用機制

元件庫的價值通常藏在共用機制：

```txt
1. install
2. config
3. prefixCls
4. locale
5. form validation
6. popup / z-index
7. utils
8. mixins
9. directives
10. types
```

所以每次讀元件都要問：

```txt
這個元件背後有沒有共用機制？
```

---

### 19.4 錯誤四：讀完沒有產出

讀完不產出，等於沒有真正學會。

每次讀完至少要產出一種：

```txt
1. API 表
2. 狀態流圖
3. 事件流圖
4. 設計模式整理
5. 最小仿寫版本
6. 企業封裝方案
7. 測試案例清單
```

---

## 20. 最小讀碼閉環

每次讀一個元件，建議完成這個閉環：

```txt
官方 Demo
↓
API 表
↓
源碼入口
↓
資料流
↓
事件流
↓
樣式 class
↓
共用機制
↓
測試 / Demo 反推
↓
設計模式
↓
仿寫 or 企業封裝
```

如果只做到前半段：

```txt
Demo → API → 源碼
```

那只是「理解」。

如果做到後半段：

```txt
設計模式 → 仿寫 → 企業封裝
```

才是真正把開源專案轉化成自己的能力。

## 22. 總結：這份清單真正要訓練什麼？

這份問題清單不是為了讓你變成「問很多問題的人」，而是訓練你具備三種能力。

### 22.1 第一種能力：讀懂元件表層 API

你要能看出：

```txt
這個元件讓使用者怎麼用？
它暴露了哪些能力？
它的 API 是否清楚？
```

---

### 22.2 第二種能力：讀懂內部設計決策

你要能看出：

```txt
為什麼需要這個狀態？
為什麼需要這個 watch？
為什麼這裡用 slot 而不是 prop？
為什麼這裡用 provide / inject？
為什麼這裡抽成 utils / mixin？
```

---

### 22.3 第三種能力：轉化成自己的工程能力

最後你要能做到：

```txt
我不只是知道 View UI Plus 怎麼寫，
我還能把它轉化成：

1. 自己的 Vue 3 元件設計能力
2. 自己的企業後台封裝能力
3. 自己的迷你元件庫架構能力
4. 自己的讀碼與重構判斷能力
```

這才是閱讀 View UI Plus 最有價值的地方。

## 23. 本篇最重要的核心句

```txt
讀元件庫時，不要只問「這段程式碼做了什麼」，
而要問「這段程式碼在元件 API、狀態同步、互動流程、共用機制、企業封裝中扮演什麼角色」。
```

如果你之後每讀一個元件都能照這個方向思考，你就不只是使用 View UI Plus，而是在用 View UI Plus 訓練自己的元件庫設計能力。
