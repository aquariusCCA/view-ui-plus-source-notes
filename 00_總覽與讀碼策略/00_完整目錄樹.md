# 00_完整目錄樹

> 這份文件不是單純列出檔案名稱，而是整套 **View UI Plus 源碼筆記系統的導航地圖**。  
> 目標不是「把所有資料夾都建出來」，而是讓每一篇筆記都能對應到真實源碼、真實問題、真實可學走的設計。

---

## 1. 這份目錄樹的定位

這份目錄樹負責回答四件事：

1. **我要讀哪一層源碼？**  
   例如入口、元件、指令、工具函式、樣式、型別、範例、打包流程。

2. **每一層源碼應該產出什麼筆記？**  
   不是只寫「這是 Button」，而是要寫清楚：原始碼位置、核心流程、依賴關係、可學走的設計。

3. **哪些內容要先讀，哪些內容可以晚點讀？**  
   元件庫讀碼不能從頭到尾硬啃，應該先讀入口與代表性元件，再讀複雜元件與工程配套。

4. **之後遇到問題時，要怎麼回查？**  
   例如遇到全域 API、表單驗證、彈層掛載、Table 狀態管理時，可以快速找到對應筆記。

## 2. 目錄設計原則

### 原則一：源碼對齊，而不是憑空分類

每一篇正式筆記都必須標明：

```md
## 原始碼位置
- src/index.js
- src/components/button/button.vue
- src/styles/components/button.less
```

如果某篇筆記無法對應到至少一個真實檔案，就只能放在「總結 / 專題 / 仿寫」區，不能假裝是源碼分析。

---

### 原則二：先讀架構，再讀元件，再讀專題

View UI Plus 這類 UI 元件庫不能直接從某個元件開始亂讀，建議順序是：

```txt
專案定位
  ↓
入口 install 機制
  ↓
components/index.js 匯出清單
  ↓
代表性簡單元件 Button / Alert / Tag
  ↓
表單元件 Input / Form / Select
  ↓
命令式元件 Message / Modal / Notice
  ↓
複雜元件 Table / Tree / Upload
  ↓
樣式系統 / mixins / utils / directives / types
  ↓
仿寫與重構
```

---

### 原則三：分類是為了學習，真實路徑仍以源碼為準

筆記中會把元件分成「基礎通用、布局、表單、資料展示、導航、回饋、業務型」等分類。  
但 View UI Plus 原始碼中的元件大多都集中在：

```txt
src/components/*
```

所以每一個元件筆記都應該同時保留：

```md
## 學習分類
表單與輸入元件

## 真實源碼路徑
src/components/input/input.vue
src/components/input/index.js
src/styles/components/input.less
```

---

### 原則四：不要產生大量空殼筆記

如果目前還沒有讀過某個元件，不要只建立一個空的 README。  
更好的做法是：

```txt
先建立：
- 00_總覽與讀碼策略
- 03_src_入口與插件機制
- 04_src-components_元件總覽
- Button / Input / Form / Message / Modal / Table 等代表性元件

等真的讀到其他元件時，再補對應筆記。
```

---

## 3. 閱讀優先級標記

| 標記 | 意義 | 建議處理方式 |
|---|---|---|
| `P0` | 第一輪必讀 | 要寫成完整筆記，可以回查與仿寫 |
| `P1` | 第二輪核心 | 讀完代表性元件後再補 |
| `P2` | 有需求再讀 | 使用到或遇到問題時再深入 |
| `P3` | 不主讀 | 只理解用途，不需要逐行分析 |

## 5. 每個元件筆記的最低完成標準

不要只寫「這個元件用來顯示按鈕」。一篇合格的元件源碼筆記至少要有以下內容：

```md
# Button 元件源碼分析

## 1. 元件定位
Button 解決什麼 UI 問題？在元件庫中屬於哪一類？

## 2. 原始碼位置
- src/components/button/index.js
- src/components/button/button.vue
- src/styles/components/button.less

## 3. 對外 API
### Props
### Emits
### Slots
### Methods / Expose

## 4. 內部狀態
哪些資料是由 props 來？哪些是 computed？哪些是內部 state？

## 5. 核心流程
從使用者傳入 props 到最後渲染 DOM，中間經過哪些判斷？

## 6. 與其他模組的關係
有沒有用到 mixins、utils、directives、locale、globalConfig？

## 7. 樣式對照
DOM class 如何對應到 Less 檔案？

## 8. 可學走的設計
這個元件有哪些設計可以帶回自己的專案？

## 9. 我還沒看懂的問題
保留問題，之後回查。
```

```md
## 1. 涉及檔案
```

## 8. 最終目標

這套筆記最後不只是「View UI Plus 讀碼筆記」，而是要沉澱出以下能力：

