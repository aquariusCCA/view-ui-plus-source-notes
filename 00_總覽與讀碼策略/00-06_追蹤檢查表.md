## 本節導讀

這篇是 View UI Plus 源碼閱讀時的「追蹤檢查表」。前面幾節回答了「在哪裡讀、按什麼路線讀、用什麼方法讀、概念在哪裡查」；這一節回答的是：

> 讀到什麼程度，才算這段源碼真的追完了？

本篇不負責解釋每個元件的細節，而是提供一組可重複使用的檢查表，幫助你在閱讀時判斷：

- 是否已確認入口，而不是只看到檔名。
- 是否已區分定義位置、註冊位置與生效位置。
- 是否已追到資料流、狀態流、事件流與副作用。
- 是否已對照樣式、型別、範例與測試。
- 是否知道哪些結論是已確認，哪些仍然只是合理推論或待驗證。

如果 `00-03_讀碼方法.md` 是操作順序，`00-04_元件分析模板.md` 是筆記格式，那本篇就是每次讀完後的驗收標準。

## 建議回查情境

當你遇到以下情境，可以回到這篇：

- 讀完一段源碼後，不確定自己是否真的理解。
- 想檢查一篇元件分析筆記是否達到可回查標準。
- 想確認某個結論是否有足夠證據標成「已確認」。
- 追大型元件時分支越來越多，需要決定哪些支線暫緩。
- 寫後續章節時，需要用統一標準判斷每個模組是否已追到閉環。

## 證據標記

本篇延續 00 章的證據標記：

| 標記 | 意義 |
|---|---|
| 已確認 | 已從目前倉庫中的原始碼、設定檔、型別、範例或測試直接讀到 |
| 合理推論 | 由已確認事實推導出的閱讀策略或設計理解 |
| 待驗證 | 後續深入單一元件、執行範例或測試後才能確認 |
| 證據衝突 | 原始碼、型別、範例、測試或產物之間出現不一致 |

## 0. 五分鐘快速檢查

如果只是快速判斷「這段是否能先收工」，先看這張表。

| 檢查項 | 合格標準 | 未完成時怎麼辦 |
|---|---|---|
| 問題是否明確 | 能用一句話寫出本輪要回答什麼 | 回到 `00-03` 第 1 節縮小問題 |
| 入口是否確認 | 知道第一個入口檔與它的角色 | 回到 `00-05` 快速查找表找入口 |
| 邊界是否明確 | 知道本輪只追哪條線，哪些不追 | 把支線寫入待驗證或暫緩 |
| 定義 / 註冊 / 生效是否分清 | 至少能指出目前已確認到哪一層 | 補查 `src/index.js`、`components/index.js`、examples 或 tests |
| 主流程是否能畫出 | 能用 5 到 8 行 text flow 表示流程 | 回頭追狀態、事件、渲染或 service instance |
| 外部證據是否至少有一種 | 已對照 types、examples、tests、build output 中至少一種 | 補查對應外部證據 |
| 證據分級是否標好 | 已確認、合理推論、待驗證、證據衝突不混用 | 重寫結論，不把推論寫成事實 |
| 下一步是否明確 | 知道接下來讀哪個檔或暫停理由 | 補上下一步問題 |

快速判斷：

| 結果 | 意義 |
|---|---|
| 8 項都完成 | 本輪可收工，適合整理成筆記 |
| 6 到 7 項完成 | 可暫停，但要留下待驗證問題 |
| 少於 6 項完成 | 目前還只是瀏覽，不算追完 |

## 1. 使用方式

本篇有三種用法：

| 用法 | 適用時機 | 使用章節 |
|---|---|---|
| 快速驗收 | 讀完一小段源碼，想知道能不能停 | 第 0 節、第 3 節 |
| 正式追蹤 | 要寫單一元件或模組筆記 | 第 2 節到第 14 節 |
| 品質評分 | 想判斷一篇筆記是否滿分 | 第 15 節到第 17 節 |

建議流程：

```text
先填任務卡
  -> 選對任務類型
  -> 做通用檢查
  -> 補專用檢查
  -> 標記證據分級
  -> 決定停止 / 深入 / 暫緩
```

## 2. 讀碼任務卡

每次追源碼前，先填這張卡。這張卡的目的是防止一開始就失焦。

```md
## 讀碼任務卡

- 本輪問題：
- 閱讀類型：入口 / 元件 / 命令式 API / 樣式 / 表單 / directive / locale / types / examples / tests / build
- 第一入口：
- 本輪邊界：
- 不追的支線：
- 期望產出：
- 停止條件：
```

填寫範例：

```md
## 讀碼任務卡

- 本輪問題：Button 的 `to` 如何影響渲染與跳轉？
- 閱讀類型：元件 + mixin
- 第一入口：`src/components/button/index.js`
- 本輪邊界：只追 Button 的 link 行為，不追 ButtonGroup
- 不追的支線：router 內部實作、完整 Less 細節
- 期望產出：Button link 行為流程圖
- 停止條件：能說明 `to -> tagName/tagProps -> handleCheckClick` 的流程
```

## 3. 通用追蹤檢查表

所有閱讀任務都先做這一組檢查。

### 3.1 問題與邊界

| 檢查項 | 完成標準 |
|---|---|
| 已把問題縮成一句可追問題 | 不是「讀 Button」，而是「Button 如何決定渲染成 `<button>` 或 `<a>`」 |
| 已選定閱讀類型 | 入口、元件、命令式 API、樣式、表單、build 等 |
| 已確認第一入口 | 入口有明確檔案路徑 |
| 已設定閱讀邊界 | 知道本輪追到哪裡停 |
| 已列出暫緩支線 | 大型元件分支、外部庫、非主線樣式等已記錄 |

### 3.2 證據分級

| 檢查項 | 完成標準 |
|---|---|
| 已確認的結論都有源碼位置 | 每個重要結論能指到檔案 |
| 合理推論有推論依據 | 寫出是由哪些已確認事實推導 |
| 待驗證問題有缺口說明 | 寫清楚缺的是 source、example、test，還是 runtime 驗證 |
| 證據衝突有衝突來源 | 寫清楚是哪兩個來源不一致 |
| 沒有把命名直覺寫成已確認 | 例如看到 `index.js` 不直接說它是實作入口 |

### 3.3 定義 / 註冊 / 生效

| 位置 | 必查問題 | View UI Plus 常見證據 |
|---|---|---|
| 定義位置 | 能力在哪裡被寫出來？ | `.vue`、service `index.js`、mixin、directive |
| 註冊位置 | 能力在哪裡被掛進系統？ | `src/components/index.js`、`src/index.js`、provide / inject |
| 生效位置 | 哪個使用場景真的觸發它？ | `examples/routers/<component>.vue`、`test/unit/specs`、render / event |

合格標準：

- 若只確認定義位置，只能說「這段能力存在」。
- 若確認定義與註冊位置，可以說「它進入對外 API 或系統」。
- 若再確認生效位置，才能說「這個使用場景會觸發它」。

## 4. 入口與安裝追蹤檢查表

適用於追 `app.use(ViewUIPlus)`、`install`、全域註冊、全域配置。

| 檢查項 | 完成標準 | 常見入口 |
|---|---|---|
| 已確認 build 入口 | 知道 library entry 是否指向 `src/index.js` | [`vite.config.js`](../ViewUIPlus-SourceCode/vite.config.js) |
| 已確認套件入口 | 知道 `main`、`typings`、`files` | [`package.json`](../ViewUIPlus-SourceCode/package.json) |
| 已確認 `install` 定義 | 找到 `install(app, opts)` | [`src/index.js`](../ViewUIPlus-SourceCode/src/index.js) |
| 已確認 locale 處理 | `opts.locale` / `opts.i18n` 是否進入 locale 系統 | `src/index.js`、[`src/locale/index.js`](../ViewUIPlus-SourceCode/src/locale/index.js) |
| 已確認元件註冊 | 是否遍歷 `ViewUI` 並呼叫 `app.component` | `src/index.js` |
| 已確認指令註冊 | 是否遍歷 `directives` 並呼叫 `app.directive` | `src/index.js` |
| 已確認 `$VIEWUI` | 全域配置寫到哪裡，包含哪些 key | `src/index.js`、[`types/index.d.ts`](../ViewUIPlus-SourceCode/types/index.d.ts) |
| 已確認命令式 API 掛載 | `$Message`、`$Modal` 等掛到哪裡 | `src/index.js` |
| 已確認 examples 觸發 | examples 是否真的 `app.use(ViewUIPlus)` | [`examples/main.js`](../ViewUIPlus-SourceCode/examples/main.js) |

停止條件：

```text
package / build 入口
  -> src/index.js install
  -> locale
  -> components
  -> directives
  -> $VIEWUI
  -> globalProperties
```

能畫出上面流程，就可以停止入口安裝線。

## 5. 元件註冊追蹤檢查表

適用於確認某個元件是否對外可用。

| 檢查項 | 完成標準 |
|---|---|
| 已找到元件資料夾 | `src/components/<component>/` 存在 |
| 已讀單元件 `index.js` | 判斷它是 re-export、群組入口、service，還是底層支撐 |
| 已確認主實作檔 | 找到 `.vue`、service `index.js` 或其他主檔 |
| 已確認集中匯出 | 元件出現在 [`src/components/index.js`](../ViewUIPlus-SourceCode/src/components/index.js) |
| 已確認進入 `ViewUI` | `src/index.js` 展開 `components` 或補別名 |
| 已確認全域註冊 | `install` 中會被 `app.component` 註冊 |
| 已確認是否有命令式掛載 | 如 `Message`、`Modal` 是否另掛到 `globalProperties` |
| 已確認 types 是否匯出 | `types/viewuiplus.components.d.ts` 或對應 type 檔 |
| 已確認 examples / tests | 是否有使用或測試證據 |

常見降分點：

- 只看到 `button.vue` 就說 Button 對外可用。
- 只看到 `src/components/index.js` 匯出，就忽略 `install` 註冊。
- 把 `Message` 當成普通 `.vue` 元件讀，沒有追 service。

## 6. 單一元件追蹤檢查表

適用於 Button、Input、Select、Table 等元件分析。

### 6.1 API 檢查

| 檢查項 | 完成標準 |
|---|---|
| 已列出 props | 包含元件本身與 mixin 補入的 props |
| 已列出 emits / `$emit` | 知道事件名稱、觸發位置、payload |
| 已列出 slots | 知道 default / named / scoped slot 是否存在 |
| 已列出 public methods | 若有 ref method 或 service method，需記錄 |
| 已對照 types | 知道 types 是否和實作一致 |
| 已對照 examples | 知道常見用法是否出現 |
| 已對照 tests | 知道哪些 API 被測試固定 |

### 6.2 狀態與計算檢查

| 檢查項 | 完成標準 |
|---|---|
| 已讀 data | 知道內部狀態初始值 |
| 已讀 computed | 知道 props / state 如何轉成 class、渲染分支或輸出 |
| 已讀 watch | 知道外部輸入改變時如何同步 |
| 已讀 lifecycle | 知道 mounted / beforeUnmount 是否有註冊、清理、副作用 |
| 已追 provide / inject | 如果元件有上下文協作，知道來源與接收者 |
| 已追 mixins | 知道哪些 API 或方法不是元件本身定義 |

### 6.3 渲染與事件檢查

| 檢查項 | 完成標準 |
|---|---|
| 已確認根節點或 render tag | 知道元件渲染成什麼 |
| 已確認主要渲染分支 | 例如 loading、disabled、multiple、transfer |
| 已確認事件入口 | DOM event 或子元件 event 從哪裡進入 |
| 已確認事件處理方法 | method 做了 emit、改狀態、呼叫 mixin，還是呼叫 service |
| 已確認狀態如何回到 UI | state / computed 改變後如何影響 render / class |
| 已確認副作用 | DOM、router、timer、instance、form validate 等 |

單一元件合格主流程：

```text
外部輸入 props / slot / event
  -> data / computed / watch / methods
  -> emit / state / side effect
  -> render / template
  -> class / style
  -> types / examples / tests 對照
```

## 7. mixin / utils / base 追蹤檢查表

適用於元件 import 共用邏輯時。

| 檢查項 | 完成標準 |
|---|---|
| 已列出 import | 知道元件直接依賴哪些 mixins、utils、base service |
| 已判斷依賴類型 | 是 props 補充、DOM 工具、格式化工具，還是 instance factory |
| 已追和本輪問題相關的依賴 | 只追主線，不展開所有工具 |
| 已確認 mixin 補了什麼 | props、computed、methods、inject 都要檢查 |
| 已確認 utils 是否有副作用 | 純函式、DOM 操作、狀態快取要分清 |
| 已確認 base service 的建立與銷毀 | 例如 notification、popper 類底層支撐 |
| 已回到主元件流程 | 依賴追完後，要說明它如何影響主流程 |

停止條件：

- 能說明「這個依賴為目前問題提供了什麼能力」。
- 能說明「不追這個依賴會漏掉什麼」。
- 能說明「這個依賴是否需要另開專題」。

## 8. 事件流 / 狀態流追蹤檢查表

適用於 v-model、表單驗證、下拉選擇、彈層開關、命令式 API 等互動流程。

| 檢查項 | 完成標準 |
|---|---|
| 已確認觸發點 | 使用者操作、props 變化、watch、timer、service call |
| 已確認第一個處理函式 | DOM event 綁定或 method 名稱 |
| 已確認狀態是否改變 | data / computed source / external props 是否更新 |
| 已確認是否 `$emit` | 事件名稱與 payload 清楚 |
| 已確認外部是否回寫 | v-model 或父元件是否可能把值傳回來 |
| 已確認 UI 如何更新 | render、template、class、transition |
| 已確認副作用 | router、DOM、Notification、AsyncValidator、timer |
| 已確認結束條件 | close、destroy、remove、unmounted、validation callback |

觸發點
  -> 第一個處理函式
  -> 狀態 / emit / 副作用
  -> computed / watch
  -> render / style
  -> 結束條件
```

- 已確認：
- 合理推論：
- 待驗證：
````

## 9. 樣式追蹤檢查表

適用於追 `prefixCls`、class、Less、主題變數、動畫。

| 檢查項 | 完成標準 |
|---|---|
| 已找到元件 class 來源 | `prefixCls`、computed class、template class |
| 已確認 class 條件 | 哪些 props / state 會加上 class |
| 已找到 component Less | `src/styles/components/<component>.less` 或替代來源 |
| 已確認 Less selector | selector 和 class 能對上 |
| 已追變數來源 | `custom.less` 或相關變數檔 |
| 已追 mixin | Less mixin 來源清楚 |
| 已追 animation | transition name 對應到 animation 檔 |
| 已確認樣式入口 | [`src/styles/index.less`](../ViewUIPlus-SourceCode/src/styles/index.less) 是否匯入到總樣式 |

常見待驗證：

- 找不到同名 Less 檔時，可能樣式來自 common、animation 或其他元件 Less。
- class 在 render 裡出現，不代表已確認樣式效果，仍要追 selector。

## 11. 命令式 API 追蹤檢查表

適用於 Message、Notice、Modal、LoadingBar、ImagePreview 等。

| 檢查項 | 完成標準 |
|---|---|
| 已確認全域掛載 | `src/index.js` 是否掛到 `app.config.globalProperties` |
| 已確認 service 入口 | `src/components/<service>/index.js` |
| 已確認 API 方法 | `info`、`success`、`confirm`、`destroy` 等 |
| 已確認 instance 建立 | `newInstance`、singleton、factory 或 mount 方式 |
| 已確認呼叫流程 | 外部 method 如何轉成 instance method |
| 已確認 config | defaults 是否可被修改 |
| 已確認 remove / destroy | 清理何時發生，是否會重置 instance |
| 已確認 DOM / side effect | 是否建立通知、彈窗、loading、timer |
| 已對照 types / examples / tests | 外部契約與行為證據是否一致 |

命令式 API 主流程模板：

```text
this.$Xxx.method(options)
  -> globalProperties.$Xxx
  -> service method
  -> getInstance / newInstance
  -> instance.show / notice / remove
  -> close / destroy / callback
```

## 12. 表單協作追蹤檢查表

適用於 Form、FormItem、Input、Select、InputNumber、DatePicker 等表單控件。

| 檢查項 | 完成標準 |
|---|---|
| 已確認 Form provide | `FormInstance` 來源清楚 |
| 已確認 FormItem inject | `FormItem` 如何取得 Form |
| 已確認 FormItem provide | `FormItemInstance` 是否提供給控件 |
| 已確認控件 mixin | 控件是否使用 `mixins/form` |
| 已確認欄位註冊 | `addField` / `removeField` 時機 |
| 已確認 rules 來源 | Form rules、FormItem rules、required |
| 已確認 trigger | blur / change 如何過濾規則 |
| 已確認 AsyncValidator | descriptor、model、callback |
| 已確認 validation state | `validateState`、`validateMessage` 如何影響 UI |
| 已確認 reset | `initialValue` 與 model 回寫 |
| 已確認 on-validate | Form 是否 emit 驗證結果 |

表單驗證主流程：

```text
控件 blur / change
  -> mixins/form.handleFormItemChange
  -> FormItem.formBlur / formChange
  -> FormItem.validate(trigger)
  -> AsyncValidator
  -> validateState / validateMessage
  -> Form emit on-validate
```

## 13. directive 追蹤檢查表

適用於 `resize`、`line-clamp`、style 快捷指令、click outside、transfer DOM 等。

| 檢查項 | 完成標準 |
|---|---|
| 已確認是否全域註冊 | `src/index.js` 的 `directives` 物件是否包含它 |
| 已確認 directive lifecycle | mounted、updated、unmounted 是否清楚 |
| 已確認 binding value | directive 期待什麼值 |
| 已確認 DOM 副作用 | 修改 style、監聽 resize、搬移 DOM、事件監聽 |
| 已確認清理邏輯 | unmounted 是否 remove listener 或還原 DOM |
| 已確認局部使用位置 | 若非全域註冊，哪些元件 import 它 |
| 已確認第三方依賴 | 例如 resize detector、click outside lib |

常見誤讀：

- `src/directives/` 中存在檔案，不代表它一定全域可用。
- directive 的核心不是 props，而是 DOM lifecycle 與副作用清理。

## 15. 追蹤完成度分級

用這張表判斷本輪閱讀品質。

| 分數 | 狀態 | 判斷標準 |
|---|---|---|
| 10 / 10 | 可交接 | 入口、邊界、定義 / 註冊 / 生效、主流程、外部證據、證據分級、待驗證都完整 |
| 9 / 10 | 可回查 | 主流程完整，少數外部證據或分支可後續補 |
| 8 / 10 | 可暫停 | 已能回答本輪問題，但待驗證較多 |
| 6 到 7 / 10 | 初步理解 | 有入口與部分流程，但缺少註冊、生效或外部對照 |
| 4 到 5 / 10 | 瀏覽階段 | 只讀到局部檔案，無法說清完整流程 |
| 3 以下 | 未追蹤 | 只有檔名、印象或命名推測 |

滿分 10 分必要條件：

- 問題明確。
- 入口正確。
- 邊界清楚。
- 已分清定義、註冊、生效。
- 主流程能畫出。
- 依賴與副作用沒有漏掉。
- 樣式、型別、範例、測試至少按需要對照。
- 已標記已確認、合理推論、待驗證、證據衝突。
- 已留下下一步。

## 16. 停止條件

讀碼不是追得越深越好。達到本輪目的後要停止，把支線交給後續筆記。

| 任務 | 可以停止的條件 |
|---|---|
| 入口安裝 | 能說明 `app.use` 到 `install` 的主要動作 |
| 元件註冊 | 能說明單元件如何進入 `components/index.js` 與 `install` |
| 單一元件 | 能說明 API、狀態、事件、渲染、樣式主流程 |
| 命令式 API | 能說明全域掛載、service、instance、remove / destroy |
| 表單協作 | 能說明 Form、FormItem、控件、AsyncValidator 如何串起來 |
| 樣式 | 能把 class 對到 Less、變數、mixin 或動畫 |
| types | 能標記實作與型別一致、不一致或待驗證 |
| examples | 能確認用法證據，但不把 examples 當完整規格 |
| tests | 能列出測試固定行為與測試缺口 |
| build | 能畫出輸入、處理、輸出 |

如果繼續追只是在展開支線，應停止並記錄：

```md
## 暫緩支線

- 支線：
- 暫緩理由：
- 目前證據：
- 之後回來時的第一入口：
```

## 17. 常見不合格情況

| 不合格情況 | 為什麼不合格 | 修正方式 |
|---|---|---|
| 只讀到一個 `.vue` 檔 | 無法確認對外註冊與使用場景 | 補查 `index.js`、`components/index.js`、`src/index.js` |
| 只列 props | 無法知道 props 如何生效 | 補 computed、watch、methods、render、style |
| 只看 `$emit` | 不知道外部是否處理或回寫 | 補查 v-model、父層、examples、tests |
| 只看 class | 不知道樣式是否存在 | 補 Less selector 與樣式入口 |
| 只看 types | types 不是執行時真相 | 補實作、examples、tests |
| 只看 examples | examples 不是完整規格 | 補實作與測試 |
| 只看 service method | 命令式 API 缺少 instance 與銷毀 | 補 newInstance、remove、destroy |
| 只看到 directive 檔 | 不知道是否全域註冊 | 補查 `src/index.js` 的 `directives` |
| 追大型元件所有分支 | 主流程容易失焦 | 先完成主流程，把支線列入待驗證 |
| 沒有證據分級 | 讀者分不清事實和推論 | 按已確認、合理推論、待驗證、證據衝突重寫 |

## 本節小結

追蹤檢查表的核心不是要求每次都讀到最深，而是讓每次閱讀都有可驗證的收斂點：

> 先確定問題與入口，再分清定義、註冊與生效位置；接著追主流程、依賴、副作用與外部證據；最後用已確認、合理推論、待驗證與證據衝突標記所有結論。

當一段源碼能被這張檢查表驗收，就代表它已經從「看過」變成「可回查、可交接、可延伸」的理解。
