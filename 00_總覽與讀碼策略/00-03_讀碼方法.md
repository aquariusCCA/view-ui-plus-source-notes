## 本節導讀

`00-01_源碼總地圖` 解決「檔案在哪裡」，`00-02_讀碼路線` 解決「先讀哪條線」。這一節解決「每次打開一段源碼時，具體怎麼讀」。

View UI Plus 是一個 Vue 3 元件庫。讀它的源碼時，不要只看單一 `.vue` 檔，也不要只看 `index.js` 的匯出。比較穩定的方法是：

```text
先定問題
  -> 找入口
  -> 判斷定義 / 註冊 / 生效位置
  -> 追依賴
  -> 分析 API
  -> 分析狀態與事件
  -> 分析渲染
  -> 對照樣式
  -> 對照型別 / 範例 / 測試
  -> 標記證據等級與待驗證問題
```

這套方法的目標不是一次讀很多檔案，而是每次都形成一個可回查、可交接、可延伸的最小閉環。

## 關鍵字

## 建議回查情境

當你遇到以下情境，可以回到這篇：

- 打開某個元件後不知道先看 props、template、methods，還是樣式。
- 想追一個功能，但依賴越追越多，需要收斂。
- 想判斷一段程式是定義、註冊，還是實際生效位置。
- 想把 `.vue`、`index.js`、Less、types、examples、tests 串成一條證據鏈。
- 想避免把命名直覺或框架慣例誤寫成已確認結論。

## 0. 這篇怎麼用

本篇是「正在讀源碼時」放在旁邊使用的操作卡。它不替代 `00-02_讀碼路線.md`，也不替代 `00-04_元件分析模板.md`。

| 文件 | 負責回答 | 使用時機 |
|---|---|---|
| `00-02_讀碼路線.md` | 先讀哪條線、下一步去哪裡、讀到哪裡停 | 開始一輪學習或迷路時 |
| `00-03_讀碼方法.md` | 打開一段源碼後，具體按什麼順序分析 | 正在追一個問題或一段流程時 |
| `00-04_元件分析模板.md` | 如何把單一元件分析整理成可複製的筆記 | 要正式沉澱元件筆記時 |

### 0.1 三十分鐘讀碼閉環

如果只有三十分鐘，建議照這個節奏完成一個最小閉環：

| 時間 | 動作 | 產出 |
|---|---|---|
| 5 分鐘 | 把問題縮小，選閱讀模式 | 一句可追問題 |
| 5 分鐘 | 找最保守入口，判斷它是定義、註冊還是使用入口 | 入口檔與邊界 |
| 10 分鐘 | 追直接依賴、API、狀態、事件或 build 流程 | 主流程草圖 |
| 5 分鐘 | 對照樣式、型別、範例或測試中的至少一種 | 外部證據 |
| 5 分鐘 | 標記已確認、合理推論、待驗證、證據衝突 | 可回查記錄 |

合理推論：短時間讀碼的重點不是追完所有分支，而是確保本輪結論有入口、有證據、有停止點。

### 0.2 最小證據鏈

一個結論要寫進筆記，至少要能放進下面某一種證據鏈：

| 結論類型 | 最小證據鏈 |
|---|---|
| 元件可全域使用 | `src/components/<component>/index.js` -> `src/components/index.js` -> `src/index.js install` |
| 命令式 API 可用 | service `index.js` -> `src/components/index.js` -> `app.config.globalProperties.$Xxx` |
| prop / event 生效 | `.vue` props / emits -> method / computed / render -> example 或 test |
| class / 樣式生效 | `prefixCls` / class 字串 -> component Less -> `src/styles/index.less` |
| build 輸出成立 | `package.json` script -> build config / script -> output path |
| 型別是對外契約 | `types/index.d.ts` 或 `types/<component>.d.ts` -> 實作對照 -> example / test 對照 |

若證據鏈中斷，就不要把結論寫成「已確認」；應標成「合理推論」或「待驗證」。

## 1. 讀碼前先定問題

不要用「我要讀 Button」當成問題，這太泛。比較好的問題是：

| 模糊問題 | 改成可追的問題 |
|---|---|
| 讀 Button | Button 如何決定渲染成 `<button>` 還是 `<a>`？ |
| 讀 Message | `$Message.success()` 如何建立通知並銷毀？ |
| 讀樣式 | Button 的 `ivu-btn-loading` class 如何對應 Less？ |
| 讀 Form | Input 如何通知 FormItem 做驗證？ |
| 讀 build | `npm run build:style` 如何產生 CSS？ |

可追問題通常包含四個元素：

```text
入口 + 動作 + 狀態 / 輸出 + 你要確認的結果
```

例如：

```text
入口：Button 的 click
動作：點擊帶有 to 的 Button
狀態 / 輸出：決定是否 preventDefault、是否 router push
結果：確認 Button 的跳轉邏輯在哪裡完成
```

## 2. 先判斷閱讀模式

不同問題要用不同入口，不要每次都從 `src/index.js` 開始。

| 閱讀模式 | 適用問題 | 最保守入口 |
|---|---|---|
| 工程 / build | 打包、樣式輸出、語系輸出 | [`package.json`](../ViewUIPlus-SourceCode/package.json) |
| 插件安裝 | `app.use`、全域配置、全域 API | [`src/index.js`](../ViewUIPlus-SourceCode/src/index.js) |
| 元件註冊 | 元件是否對外匯出、如何全域註冊 | [`src/components/index.js`](../ViewUIPlus-SourceCode/src/components/index.js) |
| 單一元件 | props、events、slots、渲染、樣式 | `src/components/<component>/index.js` |
| 命令式 API | `$Message`、`$Modal`、`$Notice`、`$Loading` | 對應 `src/components/<service>/index.js` |
| 樣式 | class、主題、動畫、prefix | [`src/styles/index.less`](../ViewUIPlus-SourceCode/src/styles/index.less) 或對應 component less |
| 型別 | install options、props 型別、全域屬性型別 | [`types/index.d.ts`](../ViewUIPlus-SourceCode/types/index.d.ts) |
| 範例 | 實際使用方式、Demo 場景 | [`examples/main.js`](../ViewUIPlus-SourceCode/examples/main.js)、`examples/routers/<component>.vue` |
| 測試 | 已被固定的行為規格 | [`test/unit/specs`](../ViewUIPlus-SourceCode/test/unit/specs/) |

合理推論：先選對模式，可以把閱讀範圍限制在一條問題鏈上，避免一開始就展開整個元件庫。

## 3. 找入口：不要只看檔名

找入口時，要先確認它是實作入口、轉發入口、註冊入口，還是使用入口。

以 Button 為例：

```text
src/components/button/index.js
  -> src/components/button/button.vue
```

已確認：[`src/components/button/index.js`](../ViewUIPlus-SourceCode/src/components/button/index.js) 只是把 `button.vue` import 後 default export。這代表真正的元件邏輯要進到 [`button.vue`](../ViewUIPlus-SourceCode/src/components/button/button.vue)。

以 Message 為例：

```text
src/components/message/index.js
  -> src/components/base/notification
```

已確認：[`src/components/message/index.js`](../ViewUIPlus-SourceCode/src/components/message/index.js) 不是轉發 `.vue`，而是直接定義 `info`、`success`、`warning`、`error`、`loading`、`config`、`destroy` 等方法，並透過 `Notification.newInstance` 建立通知實例。

所以入口判斷要先問：

- 這個 `index.js` 是 re-export 嗎？
- 它是否直接定義服務方法？
- 它是否只是把多個子元件組合起來？
- 它是否被 `src/components/index.js` 匯出？
- 它是否又被 `src/index.js` 掛到 `app.component` 或 `globalProperties`？

## 4. 分清三種位置

讀 View UI Plus 時，最容易誤讀的是「看到定義，就以為已經確認生效」。每次都要分清：

| 位置 | 意義 | View UI Plus 例子 |
|---|---|---|
| 定義位置 | 這段能力在哪裡被寫出來 | `Button` 的 props / computed / render 在 `button.vue` |
| 註冊位置 | 這段能力在哪裡被掛進系統 | `Button` 在 `src/components/index.js` 匯出，在 `src/index.js` 的 `install` 中被 `app.component` 註冊 |
| 生效位置 | 哪個使用場景真的觸發它 | `examples/routers/button.vue` 使用 `<Button>`，`button.spec.js` 驗證部分行為 |

以 `$Message` 為例：

| 位置 | 對應檔案 |
|---|---|
| 定義位置 | [`src/components/message/index.js`](../ViewUIPlus-SourceCode/src/components/message/index.js) |
| 註冊位置 | [`src/components/index.js`](../ViewUIPlus-SourceCode/src/components/index.js) 匯出 `Message` |
| 全域掛載位置 | [`src/index.js`](../ViewUIPlus-SourceCode/src/index.js) 掛到 `app.config.globalProperties.$Message` |
| 使用 / 驗證位置 | `examples/routers/message.vue`、`test/unit/specs/message.spec.js` |

如果只看到 `message/index.js` 的方法，還不能直接說 `$Message` 可用；必須再確認它有被匯出並掛到 `globalProperties`。

## 5. 追依賴：先追直接依賴，再追共用層

讀單一元件時，先列出直接依賴：

```text
imports
  -> components
  -> mixins
  -> utils
  -> base services
  -> styles
  -> types / examples / tests
```

以 Button 為例，已確認 [`button.vue`](../ViewUIPlus-SourceCode/src/components/button/button.vue) 直接依賴：

| 依賴 | 角色 |
|---|---|
| `vue` 的 `h`、`getCurrentInstance` | render 與讀全域配置 |
| `../icon` | loading / icon 渲染 |
| `../../utils/assist` 的 `oneOf` | props validator |
| `../../mixins/link` | `to`、`target`、router 跳轉 |
| `../../mixins/form` | 表單 disabled 與 FormItem 通知 |

追依賴時不要一次展開到底。建議順序：

1. 先確認每個依賴提供什麼能力。
2. 只追和當前問題相關的依賴。
3. 如果依賴是 mixin，要回頭確認它補了哪些 props、computed、methods。
4. 如果依賴是 base service，要確認誰建立 instance、誰銷毀 instance。

例如 Button 的 `tagProps` 使用 `linkUrl`，而 `linkUrl` 不是 `button.vue` 自己定義，是 [`src/mixins/link.js`](../ViewUIPlus-SourceCode/src/mixins/link.js) 提供。若不追 mixin，就會漏掉 Button 的連結行為。

## 6. 分析對外 API

對外 API 不只看 props。至少要看：

| 類型 | 在哪裡找 |
|---|---|
| props | `.vue` 的 `props`、mixin 的 `props`、`types/<component>.d.ts` |
| events | `.vue` 的 `emits`、`this.$emit`、types 裡的 `onXxx` |
| slots | `$slots`、render / template 中對 slot 的使用 |
| methods / service API | 元件 methods、命令式 API 的 exported object |
| install options | `src/index.js` 的 `install(app, opts)` 與 `types/index.d.ts` |
| global properties | `app.config.globalProperties` 與 Vue runtime-core 擴充型別 |

以 Button 為例：

- 已確認：`button.vue` 的 props 包含 `type`、`shape`、`size`、`loading`、`disabled`、`htmlType`、`icon`、`customIcon`、`long`、`ghost`。
- 已確認：`mixins/link.js` 補了 `to`、`replace`、`target`、`append`。
- 已確認：`button.vue` 宣告 `emits: ['click']`，並在 `handleClickLink` 中 `$emit('click', event)`。
- 已確認：[`types/button.d.ts`](../ViewUIPlus-SourceCode/types/button.d.ts) 也宣告了 Button 的對外屬性與 `onClick`。

待驗證：如果 `.vue`、mixin、types、examples 之間出現不一致，要另外記錄，不要硬合併成一個看似完整的 API。

## 7. 分析內部狀態與計算邏輯

讀元件內部時，不要直接跳到 methods。建議順序：

```text
props
  -> inject / provide
  -> data
  -> computed
  -> watch
  -> methods
  -> render / template
```

以 Button 為例：

| 區塊 | 已確認內容 | 閱讀意義 |
|---|---|---|
| props | `type`、`shape`、`size`、`loading` 等 | 使用者可控制的輸入 |
| mixins | `link`、`form` | 補上跳轉與表單協作能力 |
| computed: `showSlot` | 判斷是否存在 default slot | 影響 icon-only class |
| computed: `classes` | 組合 `ivu-btn`、`ivu-btn-primary`、`ivu-btn-loading` 等 class | 連到 Less 樣式 |
| computed: `isHrefPattern` | 依 `to` 判斷是否使用連結模式 | 影響渲染 tag |
| computed: `tagName` | 決定渲染 `button` 或 `a` | 連到測試案例 |
| computed: `tagProps` | 決定 `href/target` 或原生 `type` | 連到 link mixin |
| methods: `handleClickLink` | emit click 並呼叫 `handleCheckClick` | 連到事件與跳轉 |

合理推論：computed 通常是理解元件狀態到 UI 的關鍵橋樑。先看 computed，再看 render，會比直接看 render 更容易。

## 8. 分析事件流與副作用

事件流要追到「誰觸發、誰接收、狀態怎麼變、UI 如何更新」。

以 Button loading 測試為例：

```text
使用者 click
  -> Button render 綁定 onClick: handleClickLink
  -> handleClickLink emit click
  -> 外部 fetch 將 loading 設為 true
  -> props.loading 變化
  -> classes 出現 ivu-btn-loading
  -> render 顯示 loading Icon
```

已確認：[`test/unit/specs/button.spec.js`](../ViewUIPlus-SourceCode/test/unit/specs/button.spec.js) 測試了 click 後 loading class、Icon 數量與 loading icon class。

讀事件時要特別注意：

- `this.$emit` 只是通知外部，不代表內部狀態一定改變。
- v-model 類元件要追 `modelValue`、`update:modelValue` 或相近事件。
- mixin 裡的方法可能才是真正副作用位置。
- 命令式 API 的副作用通常是建立 DOM instance、加入通知、移除通知或 destroy。

## 9. 分析渲染：template 和 render 都要能讀

View UI Plus 中有些元件使用 `.vue` template，有些使用 render function。Button 就使用 render。

讀 render 時照這個順序：

1. 先看 `h()` 的第一個參數，確認 tag 或 component。
2. 再看 props / attrs / class / event。
3. 最後看 children / slots。

以 Button 為例：

| render 區塊 | 已確認行為 |
|---|---|
| `tag` | 依 `tagName` 決定 `button` 或 `a` |
| `slots` | loading 時放 loading Icon，有 icon 時放 Icon，有 default slot 時放 `span` |
| `class` | 使用 computed `classes` |
| `disabled` | 使用 mixin/form 來的 `itemDisabled` |
| `onClick` | 綁定 `handleClickLink` |
| `...tagProps` | 補上 `href/target` 或 `type` |

已確認：`button.spec.js` 測試了有 `to` 時渲染成 `<a>`，沒有 `to` 時渲染成 `<button>`，也測了 `<button>` 才保留原生 `type`。

## 10. 對照樣式：從 prefixCls 追到 Less

View UI Plus 很多元件使用 `prefixCls` 組 class。讀樣式不要只用肉眼猜，要沿著 class 追。

以 Button 為例：

```text
button.vue
  const prefixCls = 'ivu-btn'
  computed classes -> ivu-btn / ivu-btn-primary / ivu-btn-loading / ...

src/styles/components/button.less
  @btn-prefix-cls: ~"@{css-prefix}btn";
  .@{btn-prefix-cls} { ... }
```

讀樣式的固定步驟：

1. 在元件中找 `prefixCls` 或 class 字串。
2. 找 `src/styles/components/<component>.less`。
3. 對照狀態 class，例如 `-loading`、`-disabled`、`-ghost`、`-active`。
4. 如果 Less 使用 mixin，追到 `src/styles/mixins/`。
5. 如果 class 依賴變數，追到 `src/styles/custom.less` 或相關變數檔。
6. 如果涉及動畫，追 `src/styles/animation/`。

待驗證：並非每個元件都只有一個同名 Less 檔。有些樣式可能來自 common、mixins、animation 或其他元件樣式。

## 11. 對照型別、範例與測試

一段程式讀完後，不要只停在實作。至少做三種交叉驗證：

| 證據 | 用途 | Button 例子 |
|---|---|---|
| types | 確認對外 API 型別 | `types/button.d.ts` 宣告 `type`、`ghost`、`size`、`html-type`、`onClick` 等 |
| examples | 看實際使用方式 | `examples/routers/button.vue` 展示 type、size、shape、icon、to、ButtonGroup 等用法 |
| tests | 看已固定行為 | `button.spec.js` 測試 tag、type attribute、loading 狀態 |

交叉驗證的判斷順序：

1. 實作是執行時行為的主要證據。
2. 測試是「作者目前有意識固定」的行為證據。
3. examples 是使用方式證據，但不代表完整規格。
4. types 是 TypeScript 對外契約，但可能和實作不同步。

如果四者互相衝突，要標成「證據衝突」，不要自行修成一致敘述。

## 12. 命令式 API 的讀法

命令式 API 不適合套用一般 `.vue` 元件讀法。它要追：

```text
全域掛載
  -> service index.js
  -> instance 建立
  -> 方法包裝
  -> DOM / notification / modal 操作
  -> close / destroy
  -> config
```

以 Message 為例：

| 步驟 | 已確認內容 |
|---|---|
| 全域掛載 | `src/index.js` 掛載 `app.config.globalProperties.$Message = components.Message` |
| service 入口 | `src/components/message/index.js` |
| instance | `getMessageInstance()` 使用 `Notification.newInstance` |
| 方法包裝 | `info`、`success`、`warning`、`error`、`loading` 都呼叫 `message(type, options)` |
| notice | `notice()` 組 content、duration、render、onClose、background，呼叫 `instance.notice` |
| 手動移除 | `notice()` 回傳一個 closure，內部呼叫 `instance.remove(...)` |
| config | `config(options)` 修改 `defaults.top`、`defaults.duration`、`defaults.background` |
| destroy | `destroy()` 取得 instance 後呼叫 `instance.destroy('ivu-message')` |

命令式 API 至少要回答：

- API 是掛在 `globalProperties` 還是只 named export？
- 是否有 singleton instance？
- instance 何時建立？
- 每次呼叫會新增什麼？
- 如何 close / remove / destroy？
- 是否有全域 config？
- tests 或 examples 是否覆蓋這些行為？

## 13. build / 樣式 / 語系的讀法

如果讀的是工程流程，不要套用元件分析模板。改用「輸入 -> 處理 -> 輸出」。

| 主題 | 輸入 | 處理 | 輸出 |
|---|---|---|---|
| JS build | `src/index.js` | `vite.config.js` 的 library build | `dist/viewuiplus.min.js`、`dist/viewuiplus.min.esm.js` |
| style build | `src/styles/index.less` | `build/build-style.js` 使用 less、autoprefixer、cleanCSS | `dist/styles/viewuiplus.css` |
| fonts | `src/styles/common/iconfont/fonts/*.*` | `build/build-style.js` 的 fonts task | `dist/styles/fonts` |
| locale build | `src/locale/lang/*` | `build/vite.lang.config.js` 掃描語系檔作為 input | `dist/locale` |

工程流程讀完後，要能畫出流程，不需要一開始就背每個 build option。

## 15. 停止條件：不要無限追

讀碼不是越深越好。每次都要設停止條件。

| 閱讀任務 | 可以停止的條件 |
|---|---|
| 追入口 | 已確認定義、註冊、生效位置 |
| 追元件 | 已能說明 props -> state/computed -> render/style -> event 的主流程 |
| 追命令式 API | 已能說明全域掛載、instance 建立、呼叫、移除、destroy |
| 追樣式 | 已能把元件 class 對到 Less selector 與變數 / mixin 來源 |
| 追測試 | 已能列出測試固定了哪些行為、沒覆蓋哪些行為 |
| 追 build | 已能畫出輸入、處理、輸出 |

如果繼續追只是在展開次要分支，就先把它寫入「待驗證」或 `00-10_暫緩閱讀清單.md`。

## 16. 常見誤讀提醒

讀 View UI Plus 時，特別容易出現這些誤讀：

| 誤讀 | 修正方法 |
|---|---|
| 看到 `src/components/<name>/index.js` 就以為是真正邏輯 | 先判斷它是 re-export、群組入口，還是命令式服務 |
| 只看 `.vue`，漏掉 mixins | 先列 imports，遇到 mixin 要追它補的 props / computed / methods |
| 看到 `$emit` 就以為狀態改了 | `$emit` 只是通知外部，要追外部是否改 props 或 state |
| 看到 class 就直接猜樣式 | 從 `prefixCls` 追到 `src/styles/components` 和 mixins |
| 把 examples 當完整規格 | examples 是使用證據，不是完整測試 |
| 把 types 當執行時真相 | types 要和 `.vue`、tests、examples 對照 |
| 一開始就讀 dist | dist 是產物觀察區，不是初期理解設計的入口 |
| 追大型元件所有分支 | 先抓主流程，支線標記待驗證 |

## 本節小結

View UI Plus 的讀碼方法可以濃縮成一句話：

> 先把問題縮小，再找最保守入口；讀的時候分清定義、註冊與生效位置，沿著依賴追到最小閉環，最後用樣式、型別、範例與測試交叉驗證，並把所有結論標成已確認、合理推論或待驗證。
