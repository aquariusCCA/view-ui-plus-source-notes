## 所屬章節

`00_總覽與讀碼策略`

## 本節導讀

這一節提供後續分析單一 View UI Plus 元件時可直接複製使用的筆記模板。

前面三節已經建立：

- `00-01_源碼總地圖`：知道檔案與目錄在哪裡。
- `00-02_讀碼路線`：知道應該沿哪些主線讀。
- `00-03_讀碼方法`：知道每次讀源碼的操作順序。

這一節把方法落成固定欄位，讓每次分析元件時都能留下可回查的筆記，而不是只憑當下記憶。

## 建議回查情境

當你要分析以下內容時，可以先複製本篇模板：

- 基礎元件：例如 Button、Icon、Tag、Badge。
- 表單元件：例如 Form、FormItem、Input、Select。
- 父子協作元件：例如 Tabs、Breadcrumb、Timeline、Menu。
- 複雜互動元件：例如 Select、DatePicker、Table。
- 回饋 / 彈層元件：例如 Modal、Drawer、Tooltip、Poptip。
- 命令式 API：例如 Message、Notice、Modal、LoadingBar。
- 業務型元件：例如 Login、Exception、NumberInfo。

## 證據標記

每篇元件筆記都應明確標記證據狀態：

| 標記 | 意義 |
|---|---|
| 已確認 | 已從目前倉庫中的原始碼、型別、範例或測試直接讀到 |
| 合理推論 | 由已確認事實推導出的設計理解 |
| 待驗證 | 尚未從實作、範例或測試確認 |
| 證據衝突 | `.vue`、`index.js`、types、examples、tests 或 dist 之間不一致 |

## 0. 先選哪一份模板

不要每次都從完整模板開始。先依照元件規模與本輪目的選模板，才能避免筆記變成欄位很多但沒有主線。

| 使用情境 | 建議模板 | 本輪至少要填 |
|---|---|---|
| 第一次快速掃一個元件 | 第 4 節快速版模板 | 入口、角色、API、主流程、待驗證 |
| 要正式沉澱一篇單一元件筆記 | 第 3 節完整元件分析模板 | 定位、註冊鏈、API、狀態、事件、渲染、樣式、證據整理 |
| Message、Notice、Modal、LoadingBar 等服務 | 第 5 節命令式 API 模板 | 全域掛載、service 方法、instance 流程、銷毀與 config |
| Form、FormItem、Input、Select 等表單協作 | 第 6 節表單協作元件模板 | provide / inject、欄位註冊、驗證流程 |
| Select、DatePicker、Table 等大型元件 | 第 7 節大型互動元件模板 | 子模組地圖、主狀態、主流程、分支索引、停止條件 |

### 0.1 填寫深度分級

同一份模板可以分三種深度使用：

| 深度 | 適用時機 | 合格標準 |
|---|---|---|
| 初讀版 | 剛開始讀，還不確定是否深入 | 能說明入口、角色、主流程與下一步 |
| 回查版 | 之後可能反覆查這個元件 | API、狀態、事件、樣式、types / examples / tests 至少都有索引 |
| 深入版 | 要仿寫、重構或寫高階專題 | 能說明設計取捨、證據衝突、測試缺口與可仿寫點 |

### 0.2 不需要硬填的欄位

模板是檢查清單，不是填空作業。遇到以下情況可以明確寫「無」或「待驗證」：

- 元件沒有 public methods。
- 元件沒有對應測試檔。
- 元件不是表單控件，不需要填表單協作段落。
- 命令式 API 沒有 `.vue` 主實作，而是以 service `index.js` 為主。
- 大型元件本輪只追主流程，其他分支先放入分支索引。

## 1. 使用方式

每次分析元件時，先不要急著填完整篇。建議分三輪：

| 輪次 | 要做什麼 | 完成標準 |
|---|---|---|
| 第一輪 | 填「元件定位、入口、證據地圖」 | 知道自己正在讀哪條鏈 |
| 第二輪 | 填「API、狀態、事件、渲染、樣式」 | 能說明元件主流程 |
| 第三輪 | 填「types、examples、tests、可仿寫點、待驗證」 | 能交叉驗證並留下後續問題 |

如果元件很大，例如 Select、DatePicker、Table，不要一次追完所有分支。先完成主流程，再把次要分支記到「待驗證問題」。

## 6. 表單協作元件模板

Form、FormItem、Input、Select 這類會參與表單驗證或上下文協作的元件，額外補這段：

````md
## 表單協作分析

### 1. provide / inject

| key | provide 來源 | inject 接收者 | 作用 |
|---|---|---|---|
| `FormInstance` | `Form` | `FormItem` / 表單控件 | 取得 model、rules、disabled、validate |
| `FormItemInstance` | `FormItem` | Input / Select 等控件 | 觸發 blur / change 驗證 |

### 2. 欄位註冊

```text
FormItem mounted
  -> FormInstance.addField(this)
  -> Form.fields 保存 field
  -> Form.validate 遍歷 fields
```

### 3. 驗證流程

```text
使用者操作
  -> 控件觸發 blur / change
  -> mixinsForm.handleFormItemChange
  -> FormItem.formBlur / formChange
  -> FormItem.validate(trigger)
  -> AsyncValidator
  -> validateState / validateMessage
  -> Form emit on-validate
```

### 4. 需要確認

- 控件是否 mixin 了 `mixinsForm`？
- 何時觸發 `handleFormItemChange`？
- `disabled` 是否受 Form 全域 disabled 影響？
- reset 時如何還原 model？
- rules 改變時是否重新 validate？
````

## 7. 大型互動元件模板

````md
## 大型元件分析

### 1. 子模組地圖

| 子檔案 / 子元件 | 角色 |
|---|---|
| `<main>.vue` | 主狀態與主渲染 |
| `<child>.vue` | ... |
| `utils.js` | 資料轉換 / 輔助方法 |
| `mixin.js` | 共用邏輯 |

### 2. 主狀態

| 狀態 | 來源 | 更新位置 | 影響 |
|---|---|---|---|
| `visible` | data | `toggleMenu` | dropdown 顯示 |

### 3. 主流程

```text

  -> 使用者操作
  -> 更新狀態
  -> emit
````

## 9. 滿分元件筆記標準

一篇元件分析筆記若要達到可長期回查、可交接、可仿寫的標準，至少要符合下表。

| 面向 | 滿分標準 |
|---|---|
| 定位清楚 | 一開頭就知道元件負責什麼、屬於哪類元件、主要入口在哪裡 |
| 證據可追 | 每個重要結論都能追到 `.vue` / `index.js` / styles / types / examples / tests 中的證據 |
| 主線完整 | 能從外部輸入一路說到狀態、事件、渲染、樣式與輸出 |
| 邊界明確 | 能說清楚哪些分支本輪不追、哪些問題仍待驗證 |
| API 可查 | props、events、slots、methods 或 service methods 能快速查到 |
| 協作可懂 | mixins、utils、provide / inject、base service 的角色不被漏掉 |
| 樣式可對 | `prefixCls`、class、Less、變數與動畫能互相對應 |
| 外部契約可驗證 | types、examples、tests 與實作是否一致有明確記錄 |
| 可延伸 | 讀完後知道下一個要讀的檔案、可仿寫點與測試缺口 |

常見降分點：

- 只列 props，沒有說明 props 如何影響 computed / render / style。
- 只看 `.vue`，沒有追 `index.js`、mixin、utils 或 service。
- 把 examples 或 types 當成唯一真相，沒有和實作互相驗證。
- 對大型元件追太多支線，最後主流程反而不清楚。
- 沒有標記「待驗證」與「證據衝突」，讓讀者分不出事實和推論。

## 本節小結

元件分析模板的核心不是把所有欄位填滿，而是讓每次讀碼都能穩定回答：

> 這個元件在哪裡定義、怎麼被註冊、對外 API 是什麼、內部狀態如何驅動渲染、事件如何改變狀態、樣式從哪裡來，以及 types、examples、tests 是否支持目前理解。

下一節 `00-05_關鍵概念索引.md` 應該把閱讀過程中會反覆遇到的概念整理成索引，例如插件安裝、元件匯出、`globalProperties`、`prefixCls`、mixin、provide/inject、命令式 API、表單驗證與國際化。
