# 05_如何用 AI 輔助讀源碼

> 本篇任務：把「用 AI 讀 View UI Plus 原始碼」整理成一套可執行、可驗證、可沉澱筆記的 SOP。  
> 它不是教你把程式碼丟給 AI 然後照單全收，而是教你如何讓 AI 成為「讀碼助教、索引器、推理檢查器、筆記整理器」。

---

## 0. 本篇結論

用 AI 讀 View UI Plus，最重要的原則是：

```txt
源碼是主體，AI 是輔助。
AI 可以幫你解釋、拆流程、找關聯、整理筆記、設計仿寫練習；
但不能取代你對原始碼的定位、驗證與判斷。
```

所以不要這樣用 AI：

```txt
請解釋 View UI Plus 原始碼。
Button 怎麼寫？
Form 怎麼設計？
Table 怎麼實作？
```

這種問法太大、太空泛，AI 容易用一般 Vue 元件庫經驗回答，而不是根據 View UI Plus 的真實源碼回答。

要改成這樣：

```txt
我正在讀 View UI Plus 的 src/components/button/button.vue。
請只根據我貼上的源碼回答，不要補不存在的設計。
請按照「檔案角色、依賴關係、Props、Computed、Methods、Render 流程、與 src/index.js / components/index.js 的關係、可學走的設計」整理。
每個判斷都要指出對應的變數、函式、props、computed 或 import/export。
如果無法從源碼確認，請標記為「待驗證」。
```

一句話版本：

> **先縮小範圍，再餵源碼；先要證據，再要解釋；先做事實筆記，再做抽象總結。**

## 1. 本篇源碼核對依據

本篇的 AI 讀碼策略，是依照 View UI Plus 的真實專案結構設計，不是一般開源專案通用建議。

| 源碼位置 | 已確認角色 | 對 AI 輔助讀碼的意義 |
|---|---|---|
| `README.md` | 專案定位為 Vue 3 的 UI component library / front-end solution | 提醒 AI：這不是業務系統，而是元件庫 |
| `package.json` | 宣告 `main`、`typings`、`files`、`scripts`、dependencies | 問 AI 工程入口與發布產物時，要以它為依據 |
| `vite.config.js` | library build entry 指向 `src/index.js` | 問 AI 打包入口時，要回扣 `src/index.js` |
| `vite.config.dev.js` | 開發環境以 examples 為主要入口 | 問 AI demo 驗證時，要回扣 `examples` |
| `examples/main.js` | 建立 Vue app，`app.use(ViewUIPlus)`，設定大量 demo routes | 從使用者視角反推元件 API 與使用情境 |
| `src/index.js` | 匯出 components，定義 `install(app, opts)`，註冊 components / directives，掛載 `$VIEWUI` 與全域 API | 問 AI 元件庫入口、全域安裝、全域配置時的核心檔案 |
| `src/components/index.js` | 集中 export 對外元件 | 問 AI public component surface 時的核心檔案 |
| `src/components/button/button.vue` | 代表性基礎元件：props validator、mixins、computed classes、render function | 最適合練習 AI 單元件深讀 |
| `src/mixins` | 抽共用行為，例如 form / link | 問 AI 共用抽象與父子協作時要看這裡 |
| `src/directives` | 封裝 DOM 行為，例如 resize、line-clamp、style | 問 AI 為何不用元件而用 directive 時要看這裡 |
| `src/locale` | 管理語系、翻譯函式與 i18n 接入 | 問 AI 國際化時要看這裡 |
| `src/styles` | Less 樣式入口、元件樣式、變數與 mixins | 問 AI 視覺規格與 class 命名時要一起看 |
| `types` | 對外 TypeScript 型別宣告 | 問 AI 對外 API 契約時要一起驗證 |

## 2. AI 在讀源碼中應該扮演的 5 種角色

### 2.1 角色一：源碼翻譯器

用在你第一次看不懂某段程式碼時。

目標不是逐行翻譯，而是回答：

```txt
這段程式碼在整個檔案裡負責什麼？
它接收什麼輸入？
產生什麼輸出？
它影響哪些 UI 行為？
```

適合情境：

- 第一次讀 `button.vue`
- 第一次讀 `src/index.js`
- 第一次看到 `render()`、`h()`、`computed.classes`
- 第一次看到 `mixinsLink`、`mixinsForm`

不適合情境：

- 還沒貼源碼就問整套架構
- 問太抽象的「這樣寫好不好」
- 要 AI 直接幫你判斷整個專案品質

---

### 2.2 角色二：關聯索引器

用在你已經看懂單一檔案，但不知道它和其他檔案怎麼連起來時。

例如讀 `button.vue` 時，AI 要幫你追：

```txt
button.vue
  ├─ import Icon from '../icon'
  ├─ import { oneOf } from '../../utils/assist'
  ├─ import mixinsLink from '../../mixins/link'
  ├─ import mixinsForm from '../../mixins/form'
  ├─ 被 src/components/button/index.js 匯出
  ├─ 被 src/components/index.js 匯出成 Button
  └─ 最後被 src/index.js 安裝到 app.component('Button', ...)
```

這種「檔案關係圖」很適合讓 AI 整理，但你要要求它只能根據你提供的 `import/export` 或實際搜尋結果回答。

---

### 2.3 角色三：流程拆解器

用在你需要把一段源碼變成流程圖時。

例如讀 `src/index.js` 時，可以要求 AI 拆成：

```txt
使用者呼叫 app.use(ViewUIPlus)
  ↓
Vue 呼叫 install(app, opts)
  ↓
根據 opts.locale / opts.i18n 設定語系
  ↓
走訪 ViewUI 物件並 app.component(...) 註冊元件
  ↓
走訪 directives 物件並 app.directive(...) 註冊指令
  ↓
設定 app.config.globalProperties.$VIEWUI
  ↓
掛載 $Message / $Modal / $Notice / $Spin / $Loading 等命令式 API
```

這種輸出比「這是安裝函式」更有用，因為它能直接放進筆記。

---

### 2.4 角色四：驗證檢查器

AI 很容易把通用 Vue 知識套到 View UI Plus 上，所以你要讓它主動幫你檢查：

```txt
這個結論是否能從源碼確認？
有沒有 examples 可以佐證？
有沒有 types 可以佐證？
有沒有 styles 可以佐證？
有沒有 import/export 可以佐證？
```

任何無法確認的說法，都要標成：

```txt
待驗證
推論
可能如此，但需要再看 xxx 檔案
```

---

### 2.5 角色五：筆記整理器

當你已經貼源碼、確認流程、理解關聯後，才讓 AI 幫你整理成正式筆記。

不要一開始就叫 AI「幫我寫筆記」，因為它可能會產出漂亮但空泛的文章。

正確順序是：

```txt
源碼定位
  ↓
源碼解釋
  ↓
關聯追蹤
  ↓
流程拆解
  ↓
驗證疑點
  ↓
整理成筆記
```

## 3. AI 輔助讀碼 SOP

### Step 1：先定義這次讀碼目標

每次只讀一個明確目標。

好的目標：

```txt
我今天只讀 src/index.js，目標是看懂 app.use(ViewUIPlus) 做了什麼。
```

```txt
我今天只讀 button.vue，目標是看懂 Props、class 組裝、render 流程、link/form mixins 的用途。
```

```txt
我今天只讀 src/components/index.js，目標是看懂 View UI Plus 對外匯出哪些元件。
```

不好的目標：

```txt
我要讀 View UI Plus。
我要學會元件庫。
我要看懂全部原始碼。
```

---

### Step 2：建立「上下文卡片」

問 AI 前，先附上上下文卡片。這會大幅降低 AI 亂推論的機率。

```md
## 讀碼上下文

專案：View UI Plus
定位：Vue 3 UI component library
目前閱讀檔案：src/components/button/button.vue
本次目標：看懂 Button 的元件設計與可學走的寫法
已知關聯：
- Button 會被 src/components/index.js 匯出
- components 會被 src/index.js 匯入並註冊
- examples/main.js 會 app.use(ViewUIPlus)

請只根據我貼上的源碼回答。
如果需要其他檔案，請明確告訴我需要補哪個檔案。
```

這張卡片是關鍵。它能讓 AI 知道你不是要它泛談 Vue，而是要它進入 View UI Plus 的讀碼脈絡。

---

### Step 3：貼「最小可解釋源碼」

不要一次貼太多檔案。一次貼一個小範圍即可。

建議切法：

| 目標 | 一次貼的範圍 |
|---|---|
| 看入口 | `src/index.js` 的 import、directives、ViewUI、install |
| 看元件匯出 | `src/components/index.js` 的 export 清單 |
| 看 Button | `button.vue` 的 script 區塊 |
| 看樣式 | 該元件 Less + class 命名來源 |
| 看 mixin | 單一 mixin 檔案 + 使用它的元件片段 |
| 看 Form 協作 | `form.vue`、`form-item.vue`、`mixins/form.js` 分批貼 |
| 看命令式 API | `message` / `modal` 相關入口與呼叫流程分批貼 |

最小可解釋源碼的標準：

```txt
足夠回答本次問題，但不要大到讓 AI 開始省略或泛化。
```

---

### Step 4：要求 AI 先做「源碼事實整理」

第一次回答不要急著要「設計思想」。先要求它列事實。

模板：

```txt
請先不要評論好壞，也不要延伸到其他 UI library。
請只根據這段源碼整理「源碼事實」。

請輸出：
1. 這個檔案 / 片段的角色
2. import 了哪些東西，各自用途是什麼
3. export 了什麼
4. 定義了哪些核心變數 / 物件 / 函式
5. 主要流程順序
6. 哪些地方目前無法從這段源碼確認，需要補其他檔案
```

為什麼先做事實？

因為你要先知道 AI 有沒有看懂你貼的程式碼。若它連 import/export 都整理錯，後面的設計分析就不能信。

---

### Step 5：再要求 AI 做「流程圖」

當事實整理正確後，再要求它把流程串起來。

模板：

```txt
請把剛才的源碼事實整理成流程圖。

要求：
- 用文字箭頭表示，不要只寫段落
- 每個節點都要能對應到源碼中的變數、函式或方法
- 如果流程有分支，請標出分支條件
- 如果有外部依賴，請標成「需要補檔案驗證」
```

範例輸出格式：

```txt
install(app, opts)
  ↓
判斷 install.installed，避免重複安裝
  ↓
若 opts.locale 存在，呼叫 localeFile.use(opts.locale)
  ↓
若 opts.i18n 存在，呼叫 localeFile.i18n(opts.i18n)
  ↓
Object.keys(ViewUI).forEach(...)
  ↓
app.component(key, ViewUI[key])
  ↓
Object.keys(directives).forEach(...)
  ↓
app.directive(key, directives[key])
  ↓
app.config.globalProperties.$VIEWUI = {...}
```

---

### Step 6：要求 AI 做「關聯檔案追蹤」

看懂單檔後，立刻問它上下游。

模板：

```txt
請根據這段源碼的 import/export，幫我整理它的上下游關係。

請分成：
1. 這個檔案依賴誰
2. 這個檔案被誰使用
3. 哪些依賴是元件庫共用層
4. 哪些依賴是對外 API
5. 哪些關係目前只是推論，需要用搜尋或其他檔案驗證
```

範例：

```txt
button.vue
  上游依賴：Icon、oneOf、mixinsLink、mixinsForm
  下游匯出：button/index.js → components/index.js → src/index.js
  共用層：utils/assist、mixins/link、mixins/form
  對外 API：Button 元件本身、props、emits、slots
  待驗證：styles/components/button.less、types/button.d.ts、examples/routers/button.vue
```

---

### Step 7：要求 AI 做「可學走的設計」

只有在事實、流程、關聯都整理完後，才做抽象。

模板：

```txt
請根據剛才的源碼事實與流程，整理出可以學走的設計。

請注意：
- 只整理能從源碼確認的設計
- 不要把一般 Vue 最佳實踐硬套進來
- 每個設計都要附上對應源碼依據
- 請區分「可以直接學走」與「只適合大型元件庫」
```

範例分類：

| 設計 | 是否適合學走 | 使用場景 |
|---|---|---|
| `install(app, opts)` 集中安裝 | 適合 | 自己寫元件庫 / 公司內部 UI 套件 |
| `components/index.js` 集中 export | 適合 | 統一管理對外元件 |
| props validator + `oneOf` | 適合 | 限制元件 props 值域 |
| 全域 `$VIEWUI` 配置 | 視情況 | 大型元件庫可用，小型專案不要濫用 |
| render function 組裝 Button | 視情況 | 需要動態 tag / slots / props 合成時 |
| 大量全域註冊所有元件 | 視情況 | 文件 demo 方便，但正式業務專案要考慮按需載入 |

---

### Step 8：最後才整理成正式筆記

正式筆記不應該只是 AI 解釋文，而要有固定格式。

```md
# 檔案名稱

## 1. 原始碼位置

## 2. 這個檔案負責什麼

## 3. 核心流程

## 4. 關鍵源碼事實

## 5. 與其他檔案的關係

## 6. 可以學走的設計

## 7. 我自己的理解

## 8. 待驗證問題

## 9. 後續要回查的檔案
```

如果一篇筆記沒有「原始碼位置」與「關鍵源碼事實」，就不算合格的源碼筆記。

```txt
7. 與父子元件的關係
8. 與 src/components/index.js / src/index.js 的關係
9. 可學走的設計
10. 待驗證檔案

請把「源碼能確認的事實」和「你的推論」分開寫。
```

### 4.4 讀 Button 的專用模板

```txt
我正在讀 View UI Plus 的 Button 元件：src/components/button/button.vue。

請根據我貼上的源碼分析：
```

## 5. AI 回答的驗收標準

一個合格的 AI 讀碼回答，必須符合下面 8 條。

| 標準 | 合格回答應該做到 | 不合格回答常見樣子 |
|---|---|---|
| 有源碼位置 | 明確指出 `src/index.js`、`button.vue` 等檔案 | 只說「這個元件」 |
| 有具體符號 | 提到 `install`、`ViewUI`、`directives`、`classes`、`tagName` 等 | 只說「處理邏輯」 |
| 有流程 | 能用箭頭串起執行順序 | 只有概念段落 |
| 有關聯 | 能說明 import/export、上下游 | 只看單一片段 |
| 有邊界 | 能標出無法確認的地方 | 把推論講成事實 |
| 有驗證 | 會提醒看 examples / types / styles | 完全不提如何驗證 |
| 有抽象 | 能整理可學走的設計 | 只翻譯程式碼 |
| 有限制 | 能說哪些設計不一定適合搬到業務專案 | 所有設計都說很好 |

簡單判斷法：

```txt
如果 AI 的回答拿掉「View UI Plus」四個字後，還可以套用到任何 Vue 元件庫，
那這個回答通常不夠貼近源碼。
```

## 6. 防止 AI 幻覺的 10 條規則

### 規則 1：沒有貼源碼，就不要要求 AI 下定論

可以問：

```txt
請告訴我需要看哪些檔案。
```

不要問：

```txt
請直接告訴我 Button 完整設計。
```

---

### 規則 2：要求 AI 區分「事實 / 推論 / 待驗證」

固定加這句：

```txt
請把回答分成「源碼事實」「你的推論」「待驗證問題」。
```

這可以有效避免 AI 把一般經驗說成 View UI Plus 的真實設計。

---

### 規則 3：要求 AI 指出依據

固定加這句：

```txt
每個重要結論都請指出對應的變數、函式、props、computed、methods、render 區塊或 import/export。
```

---

### 規則 4：不要只貼片段，要補上下文

錯誤做法：

```txt
這段 code 是什麼意思？
```

正確做法：

```txt
這段 code 來自 View UI Plus 的 button.vue，位於 computed.classes。
請解釋它如何根據 props 組裝 class，並指出每個 class 對應哪個 props。
```

---

### 規則 5：遇到「好像」時，立刻要求回源碼

如果 AI 說：

```txt
這裡可能是為了支援表單驗證。
```

你要追問：

```txt
請指出源碼中哪個 import、mixin、inject、emit 或 method 支援這個判斷。
如果目前無法確認，請列為待驗證，並告訴我下一個要看的檔案。
```

---

### 規則 6：不要讓 AI 直接重構原始碼

讀碼階段不要一開始就問：

```txt
請幫我改成 Composition API。
```

應該先問：

```txt
請先說明原始碼為什麼這樣設計。
再另外列出如果改成 Composition API，哪些行為必須保持不變。
```

否則 AI 可能會產生看起來現代、但破壞原本 API 的版本。

---

### 規則 7：examples 是驗證 API 的重要來源

元件源碼告訴你「它怎麼實作」，examples 告訴你「它預期怎麼被使用」。

讀任何元件時都要問：

```txt
這個元件有沒有 examples？
examples 展示了哪些 props / slots / events？
examples 是否驗證了我對源碼的理解？
```

---

### 規則 8：types 是對外 API 的契約，不等於完整實作

型別檔很重要，但不能只看型別檔。

正確問法：

```txt
請對照 types 與元件源碼，區分「型別宣告的 API」與「源碼實際實作的行為」。
```

---

### 規則 9：styles 必須和 class 組裝一起看

很多 UI 元件的 props 最後不是改 JS 行為，而是改 class。

例如 Button 的 `type`、`size`、`long`、`loading`、`ghost` 這類狀態，就應該同時看：

```txt
button.vue 的 computed.classes
  ↕
styles 中 Button 對應的 Less class
```

---

### 規則 10：把沒看完的東西寫進問題索引

AI 協助讀碼最大的價值之一，是幫你保留未完成線索。

每次讀完都要產出：

```md
## 待驗證問題

- [ ] mixinsForm 的 itemDisabled 來源是什麼？
- [ ] Button 的 types 是否完整描述所有 props？
- [ ] examples 是否展示 loading + icon 同時存在的情境？
- [ ] Button 樣式檔中是否有 ivu-btn-icon-only？
```

這些問題要同步放到 `99_問題回查索引.md`。

## 7. 常見錯誤問法與修正版

| 錯誤問法 | 問題 | 修正版 |
|---|---|---|
| 請解釋 View UI Plus | 範圍太大 | 我正在讀 `src/index.js`，請解釋 `install(app, opts)` 流程 |
| Button 怎麼寫？ | 沒有範圍 | 我貼上 `button.vue` script，請分析 props / computed / render |
| 這是不是設計模式？ | 太早抽象 | 請先根據源碼整理它解決了什麼重複問題 |
| 幫我重構成 Composition API | 容易偏離原 API | 請先列出原實作必須保持的行為，再給簡化仿寫版 |
| 這段好不好？ | 會變主觀評論 | 請從可維護性、API 穩定性、元件庫需求三個角度分析，並指出源碼依據 |
| 幫我整理筆記 | 太空泛 | 請依照「位置、角色、流程、關聯、可學走、待驗證」整理 |
| 這裡為什麼這樣寫？ | 缺上下文 | 這段來自 `computed.classes`，請說明它如何把 props 映射成 class |
| 它和 Element Plus 一樣嗎？ | 容易跳出源碼 | 先只分析 View UI Plus 源碼；比較留到最後，且只比較已確認設計 |

## 8. 用 AI 讀 Button 的完整示範流程

### 8.1 第一次提問：定位

```txt
我正在讀 View UI Plus 的 Button 元件。
請根據我貼上的 button.vue script 區塊，先整理源碼事實：
1. import 了什麼
2. name / mixins / components / emits 是什麼
3. props 有哪些
4. computed 有哪些
5. methods 有哪些
6. render 做了什麼
7. 哪些地方需要補其他檔案才能確認
```

預期 AI 要回答出：

```txt
- import Icon
- import oneOf
- import mixinsLink
- import mixinsForm
- name: 'Button'
- mixins: [mixinsLink, mixinsForm]
- emits: ['click']
- props 包含 type / shape / size / loading / disabled / htmlType / icon / customIcon / long / ghost
- computed 包含 showSlot / classes / isHrefPattern / tagName / tagProps
- methods 包含 handleClickLink
- render 會根據 tagName 建立 button 或 a，並組裝 Icon / slot / class / disabled / onClick / tagProps
```

---

### 8.2 第二次提問：流程

```txt
請把 Button 從 props 到 render 的流程畫成文字流程圖。
要求每個節點都對應到源碼中的 props、computed、method 或 render 區塊。
```

預期流程：

```txt
props 傳入 type / size / loading / icon / customIcon / long / ghost / to / htmlType
  ↓
computed.showSlot 判斷是否有 default slot
  ↓
computed.classes 根據 props 組出 ivu-btn 相關 class
  ↓
computed.isHrefPattern 根據 to 判斷是否是連結型按鈕
  ↓
computed.tagName 決定渲染 button 或 a
  ↓
computed.tagProps 決定 type 或 href / target
  ↓
render 根據 loading / icon / customIcon / slot 組裝 children
  ↓
render 回傳 h(tag, { class, disabled, onClick, ...tagProps }, slots)
```

---

### 8.3 第三次提問：關聯

```txt
請根據 Button 的 import/export，幫我列出還要補看的檔案。
請分成「必看」「可延後」「只需要查一次」。
```

預期分類：

| 分類 | 檔案 | 理由 |
|---|---|---|
| 必看 | `src/mixins/link.js` | Button 的跳轉行為依賴它 |
| 必看 | `src/mixins/form.js` | Button 的 disabled / form 關聯可能依賴它 |
| 必看 | `src/components/button/index.js` | 確認 Button 如何被單元件匯出 |
| 必看 | `src/components/index.js` | 確認 Button 如何進入 public component surface |
| 必看 | `src/index.js` | 確認 Button 如何被全域註冊 |
| 可延後 | Button 樣式檔 | 驗證 classes 對應的視覺狀態 |
| 可延後 | Button types | 驗證對外 API 契約 |
| 只查一次 | `utils/assist` | 確認 `oneOf` 的實作 |

---

### 8.4 第四次提問：抽象設計

```txt
請根據 Button 的源碼，整理可以學走的設計。
請分成：
1. 適合公司業務專案直接學的
2. 適合元件庫開發時學的
3. 不建議盲目照搬的
```

預期方向：

| 分類 | 設計 | 說明 |
|---|---|---|
| 業務專案可學 | props validator | 限制固定值，避免 class 狀態失控 |
| 業務專案可學 | computed 統一組裝 class | 避免 template 裡充滿條件判斷 |
| 業務專案可學 | loading / icon / slot 狀態互斥 | 提升 UI 狀態一致性 |
| 元件庫可學 | mixin 抽 link/form 共用能力 | 多元件共享跳轉與表單狀態 |
| 元件庫可學 | 從 `$VIEWUI.size` 讀全域配置 | 讓整套元件支援全域尺寸設定 |
| 不建議盲搬 | 大量全域配置 | 小專案容易造成隱式依賴 |
| 不建議盲搬 | render function 全部手寫 | 對一般業務元件可讀性未必較好 |

以下情況要先自己看，不要立刻丟給 AI。

```md
## 回寫狀態
```

## 13. 最短可執行路線

如果你今天只想開始，不想設計太多流程，就照下面做。

```txt
1. 開 `src/index.js`
2. 貼給 AI：import、directives、ViewUI、install
3. 問：app.use(ViewUIPlus) 做了什麼？
4. 要求 AI 輸出流程圖與源碼事實
5. 自己核對一次 `app.component`、`app.directive`、`globalProperties`
6. 整理成 `src_index_install流程.md`
7. 把不懂的 `$Message`、`localeFile`、`directives` 寫進問題索引
8. 下一篇讀 `src/components/index.js`
9. 再下一篇讀 `src/components/button/button.vue`
```

這條路線最符合 View UI Plus 的真實源碼結構：

```txt
入口安裝
  → 對外元件清單
  → 第一個基礎元件
  → 共用 mixins / utils
  → styles / types / examples 驗證
```

## 14. 本篇給你的執行規則

之後你用 AI 讀 View UI Plus，請固定遵守下面規則：

```txt
1. 每次只讀一個明確目標。
2. 問 AI 前先寫上下文卡片。
3. 貼最小可解釋源碼，不要一次塞整個專案。
4. 第一輪只要源碼事實，不要急著要設計思想。
5. 第二輪要流程圖。
6. 第三輪要上下游關係。
7. 第四輪才要可學走的設計。
8. 所有推論都要標成待驗證。
9. examples、types、styles 是驗證來源，不是附屬資料。
10. 每次讀完都要更新覆蓋表與問題索引。
```

最後總結：

> **AI 不是幫你「跳過讀碼」，而是幫你把讀碼過程變成可拆解、可驗證、可回查、可累積的系統。**
