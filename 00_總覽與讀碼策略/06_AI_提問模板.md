# 06_AI 提問模板

> 本篇任務：建立一套專門用來讀 **View UI Plus 原始碼** 的 AI 提問模板庫。  
> 它不是普通的 prompt 清單，而是要讓 AI 的回答能回扣到 `src/index.js`、`src/components/index.js`、`button.vue`、`mixins`、`directives`、`styles`、`types`、`examples` 等真實源碼位置。

---

## 0. 本篇結論

讀 View UI Plus 時，最好的 AI 提問方式不是：

```txt
請解釋 View UI Plus。
請解釋 Button。
請幫我整理元件庫架構。
```

這種問法太大，AI 很容易用一般 Vue 元件庫經驗回答，而不是根據 View UI Plus 的真實源碼回答。

更好的問法是：

```txt
我正在讀 View UI Plus 的 src/components/button/button.vue。
請只根據我貼上的源碼回答，不要補不存在的設計。
請按照「檔案角色、import 依賴、props、computed、methods、render 流程、與 components/index.js / src/index.js 的關係、可學走的設計」整理。
每個結論都要指出對應的變數、函式、props、computed、method、import/export 或 class name。
無法從源碼確認的地方請標記為「待驗證」。
```

本篇的核心原則：

```txt
先給上下文，再貼最小源碼；
先要求證據，再要求解釋；
先做源碼事實，再做抽象總結；
先確認關聯，再產出筆記。
```

---

## 1. 使用本篇模板前的固定規則

### 1.1 每次提問都要帶的 5 個欄位

每次問 AI 之前，先補上這張「上下文卡片」。

```txt
【專案】View UI Plus
【目前目標】我正在讀哪個檔案 / 哪個元件 / 哪個流程
【源碼位置】例如 src/components/button/button.vue
【我已知背景】例如這是 Vue 3 元件庫，不是業務系統
【希望輸出】例如流程圖、表格、筆記、問題清單、仿寫練習
```

範例：

```txt
【專案】View UI Plus
【目前目標】讀懂 Button 元件的核心設計
【源碼位置】src/components/button/button.vue
【我已知背景】Button 最後會經由 src/components/index.js 匯出，再由 src/index.js 全域註冊
【希望輸出】請整理成「源碼事實 / 設計意圖 / 可學走的寫法 / 待驗證問題」
```

---

### 1.2 每次都要要求 AI 遵守的回答規則

這段可以直接貼在每個 prompt 最後：

```txt
回答限制：
1. 請只根據我提供的源碼與檔案路徑回答。
2. 不要把一般 Vue 元件庫經驗直接套進 View UI Plus。
3. 每個結論都要指出對應的 import、export、props、computed、methods、render、class name、變數或函式。
4. 無法從源碼確認的內容，請標記為「待驗證」。
5. 請區分「源碼事實」、「合理推論」、「待驗證」。
6. 不要只逐行翻譯，請說明它在整個元件庫中的角色。
7. 若需要其他檔案才能確認，請列出下一步應該看的檔案。
```

這段是本篇所有模板的「安全護欄」。

---

### 1.3 不建議使用的問法

| 不建議問法 | 問題 |
|---|---|
| `請解釋這個元件` | 太空泛，容易變成 Vue 通用解釋 |
| `這個元件設計得好嗎？` | 沒有評分標準，AI 容易主觀評論 |
| `幫我整理成筆記` | 還沒先拆事實，容易產生漂亮但空泛的筆記 |
| `View UI Plus 架構是什麼？` | 範圍太大，容易漏掉入口、匯出、型別、樣式、examples |
| `請幫我重構這段程式` | 讀碼階段不該太早重構，容易離開原始碼 |

## 3. T00：建立上下文卡片模板

### 使用時機

每一次開始讀新檔案之前都先用。

### 模板

```txt
我正在讀 View UI Plus 原始碼，請先不要急著解釋程式碼。

【專案定位】
View UI Plus 是基於 Vue 3 的 UI 元件庫，不是一般業務系統。

【本次讀碼目標】
我要讀懂：{這次要讀的檔案 / 元件 / 流程}

【源碼位置】
{例如 src/components/button/button.vue}

【我已經知道的上下文】
{例如：這個元件會被 src/components/index.js 匯出，最後由 src/index.js 安裝}

【我貼給你的源碼範圍】
{例如：完整檔案 / 只貼 props / 只貼 render / 只貼 import/export}

【請你輸出】
1. 這份源碼在 View UI Plus 中的角色
2. 它依賴哪些檔案
3. 它對外提供什麼能力
4. 下一步應該補看哪些檔案
5. 哪些結論目前還不能確認

回答限制：
- 只能根據我貼的源碼回答。
- 無法確認請標記「待驗證」。
- 不要直接套用一般 Vue 元件庫的經驗。
```

### 好輸出的判斷標準

AI 的回答至少應該包含：

```txt
檔案角色
依賴關係
對外能力
下一步源碼
待驗證項目
```

如果 AI 只講「這是一個 Vue 元件」，代表 prompt 還不夠精準。

## 4. T01：解釋單一檔案模板

### 使用時機

適合第一次讀某個檔案，例如：

```txt
src/index.js
src/components/index.js
src/components/button/button.vue
src/mixins/form.js
src/directives/resize.js
src/locale/index.js
```

### 模板

```txt
請幫我分析 View UI Plus 的這個檔案：

【源碼位置】
{貼檔案路徑}

【源碼內容】
{貼源碼}

請按照以下格式回答：

## 1. 檔案定位
- 這個檔案在 View UI Plus 中負責什麼？
- 它是入口檔、元件檔、共用工具、樣式、型別、demo，還是建置設定？

## 2. 對外輸入
- 它接收哪些 props、options、參數、import、全域設定？

## 3. 對外輸出
- 它 export 什麼？
- 它註冊什麼？
- 它掛載什麼？
- 它影響哪些 UI 行為？

## 4. 核心流程
請用條列流程說明，不要只逐行翻譯。

## 5. 關聯檔案
請根據 import/export 或明確線索列出它和哪些檔案有關。

## 6. 可學走的設計
請說明這個檔案有哪些設計可以帶回自己的 Vue / Java 全端專案。

## 7. 待驗證問題
請列出需要繼續看其他檔案才能確認的問題。

回答限制：
- 每個結論都要回扣到源碼中的變數、函式、import/export、props、computed、methods 或 class name。
- 無法確認請標記「待驗證」。
```

### 適合產出的筆記標題

```txt
src_index_js_插件入口分析.md
button_vue_檔案角色分析.md
mixins_form_js_共用表單狀態分析.md
```

## 5. T02：讀 `src/index.js` 入口安裝流程模板

### 使用時機

當你要讀懂：

```txt
app.use(ViewUIPlus)
install(app, opts)
全域元件註冊
全域 directive 註冊
$VIEWUI
$Message / $Modal / $Notice / $Spin / $Loading
```

### 模板

```txt
我正在讀 View UI Plus 的 src/index.js。
目標是看懂 app.use(ViewUIPlus) 背後到底做了什麼。

【已知上下文】
- package.json 的 main 指向 dist/viewuiplus.min.js
- vite.config.js 的 library build entry 指向 src/index.js
- examples/main.js 會 import ViewUIPlus from '../src/index' 並 app.use(ViewUIPlus)

【源碼】
{貼 src/index.js}

請按照以下格式回答：

## 1. src/index.js 在整個專案中的定位
請說明它為什麼是元件庫入口，而不是普通工具檔。

## 2. install(app, opts) 的完整流程
請用流程圖文字描述：
使用者 app.use(ViewUIPlus) 之後，Vue 會怎麼呼叫 install？install 裡面依序做了什麼？

## 3. components 的註冊方式
請說明：
- components 從哪裡來？
- ViewUI 物件為什麼要展開 components？
- iButton、iTable、iInput 這類別名的目的可能是什麼？

## 4. directives 的註冊方式
請說明 directives 物件包含哪些 directive，以及怎麼被 app.directive 註冊。

## 5. $VIEWUI 全域設定
請整理 $VIEWUI 下面有哪些設定區塊，例如 size、capture、transfer、cell、menu、modal、tabs、select 等。
請說明哪些元件可能會讀取這些設定。

## 6. 全域命令式 API
請整理 app.config.globalProperties 上掛了哪些 API，例如 $Message、$Modal、$Notice 等。
請區分它們和普通元件註冊的差異。

## 7. 對後續讀碼的影響
讀完 src/index.js 之後，下一步應該看哪些檔案？

回答限制：
- 請只根據源碼回答。
- 不要假設沒有出現在 src/index.js 的全域 API。
- 無法從本檔確認的地方請標記「待驗證」。
```

### 這個模板的產出標準

最後應該得到這種流程：

```txt
app.use(ViewUIPlus)
  ↓
Vue 呼叫 install(app, opts)
  ↓
處理 opts.locale / opts.i18n
  ↓
Object.keys(ViewUI).forEach(app.component)
  ↓
Object.keys(directives).forEach(app.directive)
  ↓
設定 app.config.globalProperties.$VIEWUI
  ↓
掛載 $Spin / $Loading / $Message / $Notice / $Modal 等命令式 API
```

## 6. T03：讀 `src/components/index.js` 元件匯出總表模板

### 使用時機

當你要整理 View UI Plus 對外公開了哪些元件。

### 模板

```txt
我正在讀 View UI Plus 的 src/components/index.js。
目標是看懂這個檔案如何定義元件庫的 public component surface。

【源碼】
{貼 src/components/index.js}

請按照以下格式回答：

## 1. 這個檔案的角色
請說明它如何集中 export 對外元件。

## 2. 匯出清單分類
請把 export 的元件粗略分類，例如：
- 基礎元件
- 表單元件
- 導航元件
- 回饋元件
- 資料展示元件
- 佈局元件
- 命令式 / 工具型能力
- 業務型 / Pro 型元件

## 3. 命名規則觀察
請觀察元件資料夾命名與 export name 的關係。
例如 button -> Button、button-group -> ButtonGroup。

## 4. 和 src/index.js 的關係
請說明 src/index.js import * as components from './components' 之後，如何使用這些 export。

## 5. 後續讀碼順序建議
請根據元件複雜度，推薦第一輪、第二輪、第三輪應該讀哪些元件。

回答限制：
- 分類可以是合理推論，但必須標記為「分類推論」。
- 不要宣稱某元件內部行為，除非已經看過該元件原始碼。
- 請列出最適合新手先讀的 5 個元件，以及不建議一開始讀的 5 個元件。
```

### 這個模板的重點

`components/index.js` 不只是清單，它代表元件庫的對外入口之一。

你要讓 AI 幫你看出：

```txt
單一元件資料夾
  ↓
元件 index.js
  ↓
src/components/index.js
  ↓
src/index.js
  ↓
app.component(...)
  ↓
使用者可在模板中使用
```

## 9. T06：讀 render function 模板

### 使用時機

View UI Plus 有些元件使用 render function，例如 Button。當你看到：

```js
render () {
  return h(...)
}
```

可以使用這個模板。

### 模板

```txt
我正在讀 View UI Plus 的 render function。
請不要只逐行翻譯，請幫我還原它最後產生的 DOM 與 UI 行為。

【元件】
{元件名稱}

【render 相關源碼】
{貼 render / computed / methods / props 相關源碼}

請按照以下格式回答：

## 1. render 的輸入
它依賴哪些 props、computed、methods、slots？

## 2. render 的分支條件
請列出所有 if / 條件判斷，例如 loading、icon、slot、tagName。

## 3. 最終 DOM 結構
請用偽 HTML 表示不同狀態下會渲染出什麼：
- 一般狀態
- loading 狀態
- icon 狀態
- 有 default slot 狀態
- link 狀態

## 4. 事件綁定
請說明 onClick 等事件最後連到哪個 method。

## 5. class / attrs / props 綁定
請說明 class、disabled、href、target、type 等屬性如何被組裝。
```

## 24. 對話流程建議：一個檔案至少問 4 輪

### 第 1 輪：定位

```txt
這個檔案在專案中負責什麼？
```

### 第 2 輪：拆解

```txt
請拆 props / computed / methods / render / import / export。
```

### 第 3 輪：關聯

```txt
請追它和其他檔案的關係。
```

### 第 4 輪：沉澱

```txt
請根據已確認事實產出正式筆記與回查索引。
```

不要第一輪就要求「幫我產生完整筆記」。那樣會讓筆記看起來完整，但其實缺少源碼證據。

## 25. 模板品質自評表

你每次問完 AI 後，可以用這張表檢查回答品質。

| 檢查項 | 合格標準 | 是否通過 |
|---|---|---|
| 有沒有指出檔案路徑 | 至少出現目前源碼位置 | □ |
| 有沒有指出具體源碼依據 | props、method、computed、import、class 等 | □ |
| 有沒有區分事實與推論 | 明確標記「源碼事實 / 合理推論 / 待驗證」 | □ |
| 有沒有說明關聯檔案 | 至少指出下一步應看檔案 | □ |
| 有沒有避免通用回答 | 沒有只講 Vue 常識 | □ |
| 有沒有產出可放進筆記的內容 | 表格、流程、清單、回查問題 | □ |
| 有沒有列出待驗證問題 | 至少 1～3 條 | □ |

若低於 5 項通過，代表 prompt 要重問。

## 26. 最推薦的實際使用順序

如果你接下來要正式讀 View UI Plus，建議照這個順序使用模板：

```txt
第一天：src/index.js
T00 → T02 → T14 → T16 → T17
```

逐步建立起來，而不是散亂地看單一元件。

## 28. 最終原則

用 AI 讀 View UI Plus 時，真正重要的不是「讓 AI 講很多」，而是讓 AI 幫你做到：

```txt
把源碼定位清楚；
把流程拆清楚；
把關聯檔案串清楚；
把 API 契約整理清楚；
把可學走的設計萃取清楚；
把不能確認的地方標清楚。
```

所以每次提問都要盯住這三件事：

```txt
有沒有源碼依據？
有沒有檔案關聯？
有沒有可沉澱成筆記？
```

只要做到這三點，AI 就不是在替你「幻想原始碼」，而是在幫你建立一套可以長期回查的 View UI Plus 源碼知識庫。
