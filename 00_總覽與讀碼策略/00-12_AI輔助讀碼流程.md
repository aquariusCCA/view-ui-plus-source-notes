# 00-12_AI輔助讀碼流程

> 所屬目錄：`00_總覽與讀碼策略`  
> 筆記定位：建立一套「用 AI 協助閱讀 View UI Plus 源碼」的標準作業流程  
> 適用範圍：View UI Plus、Vue 3 元件庫、企業後台元件封裝、開源專案讀碼  
> 筆記等級：9.5 分版本  
> 核心原則：**源碼是第一事實來源，官方文件與 Demo 是第二事實來源，AI 是輔助推理工具，不是最終答案。**

---

## 1. 這篇筆記的目的

這篇筆記不是教你「問 AI 問題」而已，而是要建立一套可以長期複用的 **AI 輔助讀碼流程**。

學習 View UI Plus 時，AI 最有價值的地方不是直接告訴你「這段程式碼是什麼意思」，而是協助你完成以下工作：

1. 快速建立源碼閱讀地圖。
2. 把零散檔案串成完整流程。
3. 反推出元件 API 設計意圖。
4. 找出元件背後的設計模式。
5. 把開源元件庫設計轉化成自己的企業後台封裝能力。
6. 幫你生成筆記模板、檢核清單、仿寫任務。
7. 協助你從「看懂」提升到「能仿寫、能重構、能應用」。

但是 AI 也有風險：

1. 它可能根據常見 Vue 元件庫經驗猜測。
2. 它可能把 Element Plus、Ant Design Vue、Naive UI 的設計混進 View UI Plus。
3. 它可能在沒有看到源碼的情況下做過度推論。
4. 它可能忽略版本差異。
5. 它可能用漂亮的文字包裝錯誤理解。

所以這篇的核心目標是：

```txt
建立一套「AI 協助，但由源碼驗證」的讀碼流程。
```

---

## 2. AI 在讀碼中的正確定位

### 2.1 AI 不是答案機，而是讀碼助手

讀 View UI Plus 時，AI 的角色應該是：

| 角色 | 說明 |
|---|---|
| 導讀者 | 幫你建立閱讀順序 |
| 解釋者 | 幫你拆解陌生程式碼 |
| 追問者 | 幫你提出應該觀察的問題 |
| 整理者 | 幫你把理解轉成結構化筆記 |
| Reviewer | 幫你檢查理解是否有漏洞 |
| 教練 | 幫你安排仿寫與重構任務 |

AI 不應該是：

| 不該扮演的角色 | 問題 |
|---|---|
| 權威來源 | AI 說的必須回到源碼驗證 |
| 代替你讀碼的人 | 你不自己追蹤源碼，就無法形成能力 |
| 單純摘要工具 | 只摘要不追流程，學不到架構 |
| 自動筆記生成器 | 筆記必須有你自己的理解與判斷 |
| 替代練習的人 | 只讀不仿寫，最後仍然不會設計元件 |

---

## 3. AI 輔助讀碼的四層資料來源

讀碼時要把資料來源分成四層，不要混在一起。

```txt
第一層：View UI Plus 源碼
第二層：官方文件 / Demo / examples / test
第三層：AI 分析與推理
第四層：你自己的筆記與仿寫成果
```

### 3.1 第一層：源碼

源碼是最重要的事實來源。

你要優先確認：

1. 入口檔在哪裡。
2. 元件主檔在哪裡。
3. 樣式檔在哪裡。
4. 型別宣告在哪裡。
5. 是否有測試。
6. 是否有 examples。
7. 是否有 utils / mixins / directives 依賴。
8. 是否有全域配置依賴。
9. 是否有父子元件協作。

如果 AI 的回答和源碼不一致，以源碼為準。

---

### 3.2 第二層：官方文件 / Demo / examples / test

官方文件與 Demo 可以幫你理解「使用者視角」。

它們適合回答：

1. 使用者怎麼使用這個元件？
2. 常見 props 是什麼？
3. 事件怎麼觸發？
4. slots 怎麼暴露？
5. 哪些行為是公開 API？
6. 哪些情境是官方預期支援的？

test 則可以反推出「元件規格」。

如果 test 裡面測了某個行為，代表這個行為通常是設計者認為重要的規格。

---

### 3.3 第三層：AI 分析與推理

AI 適合做：

1. 幫你把源碼翻譯成白話。
2. 幫你整理成流程圖。
3. 幫你將 props 分類。
4. 幫你推導父子元件協作方式。
5. 幫你反推出設計模式。
6. 幫你設計仿寫任務。
7. 幫你檢查筆記是否缺少重點。

AI 不適合單獨做：

1. 判斷真實源碼行為。
2. 猜測沒有提供的檔案內容。
3. 判定版本差異。
4. 替你確認 bug。
5. 直接產生最終筆記而不經驗證。

---

### 3.4 第四層：你的筆記與仿寫成果

最終產出不是 AI 回答，而是你的筆記。

高品質筆記應該包含：

1. 源碼位置。
2. 使用範例。
3. API 設計。
4. 狀態流。
5. 事件流。
6. 父子協作。
7. 樣式系統。
8. 型別設計。
9. 測試反推。
10. 可仿寫任務。
11. 企業後台應用場景。
12. 你自己的理解。

## 4. AI 輔助讀碼總流程

建議固定採用以下流程：

```txt
Step 1：選定閱讀目標
Step 2：收集源碼上下文
Step 3：建立源碼地圖
Step 4：請 AI 做第一輪導讀
Step 5：自己追一次流程
Step 6：請 AI 協助拆解 API / 狀態 / 事件
Step 7：回源碼驗證 AI 說法
Step 8：請 AI 幫你反推出設計模式
Step 9：整理成筆記
Step 10：請 AI 幫你審查筆記
Step 11：產出仿寫任務
Step 12：實作迷你版並回頭修正筆記
```

這套流程要反覆使用。

不要每次都臨時亂問，否則你的筆記風格會不一致，回查價值會下降。

---

## 5. Step 1：選定閱讀目標

每次讀碼前，先明確定義本次目標。

不要問：

```txt
幫我看一下 View UI Plus 的 Form。
```

要改成：

```txt
我這次要閱讀 View UI Plus 的 Form / FormItem 協作機制。
我的目標是理解：
1. Form 如何收集 FormItem
2. FormItem 如何接收 prop / rules
3. validate / resetFields / clearValidate 如何設計
4. 表單驗證如何和 async-validator 串接
5. 這套設計如何轉化成企業後台 SearchForm / DialogForm
```

這樣 AI 才知道你要的是「架構理解」，不是淺層摘要。

---

## 6. Step 2：收集源碼上下文

向 AI 提問前，先準備足夠上下文。

### 6.1 單一元件最小上下文

閱讀一個元件時，至少提供：

```txt
1. 元件名稱
2. View UI Plus 版本
3. 官方使用範例
4. 元件主檔路徑
5. 相關子元件路徑
6. 樣式檔路徑
7. 型別檔路徑
8. examples / test 路徑
9. 你目前卡住的問題
```

### 6.2 橫向機制最小上下文

閱讀橫向機制時，至少提供：

```txt
1. 機制名稱
2. 使用者如何觸發這個機制
3. 入口檔
4. 核心資料結構
5. 涉及元件列表
6. 涉及工具函式
7. 是否涉及 globalProperties
8. 是否涉及 provide / inject
9. 是否涉及 mixins / directives
10. 你希望最後產出的筆記格式
```

```txt
3. 哪些檔案一定要精讀
4. 哪些檔案可以略讀
5. 哪些檔案需要搭配 examples / test 反推
6. 目前還缺少哪些上下文
7. 不能根據現有資訊直接下結論的地方
```

## 8. Step 4：請 AI 做第一輪導讀

第一輪導讀的目的是「建立理解框架」，不是細節定案。

### 8.1 第一輪導讀 Prompt

```txt
我正在閱讀 View UI Plus 的【元件名稱】元件。

我的目標不是只會使用它，而是理解它作為 Vue 3 元件庫元件的設計方式。

請你根據我提供的源碼，先做第一輪導讀：

1. 這個元件的定位是什麼？
2. 它解決什麼 UI / 互動問題？
3. 它對外暴露哪些 API？
4. 它內部可能有哪些狀態？
5. 它和其他元件有什麼協作？
6. 它背後可能涉及哪些設計模式？
7. 後續精讀時應該優先看哪些地方？
8. 哪些判斷需要我回源碼確認？

請注意：
- 不要補充沒有出現在源碼中的功能。
- 如果你是推測，請標記「推測」。
- 每個重要結論請盡量對應到具體程式碼位置。
```

## 9. Step 5：自己追一次流程

AI 導讀後，不要直接把回答貼進筆記。

你要自己追一次流程。

建議用這個順序：

```txt
1. 從官方 Demo 看使用方式
2. 從元件入口看 export
3. 從 props 看 API
4. 從 emits 看事件
5. 從 template 看 DOM 結構
6. 從 data / setup / computed 看狀態
7. 從 methods / watch 看行為
8. 從 imports 看依賴
9. 從 styles 看 class 與 prefixCls
10. 從 examples / test 看規格
```

追完後，把你的理解再丟給 AI 檢查。

---

## 10. Step 6：請 AI 拆解 API 設計

API 設計是讀元件庫最重要的部分之一。

不要只記「有哪些 props」，而是要問：

```txt
為什麼這個元件需要這些 props？
哪些是樣式控制？
哪些是狀態控制？
哪些是行為控制？
哪些是資料控制？
哪些是擴展點？
```

### 10.1 Props 分類 Prompt

```txt
請根據以下 View UI Plus 元件源碼，幫我整理 props API。

請不要只列表，請按照以下類型分類：

1. 樣式型 props
2. 狀態型 props
3. 行為型 props
4. 資料型 props
5. 表單協作型 props
6. 彈層 / 定位型 props
7. 兼容舊 API 或語意別名 props
8. 可能可以簡化或重構的 props

每個 prop 請整理：

- prop 名稱
- 型別
- 預設值
- 使用者視角的用途
- 內部源碼如何使用
- 是否會影響 class
- 是否會影響狀態
- 是否會觸發事件
- 是否適合企業後台封裝
```

### 10.2 Emits / Events 分析 Prompt

```txt
請根據以下源碼，幫我分析這個元件的事件設計。

請整理：

1. 對外 emits / event 有哪些
2. 每個事件的觸發時機
3. 每個事件的參數
4. 哪些事件和 v-model 有關
5. 哪些事件是使用者互動觸發
6. 哪些事件是內部狀態變更觸發
7. 是否存在事件命名不一致或歷史相容設計
8. 如果我要仿寫這個元件，事件設計應該怎麼簡化
```

### 10.3 Slots 分析 Prompt

```txt
請根據以下源碼，幫我分析 slots 設計。

請整理：

1. 這個元件暴露哪些 slots
2. 每個 slot 的使用情境
3. slot 是增加內容彈性，還是覆蓋預設渲染？
4. slot 是否和 props 形成替代關係？
5. slot 是否有 scoped slot 資料？
6. slot 如何影響元件的可擴展性？
7. 企業後台封裝時，哪些 slot 應該保留？
```

## 11. Step 7：拆解狀態流與事件流

讀元件庫時，真正的重點不是「程式碼每行做什麼」，而是狀態如何流動。

### 11.1 狀態流分析 Prompt

```txt
請根據以下源碼，幫我畫出這個元件的狀態流。

請整理：

1. 外部傳入狀態：props / v-model
2. 內部維護狀態：data / ref / reactive
3. 衍生狀態：computed
4. 監聽狀態：watch
5. 狀態變更入口：methods / event handlers
6. 狀態輸出口：emit / callback / class / DOM
7. 狀態同步策略：受控、非受控或混合
8. 可能的邊界情況
9. 如果改用 Composition API，狀態可以怎麼重構
```

### 11.2 事件流分析 Prompt

```txt
請根據以下源碼，幫我整理事件流。

請用以下格式輸出：

使用者操作
→ 觸發 DOM event
→ 呼叫內部 method
→ 修改內部狀態
→ emit 對外事件
→ 父層更新 props / v-model
→ 元件重新渲染

請同時指出：
1. 哪些流程是同步的
2. 哪些流程可能是非同步的
3. 哪些流程會被 debounce / throttle / nextTick 影響
4. 哪些流程需要測試案例驗證
```

## 12. Step 8：拆解父子協作與跨元件機制

View UI Plus 這類元件庫中，很多真正有價值的設計不在單一元件，而在父子協作。

常見例子：

1. `Form` / `FormItem`
2. `Select` / `Option`
3. `Menu` / `MenuItem` / `Submenu`
4. `Dropdown` / `DropdownMenu` / `DropdownItem`
5. `Table` / column config
6. `Tabs` / `TabPane`
7. `CheckboxGroup` / `Checkbox`
8. `RadioGroup` / `Radio`
9. `Collapse` / `Panel`
10. `Steps` / `Step`

### 12.1 父子協作分析 Prompt

```txt
我正在分析 View UI Plus 的【父元件】與【子元件】協作。

請幫我根據源碼分析：

1. 父元件提供了什麼上下文？
2. 子元件如何取得父元件資料？
3. 使用的是 props 傳遞、provide/inject、事件回傳、還是 mixin？
4. 子元件是否會向父元件註冊自己？
5. 父元件是否會管理子元件集合？
6. 父元件是否會統一觸發子元件方法？
7. 子元件銷毀時是否需要解除註冊？
8. 這套設計的優點是什麼？
9. 這套設計在企業後台元件封裝中可以怎麼應用？
```

### 12.2 Form / FormItem 專題 Prompt

```txt
我正在閱讀 View UI Plus 的 Form / FormItem 驗證系統。

請你幫我分析：

1. Form 如何保存 model / rules
2. FormItem 如何取得自己的 prop
3. FormItem 如何向 Form 註冊
4. validateField / validate / resetFields / clearValidate 的流程
5. async-validator 如何接入
6. 觸發驗證的時機有哪些
7. blur / change trigger 如何設計
8. 錯誤訊息如何儲存與顯示
9. 巢狀欄位如何處理
10. 企業後台 DialogForm / SearchForm 可以怎麼借鑑
```

## 13. Step 9：拆解命令式 API

`Message`、`Notice`、`Modal`、`LoadingBar`、`Spin` 這類元件通常不是單純 template 使用，而是可以透過命令式 API 呼叫。

例如：

```js
this.$Message.success('成功')
this.$Modal.confirm({...})
```

這類設計非常值得學。

### 13.1 命令式 API 分析 Prompt

```txt
我正在閱讀 View UI Plus 的【Message / Notice / Modal / LoadingBar】命令式 API。

請你根據源碼分析：

1. 使用者如何呼叫這個 API？
2. API 是如何掛載到 globalProperties？
3. 內部如何建立 Vue component instance？
4. 是否使用 singleton instance？
5. 是否動態建立 DOM container？
6. 多個 message / modal 如何管理？
7. close / destroy 如何設計？
8. duration / timer 如何處理？
9. z-index / transfer / body append 如何處理？
10. 這種模式和一般元件使用方式有什麼差異？
11. 如果我要做企業後台的 useMessage / useDialog，應該如何簡化？
```

### 13.2 命令式 API 仿寫 Prompt

```txt
請根據我剛剛閱讀的 View UI Plus Message 設計，幫我設計一個 MiniMessage。

需求：

1. 使用 Vue 3
2. 支援 success / error / warning / info
3. 支援 message 文字
4. 支援 duration 自動關閉
5. 支援手動 close
6. 支援多個 message 排列
7. 支援全域呼叫 MiniMessage.success('xxx')
8. 不需要完全照抄 View UI Plus，只保留核心設計

請輸出：
1. 最小架構
2. 檔案結構
3. 核心程式碼
4. 設計說明
5. 可以進一步擴充的地方
```

## 14. Step 10：拆解樣式系統與 prefixCls

讀元件庫不能只看 JS / Vue 檔案，樣式系統也很重要。

你要觀察：

1. class 命名規則。
2. prefixCls 如何產生。
3. props 如何影響 class。
4. 狀態如何反映在 class。
5. size / type / disabled / active / error 如何反映到樣式。
6. Less 變數如何組織。
7. 樣式與元件邏輯是否耦合過高。
8. 企業後台封裝時是否應該保留同樣 class 策略。

### 14.1 樣式系統分析 Prompt

```txt
請根據以下 View UI Plus 元件源碼與樣式檔，幫我分析樣式系統。

請整理：

1. prefixCls 是什麼
2. class 命名規則
3. 哪些 props 會影響 class
4. 哪些狀態會影響 class
5. 樣式是否依賴父層 class
6. 是否有 BEM 類似命名
7. Less 變數如何被使用
8. 樣式和元件邏輯的耦合程度
```

## 22. AI 輔助讀碼的防幻覺規則

### 規則 1：沒有源碼，不下結論

如果沒有提供源碼，只能說：

```txt
根據常見 Vue 元件庫設計，可能是……
但 View UI Plus 實際情況需要回源碼確認。
```

---

### 規則 2：每個重要結論要有依據

正式筆記中，重要結論最好附上：

```txt
來源：
- 檔案路徑
- 函式名稱
- props 名稱
- method 名稱
- class 名稱
- test case 名稱
```

---

### 規則 3：推測要明確標記

筆記中可以保留推測，但要標明：

```txt
【推測】
【待驗證】
【已由源碼確認】
【由 examples 反推】
【由 test 反推】
```

---

### 規則 4：不要一次貼太多源碼

AI 更適合分析「有邊界的上下文」。

建議一次貼：

```txt
1. 一個元件的 props 區塊
2. 一個 method
3. 一段 watch
4. 一個核心流程
5. 一組父子元件互動
```

不要一次貼整個 repo 然後要求它總結。

---

### 規則 5：先問結構，再問細節

不建議一開始就問：

```txt
這行是什麼意思？
```

建議先問：

```txt
這段在整個元件流程中的角色是什麼？
```

理解結構後，再回頭看細節。


### 29.3 每篇都要有「未來回查問題」

```md
## 未來回查問題

1. View UI Plus Button 的 loading 是如何影響 click 行為的？
2. ButtonGroup 如何影響 Button 的 size？
3. Button 的 type 是如何轉成 class 的？
4. 如果我要封裝 PermissionButton，可以借鑑 Button 哪些設計？
```

這樣未來你問 AI：

```txt
幫我找 View UI Plus Button loading 和 click 行為的筆記
```

AI 比較容易定位。

## 30. 常見錯誤

### 錯誤 1：只問「這段程式碼是什麼意思」

這會讓 AI 只做翻譯。

應改問：

```txt
這段程式碼在整個元件流程中扮演什麼角色？
它接收什麼輸入，產生什麼輸出，修改什麼狀態？
```

---

### 錯誤 2：沒有提供檔案路徑

沒有檔案路徑，未來筆記很難回查。

每次提問都應該包含：

```txt
檔案路徑：
元件名稱：
閱讀目標：
目前問題：
```

---

### 錯誤 3：把 AI 推測當成源碼事實

任何沒有源碼依據的內容，都要標記：

```txt
待驗證
```

---

### 錯誤 4：只讀單一檔案

元件庫通常是跨檔案設計。

一個元件通常至少要看：

```txt
1. index / export
2. main component
3. sub components
4. styles
5. types
6. examples
7. tests
8. utils / mixins
```

---

### 錯誤 5：讀完沒有仿寫

只讀不仿寫，很容易以為自己懂了。

每讀完一個核心元件，至少做一個 Mini 版本。

## 33. 本篇總結

AI 可以大幅提升讀 View UI Plus 的效率，但前提是你不能把 AI 當成權威答案。

真正有效的流程是：

```txt
你定義目標
→ 你提供源碼
→ AI 幫你建立理解框架
→ 你回源碼驗證
→ AI 幫你整理與追問
→ 你寫成筆記
→ AI 幫你審查
→ 你做仿寫
```

這樣使用 AI，才不會變成「看似學很快，實際沒有內化」。

對你來說，最重要的不是把 View UI Plus 每個元件都看完，而是透過 AI 和源碼協作，逐步建立以下能力：

```txt
1. 看懂企業級 Vue 元件庫架構
2. 看懂元件 API 設計
3. 看懂狀態與事件流
4. 看懂父子元件協作
5. 看懂全域配置與命令式 API
6. 看懂樣式系統與型別設計
7. 能仿寫迷你元件庫
8. 能把設計轉化成企業後台元件封裝
```

這才是學習 View UI Plus 的真正價值。
