## 所屬章節

`00_總覽與讀碼策略`

## 本節導讀

這篇把 View UI Plus 源碼學習拆成可驗收的階段成果。前面幾篇已經建立了源碼地圖、讀碼路線、讀碼方法、元件分析模板、關鍵概念索引、追蹤檢查表與元件選讀策略；本篇要回答的是：

1. 每個階段讀到什麼程度才算完成？
2. 每個階段應該留下哪些可回查產物？
3. 讀完後應該能回答哪些問題？
4. 讀完後應該具備哪些仿寫或重構能力？
5. 如何判斷目前只是「看過」，還是真的達到 10 / 10？

本篇的核心觀念是：

> 里程碑不是時間表，而是驗收點。  
> 一個階段是否完成，不看花了多久，而看是否已經留下可回查、可驗證、可交接、可延伸的成果。

如果 `00-02_讀碼路線.md` 是「往哪裡走」，`00-06_追蹤檢查表.md` 是「單次讀碼是否合格」，那本篇就是「整輪學習如何分段通關」。

## 1. 里程碑使用方式

每進入一個階段前，先填一張任務卡。它的作用是避免把「讀很多」誤當成「完成」。

```md
## 里程碑任務卡

- 本階段：
- 對應章節：
- 第一入口：
- 本階段邊界：
- 必交產物：
- 必答問題：
- 可仿寫能力：
- 不追的支線：
- 外部證據：
- 本階段分數：
- 下一步：
```

填寫範例：

```md
## 里程碑任務卡

- 本階段：第 2 階段，入口、插件、註冊、全域配置
- 對應章節：06、07、08、09
- 第一入口：ViewUIPlus-SourceCode/src/index.js
- 本階段邊界：只追 install、components 註冊、directives 註冊、globalProperties，不追每個元件內部實作
- 必交產物：install 流程圖、元件註冊鏈、全域 API 清單
- 必答問題：app.use(ViewUIPlus) 後發生什麼事？
- 可仿寫能力：能寫出一個簡化版 Vue 3 元件庫 install 函式
- 不追的支線：每個命令式 API 的完整 service instance 細節
- 外部證據：package.json、vite.config.js、src/index.js、types/index.d.ts、examples/main.js
- 本階段分數：待評分
- 下一步：若達到 9 分以上，進入第 3 階段讀 Button 與 Layout
```

建議每完成一個階段，就把任務卡摘要同步到根目錄的 [`學習追蹤表.md`](../學習追蹤表.md)。

## 2. 滿分 10 分通用標準

本篇採用 `00-06_追蹤檢查表.md` 的分級標準，但把它擴大到整個階段。

| 分數 | 階段狀態 | 判斷標準 |
|---|---|---|
| 10 / 10 | 可交接、可延伸 | 本階段入口、邊界、主流程、源碼證據、外部對照、待驗證、仿寫能力都完整 |
| 9 / 10 | 可回查 | 主流程完整，少量支線或外部證據可後續補 |
| 8 / 10 | 可暫停 | 已能回答本階段核心問題，但待驗證較多 |
| 6 到 7 / 10 | 初步理解 | 有入口與局部流程，但沒有形成完整閉環 |
| 4 到 5 / 10 | 瀏覽階段 | 看過部分章節或檔案，但無法說清整體流程 |
| 3 以下 | 未完成 | 只有印象、檔名或零散摘要 |

滿分 10 分必要條件：

- 已明確列出本階段問題，而不是只寫「讀完某章」。
- 已確認第一入口，且能說明為什麼從這裡開始。
- 已定義本階段邊界，知道哪些支線先不追。
- 已完成主流程圖或流程表。
- 已留下源碼位置或章節位置，不只留下口頭結論。
- 已對照至少一種外部證據：types、examples、tests、dist、build output。
- 已分清已確認、合理推論、待驗證與證據衝突。
- 已留下可回查產物，例如流程圖、索引表、模板、問題卡或仿寫筆記。
- 已能回答本階段必答問題。
- 已能說出下一階段第一入口與停止條件。

## 5. 第 2 階段：入口、插件、註冊與全域配置

第 2 階段回答的是：使用者呼叫 `app.use(ViewUIPlus)` 後，View UI Plus 內部做了什麼。

| 項目 | 內容 |
|---|---|
| 對應章節 | `06_src_入口與插件機制`、`07_元件註冊機制`、`08_全域配置與globalProperties`、`09_元件庫整體設計模式` |
| 第一入口 | [`src/index.js`](../ViewUIPlus-SourceCode/src/index.js) |
| 輔助入口 | [`src/components/index.js`](../ViewUIPlus-SourceCode/src/components/index.js)、[`types/index.d.ts`](../ViewUIPlus-SourceCode/types/index.d.ts)、[`examples/main.js`](../ViewUIPlus-SourceCode/examples/main.js) |
| 核心問題 | install 如何處理 locale、components、directives、globalProperties 與命令式 API？ |
| 必交產物 | `install` 流程圖、元件註冊鏈、globalProperties 清單、插件模式總結 |

### 5.1 本階段主流程

```text
使用者 createApp(...)
  -> app.use(ViewUIPlus, opts)
  -> install(app, opts)
  -> locale 設定
  -> components 全量註冊
  -> directives 全量註冊
  -> app.config.globalProperties.$VIEWUI
  -> $Message / $Notice / $Modal / $Loading / $Date 等全域能力
```

### 5.2 必答問題

完成第 2 階段後，應能回答：

- `ViewUIPlus` 的 default export 和 named exports 各自扮演什麼角色？
- `src/components/index.js` 和 `src/index.js` 的責任差異是什麼？
- 單一元件如何從資料夾進入對外 API？
- `install(app, opts)` 如何處理 `opts.locale`、`opts.i18n`？
- 全域元件註冊與命令式 API 掛載有什麼不同？
- `$VIEWUI` 保存哪些全域配置？
- `types/index.d.ts` 如何補上全域屬性的型別證據？
- examples 如何證明 `app.use(ViewUIPlus)` 是實際使用入口？

### 5.3 10 / 10 驗收

| 檢查項 | 滿分標準 |
|---|---|
| install 入口 | 能指出 `install(app, opts)` 的位置與主流程 |
| 元件註冊 | 能說明 `components/index.js` 到 `app.component` 的鏈路 |
| 指令註冊 | 能說明 directives 如何被收集與註冊 |
| 全域配置 | 能說明 `$VIEWUI` 的來源、用途與可能被誰讀取 |
| 命令式 API | 能分辨普通元件註冊和 `$Message` 這類 service 掛載 |
| 型別對照 | 能對照 globalProperties 相關型別宣告 |
| 使用場景 | 能對照 examples 的 app 初始化 |

### 5.4 可仿寫能力

本階段完成後，應能仿寫：

```js
const components = [Button, Input]

const install = (app, options = {}) => {
  components.forEach((component) => {
    app.component(component.name, component)
  })

  app.config.globalProperties.$MY_UI = {
    size: options.size,
    transfer: options.transfer
  }

  app.config.globalProperties.$Message = Message
}

export default {
  install
}
```

重點不是程式碼長相，而是理解「元件庫入口」通常同時負責註冊、配置與命令式能力掛載。

### 6.1 建議先讀順序

不要讓每個章節只留下散文式筆記。每個階段至少要有一組固定產物。

| 產物 | 用途 | 適用階段 |
|---|---|---|
| 流程圖 | 把入口、處理、輸出串起來 | build、install、v-model、驗證、命令式 API |
| 角色表 | 分清每個檔案或模組的責任 | 目錄總覽、components、styles、types |
| API 表 | 整理 props、events、slots、methods | 單一元件、命令式 API |
| 狀態表 | 整理 data、computed、watch、provide / inject | 表單、選擇器、導航、彈層 |
| 對照表 | 連接 source、styles、types、examples、tests | 所有代表元件 |
| 暫緩支線表 | 防止大型元件失焦 | Select、DatePicker、Table、Upload |
| 問題卡 | 收納待驗證、證據衝突、下一入口 | 所有階段 |
| 仿寫卡 | 把讀碼結果轉成實作能力 | 第 3 階段之後 |
| 評分表 | 判斷是否達到 8、9、10 分 | 每個階段結束 |

## 13. 階段完成記錄模板

每完成一個階段，可以用這份模板沉澱。

````md
## 階段完成記錄

- 階段：
- 日期：
- 對應章節：
- 第一入口：
- 已讀源碼：
- 產出筆記：
- 對照證據：
- 暫緩支線：
- 待驗證問題：
- 可仿寫能力：
- 本階段評分：
- 下一階段入口：

## 主流程

```text
入口
  -> ...
  -> 輸出 / UI / 副作用
```

## 我現在能回答的問題

- ...

## 我現在還不能回答的問題

- ...

## 下一步

- ...
````

建議每次完成後，把「本階段評分」寫得保守一點。若缺少源碼位置、外部證據或待驗證記錄，不要標 10 / 10。

## 14. 從問題反推所在階段

如果你不是按章節順讀，而是遇到問題才回查，可以用這張表判斷應該回到哪個里程碑。

| 目前問題 | 所在階段 | 優先入口 |
|---|---|---|
| 不知道專案從哪裡開始讀 | 第 0 階段 | `00-01_源碼總地圖.md` |
| 不知道 build 如何產出 dist | 第 1 階段 | `package.json`、`vite.config.js`、`build/` |
| 不知道 `app.use` 做了什麼 | 第 2 階段 | `src/index.js` |
| 不知道某個元件是否對外可用 | 第 2 或第 3 階段 | `src/components/index.js`、單元件 `index.js` |
| 不知道 props 如何影響 UI | 第 3 階段 | 單元件 `.vue`、styles、types |
| 不知道 v-model 怎麼同步 | 第 4 階段 | Input、Select、DatePicker |
| 不知道表單驗證如何運作 | 第 4 階段 | Form、FormItem、`mixins/form` |
| 不知道 Table 主流程怎麼拆 | 第 4 階段 | Table 子模組地圖 |
| 不知道 Menu / Tabs active 怎麼協作 | 第 5 階段 | Menu、Tabs |
| 不知道 Tooltip / Poptip 怎麼定位 | 第 5 階段 | Tooltip、Poptip、Popper |
| 不知道 Message / Modal 怎麼命令式呼叫 | 第 5 或第 7 階段 | Message、Notice、Modal、LoadingBar |
| 不知道 class 從哪裡來 | 第 6 階段 | `src/styles/index.less`、component less |
| 不知道 types 和 source 是否一致 | 第 6 階段 | `types/`、source、examples、tests |
| 想抽出可復用設計模式 | 第 7 階段 | 代表元件筆記、高階專題 |
| 想安排下一輪 | 第 8 階段 | `99_總結`、問題索引、暫緩清單 |

## 15. 常見不合格情況

| 不合格情況 | 問題 | 修正方式 |
|---|---|---|
| 把章節讀完當成完成 | 讀完不等於能回查 | 補必答問題與主流程圖 |
| 只留下摘要 | 之後無法追證據 | 補源碼位置、外部對照與證據分級 |
| 每個階段都標 10 分 | 評分失去區分度 | 用第 2 節逐項檢查 |
| 只追元件本體 | 缺少樣式、型別、範例、測試 | 補第 6 階段的支撐系統對照 |
| 大型元件追太深 | 主流程失焦 | 先寫暫緩支線表 |
| 高階總結太早 | 沒有代表元件證據 | 至少讀完 Button、Input、Form、Select、Message、Table 再抽象 |
| 只有問題，沒有下一入口 | 問題索引無法行動 | 每個問題都補「之後回來時第一入口」 |
| 仿寫只停在想法 | 無法驗證理解 | 做一個最小可運行片段或明確實作任務 |

## 本節小結

這份里程碑的重點不是把學習變成僵硬排程，而是讓每個階段都有清楚的完成定義。

一個階段達到 10 / 10，代表你不只看過檔案，而是能交代：

```text
入口
  -> 邊界
  -> 主流程
  -> 證據
  -> 外部對照
  -> 待驗證
  -> 可仿寫能力
  -> 下一步
```

只要每個階段都按這個標準沉澱，這套筆記就不只是閱讀紀錄，而會逐步變成可回查的 View UI Plus 源碼地圖，以及可遷移到自己專案的元件庫設計手冊。
