# 01-02_版本選擇策略：讀源碼要選哪個版本？

> 所屬章節：`01_專案啟動與版本選擇`  
> 筆記定位：讀碼前的版本決策筆記  
> 適用專案：View UI Plus / Vue 3 元件庫 / 開源前端工程  
> 建議閱讀時機：正式 clone 專案、安裝依賴、開始讀 `package.json` 前  
> 建立日期：2026-05-04

---

## 1. 本篇要解決的問題

讀開源專案時，很多人一開始就直接看 `master` 或 `main` 分支，結果很容易遇到以下問題：

1. 今天看的源碼和 npm 使用者實際安裝的版本不一致。
2. 筆記寫到一半，官方又更新了程式碼，導致筆記對不上。
3. 文檔站、GitHub、npm、examples 專案之間的版本可能不同步。
4. 啟動專案時出現依賴錯誤，卻不知道是自己環境問題，還是版本本身問題。
5. 後續想回查筆記時，已經不知道當初是基於哪個 commit 寫的。

所以本篇的核心問題是：

> **讀 View UI Plus 源碼時，到底應該選哪一個版本作為學習基準？**

---

## 2. 本篇結論先講

如果你的目標是「系統性學習 View UI Plus 的元件庫設計」，建議採用以下策略：

> **優先選擇 npm 最新穩定版對應的 GitHub tag 或 commit 作為讀碼基準。**

以 2026-05-04 查核結果來看：

| 項目 | 查核結果 |
|---|---|
| npm package | `view-ui-plus` |
| npm latest version | `1.3.24` |
| npm last published | 2026-03-05 |
| GitHub package.json version | `1.3.24` |
| 專案定位 | 基於 Vue.js 3 的 UI 元件庫 |
| 官方安裝方式 | `npm install view-ui-plus --save` |

因此，本筆記系列可暫定：

```text
讀碼候選版本：view-ui-plus@1.3.24
```

但正式開始讀碼前，還要再補上：

```text
Git branch/tag：待確認
commit hash：待確認
Node.js 版本：待確認
package manager 版本：待確認
```

> 重點：  
> `1.3.24` 是 npm 與 GitHub `package.json` 層級看到的版本號。  
> 真正能保證筆記可回溯的是 **Git commit hash**，不是單純的版本號。

---

## 3. 為什麼讀源碼前一定要先選版本？

### 3.1 因為源碼會變

開源專案的 `master` / `main` 分支通常代表「目前開發狀態」，它不一定等於：

1. npm 上使用者安裝到的版本。
2. 官方文檔目前展示的版本。
3. examples 專案正在使用的版本。
4. 你筆記截圖或分析時的版本。

如果不固定版本，你後面寫的筆記可能會出現這種狀況：

```text
筆記寫：Button 元件的入口在 A 檔案
後來看：Button 元件的入口已經改到 B 檔案
```

這不是你讀錯，而是版本沒有鎖定。

---

### 3.2 因為讀碼筆記需要可回放

好的讀碼筆記應該具備「可重現性」。

也就是未來你自己或別人看到筆記時，可以根據你記錄的版本資訊重新回到當時的源碼狀態。

最少應該記錄：

```text
package name
package version
Git branch / tag
commit hash
Node.js version
package manager version
install command
start command
```

---

### 3.3 因為版本會影響架構判斷

元件庫的版本變化，可能會影響很多架構判斷，例如：

1. 是否支援 SSR。
2. 是否支援 Nuxt。
3. 是否補充 TypeScript 宣告檔。
4. 是否新增 Composition API 寫法。
5. 是否調整打包工具。
6. 是否重構樣式系統。
7. 是否改動元件註冊方式。
8. 是否調整表單驗證邏輯。

如果你沒有固定版本，就很難判斷：

> 這是 View UI Plus 的核心設計，還是某個版本過渡期的寫法？

## 4. 常見的版本來源

讀 View UI Plus 時，通常會接觸到以下版本來源。

---

### 4.1 GitHub `master` / `main` 分支

這是最容易直接看到的源碼版本。

優點：

1. 最新。
2. 最容易 clone。
3. 可以看到目前開發狀態。
4. 適合追蹤最新改動。

缺點：

1. 不一定穩定。
2. 不一定等於 npm latest。
3. 筆記容易因後續 commit 失效。
4. 可能存在尚未發布的改動。

適合情境：

```text
想看目前專案最新長什麼樣子
想追 bug fix
想研究近期改動
想對照 issue / PR
```

不適合情境：

```text
第一次系統性讀碼
想寫長期可回查筆記
想建立穩定學習基準
```

---

### 4.2 npm 最新穩定版

npm 版本代表一般使用者實際安裝到的版本。

優點：

1. 最接近使用者真實使用場景。
2. 通常是已發布版本。
3. 可以透過 `npm install view-ui-plus` 取得。
4. 適合從使用者視角反推源碼。

缺點：

1. npm 套件通常是打包後產物，不一定完整保留開發源碼結構。
2. 如果沒有對應 Git tag，仍需找出發布時的 commit。
3. npm latest 也可能不是你最想研究的歷史版本。

適合情境：

```text
想知道使用者目前裝到什麼版本
想從 API 使用方式反推源碼
想研究 dist / types / package exports
```

---

### 4.3 GitHub tag

Git tag 通常代表某個穩定版本的源碼快照。

優點：

1. 適合讀碼。
2. 可重現。
3. 和版本號容易對應。
4. 筆記不容易失效。

缺點：

1. 有些專案 tag 不完整。
2. 有些專案 npm 版本與 tag 命名不一致。
3. 有些 release notes 不完整。

適合情境：

```text
第一次系統性讀碼
建立長期筆記
搭配 package.json 分析
搭配 release notes 分析
```

這是本筆記系列最推薦的方式。

---

### 4.4 Git commit hash

commit hash 是最精準的版本定位方式。

優點：

1. 最精準。
2. 最可回溯。
3. 不依賴 tag 是否存在。
4. 適合寫嚴謹筆記。

缺點：

1. 不如版本號好讀。
2. 初學者比較不容易記。
3. 需要額外記錄在筆記中。

建議做法：

```bash
git rev-parse HEAD
```

然後把輸出結果記錄到：

```text
01-10_版本鎖定與讀碼基準紀錄.md
```

---

### 4.5 官方文檔站

官方文檔站通常代表「官方希望使用者怎麼使用這個元件庫」。

優點：

1. 適合確認 API 用法。
2. 適合從使用者視角切入。
3. 適合確認安裝方式。
4. 適合確認元件 props / events / slots。

缺點：

1. 文檔不一定明確標示版本。
2. 文檔更新可能和 GitHub / npm 不完全同步。
3. 文檔是使用說明，不是源碼依據。

適合情境：

```text
查元件怎麼用
確認全量引入與按需引入
確認元件 API
從 API 反推源碼設計
```

---

### 4.6 官方 examples / template 專案

例如官方提供的 TypeScript + Vite 範例專案。

優點：

1. 可以看到真實使用方式。
2. 可以理解 main.ts 如何註冊 View UI Plus。
3. 可以理解 Vue 3 + Router + Pinia + Vite 的整合方式。
4. 適合從「專案使用者」角度學習。

缺點：

1. 範例專案可能沒有使用最新 View UI Plus。
2. 範例專案不是 View UI Plus 源碼本體。
3. 不適合作為元件庫架構分析的唯一依據。

適合情境：

```text
想知道企業後台專案怎麼引入元件庫
想看 View UI Plus 在 Vite 專案中的實際使用
想模仿一個小型後台模板
```

## 5. 版本選擇決策矩陣

| 讀碼目標 | 推薦版本來源 | 原因 |
|---|---|---|
| 第一次系統性讀 View UI Plus | npm latest 對應 Git tag / commit | 穩定、接近使用者、可回溯 |
| 想看最新開發狀態 | GitHub `master` / `main` | 最新，但不一定穩定 |
| 想研究某次新增功能 | 對應 release / tag | 可以對照 release notes |
| 想研究 bug 修復 | issue / PR 對應 commit | 精準定位改動 |
| 想學元件庫整體架構 | 穩定版 tag | 避免被開發中變動干擾 |
| 想學打包產物 | npm package | npm 套件最接近使用者安裝結果 |
| 想學歷史演進 | 多版本 tag diff | 適合進階讀碼 |
| 想仿寫迷你元件庫 | 穩定版 source + npm API | 同時看設計與使用方式 |

## 6. 對你目前最適合的策略

你的目標不是單純「會用 View UI Plus」，而是：

```text
透過 View UI Plus 學習 Vue 3 元件庫設計
建立可回查筆記
補強前端底層能力
未來能仿寫與封裝企業後台元件
```

因此建議採用：

```text
主讀版本：npm latest 對應的 Git tag / commit
輔助版本：官方文檔站
輔助專案：官方 TypeScript + Vite template
不建議：直接以 master 當唯一讀碼基準
```

也就是：

```text
先以穩定版建立主線筆記
再用 master 看最新差異
不要一開始就追最新變動
```

## 7. 推薦讀碼版本策略

### 7.1 第一階段：固定一個穩定版本

先選定一個版本作為整個筆記系列的基準。

目前候選：

```text
view-ui-plus@1.3.24
```

然後確認 GitHub 是否存在對應 tag：

```bash
git tag -l
```

如果存在：

```bash
git checkout tags/v1.3.24 -b study-v1.3.24
```

如果不存在對應 tag，就使用目前 `package.json` 版本為 `1.3.24` 的 commit，並記錄 commit hash：

```bash
git rev-parse HEAD
```

---

### 7.2 第二階段：建立自己的讀碼分支

不要直接在 `master` 上做筆記、加註解或測試。

建議建立自己的本地分支：

```bash
git checkout -b study/view-ui-plus-1.3.24
```

這個分支可以用來：

1. 加 console。
2. 寫註解。
3. 暫時改元件。
4. 做小型實驗。
5. 對照打包結果。

但不要把這個分支當作官方源碼。

---

### 7.3 第三階段：建立版本基準紀錄

在 `01-10_版本鎖定與讀碼基準紀錄.md` 中記錄：

```md
# View UI Plus 讀碼基準

| 項目 | 內容 |
|---|---|
| package | view-ui-plus |
| npm version | 1.3.24 |
| Git branch/tag | 待確認 |
| commit hash | 待確認 |
| Node.js | 待確認 |
| package manager | 待確認 |
| install command | 待確認 |
| start command | 待確認 |
| docs url | https://www.iviewui.com |
| repository | https://github.com/view-design/ViewUIPlus |
| 筆記建立日期 | 2026-05-04 |
```

這樣後面的所有筆記都可以回到同一個基準。

## 8. 具體操作流程

### 8.1 查 npm 最新版本

```bash
npm view view-ui-plus version
```

也可以查更多資訊：

```bash
npm view view-ui-plus
```

你要特別看：

```text
version
dist-tags
published time
dependencies
peerDependencies
```

---

### 8.2 clone GitHub 專案

```bash
git clone https://github.com/view-design/ViewUIPlus.git
cd ViewUIPlus
```

---

### 8.3 查看 tag

```bash
git tag -l
```

如果 tag 太多，可以過濾：

```bash
git tag -l "v1.*"
```

---

### 8.4 切換到指定版本

如果有 `v1.3.24`：

```bash
git checkout tags/v1.3.24 -b study-v1.3.24
```

如果沒有對應 tag：

```bash
git checkout master
git pull
```

然後確認 `package.json`：

```bash
node -p "require('./package.json').version"
```

---

### 8.5 記錄 commit hash

```bash
git rev-parse HEAD
```

將結果寫入版本紀錄筆記。

---

### 8.6 記錄遠端資訊

```bash
git remote -v
```

建議記錄：

```text
origin repository
current branch
commit hash
tag
```

## 10. 不同選擇的後果

### 10.1 如果你直接讀 master

你會得到最新源碼，但可能遇到：

1. 筆記容易過期。
2. 啟動可能受最新改動影響。
3. npm 安裝版本可能對不上。
4. 文檔可能還沒更新。
5. 你很難判斷哪些是穩定設計，哪些是開發中寫法。

結論：

```text
master 可以看，但不建議當第一輪讀碼主線。
```

---

### 10.2 如果你只讀 npm package

你會很貼近使用者安裝結果，但可能遇到：

1. 看到的是打包後產物。
2. 開發目錄結構可能被壓縮或轉換。
3. 不適合完整理解元件庫工程架構。
4. 不適合研究 examples / build / test / source organization。

結論：

```text
npm package 適合分析使用者端入口、dist、types，不適合作為唯一源碼依據。
```

---

### 10.3 如果你讀 Git tag

你會得到最適合寫筆記的版本基準。

優勢：

1. 穩定。
2. 可回溯。
3. 容易和版本號對應。
4. 適合長期筆記。

結論：

```text
Git tag 是第一輪系統性讀碼的最佳選擇。
```

---

### 10.4 如果你讀特定 commit

這是最嚴謹的方式。

適合：

1. 沒有對應 tag。
2. 要精準對照某個 bug。
3. 要精準對照某篇筆記。
4. 要固定學習快照。

結論：

```text
commit hash 是筆記可重現性的最終依據。
```

## 12. 本章建議採用的版本決策

### 12.1 推薦結論

本筆記系列建議採用：

```text
view-ui-plus@1.3.24
```

作為第一輪讀碼候選版本。

但要注意，這不是最終完整基準。  
最終完整基準應該長這樣：

```text
package：view-ui-plus
version：1.3.24
Git tag：待確認
commit hash：待確認
Node.js：待確認
package manager：待確認
```

---

### 12.2 為什麼不直接寫死 `master`？

因為 `master` 是流動的。

今天的 `master` 和一個月後的 `master` 可能不一樣。  
如果你的筆記寫：

```text
依據 master 分支分析
```

這個說法其實不夠精準。

更好的寫法是：

```text
依據 ViewUIPlus commit xxxxxxx 分析
package.json version = 1.3.24
```

---

### 12.3 為什麼版本號還不夠？

因為版本號只能表示「發布版本」，不能百分之百表示你手上的 source tree。

例如：

```text
version = 1.3.24
```

仍然需要知道：

```text
哪個 commit？
哪個 tag？
是不是發布後又有修改？
是不是本地改過？
```

所以嚴格來說：

```text
版本號是人看的
commit hash 是機器可回溯的
```


這樣可以避免以後筆記和源碼對不起來。

## 14. 版本選擇時常見錯誤

### 錯誤一：以為最新就是最好

最新版本不一定最適合學習。

對初學讀碼來說，最重要的是：

```text
穩定
可啟動
可回溯
文檔相對完整
```

不是一味追最新。

---

### 錯誤二：只看 GitHub 不看 npm

GitHub 是源碼視角，npm 是使用者視角。

元件庫讀碼應該同時看：

```text
GitHub source code
npm package
official docs
example project
```

這樣才能知道：

```text
源碼怎麼寫
使用者怎麼用
打包後怎麼輸出
文檔怎麼呈現
```

---

### 錯誤三：沒有記錄 commit hash

這是最常見也最嚴重的錯誤。

如果沒有 commit hash，你未來只知道自己讀過 `1.3.24`，但不一定知道當時看的具體 source tree。

建議每一輪正式讀碼都執行一次：

```bash
git rev-parse HEAD
```

---

### 錯誤四：一開始就研究版本差異

第一輪讀碼不建議馬上研究多版本 diff。

更好的順序是：

```text
第一輪：讀穩定版整體架構
第二輪：讀核心元件
第三輪：讀高階專題
第四輪：再研究版本演進
```

---

### 錯誤五：文檔站和源碼混著看，卻沒有標註來源

有些內容是文檔說的，有些內容是源碼看出來的。

筆記中最好區分：

```text
官方文檔觀察
源碼觀察
個人推論
仿寫建議
```

這樣你的筆記會更嚴謹。
