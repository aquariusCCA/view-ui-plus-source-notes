## 1. 本篇要解決的問題

上一篇你已經從使用者視角完成了：

```text
建立 Vue 3 專案
安裝 view-ui-plus
引入 ViewUIPlus
引入 viewuiplus.css
app.use(ViewUIPlus)
使用 Button / Input / Form / Message
確認 dev / build 是否成功
```

但這些只是「會用」。

如果你的目標是透過 View UI Plus 學習元件庫設計，你還需要進一步轉換成「會讀」。

也就是要從：

```text
我知道怎麼用 Button
```

轉成：

```text
我知道 Button 是怎麼被註冊、怎麼定義 props、怎麼產生 class、怎麼載入樣式、怎麼輸出型別、怎麼被打包到 dist 裡。
```

本篇核心問題是：

> **使用者看到的是 API；源碼讀碼要看到 API 背後的工程結構、註冊機制、狀態設計、樣式系統與型別輸出。**

---

## 2. 本篇結論先講

View UI Plus 官方定位是基於 Vue.js 3 的企業級 UI 元件庫與前端解決方案。使用者通常會透過：

```bash
npm install view-ui-plus --save
```

安裝，並透過：

```ts
import ViewUIPlus from 'view-ui-plus'
import 'view-ui-plus/dist/styles/viewuiplus.css'

app.use(ViewUIPlus)
```

完成全量引入與樣式引入。

在使用者視角，這只是三個動作：

```text
安裝
引入
使用
```

但在源碼讀碼視角，這三個動作背後至少會牽出九條主線：

```text
1. package.json 如何定義 npm 入口
2. dist 產物如何對應 main
3. types 如何對應 TypeScript 提示
4. install 方法如何設計
5. components 如何批次註冊
6. directives 是否一起註冊
7. globalProperties 是否掛載全域 API
8. styles 如何從 source 編譯到 dist
9. 每個元件的 props / emits / slots 如何設計
```

一句話：

> **使用者視角看「怎麼用」；源碼視角看「為什麼可以這樣用」。**

---

## 3. 使用者視角是什麼？

使用者視角，就是一般前端工程師在業務專案中使用 View UI Plus 時的角度。

他通常關心：

```text
套件怎麼裝？
main.ts 怎麼寫？
CSS 要不要引入？
元件名稱是什麼？
props 怎麼傳？
事件怎麼綁？
v-model 怎麼用？
表單怎麼驗證？
Message / Modal 怎麼呼叫？
打包會不會成功？
樣式會不會跑掉？
TypeScript 有沒有提示？
```

使用者不太關心：

```text
install 方法怎麼寫
元件如何批次註冊
樣式如何經過 Gulp / Less 處理
package.json 的 files 為什麼包含 dist / src / types
types/index.d.ts 如何設計
Message 是否用 render / createVNode 建立
FormItem 如何向 Form 註冊自己
```

使用者視角的目標是：

```text
快速、穩定、正確地把元件用進專案。
```

---

## 4. 源碼讀碼視角是什麼？

源碼讀碼視角，就是你把自己切換成「元件庫設計者」或「維護者」。

你關心的不只是：

```text
這個元件怎麼用？
```

而是：

```text
這個用法是怎麼被設計出來的？
```

源碼視角會問：

```text
為什麼使用者可以直接寫 <Button />？
為什麼 app.use(ViewUIPlus) 後所有元件都能用？
為什麼需要引入 viewuiplus.css？
為什麼 Message 可以 Message.success(...)？
為什麼 FormItem 可以知道自己屬於哪個 Form？
為什麼 Input 可以支援 v-model？
為什麼 Table 的 API 會長成 columns + data？
為什麼 package.json 同時有 main、typings、files？
為什麼專案同時有 src、dist、types？
```

源碼讀碼視角的目標是：

```text
理解元件庫作者如何設計 API、封裝複雜度、輸出套件、維護一致性，並把這些能力轉化成自己的前端工程能力。
```

---

## 5. 兩種視角的核心差異

| 面向 | 使用者視角 | 源碼讀碼視角 |
|---|---|---|
| 主要問題 | 這個元件怎麼用？ | 這個元件為什麼可以這樣用？ |
| 關心對象 | API、Demo、文檔 | source、install、props、types、styles、build |
| 操作入口 | `npm install`、`main.ts`、`App.vue` | `package.json`、`src`、`types`、`dist`、`build` |
| 成功標準 | 畫面能跑、樣式正常、功能有效 | 能說明 API 背後的源碼結構 |
| 看 Button | `<Button type="primary" />` | props、class、style、install、type declaration |
| 看 Form | rules 怎麼寫 | Form / FormItem 通訊與驗證流程 |
| 看 Message | `Message.success()` | 命令式渲染、實例管理、全域方法 |
| 看 CSS | 引入一行 CSS | Less 來源、prefixCls、build:style、dist styles |
| 看 TypeScript | IDE 有提示 | `typings`、`types/index.d.ts`、元件型別宣告 |
| 看 build | `pnpm build` 成功 | `main`、`files`、Vite、Gulp、產物結構 |

## 6. 從「安裝」轉成源碼問題

### 6.1 使用者視角

使用者看到的是：

```bash
npm install view-ui-plus --save
```

或：

```bash
pnpm add view-ui-plus
```

使用者只需要知道：

```text
套件名稱是 view-ui-plus
安裝後可以 import
```

---

### 6.2 源碼讀碼視角

源碼讀碼要問：

```text
package.json 的 name 是什麼？
version 是什麼？
main 指向哪裡？
typings 指向哪裡？
files 發布哪些目錄？
dependencies 和 devDependencies 如何區分？
這個套件發布到 npm 時包含哪些內容？
```

目前 View UI Plus 本體 `package.json` 可看到：

```json
{
  "name": "view-ui-plus",
  "version": "1.3.24",
  "main": "dist/viewuiplus.min.js",
  "typings": "types/index.d.ts",
  "files": [
    "dist",
    "src",
    "types"
  ]
}
```

這代表：

```text
使用者 import 'view-ui-plus' 時，會和 dist 入口有關。
TypeScript 找型別時，會和 types/index.d.ts 有關。
npm 發布時，dist / src / types 是核心發布目錄。
```

## 7. 從「import ViewUIPlus」轉成源碼問題

### 7.1 使用者視角

使用者在 `main.ts` 中寫：

```ts
import ViewUIPlus from 'view-ui-plus'
```

使用者只知道：

```text
我引入了一個元件庫物件。
```

---

### 7.2 源碼讀碼視角

源碼讀碼要問：

```text
這個 default export 到底是什麼？
它來自 source 的哪個檔案？
它是否符合 Vue plugin 格式？
它是否包含 install 方法？
install 方法接收哪些參數？
它是否同時 export 單一元件？
```

如果最終使用者可以這樣寫：

```ts
app.use(ViewUIPlus)
```

代表 `ViewUIPlus` 至少要符合 Vue plugin 的使用方式。

也就是源碼中大概率會存在類似概念：

```ts
const install = (app) => {
  // register components
}

export default {
  install
}
```

實際源碼要在後續 `06_src_入口與插件機制` 中確認。

## 23. 不要用錯視角

### 23.1 錯誤一：用使用者視角讀源碼

錯誤表現：

```text
只知道這個 prop 怎麼用
但不知道 prop 在哪裡定義
不知道預設值在哪裡
不知道樣式如何對應
不知道事件如何 emit
```

這樣只是會用，不是會讀。

---

### 23.2 錯誤二：用源碼視角寫業務

錯誤表現：

```text
業務只是要一個 Button
你卻想研究 Button 所有源碼才敢用
```

這樣效率太低。

在業務中：

```text
先會用
再按需要深入
```

在學習中：

```text
先用一次
再系統性深入
```

---

### 23.3 錯誤三：直接從 source 第一行開始讀

不建議：

```text
打開 src 從第一個檔案開始逐行讀
```

更好的做法：

```text
先找一個使用者 API
再反查它的源碼路徑
再讀相關鏈路
```

例如：

```text
<Button type="primary" />
    ↓
Button source
    ↓
props
    ↓
class
    ↓
style
    ↓
types
    ↓
install
```

---

### 23.4 錯誤四：把文檔當成源碼結論

文檔是使用說明，不是最終實作。

正確流程：

```text
文檔看 API
Demo 看用法
source 看實作
types 看型別
styles 看視覺
tests / issues 看邊界
```

---

### 23.5 錯誤五：忽略 build 與 dist

元件庫不是只寫 source。

你還要理解：

```text
source 如何變成 npm package
CSS 如何變成 dist/styles/viewuiplus.css
types 如何讓使用者有 TS 提示
main 如何指向打包結果
files 如何控制發布內容
```

否則你只會看元件，不會看元件庫工程。

## 24. 哪些內容先用使用者視角，哪些內容直接讀源碼？

| 類型 | 建議切入方式 | 原因 |
|---|---|---|
| Button | 先使用者視角 | API 簡單，適合反推 props / style |
| Input | 先使用者視角 | v-model 行為要先體驗 |

| styles | 先使用者再源碼 | 先看 CSS 生效，再追 Less |
| types | 先使用者再源碼 | 先看 TS 提示，再追 d.ts |
