## 1. 本篇要解決的問題

讀開源專案最怕的不是一開始看不懂，而是：

```text
今天看的是一個版本
下週 source 改了
一個月後筆記對不起來
半年後回查時不知道當初看的 commit 是哪一個
```

View UI Plus 這類 UI 元件庫尤其需要版本鎖定，因為它同時涉及：

```text
GitHub source
npm package
dist 產物
types 宣告
官方文檔
官方範例
Vite / Vue CLI / Gulp / Less 工具鏈
Node / npm / pnpm 版本
```

如果沒有建立讀碼基準，後續你寫的每一篇筆記都可能出現：

```text
我記得當時是這樣
但現在源碼好像不一樣
不知道是我看錯，還是官方更新了
```

所以本篇核心目標是：

> **把 View UI Plus 的讀碼版本、Git commit、Node 環境、啟動指令、官方資源與本地測試結果全部固定下來，作為後續所有筆記的共同基準。**

## 3. 為什麼版本號不夠？

很多人會以為：

```text
我讀的是 view-ui-plus@1.3.24
```

這樣就夠了。

其實不夠。

因為 `1.3.24` 只能說明：

```text
package.json 或 npm 顯示的版本號
```

但它不能完整說明：

```text
你本地 clone 的是哪個 commit？
你看的是否是 tag？
你的本地分支是否有修改？
你是否切到了 master 最新狀態？
你是否在 node_modules 裡看 npm 產物？
你是否在 GitHub 網頁看 master？
```

所以嚴格來說，完整讀碼基準應該包含：

```text
package version
Git tag
Git commit hash
本地 branch
Node version
package manager version
install command
run command
build result
```

## 4. 版本鎖定的三個層級

### 4.1 第一層：package 版本

例如：

```text
view-ui-plus@1.3.24
```

這是人類最容易理解的版本。

用途：

```text
對齊 npm
對齊 package.json
對齊官方安裝
對齊使用者視角
```

限制：

```text
不能唯一定位 source tree
不能保證 tag 存在
不能保證 master 沒有後續 commit
```

---

### 4.2 第二層：Git tag

例如：

```text
v1.3.24
```

Tag 比版本號更適合讀碼，因為它通常對應某個 commit。

用途：

```text
checkout 固定版本
對照 release
建立讀碼分支
```

限制：

```text
有些專案 tag 不完整
有些版本發布到 npm 但 GitHub release / tag 沒同步
tag 名稱可能是 v1.3.24，也可能是 1.3.24
```

---

### 4.3 第三層：commit hash

例如：

```text
fcad615...
```

commit hash 是最精準的版本鎖定。

用途：

```text
唯一定位 source tree
確保筆記可回溯
未來可以重新 checkout
可以判斷本地是否和筆記一致
```

讀碼筆記中最重要的是：

```text
commit hash
```

## 5. 目前 View UI Plus 的版本狀態觀察

### 5.1 GitHub package.json

目前 GitHub `master` 的 `package.json` 中可看到：

```json
{
  "name": "view-ui-plus",
  "version": "1.3.24",
  "main": "dist/viewuiplus.min.js",
  "typings": "types/index.d.ts",
  "files": [
    "dist",
    "src",
    "types"
  ]
}
```

這代表：

```text
GitHub master 的 package.json 版本目前是 1.3.24
```

---

### 5.2 npm package

目前 npm 搜尋結果顯示：

```text
view-ui-plus latest version：1.3.24
last published：2026-03-05
```

這代表：

```text
使用者目前透過 npm install view-ui-plus 拿到的 latest 版本是 1.3.24
```

---

### 5.3 jsDelivr / CDN 版本列表

jsDelivr 顯示的版本列表中包含：

```text
view-ui-plus@1.3.24
view-ui-plus@1.3.23
view-ui-plus@1.3.22
...
```

並且 package 檔案中可看到：

```text
dist
src
types
LICENSE
package.json
README.md
```

這與 `package.json` 的 `files` 欄位一致。

---

### 5.4 GitHub Releases / Tags

GitHub Releases 搜尋結果中可看到較新的 release 顯示到 `1.3.20`，而 npm / GitHub `package.json` 顯示 `1.3.24`。

這代表你在讀碼時要特別注意：

```text
npm latest
GitHub package.json
GitHub release
GitHub tag
```

這幾個來源不一定完全同步。

因此本篇不直接斷言：

```text
一定存在 v1.3.24 tag
```

而是要求你在本地執行：

```bash
git tag -l
git tag -l "v1.3.24"
git tag -l "1.3.24"
```

確認後再記錄。

## 23. 如何比較兩個版本？

如果你未來想比較兩個版本，可以用：

```bash
git diff old_commit new_commit -- package.json
git diff old_commit new_commit -- src
git diff old_commit new_commit -- types
git diff old_commit new_commit -- build
```

也可以看統計：

```bash
git diff --stat old_commit new_commit
```

或看某個元件：

```bash
git diff old_commit new_commit -- src/components/button
```

這會支援後續高階學習：

```text
版本演進
API 調整
重構方向
bug fix 分析
```

## 24. 不要犯的錯誤

### 錯誤一：只記版本號，不記 commit

不夠。

應該記：

```text
version + tag + commit hash
```

---

### 錯誤二：把 npm latest 當成本地 source

npm latest 代表發布版本，不代表你本地 clone 的 source。

要用：

```bash
git rev-parse HEAD
```

確認。

---

### 錯誤三：把 GitHub master 當作穩定版本

`master` 是流動的。

如果要寫長期筆記，一定要記 commit。

---

### 錯誤四：本地改過 source 但沒記錄

如果你為了測試加了 `console.log`，要記錄：

```text
working tree 是否乾淨
是否有本地修改
```

執行：

```bash
git status
git diff
```

---

### 錯誤五：沒有區分本體源碼與使用者 playground

View UI Plus 本體源碼環境和你自己的 Vite playground 是兩件事。

要分開記錄：

```text
本體 source 環境
使用者 playground 環境
官方 example 環境
```

---

### 錯誤六：升級依賴後還用原筆記結論

如果你升級了：

```text
Vue
Vite
TypeScript
Less
Gulp
```

那你的環境已經不是原始讀碼基準。

必須另外記錄。

## 25. 基準紀錄的最佳實務

### 25.1 先固定，再分析

不要一邊讀，一邊還在換版本。

先固定：

```text
commit hash
Node
npm
install result
dev2 result
```

再開始讀。

---

### 25.2 每篇筆記都留基準

不要只在本篇記一次。

因為後續單篇筆記被獨立打開時，也要知道它依據哪個版本。

---

### 25.3 重要推論要標註來源

筆記中可以標註：

```text
[package.json]
[GitHub source]
[npm package]
[official docs]
[local test]
[personal inference]
```

這樣可以區分：

```text
官方事實
本地觀察
個人推論
```

---

### 25.4 錯誤也要記錄

啟動失敗、build 失敗、CSS 不生效，這些都是有價值的學習資料。

不要只記成功結果。
