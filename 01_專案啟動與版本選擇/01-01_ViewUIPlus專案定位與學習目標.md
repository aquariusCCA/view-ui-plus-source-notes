> 所屬章節：`01_專案啟動與版本選擇`

> 適用對象：已有 Vue 專案經驗，想從「會用元件庫」進一步提升到「看懂元件庫、封裝企業元件、設計可維護前端架構」的人。

## 0. 本篇核心結論

View UI Plus 不應該只被當成「一套 Vue 3 UI 元件庫」來學。

對讀碼者來說，它更適合作為一個觀察樣本，用來學習：

1. **Vue 3 元件庫如何組織工程結構**
2. **元件如何設計 props、events、slots 與樣式介面**
3. **元件庫如何提供全域安裝、單元件使用、樣式匯入與型別支援**
4. **企業後台常見元件如何被抽象成通用 API**
5. **如何從開源元件庫反推出自己公司內部元件封裝方法**

本章不是要你馬上讀懂所有源碼，而是要先建立一個正確前提：

> 你讀 View UI Plus 的目標，不是背元件怎麼寫，而是學會「元件庫作者如何設計一套可被很多專案重複使用的前端基礎設施」。

---

## 1. View UI Plus 是什麼？

### 1.1 官方定位

View UI Plus 官方倉庫將它描述為：

> An enterprise-level UI component library and front-end solution based on Vue.js 3.

也就是：

> **基於 Vue.js 3 的企業級 UI 元件庫與前端解決方案。**

這句話可以拆成三層理解。

| 關鍵字 | 解讀 |
|---|---|
| `Vue.js 3` | 它不是 Vue 2 時代的寫法，而是面向 Vue 3 生態的元件庫 |
| `UI component library` | 它提供 Button、Input、Form、Table、Modal 等可重複使用的 UI 元件 |
| `front-end solution` | 它不只是一堆元件，也包含安裝方式、樣式系統、範例、文檔、工程配置、型別支援等配套 |

所以讀 View UI Plus 時，不要只看單一元件的 template。

你應該同時觀察：

```text
元件本身
元件註冊
樣式組織
型別輸出
範例 Demo
文檔 API
建置產物
全域配置
外部使用方式
```

## 2. 為什麼這個專案值得讀？

### 2.1 它適合學「企業後台 UI 元件庫」的設計方式

View UI Plus 的使用場景偏向企業級中後台，例如：

- 表單
- 表格
- 彈窗
- 下拉選單
- 日期選擇
- 分頁
- 訊息提示
- 導航選單
- 上傳
- 驗證
- 命令式回饋元件

這些剛好也是企業系統最常出現的前端場景。

如果你的目標是提升企業系統開發能力，那麼讀這種元件庫會比單純做靜態頁面更有價值。

---

### 2.2 它適合從「使用者視角」切到「作者視角」

一般使用者只會這樣看元件庫：

```vue
<Button type="primary">送出</Button>
```

但讀源碼時，要反過來思考：

```text
為什麼 type 可以控制樣式？
Button 的 props 在哪裡定義？
Button 的 class 如何生成？
Button 如何支援 loading？
Button 如何處理 disabled？
Button 的事件如何向外暴露？
Button 如何被全域註冊？
Button 的型別如何讓 TS 使用者取得提示？
```

這種思考方式，就是你從「會用框架」走向「會設計框架級抽象」的關鍵。

---

### 2.3 它適合練習前端工程化讀碼

官方倉庫可以看到 View UI Plus 不只包含 `src`，也包含 `build`、`dist`、`examples`、`test`、`types` 等目錄。

這代表它不是單純的元件練習專案，而是一個完整元件庫工程。

你可以透過它學到：

| 面向 | 可以觀察的內容 |
|---|---|
| 工程結構 | 根目錄、package、build、dist、src、types 如何分工 |
| 元件設計 | 每個元件如何拆分 props、樣式、子元件、工具函式 |
| 插件化 | 元件庫如何提供 `install` 給 Vue app 使用 |
| 樣式系統 | Less / CSS 檔案如何組織，class 命名如何維持一致 |
| 型別系統 | 元件庫如何提供 TypeScript 宣告 |
| 範例系統 | examples / docs 如何展示元件 API |
| 測試思維 | test 目錄如何反推元件規格 |

## 3. 這個專案適合學哪些能力？

### 3.1 第一層：會用 View UI Plus

這是最低層目標。

你要能做到：

```text
1. 建立 Vue 3 專案
2. 安裝 view-ui-plus
3. 引入樣式
4. 全域註冊元件庫
5. 使用 Button / Input / Form / Table / Modal 等常見元件
6. 能看懂官方文檔中的 props、events、slots
```

這一層的重點不是源碼，而是先理解「外部使用者看到的 API 長什麼樣」。

---

### 3.2 第二層：看懂元件 API 設計

這一層開始進入讀碼。

你要能回答：

```text
1. 這個元件對外暴露哪些 props？
2. 哪些 props 是控制視覺樣式？
3. 哪些 props 是控制行為？
4. 哪些事件會 emit？
5. slot 提供了哪些擴充點？
6. 元件 API 是否容易理解？
7. API 是否和企業後台場景相符？
```

例如讀 `Input` 時，不要只看它怎麼渲染 input，而要看：

```text
clearable 如何設計？
disabled 如何傳遞？
prefix / suffix 如何支援？
v-model 如何實現？
表單驗證如何接入？
```

---

### 3.3 第三層：看懂元件內部實現

這一層才是源碼實作。

你要觀察：

```text
1. template 如何拆分？
2. computed 如何生成狀態？
3. watch 用在哪些地方？
4. emit 何時觸發？
5. className 如何組合？
6. 樣式前綴如何處理？
7. DOM 事件如何封裝？
8. 子元件如何協作？
9. 是否有 mixins / composables / utils？
```

這一層不是要逐行背，而是要整理出可重複使用的模式。

---

### 3.4 第四層：抽象成自己的企業元件封裝能力

這才是讀 View UI Plus 最重要的目標。

你最後應該能把學到的模式轉成自己的能力，例如：

```text
1. 封裝公司內部 BaseButton
2. 封裝帶業務規則的 SearchForm
3. 封裝通用 DataTable
4. 封裝命令式 Message / Confirm
5. 設計表單驗證規則
6. 設計元件 props 命名規範
7. 設計元件事件命名規範
8. 設計元件 slot 擴充點
9. 設計元件庫的 install 方法
10. 設計公司內部前端元件規範
```

## 4. 本專案對你的前端能力提升價值

> 這一節用「能力轉換」來看 View UI Plus，而不是用「元件數量」來看。

| 讀碼對象 | 表面上在看 | 實際上要學 |
|---|---|---|
| Button | 一個按鈕 | props 設計、狀態控制、class 組合 |
| Input | 一個輸入框 | v-model、事件同步、可清除、表單整合 |
| Form | 表單容器 | 驗證架構、欄位註冊、父子元件通訊 |
| Table | 表格 | 複雜資料展示、欄位配置、插槽擴充 |
| Modal | 彈窗 | 顯示控制、傳送門、遮罩、命令式呼叫 |
| Message | 提示訊息 | 動態掛載、命令式 API、佇列管理 |
| Menu | 選單 | 遞迴結構、選中狀態、路由整合思維 |
| Upload | 上傳 | 非同步狀態、檔案校驗、事件回呼 |

| Styles | 樣式系統 | prefixCls、Less 變數、主題維護 |

讀完後，你應該不是只會說：

> 我看過 Button 源碼。

而是能說：

> 我知道一個企業級 Button 元件要如何設計 props、狀態、樣式命名、loading 行為、disabled 行為、事件轉發，以及如何被全域註冊到 Vue app 中。

## 5. 本章的學習目標

`01_專案啟動與版本選擇` 這一章的目的不是深入源碼，而是建立讀碼前提。

---

### 5.1 目標一：明確知道 View UI Plus 的定位

讀完本篇後，你要能用自己的話說出：

```text
View UI Plus 是一套基於 Vue 3 的企業級 UI 元件庫與前端解決方案。
它不只提供元件，也提供安裝方式、文檔、樣式、型別、範例與工程化配套。
我讀它不是為了背元件，而是為了學習元件庫設計與企業前端封裝能力。
```

### 5.2 目標二：區分「使用者視角」與「讀碼者視角」

你要能區分：

| 視角 | 關心問題 |
|---|---|
| 使用者視角 | 怎麼安裝、怎麼引入、怎麼使用元件、API 是什麼 |
| 讀碼者視角 | API 怎麼設計、元件怎麼實作、樣式怎麼組織、全域註冊怎麼完成 |
| 封裝者視角 | 如何借鑑它，封裝公司內部元件、表單、表格、彈窗、命令式 API |

這三種視角會貫穿整個 View UI Plus 讀碼筆記。

---

### 5.3 目標三：知道這個專案最值得優先讀哪幾類內容

建議優先順序如下：

```text
1. 使用入口
2. 全域 install 機制
3. 基礎元件
4. 表單元件
5. 彈層與命令式元件
6. 表格與資料展示元件
7. 樣式系統
8. 型別宣告
9. 測試與範例
10. 建置與打包
```

原因是：

> 先看「使用者如何使用」，再看「元件如何被註冊」，最後才看「打包怎麼輸出」。

這樣比較不會一開始就迷失在 build 腳本與工程設定裡。

## 6. 不建議一開始追求的目標

讀開源元件庫最常見的錯誤，是一開始就想「全部讀懂」。

這很容易失敗。

### 6.1 不建議目標一：逐行讀完整個專案

不需要一開始逐行讀完：

```text
每一個元件
每一個樣式檔
每一個 build 腳本
每一個 type 宣告
每一個 demo
```

這樣成本太高，而且容易看完就忘。

比較好的方式是：

```text
先建立地圖
再挑代表性元件
再整理共通模式
最後回頭補細節
```

---

### 6.2 不建議目標二：把它當作 Vue 3 入門教材

View UI Plus 不是 Vue 3 基礎教學。

如果你對以下概念不熟，讀碼會很吃力：

```text
props
emits
slots
v-model
computed
watch
provide / inject
生命週期
插件 install
動態元件
Teleport
TypeScript 型別
CSS / Less
```

這些應該在讀碼過程中補強，但不要期待 View UI Plus 本身會像教材一樣從零教你。

---

### 6.3 不建議目標三：只看畫面，不看 API 設計

元件庫的價值不在畫面，而在 API 抽象。

例如同樣是一個 Button，真正值得學的是：

```text
type 如何命名？
size 如何設計？
loading 是否影響 disabled？
icon 放在 props 還是 slot？
事件是否需要轉發？
class 命名是否穩定？
樣式是否能主題化？
```

如果只看畫面，你學不到元件庫設計能力。

---

### 6.4 不建議目標四：直接照抄到公司專案

開源元件庫的程式碼不一定能直接搬到企業專案。

企業專案通常還有：

```text
權限規則
表單業務校驗
後端欄位映射
資料字典
錯誤碼處理
審核流程
登入狀態
稽核需求
跨系統整合
```

所以更好的方式是：

> 學它的設計模式，不是複製它的所有程式碼。

## 8. 建議優先研究的元件類型

View UI Plus 元件很多，不建議從頭到尾平均讀。

建議先挑「代表性強、企業場景常見、設計價值高」的元件。

---

### 8.1 第一批：基礎元件

| 元件類型 | 學習重點 |
|---|---|
| Button | props、class、loading、disabled、icon、事件轉發 |
| Icon | 字體圖示 / SVG / class 命名 / 尺寸控制 |
| Grid / Layout | 佈局抽象、響應式設計、容器結構 |

這一批適合建立源碼閱讀手感。

---

### 8.2 第二批：輸入與表單元件

| 元件類型 | 學習重點 |
|---|---|
| Input | v-model、事件同步、clearable、prefix / suffix |
| Select | 下拉、選項管理、遠端搜尋、鍵盤操作 |
| Checkbox / Radio | group 設計、value 同步、表單整合 |
| Form / FormItem | 驗證架構、欄位註冊、provide / inject |

這一批最接近企業後台日常開發。

---

### 8.3 第三批：資料展示與操作元件

| 元件類型 | 學習重點 |
|---|---|
| Table | 欄位配置、slot、自訂渲染、排序、分頁 |
| Page | 分頁狀態、事件設計、資料查詢互動 |
| Tree | 遞迴資料、節點狀態、展開收合 |
| Transfer | 多選、資料搬移、狀態同步 |

這一批適合學習複雜元件的狀態設計。

---

### 8.4 第四批：彈層與命令式元件

| 元件類型 | 學習重點 |
|---|---|
| Modal | 顯示控制、遮罩、鍵盤事件、關閉流程 |
| Drawer | 彈層方向、動畫、內容容器 |
| Message | 動態掛載、命令式 API、佇列 |
| Notice | 全域通知、配置合併、關閉管理 |
| Poptip / Tooltip | 定位、觸發方式、浮層管理 |

這一批是元件庫高階能力的核心。

## 14. 常見錯誤理解

### 14.1 錯誤一：以為會用元件庫就等於看懂元件庫

會用：

```vue
<Button type="primary">送出</Button>
```

不代表看懂：

```text
type 如何映射 class？
loading 如何影響互動？
disabled 是否阻止 click？
icon 如何渲染？
元件如何被 install？
樣式如何被打包？
```

---

### 14.2 錯誤二：以為讀源碼就是逐行看

真正有效的讀碼流程應該是：

```text
先看使用方式
再看入口
再看目錄
再看代表性元件
再抽象共通模式
最後補細節
```

---

### 14.3 錯誤三：以為開源元件可以直接搬到業務專案

企業專案的元件封裝要考慮：

```text
業務規則是否穩定
元件是否過度耦合
props 是否會膨脹
事件是否容易理解
slot 是否足夠擴充
是否能測試
是否能維護
```

所以讀 View UI Plus 的重點是「借鑑設計」，不是「複製貼上」。
