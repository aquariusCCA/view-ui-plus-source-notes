# 01-03_官方倉庫、npm、文檔站、資源入口整理

> 所屬章節：`01_專案啟動與版本選擇`  
> 筆記定位：View UI Plus 讀碼前的官方資源入口地圖  
> 適用目標：建立穩定、可回查、可驗證的資料來源清單  
> 建立日期：2026-05-04  
> 查核基準：以官方 GitHub、npm、官方文檔站、官方範例專案為主

---

## 1. 本篇要解決的問題

讀開源專案時，最常見的混亂不是「程式碼太難」，而是：

> **不知道每一個資料來源應該拿來看什麼。**

以 View UI Plus 為例，你可能同時看到：

```text
GitHub 原始碼
npm 套件頁
官方文檔站
官方安裝頁
GitHub Releases
GitHub Tags
官方 starter kit
官方 Vite 範例專案
官方 TypeScript 範例專案
CDN 版本列表
Issues / Pull Requests
```

如果沒有先整理資源入口，很容易出現以下問題：

1. 用文檔站判斷源碼結構。
2. 用 `master` 分支判斷 npm 版本。
3. 用 npm 打包產物推測開發目錄。
4. 用範例專案的寫法推斷元件庫內部設計。
5. 用舊 issue 當作目前版本的行為依據。
6. 筆記裡沒有標註資料來源，之後回查時不知道依據是什麼。

所以本篇的核心目標是：

> **建立 View UI Plus 的官方資源入口地圖，並明確定義每個入口的用途、適用場景與使用限制。**

---

## 2. 本篇結論先講

View UI Plus 讀碼時，建議把資料來源分成六層：

```text
第一層：本地 clone 後固定 commit 的源碼
第二層：GitHub 官方倉庫
第三層：npm package 與 package.json
第四層：官方文檔站與安裝頁
第五層：官方範例專案 / starter kits
第六層：Releases、Tags、Issues、PR、CDN
```

其中最重要的原則是：

> **源碼分析以 GitHub / 本地 commit 為準；使用方式以官方文檔與 npm 為準；實際整合方式以官方範例專案為輔。**

### 4.3 GitHub 倉庫的使用規則

讀 View UI Plus 時，GitHub 倉庫不能只用瀏覽器打開看看，建議你要做三件事：

```bash
git clone https://github.com/view-design/ViewUIPlus.git
cd ViewUIPlus
git rev-parse HEAD
```

然後把 commit hash 記錄到：

```text
01-09_版本鎖定與讀碼基準紀錄.md
```

原因是：

```text
GitHub 網頁上的 master / main 會變
本地固定 commit 才能保證筆記可回溯
```

---

### 4.4 GitHub 倉庫中最重要的檔案

第一輪讀碼時，建議先鎖定這幾個入口：

| 檔案 / 目錄 | 先看原因 |
|---|---|
| `README.md` | 確認專案定位、安裝方式、官方入口 |
| `package.json` | 確認版本、依賴、scripts、main、typings |
| `src/` | 確認源碼主體 |
| `src/index.*` | 確認元件庫入口與 install 機制 |
| `src/components/` | 確認元件目錄組織方式 |
| `src/styles/` | 確認樣式系統 |
| `types/` | 確認型別輸出 |
| `dist/` | 確認打包產物 |
| `build/` | 確認建置流程 |
| `examples/` | 確認內部展示與測試方式 |

---

### 4.5 不要誤用 GitHub 倉庫

GitHub 倉庫雖然是源碼主入口，但要避免以下誤用：

```text
錯誤一：看到 master 就以為是 npm latest
錯誤二：看到某段源碼就以為所有版本都一樣
錯誤三：沒有記 commit hash 就開始寫筆記
錯誤四：直接用 issue 裡的說法當作正式設計
錯誤五：沒有分清楚 source、dist、examples、docs 的角色
```


---

### 5.3 npm 對讀碼的價值

讀元件庫時，npm 不是只用來安裝套件，它還可以幫你回答：

| 問題 | npm 可以提供什麼 |
|---|---|
| 使用者安裝的套件叫什麼？ | `view-ui-plus` |
| 目前 latest 是哪個版本？ | `npm view view-ui-plus version` |
| 發布頻率如何？ | npm 版本列表與發布時間 |
| 使用者安裝後拿到什麼？ | npm package files / jsDelivr |
| main 入口是什麼？ | `package.json` 的 `main` |
| types 入口是什麼？ | `package.json` 的 `typings` |
| 是否適合做版本基準？ | 可搭配 Git tag / commit 判斷 |

---

### 5.4 建議使用的 npm 指令

查最新版本：

```bash
npm view view-ui-plus version
```

查完整套件資訊：

```bash
npm view view-ui-plus
```

查 dist-tags：

```bash
npm view view-ui-plus dist-tags
```

查所有版本：

```bash
npm view view-ui-plus versions
```

查 main / typings / files：

```bash
npm view view-ui-plus main
npm view view-ui-plus typings
npm view view-ui-plus files
```

查依賴：

```bash
npm view view-ui-plus dependencies
npm view view-ui-plus peerDependencies
```

---

### 5.5 npm 與 GitHub 的關係

npm 和 GitHub 要一起看。

```text
GitHub：看開發源碼
npm：看發布套件
```

兩者可能不同步，所以不能假設：

```text
GitHub master = npm latest
```

更精準的做法是：

```text
npm latest version
        ↓
找 GitHub 對應 tag
        ↓
如果沒有 tag，找 package.json version 對應的 commit
        ↓
記錄 commit hash
```

---

### 5.6 npm 的使用限制

npm 不適合拿來直接判斷完整工程架構，原因是：

1. npm 套件通常只保留發布所需檔案。
2. 可能沒有完整 examples。
3. 可能沒有完整 build 腳本上下文。
4. 可能看不到開發過程與 commit 歷史。
5. 打包後產物不一定保留原始碼結構。

所以：

```text
npm 適合看使用者安裝結果
GitHub 適合看開發者源碼結構
```

### 7.3 官方文檔站對讀碼的價值

文檔不是源碼，但它是讀源碼的「反推入口」。

例如你在文檔看到：

```vue
<Button type="primary">Primary</Button>
```

你後續讀源碼時就可以反推：

```text
Button 元件在哪？
type prop 如何定義？
primary 對應哪個 class？
class prefix 如何組出？
樣式在哪裡？
Button 是否透過 install 註冊？
Button 是否支援全域註冊？
```

所以讀元件庫時，最好的流程不是直接逐行讀源碼，而是：

```text
先看文檔 API
        ↓
再看使用範例
        ↓
再回到源碼找 props / emit / slots
        ↓
再看樣式與型別
        ↓
最後整理成元件設計模式
```

| `23_src-styles_樣式系統` | 查樣式引入與主題配置 |
| `28_高階專題_元件API設計` | 從 API 設計反推元件設計原則 |

---

### 7.5 不要誤用官方文檔站

文檔站不能替代源碼。

常見誤用：

```text
錯誤一：文檔寫法等於內部實作
錯誤二：文檔有這個 API，源碼一定很簡單
錯誤三：文檔沒寫的東西，源碼一定沒有
錯誤四：文檔站版本一定和 GitHub master 一致
錯誤五：只看 Demo，不看 props / events / slots 的完整表格
```

正確用法：

```text
文檔站用來理解「使用者如何使用」
GitHub 源碼用來理解「作者如何實作」
```

## 27. 常見錯誤

### 錯誤一：把文檔站當成源碼依據

文檔站是使用說明，不是實作依據。

正確做法：

```text
文檔確認 API
源碼確認實作
型別確認 TS 約束
樣式確認視覺規則
```

---

### 錯誤二：只看 GitHub，不看 npm

元件庫最終是給使用者安裝的。

所以一定要看 npm，確認：

```text
package name
version
main
typings
files
dependencies
```

---

### 錯誤三：只看 npm，不看 GitHub

npm 是發布結果，不是完整開發上下文。

如果只看 npm，你可能會漏掉：

```text
build 流程
examples
測試
commit history
issue / PR
開發目錄設計
```

---

### 錯誤四：沒有區分範例專案與元件庫本體

`view-ui-project-ts` 和 `view-ui-project-vite` 是使用者專案模板，不是 View UI Plus 本體。

它們可以用來學：

```text
怎麼用 View UI Plus
```

但不能用來判斷：

```text
View UI Plus 內部怎麼設計
```

---

### 錯誤五：用 issue 當結論

issue 是問題紀錄，不一定是官方結論。

使用 issue 時要確認：

```text
是否已修復
是否有官方回覆
是否對應目前版本
是否有 PR
是否只是使用者誤用
```
