## 2. 先建立觀念：設定檔是「工程控制面」

在元件庫專案中，設定檔不是雜物。

它們共同控制了以下事情：

```text
開發怎麼跑？
打包怎麼產生？
外部使用者引用哪個檔案？
TypeScript 如何找到型別？
程式碼格式如何統一？
哪些檔案不進 Git？
哪些檔案不進 npm？
CI / 預覽平台如何啟動？
```

所以根目錄設定檔可以視為：

> **元件庫工程的控制面。**

如果 `src/` 是元件庫的「業務核心」，那根目錄設定檔就是控制這些源碼如何被開發、檢查、打包、發布與使用的工程層。

## 4. 設定檔分類總表

| 分類 | 設定檔 | 一句話定位 | 第一輪優先級 |
|---|---|---|---:|
| 專案中樞設定 | `package.json` | 專案名稱、版本、入口、scripts、依賴、npm 發布邊界 | 高 |
| Vite 開發設定 | `vite.config.dev.js` | 用 Vite 啟動 examples，支援本地開發預覽 | 高 |
| Vue CLI 開發設定 | `vue.config.js` | 舊式 Vue CLI 開發入口，指向 examples | 中 |
| 元件庫打包設定 | `vite.config.js` | 將 `src/index.js` 打成 UMD / ES module 產物 | 高 |
| TypeScript 設定 | `tsconfig.json` | TS 編譯選項、alias、型別 include 範圍 | 中 |
| 程式碼規範 | `.editorconfig`、`.eslintrc.js`、`.eslintignore`、`tslint.json` | 格式、lint、歷史規則 | 中 |
| Git / npm 邊界 | `.gitignore`、`.gitattributes`、`.npmignore` | 控制 Git 追蹤與 npm 發布排除規則 | 中 |
| CI / 線上預覽 / 平台 | `.travis.yml`、`preview.yml`、`.inscode`、`.github` | 自動化測試、線上工作區、協作平台設定 | 低～中 |
| 文件與授權 | `README.md`、`README-CN.md`、`LICENSE`、`SECURITY.md` | 專案說明、授權、安全政策 | 低～中 |

第一輪最重要的是這 4 個：


原因是它們直接回答三個核心問題：

```text

```

## 5. 建議讀碼順序

不要照字母順序讀設定檔。建議照「工程流程」讀。


這一輪的目的，是看專案如何處理：

這一輪的目的，是看專案如何處理：

## 6. `package.json`：整個專案的中樞設定

### 6.1 一句話定位

`package.json` 是根目錄設定檔的中樞。

它同時負責：

- 專案身分；
- npm 套件資訊；
- 開發與打包 scripts；
- 依賴與開發依賴；
- 發布檔案邊界；
- 對外入口檔案。

### 6.2 在 View UI Plus 中，你要先看哪些欄位？

第一輪只看這幾個欄位即可：

```json
{
  "name": "view-ui-plus",
  "version": "1.3.24",
  "main": "dist/viewuiplus.min.js",
  "typings": "types/index.d.ts",
  "files": ["dist", "src", "types"],
  "scripts": {
    "dev": "vue-cli-service serve",
    "dev2": "vite --config vite.config.dev.js",
    "build": "npm run build:prod && npm run build:style && npm run build:lang",
    "build:style": "gulp --gulpfile build/build-style.js",
    "build:prod": "vite build",
    "build:lang": "vite build --config build/vite.lang.config.js",
    "lint": "vue-cli-service lint --fix"
  }
}
```

### 6.3 讀碼重點

你可以從 `package.json` 先得到幾個關鍵結論。

#### 結論一：這是 npm 元件庫，不只是 Vue App

因為它有：

```json
"main": "dist/viewuiplus.min.js"
```

這代表 CommonJS / 傳統解析入口會指向 `dist` 裡的打包產物。

#### 結論二：它有 TypeScript 對外型別入口

```json
"typings": "types/index.d.ts"
```

這代表 TypeScript 使用者安裝 `view-ui-plus` 後，會透過 `types/index.d.ts` 取得型別宣告。

#### 結論三：npm 發布內容主要包含三塊

```json
"files": ["dist", "src", "types"]
```

這代表 npm package 主要保留：

| 目錄 | 對使用者的意義 |
|---|---|
| `dist` | 打包後 JS / CSS / locale 等產物 |
| `src` | 原始碼，可能支援按需引用或除錯 |
| `types` | TypeScript declaration |

#### 結論四：工程流程是混合型，不是單一工具

`scripts` 顯示：

| script | 工具 | 初步角色 |
|---|---|---|
| `dev` | Vue CLI | 舊式開發入口 |
| `dev2` | Vite | Vite 開發入口 |
| `build:prod` | Vite | JS library 打包 |
| `build:style` | Gulp | 樣式打包 |
| `build:lang` | Vite + 額外 config | 語言包打包 |
| `lint` | Vue CLI ESLint | lint 自動修復 |

這說明 View UI Plus 不是純 Vite 專案，而是：

> **Vue CLI + Vite + Gulp + TypeScript + ESLint / TSLint 混合演進的元件庫工程。**

### 6.4 這裡先不要深入什麼？

本篇只定位 `package.json`，不要在這裡深入：

- 每個 dependency 的用途；
- 每個 devDependency 的版本選擇；
- 為什麼還有 Vue CLI；
- Gulp 樣式打包的完整流程；
- build 後 `dist` 內容逐檔分析。

這些留給：

```text
03_package與依賴分析
04_build_打包腳本與建置流程
05_dist_打包產物分析
```

## 7. `vite.config.dev.js`：Vite 開發模式設定

### 7.1 一句話定位

`vite.config.dev.js` 是 View UI Plus 使用 Vite 開發 examples 的設定。

它的重點不是「打包 npm 套件」，而是：

> 讓開發者可以在 `examples` 裡即時預覽與測試 `src` 中的元件庫源碼。

### 7.2 重要設定

核心設定如下：

```js
export default defineConfig({
    root: path.resolve(__dirname, 'examples'),
    plugins: [vue()],
    server: {
        port: 8080,
        open: true
    },
    resolve: {
        alias: {
            '@': path.resolve(__dirname, 'src')
        },
        extensions: ['.mjs', '.js', '.ts', '.jsx', '.tsx', '.json', '.vue']
    }
})
```

### 7.3 讀碼重點

#### 重點一：`root` 指向 `examples`

```js
root: path.resolve(__dirname, 'examples')
```

這代表 Vite 開發模式不是直接把根目錄當 app root，而是把 `examples` 當作開發展示專案。

這符合元件庫專案常見結構：

```text
src/       = 元件庫源碼
examples/  = 用來展示與測試 src 的本地站點
```

#### 重點二：alias `@` 指向 `src`

```js
alias: {
    '@': path.resolve(__dirname, 'src')
}
```

這個設定非常關鍵。

它讓 `examples` 中的程式碼可以透過 `@` 直接引用元件庫源碼，而不是引用已經打包好的 npm 產物。

也就是說，開發時的路徑關係大概是：

```text
examples/main.js
   ↓ import
src/index.js 或 src/components/**
```

這種做法可以讓元件開發更即時：

- 修改 `src` 內的元件；
- `examples` 立即看到效果；
- 不需要每次都先 build 到 `dist`。

#### 重點三：server 開在 8080

```js
server: {
    port: 8080,
    open: true
}
```

這表示 `npm run dev2` 時，Vite dev server 預設使用 8080 port，並自動打開瀏覽器。

### 7.4 這個設定檔對你有什麼學習價值？

你未來自己做迷你元件庫時，也可以模仿這個思路：

```text
packages / src       = 元件庫源碼
playground / examples = 開發展示站
vite.config.dev.js   = 專門給 playground 使用
```

這比把所有東西塞在同一個 `src` 更清楚。

### 7.5 常見誤解

| 誤解 | 正確理解 |
|---|---|
| `vite.config.dev.js` 是正式打包設定 | 不是，它偏向開發展示設定 |
| `root: examples` 代表元件源碼在 examples | 不是，examples 只是展示場 |
| alias `@` 指向 examples | 不是，它指向根目錄的 `src` |
| 修改 examples 就等於修改元件庫 | 不一定，examples 多數是 Demo / 驗證用 |

## 8. `vue.config.js`：Vue CLI 開發設定

### 8.1 一句話定位

`vue.config.js` 是 Vue CLI 時期的開發設定，主要讓 Vue CLI 以 `examples` 作為頁面入口。

### 8.2 重要設定

核心設定如下：

```js
module.exports = {
    assetsDir: '',
    pages: {
        index: {
            entry: 'examples/main.js',
            template: 'examples/index.html',
            filename: 'index.html',
            chunks: ['chunk-vendors', 'chunk-common', 'index']
        }
    },
    configureWebpack: {
        resolve: {}
    },
    devServer: {
        disableHostCheck: true
    }
}
```

### 8.3 讀碼重點

#### 重點一：入口同樣指向 `examples`

```js
entry: 'examples/main.js'
template: 'examples/index.html'
```

這代表 Vue CLI 的開發模式也是以 `examples` 作為展示工程入口。

所以 View UI Plus 至少存在兩條開發路徑：

| 指令 | 工具 | 入口 |
|---|---|---|
| `npm run dev` | Vue CLI | `examples/main.js` |
| `npm run dev2` | Vite | `examples` root |

這是典型的歷史演進痕跡。

#### 重點二：它不是目前最值得深入的設定

對你目前學習 View UI Plus 的目標來說：

- 想學元件設計：優先看 `src/components`。
- 想學開發展示架構：優先看 `vite.config.dev.js` + `examples`。
- 想了解歷史相容與舊工程工具：再看 `vue.config.js`。

### 8.4 它對企業舊專案讀碼的啟發

View UI Plus 同時保留 `vite.config.dev.js` 與 `vue.config.js`，說明開源專案也可能不是一次性完全重構。

這和企業舊專案很像：

```text
舊流程還在，新流程也加進來。
舊設定還能跑，新設定負責新場景。
```

讀碼時不要急著批判「為什麼不全部改成 Vite」。你要先判斷：

- 哪個設定目前還被 script 使用；
- 哪個設定只是歷史遺留；
- 哪個設定控制正式產物；
- 哪個設定只控制開發預覽。

## 9. `vite.config.js`：正式元件庫打包設定

### 9.1 一句話定位

`vite.config.js` 是 View UI Plus 的正式 library build 設定。

它回答的是：

> `src` 裡的元件庫源碼，最後如何輸出成 `dist` 裡的 JS bundle？

### 9.2 重要設定

核心結構如下：

```js
const config = defineConfig({
    plugins: [vue()],
    target: 'es2015',
    build: {
        outDir: path.resolve(__dirname, './dist'),
        lib: {
            entry: path.resolve(__dirname, './src/index.js'),
            name: 'ViewUIPlus'
        },
        rollupOptions: {
            context: 'globalThis',
            preserveEntrySignatures: 'strict',
            external: ['vue'],
            output: [
                {
                    format: 'umd',
                    entryFileNames: 'viewuiplus.min.js',
                    globals: { vue: 'Vue' }
                },
                {
                    format: 'es',
                    entryFileNames: 'viewuiplus.min.esm.js',
                    globals: { vue: 'Vue' }
                }
            ]
        }
    },
    resolve: {
        extensions: ['.mjs', '.js', '.ts', '.jsx', '.tsx', '.json', '.vue']
    }
})
```

### 9.3 讀碼重點

#### 重點一：入口是 `src/index.js`

```js
entry: path.resolve(__dirname, './src/index.js')
```

這是你後面讀 `06_src_入口與插件機制` 的關鍵入口。

因為元件庫對外打包時，不是從 `examples/main.js` 開始，而是從：

```text
src/index.js
```

這代表：

```text
examples/main.js = 開發展示入口
src/index.js     = 元件庫正式打包入口
```

這兩個不能混淆。

#### 重點二：輸出目錄是 `dist`

```js
outDir: path.resolve(__dirname, './dist')
```

這表示 build 後 JS bundle 會輸出到 `dist`。

因此：

```text
src/index.js
   ↓ vite build
   ↓
dist/viewuiplus.min.js
dist/viewuiplus.min.esm.js
```

#### 重點三：同時輸出 UMD 與 ES module

```js
output: [
    { format: 'umd', entryFileNames: 'viewuiplus.min.js' },
    { format: 'es', entryFileNames: 'viewuiplus.min.esm.js' }
]
```

這代表 View UI Plus 同時支援不同使用方式：

| 格式 | 檔案 | 可能使用場景 |
|---|---|---|
| UMD | `viewuiplus.min.js` | script tag、較傳統 bundle 環境 |
| ES module | `viewuiplus.min.esm.js` | 現代打包器、ESM 引用 |

#### 重點四：`vue` 被 external 掉

```js
external: ['vue']
globals: { vue: 'Vue' }
```

這代表打包 View UI Plus 時，不會把 Vue 本身打進元件庫 bundle。

原因很合理：

- Vue 是 peer-like 的基礎框架；
- 使用者專案通常已經安裝 Vue；
- 如果元件庫再打包一份 Vue，容易造成重複與版本衝突。

雖然 `package.json` 中 Vue 放在 `devDependencies`，但從 build 設定可以看出，正式 bundle 不會把 Vue 打包進去。

### 9.4 這裡先不要深入什麼？

不要在本篇深入：

- Rollup output 每個選項的完整細節；
- UMD / ESM 的底層差異；
- `manualChunks`、`inlineDynamicImports` 的打包效果；
- CSS 是否被 JS build 處理；
- 語言包 build 怎麼跑。

這些留到：

```text
04_build_打包腳本與建置流程
05_dist_打包產物分析
```

## 10. `tsconfig.json`：TypeScript 與型別設定

### 10.1 一句話定位

`tsconfig.json` 是 TypeScript 編譯與型別解析設定。

在 View UI Plus 中，它最值得注意的是：

- `compilerOptions`；
- `paths` alias；
- `include` 只包含 `types/*.ts`；
- `exclude` 排除 `node_modules`。

### 10.2 重要設定

核心內容如下：

```json
{
  "compilerOptions": {
    "target": "esnext",
    "module": "esnext",
    "strict": true,
    "jsx": "preserve",
    "importHelpers": true,
    "moduleResolution": "node",
    "experimentalDecorators": true,
    "esModuleInterop": true,
    "allowSyntheticDefaultImports": true,
    "sourceMap": true,
    "baseUrl": ".",
    "paths": {
      "@/*": ["src/*"]
    },
    "lib": ["esnext", "dom", "dom.iterable", "scripthost"]
  },
  "include": [
    "types/*.ts"
  ],
  "exclude": ["node_modules"]
}
```

### 10.3 讀碼重點

#### 重點一：`paths` 和 Vite alias 概念一致

```json
"paths": {
  "@/*": ["src/*"]
}
```

這和 `vite.config.dev.js` 的：

```js
alias: {
    '@': path.resolve(__dirname, 'src')
}
```

方向一致：`@` 都指向 `src`。

差別在於：

| 設定 | 作用對象 |
|---|---|
| `vite.config.dev.js` 的 alias | Vite / runtime module resolution |
| `tsconfig.json` 的 paths | TypeScript / IDE 型別解析 |

簡單說：

```text
Vite alias 讓程式跑得懂。
tsconfig paths 讓 TS / IDE 看得懂。
```

#### 重點二：`include` 指向 `types/*.ts`

```json
"include": ["types/*.ts"]
```

這代表這份 TS 設定的重心更偏向型別宣告相關檔案，而不是整個 `src` 原始碼都完整用 TypeScript 管理。

這也符合你在讀 View UI Plus 時會看到的情況：

- 元件實作大量是 `.vue` / `.js`；
- 對外型別則放在 `types` 目錄；
- TypeScript 在這個專案中更像是「型別合約補充」，而不是整個源碼全 TS 化。

#### 重點三：`strict: true` 不等於整個專案都是強型別 TS

`strict: true` 代表 TypeScript 對納入編譯範圍的檔案使用嚴格模式。

但因為 `include` 範圍主要是：

```text
types/*.ts
```

所以不要誤以為整個 `src` 都是嚴格 TypeScript 實作。

### 10.4 這個設定檔對你的學習價值

你之後自己做元件庫時，要分清楚兩件事：

```text
元件源碼是否用 TypeScript 寫？
對外是否提供 TypeScript 型別？
```

View UI Plus 的情況比較接近：

> 元件源碼不一定全 TS，但對外仍提供 TypeScript declaration。

這對很多老牌 Vue 元件庫很常見。

## 11. `.eslintrc.js`：ESLint 規則設定

### 11.1 一句話定位

`.eslintrc.js` 是 ESLint 的規則設定，用來控制 JavaScript / Vue 程式碼檢查。

### 11.2 重要設定

核心內容如下：

```js
module.exports = {
    root: true,
    env: {
        node: true
    },
    globals: {
        $: true,
        'BMap': true
    },
    extends: ['plugin:vue/vue3-essential'],
    parserOptions: {
        parser: 'babel-eslint',
        ecmaVersion: 6,
        sourceType: 'module'
    },
    rules: {
        'vue/script-setup-uses-vars': 'off'
    }
}
```

### 11.3 讀碼重點

#### 重點一：它使用 Vue 3 的 essential 規則

```js
extends: ['plugin:vue/vue3-essential']
```

這代表 ESLint 至少會套用 Vue 3 基礎規則。

但它不是非常嚴格的完整規範，因為 `essential` 層級主要是防止基本錯誤，不是強制大量風格規則。

#### 重點二：允許全域 `$` 與 `BMap`

```js
globals: {
    $: true,
    'BMap': true
}
```

這代表專案中可能存在對全域 `$` 或百度地圖 `BMap` 的使用。

第一輪不需要追它們在哪裡，只要知道：

> ESLint 不會把 `$` 和 `BMap` 當成未定義變數報錯。

#### 重點三：parser 還是 `babel-eslint`

```js
parser: 'babel-eslint'
```

這是歷史痕跡。現代專案更常見的是 `@babel/eslint-parser` 或其他新 parser，但這個專案保留較舊的配置。

### 11.4 對你讀碼的意義

`.eslintrc.js` 告訴你這個專案的程式碼檢查標準，但它不是理解元件設計的主線。

第一輪你只要知道：

```text
.eslintrc.js = Vue 3 基礎 lint 規則 + 一些全域變數允許 + 歷史 parser 設定。
```

## 13. `.editorconfig`：跨編輯器格式統一

### 13.1 一句話定位

`.editorconfig` 用來在不同編輯器 / IDE 中統一基本格式。

例如：

```ini
root = true

[*]
charset = utf-8
end_of_line = lf
indent_size = 4
indent_style = space
insert_final_newline = true
trim_trailing_whitespace = true

[*.md]
trim_trailing_whitespace = false
```

### 13.2 讀碼重點

View UI Plus 的 `.editorconfig` 表示：

| 設定 | 意義 |
|---|---|
| `charset = utf-8` | 使用 UTF-8 |
| `end_of_line = lf` | 換行符使用 LF |
| `indent_size = 4` | 預設 4 空格縮排 |
| `indent_style = space` | 使用空格，不使用 tab |
| `insert_final_newline = true` | 檔案結尾保留換行 |
| `trim_trailing_whitespace = true` | 移除行尾空白 |
| `[*.md] trim_trailing_whitespace = false` | Markdown 保留行尾空白 |

### 13.3 對你仿寫專案的價值

`.editorconfig` 很適合你以後放進自己的專案。

原因是它不依賴特定 IDE，對 VSCode、WebStorm 等都有幫助。

你自己的元件庫可以先用類似設定：

```ini
root = true

[*]
charset = utf-8
end_of_line = lf
indent_style = space
indent_size = 2
insert_final_newline = true
trim_trailing_whitespace = true

[*.md]
trim_trailing_whitespace = false
```

是否用 2 空格或 4 空格不是重點，重點是團隊一致。

## 15. `.npmignore`：npm 發布排除規則

### 15.1 一句話定位

`.npmignore` 控制哪些檔案不要被發布到 npm package。

View UI Plus 的 `.npmignore` 包含：

```text
.*
*.md
*.yml
build/
node_modules/
test/
gulpfile.js
```

### 15.2 讀碼重點

這個設定回答的是：

> 使用者 `npm install view-ui-plus` 時，哪些東西不應該被帶進 package？

它排除了：

| 規則 | 意義 |
|---|---|
| `.*` | 排除 dotfiles，例如 `.github`、`.eslintrc.js` 等 |
| `*.md` | 排除 Markdown 文件 |
| `*.yml` | 排除 YAML 設定 |
| `build/` | 排除建置腳本 |
| `node_modules/` | 排除依賴資料夾 |
| `test/` | 排除測試 |
| `gulpfile.js` | 排除 gulpfile |

### 15.3 和 `package.json` 的 `files` 一起看

`package.json` 裡有：

```json
"files": ["dist", "src", "types"]
```

所以判斷 npm 發布邊界時，要先看 `package.json` 的 `files`。它是 allowlist 思維：主要指定哪些檔案或目錄會被納入 package。

`.npmignore` 則是 ignore 規則。需要注意的是：npm 官方文件指出，如果根目錄同時存在 `files` 欄位與根目錄 `.npmignore`，根目錄 `.npmignore` 不會覆蓋 `files` 欄位；但子目錄中的 `.npmignore` 可以影響對應子目錄。

所以比較精準的理解是：

```text
package.json files = 主要發布 allowlist。
.npmignore = 發布 ignore 規則，但根目錄 .npmignore 不能覆蓋 files allowlist。
```

放回 View UI Plus 來看，可以先抓大方向：

```text
npm package 的核心發布範圍圍繞 dist / src / types，
工程設定、測試、build 腳本、文件與平台設定不是使用者理解元件庫 API 的主線。
```

這對元件庫很重要，因為使用者真正需要的是：

- 可引用的產物；
- 必要的原始碼；
- 型別宣告；
- package metadata。

不是整個 GitHub 倉庫。

### 15.4 和 `.gitignore` 的差別

| 檔案 | 控制對象 | 問題 |
|---|---|---|
| `.gitignore` | Git 版本控制 | 哪些檔案不要進 Git？ |
| `.npmignore` | npm 發布包 | 哪些檔案不要進 npm？ |

這兩個檔案容易混淆，但它們的目的完全不同。

- `preview.yml` 跑的是 `npm run dev`，也就是 Vue CLI；
- `vite.config.dev.js` 對應的是 `npm run dev2`；
- 不同啟動方式可能使用不同 dev server 設定。

這再次說明：

> View UI Plus 有多條開發 / 預覽路徑，不是單一工具鏈。

## 27. 從「歷史演進」角度看設定檔

View UI Plus 根目錄設定檔有很明顯的歷史演進痕跡。

你可以觀察到：

| 現象 | 可能代表的歷史演進 |
|---|---|
| 同時有 `vue.config.js` 和 `vite.config.dev.js` | 從 Vue CLI 過渡到 Vite，但舊流程仍保留 |
| 同時有 ESLint 和 TSLint | TypeScript lint 規範存在歷史遷移痕跡 |
| build 同時用 Vite 和 Gulp | JS bundle 和樣式 build 分工不同 |
| Travis CI 使用 Node 8 | CI 設定偏舊，可能是早期留下 |
| `preview.yml` 與 `.inscode` | 支援線上預覽或雲端 IDE 場景 |

這對你讀企業專案很有價值。

因為企業專案常常也是：

```text
不是完全乾淨的新架構，
而是多個時期的工程設定疊加在一起。
```

你要訓練的是辨識能力：

- 哪些是主流程？
- 哪些是舊流程？
- 哪些是發布必需？
- 哪些只是輔助平台？
- 哪些可以先跳過？

## 29. 常見錯誤理解

### 29.1 錯誤一：把 `vite.config.dev.js` 和 `vite.config.js` 混在一起

錯誤理解：

```text
兩個都是 Vite 設定，所以用途差不多。
```

正確理解：

| 檔案 | 用途 |
|---|---|
| `vite.config.dev.js` | 開發展示，root 是 `examples` |
| `vite.config.js` | 正式 library build，entry 是 `src/index.js` |

這兩個設定檔服務不同流程。

---

### 29.2 錯誤二：以為 `examples` 是正式入口

錯誤理解：

```text
examples/main.js 是入口，所以元件庫從 examples 打包。
```

正確理解：

```text
examples/main.js 是開發展示入口。
src/index.js 才是正式元件庫打包入口。
```

---

### 29.3 錯誤三：看到 `vue.config.js` 就以為它是純 Vue CLI 專案

錯誤理解：

```text
有 vue.config.js，所以這是 Vue CLI 專案。
```

正確理解：

```text
它保留 Vue CLI 開發入口，
但正式 build:prod 使用 Vite，
dev2 也使用 Vite。
```

所以它是混合工具鏈，不是單一 Vue CLI 專案。

---

### 29.4 錯誤四：以為 `tsconfig.json` 代表整個源碼都是 TypeScript

錯誤理解：

```text
有 tsconfig.json，所以 src 應該全是 TypeScript。
```

正確理解：

```text
tsconfig.json 的 include 主要是 types/*.ts，
View UI Plus 的 TypeScript 更偏對外型別宣告，而不是整個源碼全 TS 化。
```

---

### 29.5 錯誤五：只看 `.npmignore`，不看 `package.json files`

錯誤理解：

```text
npm 發布內容只由 .npmignore 決定。
```

正確理解：

```text
要一起看 package.json 的 files 與 .npmignore。
files 是主要 allowlist。
根目錄 .npmignore 不能覆蓋 files 欄位，
但 .npmignore 仍是理解發布排除意圖的重要線索。
```

## 30. 你未來仿寫迷你元件庫時，可以怎麼借鑑？

如果你之後要做 `37_迷你元件庫實作`，可以借鑑 View UI Plus 的分類思路，但不要完全照抄它的歷史包袱。

### 30.1 建議你的迷你元件庫設定檔

比較現代、乾淨的版本可以是：

```text
my-ui-library/
├── package.json
├── vite.config.ts
├── tsconfig.json
├── tsconfig.build.json
├── eslint.config.js
├── .editorconfig
├── .gitignore
├── README.md
├── src/
├── playground/
├── dist/
└── types/
```

### 30.2 對應 View UI Plus 的概念

| 你的迷你元件庫 | View UI Plus 類似概念 |
|---|---|
| `playground/` | `examples/` |
| `src/index.ts` | `src/index.js` |
| `vite.config.ts` | `vite.config.js` + `vite.config.dev.js` |
| `dist/` | `dist/` |
| `types/` | `types/` |
| `eslint.config.js` | `.eslintrc.js` / `tslint.json` |

### 30.3 不建議照抄的地方

不建議你照抄：

- 同時維持 Vue CLI + Vite；
- 使用 TSLint；
- 使用過舊的 Node CI 版本；
- 把 JS build、style build、locale build 拆得太早太複雜。

你可以學它的「架構概念」，但工程工具要用較現代的選擇。

## 31. 本篇總結

這篇你要帶走的不是每個設定檔的全部細節，而是分類能力。

### 31.1 一句話總結

> View UI Plus 的根目錄設定檔可以看成一張工程控制地圖：`package.json` 是中樞，Vite / Vue CLI 控制開發與打包，TypeScript / ESLint 控制品質與型別，Git / npm / CI / preview 控制發布與平台流程。

### 31.2 最重要的 5 個設定檔

第一輪最值得優先掌握：

```text
package.json
vite.config.dev.js
vite.config.js
vue.config.js
tsconfig.json
```

### 31.3 最重要的 5 個判斷

| 判斷 | 結論 |
|---|---|
| 專案主中樞 | `package.json` |
| 開發展示入口 | `examples` |
| Vite 開發設定 | `vite.config.dev.js` |
| 正式打包入口 | `src/index.js` |
| 對外產物 | `dist` + `types` |

### 31.4 讀碼心法

讀設定檔時，不要問：

```text
這一行語法是什麼？
```

先問：

```text
這個設定檔控制哪一段流程？
它是開發、打包、型別、規範、發布，還是平台設定？
它和 package.json 的哪個 script 或欄位有關？
```

只要你能先分類，後面深入每個設定檔就不會迷路。
