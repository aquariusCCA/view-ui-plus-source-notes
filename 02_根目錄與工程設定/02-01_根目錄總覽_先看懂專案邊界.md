# 02-01_根目錄總覽_先看懂專案邊界

> 章節位置：`02_根目錄與工程設定`  
> 筆記定位：先建立 View UI Plus 專案的「工程邊界地圖」  
> 適合閱讀時機：第一次打開 View UI Plus 原始碼、還不知道每個根目錄檔案在做什麼時  

---

## 1. 本篇筆記要解決什麼問題？

當你第一次打開 View UI Plus 這類元件庫專案時，很容易陷入兩種錯誤讀法：

1. 直接衝進 `src/components` 看元件實作  
2. 直接打開 `package.json` 看 scripts 和 dependencies

這兩種讀法都不是錯，但如果你還沒有先看懂「根目錄邊界」，後面會很容易迷路。

因為 View UI Plus 不是一般後台專案，而是**元件庫工程**。它同時要處理：

- 元件原始碼開發
- 範例站與 Demo 開發
- 元件庫打包
- 樣式打包
- 語言包打包
- 型別宣告
- npm 發布邊界
- lint / test / CI / preview 等工程設定

所以本篇的核心任務是：

> 先把根目錄看成一張「工程地圖」，理解每個資料夾與設定檔負責哪一段流程，而不是急著逐行讀完每個設定檔。

## 3. View UI Plus 的專案定位

View UI Plus 官方描述是：

> A high quality UI components Library with Vue.js 3

也就是說，它的本質不是「一個 Vue App」，而是：

> 一個基於 Vue 3 的企業級 UI 元件庫工程。

這個定位非常重要，因為它會影響你怎麼看根目錄。

### 一般 Vue 專案的根目錄重點

一般業務型 Vue 專案通常關心：

- `src/views`
- `src/router`
- `src/store`
- `src/api`
- `.env`
- `vite.config.ts`
- `package.json`

它的重點是「把一個應用跑起來」。

### 元件庫專案的根目錄重點

View UI Plus 這種元件庫專案更關心：

- 元件源碼在哪裡
- 元件如何統一匯出
- 元件如何被範例站引用
- 元件庫如何被打包成 npm 套件
- CSS / Less 樣式如何打包
- 使用者安裝後會拿到哪些檔案
- TypeScript 型別如何提供給外部使用者
- 文件與 demo 如何支撐元件開發

所以你看根目錄時，不能只用「前端專案」角度看，而要用「元件庫產品」角度看。

## 4. 根目錄第一印象：它不是純 Vite 專案

從根目錄與 `package.json` 可以看出，View UI Plus 同時存在多種工程工具：

| 工具 / 機制 | 觀察到的位置 | 初步判斷 |
|---|---|---|
| Vue CLI | `package.json` 的 `dev`、`lint`、`vue.config.js` | 保留早期開發與 lint 流程 |
| Vite | `package.json` 的 `dev2`、`build:prod`、`vite.config.js`、`vite.config.dev.js` | 用於較新的開發 / 打包流程 |
| Gulp | `package.json` 的 `build:style`、`build/build-style.js` | 負責樣式建置 |
| Rollup | 透過 Vite build 底層的 `rollupOptions` | 負責 library bundle 的細節輸出 |
| TypeScript | `tsconfig.json`、`types/`、`typings` 欄位 | 對外提供型別宣告 |
| ESLint / TSLint | `.eslintrc.js`、`.eslintignore`、`tslint.json` | 程式碼規範與歷史規則 |
| Travis CI | `.travis.yml` | 舊式 CI 設定 |
| Preview 設定 | `preview.yml` | 線上工作區或預覽環境設定 |

這裡可以先得到一個重要結論：

> View UI Plus 不是一個乾淨的「新專案範本」，而是一個經過長期演進的元件庫工程。讀碼時要接受它有歷史包袱，不要期待每個設定都完全現代化。

這對你很重要，因為企業舊專案通常也會有類似情況：  
新工具與舊工具並存、build 流程混合、設定檔有歷史痕跡。

> 這個資料夾 / 檔案在整個元件庫工程中負責哪一段？

## 6. 用「五大邊界」理解根目錄

根目錄可以分成五大邊界。

---

### 6.1 源碼邊界：真正的元件庫內容在哪？

| 路徑 | 初步責任 |
|---|---|
| `src/` | 元件庫主體原始碼 |
| `types/` | 對外提供的 TypeScript 型別宣告 |
| `assets/` | 專案使用的資源檔，可能和展示、文件、樣式相關 |

這一組是你未來真正要讀的核心。

但在本章不要急著深入，先知道：

- `src/` 是元件實作與插件入口的主要所在
- `types/` 是使用者在 TypeScript 專案中獲得型別提示的來源
- `assets/` 不一定是元件邏輯核心，通常是輔助資源

### 讀碼提醒

對元件庫來說，`src/` 不是單純的業務程式碼資料夾，而是「套件原始碼」。  
它最後會透過 build 流程輸出到 `dist/`，並透過 npm 發布給使用者。

---

### 6.2 開發展示邊界：元件如何被開發與測試展示？

| 路徑 | 初步責任 |
|---|---|
| `examples/` | 範例站、Demo、開發時預覽元件 |
| `vite.config.dev.js` | Vite 開發模式設定，root 指向 `examples` |
| `vue.config.js` | Vue CLI 開發設定，入口也指向 `examples` |

這裡是 View UI Plus 很值得觀察的地方。

`vite.config.dev.js` 將 Vite 的開發 root 設定為 `examples`，並透過 alias 讓 `examples` 能引用到 `src` 原始碼：

```js
root: path.resolve(__dirname, 'examples')

alias: {
    '@': path.resolve(__dirname, 'src')
}
```

這代表開發時的心智模型是：

```txt
examples/ 充當開發展示站
    ↓
透過 alias 直接引用 src/ 原始碼
    ↓
修改元件後可以在 Demo 中立即觀察效果
```

這是很多元件庫會採用的模式：  
**用一個 examples app 作為 playground，直接吃元件庫源碼，而不是吃打包後的 npm 產物。**

### 為什麼這很重要？

如果你未來要自己做迷你元件庫，也可以學這個結構：

```txt
my-ui-lib/
├── src/        # 元件庫源碼
├── examples/   # 本地開發展示站
└── vite.config.dev.ts
```

開發時讓 `examples` 直接引用 `src`，可以避免每改一次元件都要重新 publish 或重新打包。

---

### 6.3 建置與發布邊界：原始碼如何變成可安裝套件？

| 路徑 / 檔案 | 初步責任 |
|---|---|
| `build/` | 額外建置腳本，例如樣式、語言包 |
| `dist/` | 打包後的可發布產物 |
| `vite.config.js` | library build 設定 |
| `.npmignore` | npm 發布時排除哪些檔案 |
| `package.json` | 定義 main、typings、files、scripts |

這一組先不用深入，但你要先建立一個流程感：

```txt
src/ 原始碼
    ↓
vite.config.js / build 腳本
    ↓
dist/ 打包產物
    ↓
package.json + .npmignore 控制 npm 發布邊界
    ↓
使用者 npm install view-ui-plus
```

這就是元件庫工程最核心的產品化流程。

---

### 6.4 工程品質邊界：程式碼如何維持一致？

| 檔案 | 初步責任 |
|---|---|
| `.editorconfig` | 統一不同編輯器的縮排、換行、空白規則 |
| `.eslintrc.js` | JavaScript / Vue lint 規則 |
| `.eslintignore` | ESLint 忽略範圍 |
| `tslint.json` | TypeScript lint 規則，帶有歷史痕跡 |
| `tsconfig.json` | TypeScript 編譯與型別解析設定 |
| `test/` | 測試程式碼 |
| `.travis.yml` | CI 測試設定 |

這一組不是元件功能本身，但會影響專案能不能長期維護。

你在讀企業專案時也會看到類似分層：

```txt
功能程式碼
    +
工程規範
    +
自動化檢查
    =
可維護專案
```

所以根目錄設定檔不是雜物，而是專案工程紀律的外顯。

## 9. `package.json` 在本篇只看四個點

`package.json` 很重要，但本篇不要急著分析 dependencies。  
在 `02-01` 只看四個「邊界資訊」就好。

### 9.1 套件名稱

```json
"name": "view-ui-plus"
```

代表 npm 使用者安裝時會使用：

```bash
npm install view-ui-plus
```

### 9.2 主入口

```json
"main": "dist/viewuiplus.min.js"
```

代表 CommonJS / 傳統解析方式會優先指向 `dist/viewuiplus.min.js`。

### 9.3 型別入口

```json
"typings": "types/index.d.ts"
```

代表 TypeScript 專案會從 `types/index.d.ts` 取得型別宣告。

### 9.4 發布檔案範圍

```json
"files": [
    "dist",
    "src",
    "types"
]
```

這裡非常值得注意。

它代表 npm package 主要包含：

```txt
dist/    # 打包產物
src/     # 原始碼
types/   # 型別宣告
```

這和 `.npmignore` 一起構成「發布邊界」。

## 10. `.npmignore` 告訴你什麼是「不給使用者看的」

`.npmignore` 目前排除了：

```txt
.*
*.md
*.yml
build/
node_modules/
test/
gulpfile.js
```

這代表從 npm 發布角度看，很多開發用檔案都不需要給使用者。

你可以這樣理解：

| 檔案 / 資料夾 | 為什麼不發布 |
|---|---|
| `.*` | 大多數 dotfile 是開發 / 平台設定 |
| `*.md` | README 類文件不一定要進入 package 內容 |
| `*.yml` | CI / preview 設定不需要給使用者 |
| `build/` | 建置腳本通常不需要給使用者 |
| `node_modules/` | 永遠不應該發布 |
| `test/` | 測試只服務維護者 |
| `gulpfile.js` | 建置工具腳本，不是執行時產物 |

這裡可以建立一個元件庫觀念：

> 根目錄裡的東西，不等於 npm 使用者最後會拿到的東西。

這點非常重要。  
你讀元件庫時，要同時分清楚：

1. 倉庫內容  
2. 開發時內容  
3. 打包後內容  
4. npm 發布後內容  
5. 使用者實際 import 的內容

## 11. 為什麼同時有 `vite.config.dev.js`、`vite.config.js`、`vue.config.js`？

這是本章最值得留意的工程現象。

---

### 11.1 `vite.config.dev.js`：給開發展示用

核心特徵：

```js
root: path.resolve(__dirname, 'examples')

server: {
    port: 8080,
    open: true
}

alias: {
    '@': path.resolve(__dirname, 'src')
}
```

它的定位是：

> 使用 Vite 啟動 examples，讓 examples 可以直接引用 src 原始碼。

---

### 11.2 `vite.config.js`：給元件庫打包用

核心特徵：

```js
build: {
    outDir: path.resolve(__dirname, './dist'),
    lib: {
        entry: path.resolve(__dirname, './src/index.js'),
        name: 'ViewUIPlus'
    },
    rollupOptions: {
        external: ['vue']
    }
}
```

它的定位是：

> 把 `src/index.js` 當作元件庫入口，打包輸出到 `dist/`。

這裡的 `external: ['vue']` 非常重要。  
它代表 Vue 不被打包進 View UI Plus 的 bundle，而是由使用者專案提供。

這是元件庫常見設計：

```txt
元件庫依賴 Vue
但不把 Vue 內嵌進元件庫產物
避免使用者專案出現多份 Vue
```

---

### 11.3 `vue.config.js`：舊的 Vue CLI 開發入口

核心特徵：

```js
pages: {
    index: {
        entry: 'examples/main.js',
        template: 'examples/index.html',
        filename: 'index.html'
    }
}
```

它的定位是：

> 使用 Vue CLI 時，也把 examples 當作開發入口。

所以你可以得到一個判斷：

```txt
dev  → Vue CLI → examples/main.js
dev2 → Vite    → examples/
build:prod → Vite library build → src/index.js → dist/
```

這代表此專案保留了 Vue CLI 與 Vite 並存的過渡痕跡。

## 14. 根目錄的四個重要判斷

讀完第一輪，你應該能做出四個判斷。

---

### 判斷一：這個專案的核心源碼在哪？

答案：

```txt
src/
```

但它不是孤立存在，而是被以下設定串起來：

```txt
src/
├── 被 examples 開發展示引用
├── 被 vite.config.js 當作 library entry
├── 被 package.json files 納入發布範圍
└── 被 types/ 補足型別能力
```

---

### 判斷二：這個專案開發時從哪裡啟動？

答案有兩條：

```txt
npm run dev  → vue-cli-service serve → vue.config.js → examples/main.js
npm run dev2 → vite --config vite.config.dev.js → examples/
```

現代閱讀時可以優先關注 Vite 這條，但 Vue CLI 那條也不能完全忽略，因為它代表專案歷史與某些工具鏈仍可能依賴舊設定。

---

### 判斷三：這個專案打包時從哪裡進入？

答案：

```txt
src/index.js
```

`vite.config.js` 的 library entry 指向 `src/index.js`。  
這代表後續看 `06_src_入口與插件機制` 時，應該優先找 `src/index.js`，而不是隨機打開某個元件。

---

### 判斷四：這個專案發布時主要給使用者什麼？

答案：

```txt
dist/
src/
types/
```

這由 `package.json` 的 `files` 欄位決定。  
再搭配 `.npmignore`，可以理解哪些東西只服務維護者，哪些東西才會進入 npm package。

## 16. 讀碼陷阱

---

### 陷阱一：把 `examples/` 誤認為正式產物

`examples/` 是開發展示站，不是元件庫使用者最終 import 的主體。

使用者真正關心的是：

```txt
dist/
types/
package.json main / typings
```

---

### 陷阱二：看到 `dist/` 就開始讀壓縮後程式碼

`dist/` 是產物，不是第一輪讀碼重點。

第一輪應該先讀：

```txt
src/ → build 設定 → dist/
```

而不是反過來從 dist 讀回 src。

---

### 陷阱三：以為 `vite.config.dev.js` 和 `vite.config.js` 是同一件事

它們職責不同：

| 檔案 | 職責 |
|---|---|
| `vite.config.dev.js` | 開發 examples |
| `vite.config.js` | 打包元件庫 |

看錯這點，會導致你搞混：

- 開發入口
- 打包入口
- examples app
- library package

---

### 陷阱四：忽略 `.npmignore`

很多人讀開源專案只看 `src` 和 `package.json`，但元件庫一定要看 `.npmignore`。

因為它會告訴你：

> 哪些檔案只是維護者需要，哪些檔案才可能進入 npm package。

---

### 陷阱五：用「新專案潔癖」批判舊設定

View UI Plus 同時有 Vue CLI、Vite、Gulp、TSLint、ESLint、Travis，這代表它是長期演進的專案。

讀這種專案時，與其問：

> 為什麼不用最新最佳實踐？

不如先問：

> 這個設定現在還服務哪條流程？  
> 它是核心流程，還是歷史遺留？  
> 它能不能先放著不讀？

這種讀法更接近企業舊系統維護的真實情境。

### 19.1 看懂企業專案邊界

```txt
舊工具 + 新工具
歷史設定 + 新增設定
多套 build 流程

```

View UI Plus 根目錄可以訓練你看懂這種混合工程。

### 19.2 學會區分「開發入口」和「產品入口」

這是元件庫工程很重要的能力：

| 類型 | 範例 |
|---|---|
| 開發入口 | `examples/main.js`、`vite.config.dev.js` |
| 打包入口 | `src/index.js`、`vite.config.js` |
| 使用者入口 | `dist/viewuiplus.min.js`、`types/index.d.ts` |

很多初學者會把這三個混在一起。  
你能分清楚，就已經比單純會寫 Vue component 更進一步。

---

### 19.3 學會看 npm 發布邊界

在公司做共用元件庫、SDK、工具包時，你一定會遇到：

- 哪些檔案要發 npm？
- 哪些檔案只是內部開發用？
- main / module / types / exports 怎麼設？
- dist 和 src 是否都要發布？
- 測試與 build 腳本要不要發布？

View UI Plus 的 `package.json` 與 `.npmignore` 就是很好的觀察素材。
