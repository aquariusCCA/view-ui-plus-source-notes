# 02-02_根目錄資料夾責任分工

> 所屬章節：`02_根目錄與工程設定`  
> 專案：View UI Plus  
> 筆記定位：從「資料夾責任」角度看懂 View UI Plus 的工程邊界  
> 建議閱讀時間：30～45 分鐘  

---

## 0. 這篇筆記要解決什麼問題？

當你第一次打開 View UI Plus 這種元件庫專案時，很容易直接衝進 `src/components` 看元件實作。

但如果沒有先理解根目錄資料夾的責任分工，後面讀碼會很容易混亂：

- 不知道哪些是「真正元件庫源碼」；
- 不知道哪些是「開發展示用 Demo」；
- 不知道哪些是「打包產物」；
- 不知道哪些是「建置流程輔助腳本」；
- 不知道哪些是「型別宣告」；
- 不知道哪些資料夾會被發布到 npm；
- 不知道改某個地方會影響開發、打包、文件、測試還是發布。

所以這篇的目的不是深入每個資料夾，而是先建立一張清楚的工程地圖：

> **先知道每個根目錄資料夾負責什麼，再決定後續要往哪裡深入。**

## 2. View UI Plus 根目錄資料夾總覽

根據 View UI Plus GitHub 倉庫目前的根目錄，主要資料夾包含：

```text
ViewUIPlus/
├── .github/
├── assets/
├── build/
├── dist/
├── examples/
├── src/
├── test/
└── types/
```

這 8 個資料夾可以先分成 6 類：

| 類型 | 資料夾 | 一句話定位 |
|---|---|---|
| 倉庫協作 / 平台設定 | `.github` | GitHub 相關協作、Issue、PR、Actions 或社群設定可能放置的位置 |
| 專案展示 / 文件資源 | `assets` | 放專案展示圖、README 圖片、文件資源等靜態素材的可能位置 |
| 建置流程 | `build` | 放樣式建置、語言包建置、輔助打包腳本 |
| 發布產物 | `dist` | 元件庫打包後給使用者引用的產物 |
| 開發展示 | `examples` | 本地開發與 Demo 預覽用的範例工程 |
| 元件庫源碼 | `src` | View UI Plus 的核心源碼，包含入口、元件、指令、工具、樣式、語系等 |
| 測試 | `test` | 測試程式與測試輔助資源 |
| 型別宣告 | `types` | TypeScript 使用者會讀到的 `.d.ts` 型別宣告 |

先記這句話：

> **`src` 是元件庫的源碼中心，`examples` 是開發展示場，`build` 是建置工具區，`dist` 是發布產物區，`types` 是 TypeScript 對外合約。**

## 3. 根目錄資料夾責任分工總表

| 資料夾 | 是否屬於核心源碼？ | 是否會影響 npm 使用者？ | 是否適合第一輪深入？ | 主要責任 |
|---|---:|---:|---:|---|
| `.github` | 否 | 間接影響 | 否 | GitHub 協作、流程、模板、可能的 CI 設定 |
| `assets` | 否 | 通常不直接影響 | 否 | README、文件、展示用圖片與靜態資源 |
| `build` | 否，但影響產物 | 是，透過產物間接影響 | 第二輪 | 樣式、語言包、打包輔助流程 |
| `dist` | 否，是產物 | 是，直接影響 | 第二輪 | 發布後使用者實際引用的 JS / CSS / locale 等產物 |
| `examples` | 否，是展示工程 | 通常不直接影響 | 第一輪可快速看 | 本地開發、元件 Demo、功能驗證 |
| `src` | 是 | 是，透過打包結果影響 | 第一輪重點 | 元件庫核心源碼 |
| `test` | 否，但反推規格 | 間接影響 | 後期深入 | 測試案例、元件行為驗證 |
| `types` | 是，對 TS 使用者是合約 | 是，直接影響 TS 體驗 | 第二輪 | TypeScript 型別宣告 |

這張表可以用來快速判斷：

- 想理解元件怎麼寫：看 `src`。
- 想知道元件怎麼跑起來：看 `examples`。
- 想知道怎麼打包：看 `build` 與 `vite.config.js`。
- 想知道使用者最後拿到什麼：看 `dist`。
- 想知道 TypeScript 提示怎麼來：看 `types`。
- 想知道元件規格如何被驗證：看 `test`。

## 4. `.github`：GitHub 協作與平台流程區

### 4.1 一句話定位

`.github` 通常是 GitHub 專用的協作設定區，不是元件庫本身的執行源碼。

### 4.2 你應該怎麼看它？

第一輪讀碼時，`.github` 不需要優先深入。

你只需要先知道：

```text
.github/ 不是 View UI Plus 元件的實作位置。
.github/ 主要服務於 GitHub 平台上的協作流程。
```

常見可能內容包含：

- Issue template；
- Pull Request template；
- GitHub Actions workflow；
- funding / security / contribution 相關設定；
- 自動化檢查流程設定。

實際專案要以倉庫內檔案為準。

### 4.3 對你讀碼的意義

`.github` 的價值在於理解開源專案協作流程，而不是理解元件庫架構。

對你目前「透過 View UI Plus 學元件庫設計」的目標來說，它的優先級較低。

### 4.4 什麼時候需要看 `.github`？

以下情境才需要看：

| 情境 | 是否需要看 `.github` |
|---|---:|
| 想學 Button、Input、Modal 怎麼寫 | 不需要 |
| 想知道 PR 流程 | 需要 |
| 想知道 CI 是否會跑 lint / test / build | 需要 |
| 想貢獻 View UI Plus 原始碼 | 需要 |
| 想模仿開源專案的 GitHub 協作設定 | 需要 |

### 4.5 學習建議

第一輪只標記即可：

```text
.github = 開源協作與平台設定，不是元件庫主線。
```

等你進入「工程化 / 開源協作」階段，再回頭研究。

## 5. `assets`：展示與文件靜態資源區

### 5.1 一句話定位

`assets` 通常用來放專案展示、README、文件或宣傳相關的靜態資源。

### 5.2 它和 `src` 裡的資源有什麼差別？

這點很重要。

在元件庫專案裡，根目錄的 `assets` 通常不等於元件運行時資源。

你可以先用這樣的邏輯區分：

| 位置 | 可能用途 | 是否屬於元件庫核心源碼？ |
|---|---|---:|
| `assets` | README 圖片、文件展示圖、專案宣傳素材 | 通常不是 |
| `src/styles` | 元件樣式原始碼 | 是 |
| `src/components/**` | 元件實作與局部資源 | 是 |
| `dist/styles` | 打包後 CSS 產物 | 是，對使用者有影響 |

### 5.3 對你讀碼的意義

`assets` 不是你學元件架構的第一主線。

你可以把它理解成：

```text
assets = 專案對外展示、文件或說明用資源。
```

### 5.4 什麼時候要看 `assets`？

| 情境 | 是否需要看 `assets` |
|---|---:|
| 想知道元件 props 怎麼設計 | 不需要 |
| 想知道元件 CSS 怎麼組織 | 不需要，應看 `src/styles` |
| 想知道 README 圖片從哪裡來 | 需要 |
| 想模仿開源專案的文件展示方式 | 可以看 |
| 想整理自己的學習筆記素材 | 可以看 |

### 5.5 常見誤解

不要看到 `assets` 就以為它等於 Vue 專案中的 `src/assets`。

在一般業務專案中：

```text
src/assets 通常是應用內會使用的圖片、字型、SVG 等。
```

但在元件庫根目錄中：

```text
assets 更可能是倉庫文件、展示、品牌或說明相關素材。
```

具體仍要以實際檔案內容確認。

## 6. `build`：建置流程輔助區

### 6.1 一句話定位

`build` 是 View UI Plus 的建置輔助腳本區，負責支援樣式、語言包或其他打包流程。

### 6.2 從 `package.json` 看 `build` 的角色

View UI Plus 的 `package.json` 中有這些腳本：

```json
{
  "scripts": {
    "build": "npm run build:prod && npm run build:style && npm run build:lang",
    "build:style": "gulp --gulpfile build/build-style.js",
    "build:prod": "vite build",
    "build:lang": "vite build --config build/vite.lang.config.js"
  }
}
```

從這裡可以看出：

| 腳本 | 相關位置 | 責任 |
|---|---|---|
| `build:prod` | `vite.config.js` | 打包元件庫 JS 主產物 |
| `build:style` | `build/build-style.js` | 打包 / 處理樣式 |
| `build:lang` | `build/vite.lang.config.js` | 打包語言包 |

所以 `build` 不是元件源碼本身，而是：

> **把源碼轉成發布產物的工程流程區。**

### 6.3 對你讀碼的意義

`build` 會回答這類問題：

- Less 樣式怎麼被編譯成 CSS？
- 語言包怎麼輸出到 `dist`？
- 元件庫有哪些非 JS 主產物？
- 為什麼 `dist/styles/viewuiplus.css` 會存在？
- 為什麼 `dist/locale/*` 可能會存在？

但這些問題不是本篇要深入解決的，本篇只先建立定位。

### 6.4 `build` 和 `dist` 的關係

可以這樣理解：

```text
src / styles / locale 等原始碼
        ↓
      build 腳本
        ↓
      dist 發布產物
```

`build` 是「加工流程」，`dist` 是「加工結果」。

### 6.5 什麼時候要看 `build`？

| 情境 | 是否需要看 `build` |
|---|---:|
| 想知道 Button 元件怎麼實作 | 不優先 |
| 想知道樣式如何從 Less 變 CSS | 需要 |
| 想知道語言包如何被打包 | 需要 |
| 想學元件庫 build pipeline | 需要 |
| 想仿寫自己的迷你元件庫 | 第二階段需要 |

### 6.6 初學讀碼建議

第一輪不要一開始就陷入 `build`。

建議順序是：

```text
先看 src 入口與元件設計
再看 examples 如何驗證元件
最後回頭看 build 如何輸出產物
```

如果太早看 `build`，你可能會被 Gulp、Less、Vite library mode、Rollup output 設定分散注意力。

## 7. `dist`：打包後發布產物區

### 7.1 一句話定位

`dist` 是 View UI Plus 打包完成後給使用者引用的產物資料夾。

### 7.2 從 `package.json` 看 `dist` 的地位

`package.json` 中有兩個關鍵設定：

```json
{
  "main": "dist/viewuiplus.min.js",
  "files": [
    "dist",
    "src",
    "types"
  ]
}
```

這代表：

| 欄位 | 意義 |
|---|---|
| `main` | CommonJS / 傳統 Node resolution 會優先指向 `dist/viewuiplus.min.js` |
| `files` | npm 發布時會包含 `dist`、`src`、`types` |

也就是說，`dist` 不是隨便產生的資料夾，而是元件庫對外發布的重要部分。

### 7.3 從 `vite.config.js` 看 `dist` 的來源

`vite.config.js` 中的 library build 設定包含：

```js
build: {
  outDir: path.resolve(__dirname, './dist'),
  lib: {
    entry: path.resolve(__dirname, './src/index.js'),
    name: 'ViewUIPlus'
  }
}
```

這代表：

```text
src/index.js
   ↓ Vite build
./dist
```

也就是說：

> **`dist` 是由 `src/index.js` 作為 library entry 打包後產生的結果。**

### 7.4 `dist` 對使用者的意義

使用者在專案中可能會這樣使用 View UI Plus：

```js
import ViewUIPlus from 'view-ui-plus'
import 'view-ui-plus/dist/styles/viewuiplus.css'
```

這種 CSS 引入路徑就會碰到 `dist`。

所以 `dist` 對使用者有直接影響：

- JS bundle 是否正常；
- ESM / UMD 產物是否正確；
- CSS 是否存在；
- locale 是否可引用；
- 使用者打包工具是否能解析。

### 7.5 `dist` 和 `src` 的關係

你可以把它們想成：

| 資料夾 | 角色 | 是否應該手動修改？ |
|---|---|---:|
| `src` | 原始碼 | 是，開發者主要修改這裡 |
| `dist` | 打包產物 | 通常不應直接手改，應由 build 流程產生 |

### 7.6 初學常見誤解

#### 誤解一：`dist` 裡的程式碼比較重要，所以要先讀 `dist`

不建議。

`dist` 是打包後的結果，通常經過壓縮、轉譯或格式包裝，不適合初學者拿來理解元件設計。

真正要學元件架構，應該讀 `src`。

#### 誤解二：`dist` 不重要，反正是產物

也不完全對。

對元件庫來說，`dist` 是使用者實際接觸的產物。當你想理解「npm 套件發出去後長什麼樣子」時，`dist` 很重要。

### 7.7 什麼時候要看 `dist`？

| 情境 | 是否需要看 `dist` |
|---|---:|
| 學元件內部實作 | 不優先 |
| 檢查打包結果 | 需要 |
| 理解 npm 使用者拿到什麼 | 需要 |
| 排查 CSS 引入問題 | 需要 |
| 學 library mode 輸出格式 | 需要 |

## 8. `examples`：本地開發與 Demo 展示區

### 8.1 一句話定位

`examples` 是開發 View UI Plus 時用來啟動本地頁面、展示元件、測試互動效果的範例工程。

### 8.2 從 `vite.config.dev.js` 看 `examples` 的角色

`vite.config.dev.js` 中有一個非常關鍵的設定：

```js
export default defineConfig({
  root: path.resolve(__dirname, 'examples'),
  plugins: [vue()],
  server: {
    port: 8080,
    open: true
  },
  resolve: {
    alias: {
      '@': path.resolve(__dirname, 'src')
    }
  }
})
```

這裡有兩個重點：

| 設定 | 意義 |
|---|---|
| `root: examples` | Vite 開發模式會把 `examples` 當成開發應用根目錄 |
| `alias @ -> src` | Demo 可以直接引用元件庫源碼 |

這代表 `examples` 不是普通資料夾，而是 View UI Plus 的本地開發舞台。

### 8.3 `examples` 和 `src` 的關係

可以這樣理解：

```text
examples = 使用 / 展示元件的地方
src      = 實作 / 提供元件的地方
```

也就是：

```text
examples 透過 alias 直接引用 src
```

這對讀碼很重要，因為你可以透過 examples 找到某個元件的使用方式，再回到 src 看實作。

### 8.4 推薦讀碼方式

如果你要讀一個元件，例如 `Button`、`Input`、`Modal`，可以用這個順序：

```text
1. 先在 examples 找到 Demo 用法
2. 看它傳了哪些 props、events、slots
3. 回到 src/components 找實作
4. 再看 types 是否有對應型別宣告
5. 最後看 test 是否有行為規格
```

這樣比直接衝進 `src/components` 更有效。

### 8.5 `examples` 對前端學習的價值

對你目前想透過開源專案學前端與元件庫設計來說，`examples` 很值得看。

因為它可以幫你回答：

- 元件作者希望使用者怎麼使用元件？
- 哪些 props 是常見使用情境？
- 哪些元件需要 slot？
- 哪些元件有命令式 API？
- 哪些元件需要配合全域配置？
- 哪些元件的互動流程比較複雜？

### 8.6 什麼時候要看 `examples`？

| 情境 | 是否需要看 `examples` |
|---|---:|
| 第一次理解元件使用方式 | 很需要 |
| 想看元件內部狀態設計 | 先看 `examples`，再看 `src` |
| 想學 Demo 如何組織 | 需要 |
| 想分析打包流程 | 不優先 |
| 想確認某個元件 API 是否好用 | 需要 |

### 8.7 注意：`examples` 不是元件庫本體

雖然 `examples` 很重要，但它不是元件庫本體。

不要把 Demo 裡的業務寫法誤認為元件庫內部設計。

你應該區分：

| 層次 | 代表資料夾 | 關注點 |
|---|---|---|
| 使用層 | `examples` | 如何使用元件 |
| 實作層 | `src` | 元件如何設計與實作 |
| 型別層 | `types` | 對外 API 如何被 TypeScript 描述 |
| 產物層 | `dist` | 使用者最後引用什麼 |

## 9. `src`：元件庫核心源碼區

### 9.1 一句話定位

`src` 是 View UI Plus 的核心源碼區，是你學元件庫設計最重要的資料夾。

### 9.2 從 `vite.config.js` 看 `src` 的地位

`vite.config.js` 中指定 library entry：

```js
lib: {
  entry: path.resolve(__dirname, './src/index.js'),
  name: 'ViewUIPlus'
}
```

這代表 `src/index.js` 是元件庫打包入口。

你可以先記：

```text
src/index.js = View UI Plus 對外安裝與匯出的總入口之一
```

後面 `06_src_入口與插件機制` 會深入分析它。

### 9.3 `src` 內通常要關注哪些東西？

雖然本篇不深入 `src`，但第一輪你可以先建立以下分類：

| 類型 | 可能位置 | 你要關心的問題 |
|---|---|---|
| 元件 | `src/components` | 每個 UI 元件如何拆分、props 如何設計、事件如何設計 |

## 11. `types`：TypeScript 型別宣告區

### 11.1 一句話定位

`types` 是 View UI Plus 對 TypeScript 使用者提供型別宣告的地方。

### 11.2 從 `package.json` 看 `types` 的地位

`package.json` 中有：

```json
{
  "typings": "types/index.d.ts",
  "files": [
    "dist",
    "src",
    "types"
  ]
}
```

這代表：

| 欄位 | 意義 |
|---|---|
| `typings` | TypeScript 會從 `types/index.d.ts` 找型別入口 |
| `files` | npm 發布會包含 `types` 資料夾 |

因此 `types` 不是無關資料夾，而是元件庫對 TypeScript 使用者的正式合約。

### 11.3 `types` 和 `src` 的差別

| 資料夾 | 角色 | 內容重點 |
|---|---|---|
| `src` | 實作 | 元件實際怎麼運作 |
| `types` | 宣告 | 使用者在 TS 裡看到哪些型別、props、事件、插件型別 |

你可以這樣理解：

```text
src   = 元件庫做了什麼
 types = 元件庫承諾給 TypeScript 使用者什麼
```

### 11.4 對你學習元件庫 API 設計的價值

`types` 非常值得看，尤其你正在學 TypeScript、Vue、元件庫設計。

它能幫你理解：

- 元件 props 如何被型別描述；
- 插件 install 如何對外宣告；
- 全域方法如何掛載到 Vue instance；
- `$Message`、`$Modal`、`$Notice` 這類命令式 API 如何對 TS 使用者可見；
- 元件庫如何提供比較完整的開發體驗。

### 11.5 什麼時候要看 `types`？

| 情境 | 是否需要看 `types` |
|---|---:|
| 看元件內部實作 | 不一定 |
| 學 TypeScript 元件庫設計 | 很需要 |
| 排查 TS 型別錯誤 | 必看 |
| 想知道 API 對外長相 | 很需要 |
| 想仿寫自己的元件庫 | 必看 |

### 11.6 和後續章節關係

你後面有：

```text

31_高階專題_命令式API設計
```

這幾章都會和 `types` 有密切關係。

所以本篇只需要先記住：

> **`types` 是對 TypeScript 使用者的 API 合約，不是單純補充文件。**

## 14. 第一輪讀碼建議順序

如果你的目標是學元件庫設計，而不是馬上修改專案，建議順序如下：

```text
1. 先看根目錄資料夾分工
2. 看 package.json scripts，知道 dev / build 怎麼跑
3. 看 vite.config.dev.js，理解 examples 如何作為開發入口
4. 看 examples，找一兩個元件 Demo
5. 回到 src，看元件實作
6. 看 src/index.js，理解插件 install 與元件註冊
7. 看 types，理解對外 API 型別合約
8. 看 build，理解樣式與語言包怎麼輸出
9. 看 dist，確認最後發布產物
10. 看 test，用測試反推規格
```

這個順序的好處是：

- 先有地圖，再讀細節；
- 先看使用方式，再看實作；
- 先理解源碼，再理解打包；
- 先懂元件設計，再看測試與型別。

## 15. 不建議的讀碼順序

以下讀碼方式不建議：

### 15.1 一開始就讀 `dist`

問題：

- `dist` 是打包後產物；
- 結構可能被轉譯或包裝；
- 不容易看出原始設計意圖。

更好的方式：

```text
先讀 src，再用 dist 驗證打包結果。
```

### 15.2 一開始就讀所有 `build` 腳本

問題：

- 容易被 Gulp、Less、Vite、Rollup 設定分散注意力；
- 還沒理解元件庫源碼前，很難知道 build 腳本為什麼這樣設計。

更好的方式：

```text
先知道 build 負責什麼，等看完 src/styles、src/locale 後再深入 build。
```

### 15.3 只看 `examples`，不看 `src`

問題：

- 你只會知道元件怎麼用；
- 不會知道元件如何設計；
- 學不到元件庫內部架構。

更好的方式：

```text
examples 看使用方式，src 看設計與實作。
```

### 15.4 只看 `src`，完全不看 `types`

問題：

- 你會忽略 TypeScript 使用者的開發體驗；
- 無法完整理解元件庫對外 API 合約；
- 仿寫元件庫時容易缺少型別設計。

更好的方式：

```text
src 看實作，types 看對外合約。
```

## 16. 根目錄資料夾分工的核心觀念

### 16.1 觀念一：元件庫專案不是一般業務專案

一般 Vue 業務專案常見結構可能是：

```text
src/
├── views/
├── router/
├── store/
├── components/
└── assets/
```

但 View UI Plus 是元件庫，不是一般應用。

所以它更關心：

- 元件如何對外註冊；
- 元件 API 如何設計；
- 樣式如何輸出；
- 型別如何提供；
- library build 如何配置；
- Demo 如何展示；
- npm 使用者如何引用。

因此它的根目錄分工會更像：

```text
src      = library source
examples = demo app
build    = build pipeline
dist     = published artifacts
types    = declaration files
```

### 16.2 觀念二：`examples` 是應用，`src` 是庫

這句非常重要：

```text
examples 是用來消費 View UI Plus 的應用
src 是被 examples 消費的元件庫源碼
```

如果你把 `examples` 當成元件庫本體，就會讀偏。

### 16.3 觀念三：`types` 也是產品的一部分

對 JavaScript 使用者來說，`dist` 很重要。

對 TypeScript 使用者來說，`types` 同樣重要。

一個現代元件庫如果型別不好，使用體驗會大幅下降。

所以你之後學元件 API 設計時，要同時看：

```text
src 實作
+ examples 用法
+ types 型別
+ test 規格
```

### 16.4 觀念四：`build` 決定源碼如何變成產品

`src` 寫得再好，如果 `build` 沒處理好，使用者還是可能遇到：

- CSS 沒輸出；
- locale 無法引用；
- ESM / UMD 格式錯誤；
- Vue 被錯誤打包進 bundle；
- tree-shaking 體驗不佳；
- 型別沒有被發布。

所以 `build` 不是主業務邏輯，但它決定元件庫能否被正確交付。
